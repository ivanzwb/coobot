# BiosBot 功能要求

## 1. 文档目的

本文档用于定义 BiosBot P0 至 P2 阶段的功能要求，明确系统必须提供的业务能力、功能边界、优先级和验收口径。
本文档主要支撑 01-需求分析 中的需求落地，并为技术分析、概要设计、详细设计、接口设计、界面设计和测试设计提供功能基线。
本文档重点回答“系统必须具备哪些功能、哪些功能进入当前阶段、每项功能达到什么程度才算完成”。

## 2. 产品范围

系统定位为本地优先的多智能体协作平台，支持文本、图片、文档等输入，围绕 Leader Agent + Domain Agent + Skills + 知识库 + 任务追踪形成闭环。

P0 核心目标只有一个：

- 用户能够上传文本和附件，发起复杂任务，看到系统如何拆解与执行，并获得可复用、可解释的最终结果。

## 3. 角色与使用对象

- 个人开发者与 AI 爱好者：关注本地部署、Agent 配置、Skills、工具与实验能力。
- 专业知识工作者：关注资料分析、结构化输出、引用依据和过程可复核。
- 小团队负责人：关注任务复盘、失败定位、知识沉淀和流程复用。

## 4. P0 功能清单

### 4.1 模型与环境配置

- 支持配置兼容 OpenAI API 的文本模型与多模态模型。（来源：N-06）
- 支持设置 API Key、Base URL、模型名称、默认模型、温度。（来源：N-06）
- 支持模型连通性测试。（来源：N-06）
- 支持基础系统配置，如任务超时、日志级别、并发上限。（来源：N-06）

### 4.2 Agent 管理

本节对应 01-需求分析 中的 N-07.1～N-07.5。

- 系统内置 Leader Agent。（来源：N-02，N-07.1）
- 支持创建、编辑、删除、启停 Domain Agent。（来源：N-02，N-07.1）
- 支持配置 Agent 名称、角色、人设、模型、温度、标签、知识库开关、绑定 Skills。（来源：N-02，N-07.1）
- Agent 配置项修改后需保存并生效，新增 Agent 需保存后通过连通性测试才能启用。（来源：N-02，N-07.1）
- Agent 只能调用已绑定的 Skills 和工具，且必须在 Skill 激活后才能使用对应工具。（来源：N-07.4，N-11）
- Agent 之间的调用必须通过系统接口进行，禁止直接相互调用或绕过系统调用工具。（来源：N-07.2，N-13）
- Agent 对于任务排队处理，同一时间只能执行一个任务，其他任务进入该 Domain Agent 的等待队列；队列顺位需可查看和解释。（来源：N-02，N-14）
- Leader Agent 负责任务意图理解、拆解与编排，选择合适的 Domain Agent 或工具链，并对多轮执行结果进行汇总、重试与对用户输出。（来源：N-02，N-03，N-07.2）
- Leader Agent 对复杂任务的拆解必须基于会话意图解析结果，且系统需记录并展示复杂任务判定摘要，至少说明拆分原因、主要目标或关键约束。（来源：N-02，N-14）
- Leader Agent 支持把一个复合任务拆解为多个 Domain 子任务，并为每个子任务分配目标 Domain Agent、入口模式和执行摘要。（来源：N-02，N-03，N-07.2）
- Leader Agent 支持在编排阶段为每个子任务标注 blocking 或 non-blocking 属性，并展示该子任务是否属于父任务最终交付所必需的执行项及其判定依据。（来源：N-02，N-03，N-14）
- Leader Agent 支持在全部必需子任务完成安排确认后，先向用户反馈“任务已安排好”及聚合 ETA；在全部必需子任务完成后，再向用户反馈最终结果。（来源：N-02，N-03，N-07.2，N-14）
- Leader Agent 与 Domain Agent 在实际执行时需遵循 ReAct（“推理 + 行动”）模式，以简短推理步骤、针对性工具调用和观察反馈交替推进；每轮观察结果都必须能够影响下一轮决策，直到形成最终结果、明确失败或人工介入结论。（来源：N-07.5，N-12，N-13，N-14）
- 与 N-07 相关的推理展示能力只允许输出 `reasoningSummary`、`observationSummary`、`actionSummary` 或等价压缩摘要，用于界面展示、审计与复盘；不允许把模型内部完整原始思维链作为接口返回、页面展示或长期存储内容。（来源：N-07.5，N-12，N-14）
- Domain Agent 聚焦于特定领域（如代码、文档、数据分析等），负责在其领域内进行推理和工具调用，不直接与用户交互。（来源：N-03，N-07.2）
- Domain Agent 支持独立承接 `queued`、`scheduled`、`event_triggered` 三类子任务入口模式，并向 Leader 返回“已入队 / 已登记定时 / 已登记触发”等安排确认。（来源：N-02，N-03，N-07.3，N-14）
- 同一任务中允许 Leader Agent 调度多个 Domain Agent 协同工作，但 Domain Agent 之间不得直接相互调用，只能通过 Leader Agent 进行协作。（来源：N-02，N-03，N-07.2）
- 支持记录并展示 Leader Agent 的路由决策摘要，包括所选 Domain Agent、Skill 或工具链及其主要原因。（来源：N-02，N-07.2，N-14）
- 支持记录并展示父任务与子任务关系、子任务所属 Domain Agent、排队顺位、触发状态、安排状态和聚合 ETA。（来源：N-02，N-03，N-07.2，N-07.3，N-14）
- 当同一任务涉及多个 Domain Agent 时，支持对无冲突步骤并行调度；涉及共享资源写入时，系统需自动串行化处理或提示等待。（来源：N-02，N-03，N-14）
- 禁止 Domain Agent 自行创建或销毁其他 Agent，所有 Agent 生命周期管理由系统和 Leader Agent 控制。（来源：N-07.1）

### 4.3 Skills 能力

- 支持全局 Skills 管理。（来源：N-11）
- Skills 以目录形式存在，包含 `SKILL.md` 和 `scripts/`。（来源：N-11）
- Agent 通过 Skill ID 引用全局 Skills。（来源：N-07.4，N-11）
- Skill 必须懒加载，初始不预注册全部工具。（来源：N-11）
- Agent 通过 `use_skill(skillId)` 激活 Skill 后才获得对应工具。（来源：N-07.4，N-11）

### 4.4 知识库能力

本节对应 01-需求分析 中的 N-09.1。

- 支持按 Agent 管理独立知识库。（来源：N-05，N-09.1）
- 支持导入文本和文件资料。（来源：N-08，N-09.1）
- 支持查看、删除、清空知识库资料。（来源：N-05，N-09.1）
- 支持知识检索命中摘要展示。（来源：N-05，N-09.1）
- 支持将任务附件显式加入知识库，不允许隐式自动入库。（来源：N-05，N-09.1）

### 4.5 多模态输入能力

- 支持文本、图片、PDF 作为任务输入。（来源：N-01，N-08）
- 支持 OCR / 文本抽取 / 文档解析。（来源：N-03，N-08）
- 支持展示上传状态、解析状态、解析摘要和失败提示。（来源：N-01，N-14）
- 支持解析失败后的重解析、忽略附件继续、降级执行。（来源：N-03，N-14）

### 4.6 会话与任务能力

#### 4.6.1 N-01 从会话输入到任务创建

- N-01.1 会话输入与 Task 创建：支持基于文本和附件创建复杂任务；输入消息写入默认会话上下文后，任务可在任务列表和详情页中被查询到。（来源：N-01，N-02）
- N-01.2 入口模式自动识别与判定摘要：支持系统根据会话文本和附件上下文自动识别立即执行、进入任务队列、定时执行或事件触发执行等任务触发模式；支持从会话内容中提取定时任务的计划执行时间或事件触发任务的预定义触发条件，并在任务列表和详情页展示对应的等待状态摘要。（来源：N-01，N-14）
- N-01.3 默认单一会话窗口主交互：支持把文本输入、多模态输入、进度显示、结果输出和权限授权统一收敛到同一会话窗口完成；任务详情页和结果页作为补充查看、跳转和复盘入口，不替代主交互面。（来源：N-01，N-04，N-13，N-14）
- N-01.4 多端显示范围与同步策略：支持不同任务入口和可扩展接入端接入同一会话主交互模型；P0 至少支持 Web 端，后续可扩展到其他接入端而不改变主交互语义；支持消息在多端间按策略同步，默认不同步，相关用户输入、多模态输入、进度显示、结果输出和权限授权信息默认只在发起该次用户输入的接入端展示，显式启用同步后才跨端分发。（来源：N-01，N-04，N-13，N-14）
- N-01.5 低置信度入口澄清与安全默认边界：当系统对入口模式判定置信度不足，或缺少定时 / 触发执行所需关键条件时，支持先创建一个可在任务列表和详情页中查询到的待澄清 / 待确认任务，再进入明确可观测的澄清、确认或安全默认处理路径，并记录判定失败原因、补救动作和最终采用的入口模式；低置信度入口创建的待澄清 / 待确认任务必须进入正式 `clarification_pending` 或等价状态；用户补充信息、系统完成确认或采用安全默认路径后，必须在原 Task 上完成状态推进与入口模式确认，不得通过重新创建第二个 Task 替代原单；若用户放弃继续澄清，原 Task 必须进入明确终态而不是无限停留。（来源：N-01，N-14）

#### 4.6.2 N-02 从任务创建到步骤编排

- N-02.1 复杂任务判定与可解释摘要：支持基于会话意图和附件上下文判定复杂任务，并展示复杂任务判定摘要，至少说明拆分原因、主要目标或关键约束。（来源：N-02，N-14）
- N-02.2 父子任务编排与 Domain 分配：支持对 `queued`、`scheduled`、`event_triggered` 等非立即执行复合任务先创建父任务与等待登记信息；仅在等待条件满足并完成 `TaskTriggerActivated`、`TaskReadyForPlanning`、`MemoryScopeConfirmed` 后，才生成正式父任务级 `TaskPlan` 并进入执行编排；支持对于复合任务自动创建 Leader 父任务与多个 Domain 子任务，并在任务详情中展示父子关系、目标 Agent、安排状态和 ETA 摘要；支持 Domain 子任务分别进入所属 Agent 队列或触发等待态，且队列顺位按 Agent 维度展示，不与其他 Agent 混淆。（来源：N-02，N-03，N-07.2，N-07.3，N-14）
- N-02.3 blocking / non-blocking 标注与改标治理：支持对 blocking / non-blocking 子任务标记形成可审计冻结事实；当该标记已用于生成安排通知、ETA 或降级交付判定后，后续若发生调整，系统必须记录明确变更事件、变更原因和影响范围，并将原安排结论或 ETA 标记为需重评估，禁止通过静默改标回溯性修正既有交付结论。（来源：N-02，N-14）
- N-02.4 安排完成与 ETA 生成：支持父任务在全部 blocking 子任务完成安排确认后先进入“已安排完成”阶段，并向用户反馈预计完成时间；支持对等待中的定时任务、事件触发任务和排队任务记录等待异常原因、最近一次重评估结果、命中的阈值依据和建议动作；当任务无法继续自动推进时，系统需把父任务或子任务推进到明确的人工介入状态，而不是保持静默等待。（来源：N-01，N-02，N-03，N-14）
- N-02.5 降级交付判定摘要：支持当仅 non-blocking 子任务失败、超时或被降级跳过时，父任务仍可进入可解释的部分完成或降级交付路径，并在结果页明确标注未满足的非关键子任务差异；父任务进入部分完成或降级交付时，必须产出结构化降级交付判定摘要，至少包含已完成的 blocking 子任务、被忽略或失败的 non-blocking 子任务、对最终结果的影响范围以及本次仍可交付的理由，并在结果页、通知或事件流中可追溯。（来源：N-02，N-03，N-04，N-14）

#### 4.6.3 N-03 从步骤执行到输出物生成

- N-03.1 子任务执行记录与结果回传：支持任务状态流转、步骤展示、整任务重试、取消任务；支持在任务详情中查看路由决策、并行步骤、自动重试和降级执行记录；支持每个已执行子任务形成执行记录并稳定回传父任务。（来源：N-02，N-03，N-14）
- N-03.2 blocking 完成后的最终输出生成：支持结果页展示主输出物、结构化结论、引用依据和执行摘要；支持在父任务结果页展示子任务完成概览、最终汇总结论以及“已安排完成通知”和“最终完成通知”的阶段性记录。（来源：N-03，N-04，N-05，N-14）
- N-03.3 non-blocking 差异对主链路结果的影响表达：支持在 non-blocking 子任务失败、超时或降级跳过时，在最终输出、结果页或执行摘要中明确标注其影响范围，而不是影响主链路结果成立。（来源：N-02，N-03，N-04，N-14）

#### 4.6.4 N-04 从输出物生成到结果交付

- N-04.1 最终输出结果页：支持对已完成且存在最终输出的任务稳定展示主输出物、结构化结论、引用依据和执行摘要。（来源：N-04，N-05）
- N-04.2 安排完成但未最终交付的阶段性结果页：支持对“仅完成安排确认但尚未形成最终输出”的父任务展示阶段性 ETA 与安排通知，并明确区分该通知与最终交付结果。（来源：N-04，N-14）
- N-04.3 无最终输出的终态摘要交付：支持对因放弃澄清、人工关闭、不可恢复失败或等价原因而合法终止且不存在最终输出物的任务，展示可重访的终态摘要页或终态结果区，明确说明关闭 / 终止原因、最后有效状态、是否存在阶段性安排信息以及后续可执行动作，并与“已有最终输出的结果交付完成”严格区分。（来源：N-04，N-14）
- N-04.4 交付后补充更新通知策略：当父任务已完成最终交付后，若 non-blocking 子任务晚到返回成功结果，系统只允许将该结果写入补充事件、执行摘要更新或结果页补充说明，并明确标记为交付后的补充信息；不得静默覆盖既有最终输出、隐式重开父任务主链路，或把补充结果伪装成初始最终交付内容；对此类补充更新，系统必须采用统一通知策略：要么仅在结果页 / 任务详情中被动展示，要么发送与首次最终完成通知严格区分的“补充更新”通知。（来源：N-03，N-04，N-14）

#### 4.6.5 N-05 从结果交付到知识沉淀与历史复盘

- N-05.1 结果加入知识库：支持从结果页将输出显式加入知识库，在知识库页查看对应文档，并支持知识检索命中摘要展示。（来源：N-05，N-09.1）
- N-05.2 历史任务复盘：支持历史会话查看、继续追问、自动生成标题，并支持从历史任务或历史会话中重新打开该任务进行复盘。（来源：N-05.2）

### 4.7 工具与环境交互能力

- 支持 Agent 通过 Skill 获得读、写、编辑、执行四类基础能力。（来源：N-03，N-11，N-13）
- 支持只读工具：`fs.read_text`、`fs.list_dir`、`fs.stat_path`。（来源：N-13）
- 支持写入工具：`fs.write_text`。（来源：N-13）
- 支持编辑工具：`fs.apply_patch`。（来源：N-13）
- 支持执行工具：`shell.exec`。（来源：N-13）
- 所有工具必须在沙箱内运行，并记录调用日志、错误码和耗时。（来源：N-13，N-14）
- 支持对读、写、编辑、执行和外部系统交互能力分别配置权限策略，如默认执行、触发权限确认、默认拒绝。（来源：N-07.4，N-13，N-14）
- 支持对权限策略采用 `deny > ask > allow` 或等价优先级进行统一裁决；命中更严格策略时必须覆盖较宽松配置，避免因 Agent、Skill 或资源多重配置产生越权执行。（来源：N-13，N-14）
- 对来源于低置信度入口判定的安全默认路径，系统仅允许确定入口模式或保持等待，不得借此绕过既有权限裁决；后续命中 `ask` 或 `deny` 的动作仍必须进入权限确认或拒绝执行流程。（来源：N-01，N-13，N-14）
- 对文件写入、文本编辑、命令执行和外部系统调用等高风险动作，系统需在执行前按权限策略发起权限请求并进入权限确认流程，同时明确展示目标对象、动作类型和潜在影响。（来源：N-03，N-13，N-14）
- 支持按 Agent、Skill、工具和目标资源配置权限范围；未授权的写入、执行或外部交互请求必须形成拒绝执行的权限决策并写入审计日志。（来源：N-07.4，N-13，N-14）
- 支持为低风险动作配置默认执行的权限策略，为高风险动作配置触发权限确认的权限策略；所有权限决策需可追踪、可查看、可复盘，并至少记录审批结果、审批人和对应任务事件。（来源：N-13，N-14）

### 4.8 Prompt 与推理能力

- 支持 Leader Prompt、Domain Prompt、Output Prompt 三类模板。（来源：N-02，N-03，N-12）
- 支持 Prompt Version、模板 ID、调用记录。（来源：N-12，N-14）
- 支持 Prompt 超长裁剪、重试上下文注入、降级执行。（来源：N-12）
- 支持以 ReAct 模式约束 Agent 执行 Prompt，明确区分简短推理、工具动作和观察结果，并要求后续决策基于最近有效观察继续推进，而不是生成脱离工具反馈的长篇静态推理。（来源：N-07.5，N-12，N-13，N-14）
- 支持把 ReAct 相关输出约束为“摘要而非原始思维链”的契约边界，确保 Prompt、接口、界面和存储层只消费可审计的压缩摘要，而不暴露完整内部推理文本。（来源：N-07.5，N-12，N-14）

### 4.9 可观测性与追踪能力

- 支持任务时间线、步骤追踪、Agent 参与摘要。（来源：N-04，N-05，N-14）
- 支持知识检索与工具调用追踪。（来源：N-05，N-09.1，N-14）
- 支持失败归因、错误分类、建议动作。（来源：N-03，N-14）
- 支持展示步骤级 Agent、工具、知识和记忆使用摘要，以及每个 Agent 的耗时与调用次数。（来源：N-04，N-09.2，N-14）
- 支持基础统计指标，如成功率、失败率、平均耗时、知识命中率、附件解析成功率。（来源：N-14）
- 支持基于稳定 `sequence`、`cursor` 或等价游标对任务时间线进行断线补拉、重复事件幂等去重和顺序恢复，保证前端恢复后的时间线与后端事实源一致。（来源：N-14）

### 4.10 记忆能力

本节对应 01-需求分析 中的 N-09.2～N-09.5。

- 支持会话级短期记忆，在同一任务内复用关键背景信息和中间结论。（来源：N-05，N-09.2）
- 每天定时总结当天的所有会话记录，剔除无关信息后写入持久记忆。（来源：N-05，N-09.3，N-10）
- 对于任务附件，文件本体存入文件系统；解析后仅将关键信息和附件文件位置写入持久记忆，并关联到对应会话。（来源：N-05，N-08，N-09.3）
- 支持按 Agent 管理持久记忆条目（如用户偏好、常用上下文），可查看、编辑、删除和清空。（来源：N-05，N-09.2）
- 支持区分会话短期记忆、Agent 持久记忆和知识库检索结果三类上下文来源，并展示其加载顺序、覆盖关系和本次任务的实际使用范围。（来源：N-09.2，N-14）
- 支持在任务执行前快速查看并选择本次可加载的记忆范围，如本会话、历史任务和手动收藏条目。（来源：N-05，N-09.2）
- 支持在任务执行前选择是否加载相关记忆，并在结果页展示使用到的记忆条目。（来源：N-05，N-09.2）
- 支持对 `queued`、`scheduled`、`event_triggered` 等等待态任务冻结入口上下文快照，并在释放执行时按既定策略使用冻结快照、最新记忆或受控合并结果；当快照失效且无法满足最小执行上下文时，系统进入降级或人工介入流程。（来源：N-09.4，N-09.5，N-14）
- 支持在结果页将本次结果一键加入记忆或从记忆中移除，并展示本次任务新增或更新的记忆条目。（来源：N-05，N-09.2，N-10）

## 5. P0 不纳入范围

- 多租户与复杂权限系统。
- DAG 工作流编排器。
- 音频、视频输入。
- 图像生成、语音合成等输出型多模态能力。
- 微服务拆分、分布式任务队列、复杂消息中间件。

## 6. P1 / P2 扩展方向

### 6.1 P1

- 更多文档格式支持。
- 更强的知识库运营能力。
- 任务模板与任务复用。
- 更丰富的统计面板与运营视图。

## 7. 验收标准

P0 验收以以下结果为准：

- 用户能完成模型配置并通过连通性测试。
- 用户能在默认单一会话窗口中上传附件并发起复杂任务，无需先显式新建会话。
- 用户能在单一会话窗口完成输入、查看进度、接收结果和处理权限授权；任务详情页与结果页只作为补充查看入口。
- 系统能把任务拆解为不少于规划、执行、汇总三级阶段。
- 用户能查看步骤、耗时、Agent、失败信息和工具摘要。
- 用户能查看父任务与子任务关系、各子任务所属 Domain Agent、排队或触发状态，以及聚合 ETA。
- 用户能查看 Agent 执行过程中形成的关键推理摘要、工具调用记录和观察结果摘要，并理解这些观察如何影响后续动作或最终结论。
- 用户能查看复杂任务判定摘要，理解系统为什么把当前请求拆分为父任务和多个子任务。
- 用户能查看入口模式判定失败或低置信度时的澄清 / 确认 / 安全默认处理结果，理解系统为何没有直接执行。
- 用户能查看低置信度入口任务进入正式 `clarification_pending` 或等价状态，并在原 Task 上完成澄清、确认或安全默认后的状态推进；放弃澄清时，原 Task 会进入明确终态而不是生成新单或长期挂起。
- 结果页能稳定展示主输出物、结构化结论、引用依据和执行摘要。
- 对不存在最终输出物但已合法终止的任务，系统仍能提供可重访的终态摘要页或终态结果区，并与真正完成最终交付的结果页明确区分。
- 用户能在任务详情或结果页看清主要路由决策、Agent 参与情况，以及自动重试或降级执行记录。
- 系统能先反馈“任务已安排好”和 ETA，再在全部必需子任务完成后反馈最终结果。
- 系统能区分 blocking 与 non-blocking 子任务，并在非关键子任务失败时给出可解释的部分完成或降级交付结果。
- 系统在 blocking / non-blocking 标记已用于安排通知、ETA 或交付判定后，不会通过静默改标回溯性修正既有结论；若发生调整，用户能看到明确变更记录和重评估结果。
- 系统在部分完成或降级交付时能展示结构化降级交付判定摘要，并在 non-blocking 子任务晚到成功时仅以补充更新方式呈现，不静默改写既有最终输出。
- 系统对晚到 non-blocking 成功结果采用统一通知策略，不会把补充更新伪装成第二次“最终完成”通知。
- 当等待中的任务长时间未释放、触发条件失效或冻结快照失效时，系统能展示等待异常原因、命中的阈值依据，并在不能自动恢复时进入明确人工介入状态。
- 用户能查看本次任务使用和写入的关键记忆条目，并对记忆加载范围进行控制。
- 系统能对写入、编辑、执行和外部系统交互按权限策略进行默认执行、触发权限确认或拒绝执行处理，并保留完整审计记录。
- 系统能在事件连接中断后基于 `sequence` 或 `cursor` 恢复完整时间线，不因重复投递产生重复状态推进。
- 在默认配置下，任务相关输入、进度、结果和权限授权只在发起端展示；启用同步策略后，相关消息才能跨端同步且不改变原时间线事实顺序。
- 系统能在解析失败、局部失败、Prompt 超长、Skill 未激活等场景下给出可执行反馈。

## 8. 功能要求结论

P0 阶段功能要求的核心不是覆盖更多能力点，而是围绕任务主链路建立可执行、可追踪、可解释的产品闭环。
