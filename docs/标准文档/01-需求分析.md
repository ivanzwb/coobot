# BiosBot 需求分析

## 1. 文档目的

本文档用于分析 BiosBot 的业务背景、用户需求、场景约束、优先级与需求来源，为后续方案设计提供依据。
本文档主要支撑 02-功能要求、03-技术分析、04-概要设计、05-详细设计 以及后续接口、数据库、界面和测试文档的编制。
本文档重点回答“为什么要做、谁来使用、核心场景是什么、需求优先级如何划分”。

## 2. 背景分析

当前通用大模型产品主要解决"单轮问答"问题，但在以下场景中存在明显不足：

| 不足场景 | 问题描述 |
|---------|---------|
| 复杂任务 | 需要拆解、执行和复盘，单轮输出不可追踪 |
| 业务资料 | 大量存在于图片、PDF、扫描件等非纯文本形式 |
| 私有知识 | 个人和小团队需要使用私有知识，但不希望把全部资料交给外部平台 |
| 多 Agent 组合 | 不同任务需要不同 Agent 和工具组合，单一模型无法稳定覆盖全部路径 |

**BiosBot 需求由此产生：**

- 需要一个**本地优先**、**可配置**、**可追踪**的多智能体工作台
- 需要让多模态输入、有学习能力，有知识库、Skills、工具和任务系统真正协同工作

## 3. 用户需求分析

### 3.1 个人开发者

| 类别 | 内容 |
|-----|------|
| **核心诉求** | 快速搭建和调试多 Agent 系统；为 Agent 绑定 Skills 和工具；读取工程文件、执行命令、生成代码或文档 |
| **核心痛点** | Agent 配置复杂且缺少统一界面；Skills 和工具缺乏规范和管理，难以复用；历史丢失，出现断片，无法复盘和积累经验；缺少统一的任务追踪和结果展示界面；工具接入方式零散，不容易复用；任务失败时难以定位是模型、工具还是上下文问题 |

### 3.2 专业知识工作者

| 类别 | 内容 |
|-----|------|
| **核心诉求** | 上传资料后由系统自动分析并生成可交付结果；明确结果依据与过程摘要；支持对图片、扫描件、年报、纪要等资料进行理解 |
| **核心痛点** | 资料预处理成本高；结果缺少可核验依据；大模型对复杂任务的过程不透明 |

### 3.3 小团队负责人

| 类别 | 内容 |
|-----|------|
| **核心诉求** | 用多 Agent 固化团队工作流；快速复盘失败任务；将高价值结果和资料沉淀为长期资产 |
| **核心痛点** | 任务失败难复盘；知识和配置难共享；缺少长期可积累的任务与资料体系 |
- 缺少长期可积累的任务与资料体系。

## 4. 场景分析

| 场景 | 描述 |
|-----|------|
| **4.1 资料分析** | 用户输入文本问题并上传图片/PDF，系统先做解析，再做分析，再生成结构化结论和草稿 |
| **4.2 工程协作** | 用户要求 Agent 阅读项目文件、修改文本文件、执行脚本并返回结果摘要 |
| **4.3 失败复盘** | 用户查看任务时间线、工具调用和错误摘要，定位失败步骤并重新执行 |
| **4.4 多 Agent 安排** | 用户输入复合任务后，Leader Agent 先拆分出多个分配给不同 Domain Agent 的子任务 |
| **4.5 多 Agent 汇总交付** | 各 Domain Agent 在各自子任务完成后把结果回传给 Leader 父任务 |
| **4.6 等待态异常与人工介入** | 当复合任务中的定时任务、事件触发任务或排队任务在各自配置的等待阈值内未能进入执行链路时 |

### 4.4 多 Agent 安排（详述）

- Leader Agent 在编排阶段需为每个子任务标注其是否属于父任务交付所必需的 **blocking** 子任务或可降级的 **non-blocking** 子任务，并在任务详情和事件流中保留该判定摘要
- 系统在全部 blocking 子任务完成安排确认后，先向用户反馈"任务已安排好"和聚合预计完成时间
- Leader Agent 随后继续接收新任务，不阻塞等待当前复合任务执行结束
- 若部分子任务因等待事件长期未命中、队列长时间未释放、触发订阅失效或冻结快照失效而无法继续推进，系统必须把父任务推进到明确可观测的**等待异常**或**人工介入**状态

### 4.5 多 Agent 汇总交付（详述）

- 若仅 non-blocking 子任务失败或被降级跳过，系统应在最终结果中**明确标记该差异**，而不是把该任务误判为主链路失败

### 4.6 等待态异常与人工介入（详述）

- 系统不能保持无反馈等待，必须记录：**等待异常原因**、**当前受阻子任务**、**最近一次重评估结果**、**命中的阈值依据**和**建议动作**
- 该阈值至少支持按 `queued`、`scheduled`、`event_triggered` 三类任务分别配置
- 若系统通过降级仍不能满足最小执行上下文，则需进入**人工介入状态**
- 允许用户在界面上查看原因，选择**继续等待**、**重新安排**或**手动关闭**

## 5. 需求拆解

本章列出的需求以 `N-xx` 编号，供功能要求、技术分析和测试用例文档引用。

### 5.1 业务主链路需求

- N-01 从会话输入到任务创建（优先级：P0，对应“从会话到任务”的入口链路）。由以下二级子需求组成：
	- N-01.1 会话输入与 Task 创建。可验收口径：用户可以在会话窗口发送消息并成功创建至少 1 个 Task，且该 Task 能在任务列表和详情页中被查询到。

	- N-01.2 入口模式自动识别与判定摘要。可验收口径：系统需根据会话输入自动识别任务入口模式，并在创建后为用户展示可解释的入口判定摘要，至少说明该任务为何被识别为 `immediate`、`queued`、`scheduled` 或 `event_triggered`，以及在等待场景下为何未立即执行。

	- N-01.3 默认单一会话窗口主交互。可验收口径：用户侧主交互面为单一会话窗口，文本输入、多模态输入、进度反馈、结果输出和权限授权默认都在该窗口内完成，任务详情页和结果页仅作为补充查看与复盘入口。

	- N-01.4 多端显示范围与同步策略。可验收口径：系统需支持不同入口和可扩展接入端，但多端消息默认不同步；只有在用户或系统明确启用同步策略后才允许跨端同步。

	- N-01.5 低置信度入口澄清与安全默认边界。可验收口径：
	  - 当系统对入口模式判定置信度不足时，应先创建待澄清/待确认 Task，置于 `clarification_pending` 状态
	  - 禁止静默猜测后直接执行，必须记录判定失败原因、补救动作和最终入口模式
	  - 用户补充信息或采用安全默认路径后，必须在原 Task 上完成状态推进
	  - 不得绕过 N-13 定义的 `deny > ask > allow` 权限裁决
	  - 待澄清/安全默认未完成前，不得进入实际执行链路
- N-02 从任务创建到步骤编排（优先级：P0，对应 Leader Agent 对任务的拆解、父子任务建立与规划）。由以下二级子需求组成：
	- N-02.1 复杂任务判定与可解释摘要
	  对于经会话意图解析判定为复杂任务的请求，系统必须输出复杂任务判定结果；该结果至少说明拆分原因、主要目标或关键约束，并在任务详情页可见、在事件流或审计链路中可追溯。
	- N-02.2 父子任务编排与 Domain 分配
	  系统生成可解释的 Leader 父任务与 Domain 子任务编排结果；子任务数量和阶段划分由任务复杂度、主要目标和约束决定，但必须足以支撑可追踪的拆分、安排和交付；对于已进入执行链路的父任务，系统生成可解释的 TaskPlan，并在任务详情页展示子任务列表、目标 Domain Agent、任务状态和顺序；对于 `queued`、`scheduled`、`event_triggered` 等等待态父任务，允许先完成任务创建、等待登记和后续释放，在释放进入执行链路后再生成正式 TaskPlan。
	- N-02.3 blocking / non-blocking 标注与改标治理
	  Leader Agent 在编排阶段必须为每个子任务标注其是否属于父任务交付所必需的 blocking 子任务或可降级的 non-blocking 子任务，并在任务详情页展示该标记及其判定依据；该标记一旦被用于生成安排通知、ETA 或降级交付判定，就必须作为可审计事实被冻结；后续若因人工介入或系统纠正而调整 blocking / non-blocking 归类，必须记录明确变更事件、变更原因和影响范围，并同步使原安排结论或 ETA 进入需重评估状态，禁止通过静默改标来回溯性修正既有交付结论。
	- N-02.4 安排完成与 ETA 生成：父任务在全部必需子任务完成安排确认后，才能形成可解释的安排完成状态与 ETA 摘要；该摘要必须可在任务详情或事件流中追溯，且不能把安排完成误解释为最终交付完成。
	- N-02.5 降级交付判定摘要：若 non-blocking 子任务失败但父任务主要目标已达成，系统需允许父任务进入可解释的降级交付或部分完成路径，而不是一律判定为主链路失败；该“主要目标已达成”的判定必须输出结构化的降级交付判定摘要，至少包含已完成的 blocking 子任务、被忽略或失败的 non-blocking 子任务、对最终结果的影响范围以及本次仍可交付的理由，并在结果页或事件流中可追溯。
- N-03 从步骤执行到输出物生成（优先级：P0，对应 Domain Agent 子任务执行、结果回传与父任务汇总）。由以下二级子需求组成：
	- N-03.1 子任务执行记录与结果回传
	  每个已执行的子任务在系统中都有执行记录，包括成功/失败、耗时、调用的 Agent/Skill/工具，且子任务结果能够稳定回传父任务。

	- N-03.2 blocking 完成后的最终输出生成
	  在全部 blocking 子任务完成后，父任务生成至少 1 个可被结果页展示的最终输出。

	- N-03.3 non-blocking 差异对主链路结果的影响表达
	  对于 non-blocking 子任务的失败、超时或降级跳过，系统需在最终输出或执行摘要中明确标记其影响范围，而不是影响父任务主链路结果的成立。
- N-04 从输出物生成到结果交付（优先级：P0，对应结果页展示与交付物生成）。由以下二级子需求组成：
	- N-04.1 最终输出结果页
	  对于已完成且存在最终输出的任务，结果页能稳定展示主输出物和结构化结论，并能通过固定 URL `/results/:taskId` 被重新访问。
	- N-04.2 安排完成但未最终交付的阶段性结果页：对于仅完成安排确认但尚未形成最终输出的父任务，结果页可以提前访问并展示阶段性 ETA 与安排通知，但必须明确标记“尚未完成最终交付”，且不得把该通知视为最终交付结果。
	- N-04.3 无最终输出的终态摘要交付：对于因用户放弃澄清、人工介入后手动关闭、不可恢复失败或等价原因而合法终止且不存在最终输出物的任务，系统仍必须提供可重访的结果或终态摘要页，明确展示关闭 / 终止原因、最后有效状态、是否存在阶段性安排信息以及后续可执行动作，并将其与“已有最终输出的结果交付完成”严格区分；只有存在最终输出物时，才视为结果交付完成，但不存在最终输出物的合法终态任务仍必须具备可审计、可复访的终态交付摘要。
	- N-04.4 交付后补充更新通知策略：若父任务已完成最终交付后，仍有 non-blocking 子任务晚到返回成功结果，系统必须把该结果写入补充事件或执行摘要更新，并明确标注为交付后的补充信息，不得静默覆盖既有最终输出、不得隐式重开父任务主链路，也不得在无明确标识的情况下把补充结果伪装成初始最终交付内容；对于该类补充更新，系统必须定义统一通知策略：要么明确仅在结果页 / 任务详情中被动展示而不再次发送“最终完成”通知，要么发送独立的“补充更新”通知并与首次最终完成通知严格区分，禁止以第二次通知伪装成首次最终交付。
- N-05 从结果交付到知识沉淀与历史复盘（优先级：P0，对应任务复盘与长期资产沉淀）。由以下二级子需求组成：
	- N-05.1 结果加入知识库：用户可以从结果页将输出显式加入知识库，并在知识库页查看对应文档。
	- N-05.2 历史任务复盘：用户可以从历史任务或历史会话中重新打开该任务进行复盘。

### 5.2 关键支撑需求

本节先对 N-07、N-09 做二级拆解，其他关键支撑需求暂保持一级编号，便于在不扩大同步面的前提下提升关键支撑能力的追踪精度。

- N-06 模型配置（优先级：P0，支撑多 Agent 与多模态能力稳定运行）。可验收口径：通过设置页完成模型配置并通过连通性测试，配置修改**实时生效无需重启服务**，且所有模型调用均走**统一适配层**。

- N-07 Agent 管理（优先级：P0，支撑 Leader / Domain Agent 的创建、配置、队列与调度）。可验收口径：

	### 1）N-07.1 Agent 生命周期与基础配置管理
	用户可在界面上创建、编辑、启停 Domain Agent；保存后的 Agent 配置能够稳定生效并通过连通性校验；**Leader Agent 始终存在且不可删除**。

	### 2）N-07.2 Leader / Domain 角色边界与协作约束
	- Leader Agent 负责**复合任务的路由、拆解、编排与汇总**
	- Domain Agent 负责**各自领域内的执行**
	- Domain Agent 之间**不得绕过 Leader** 直接相互调用
	- Leader 在父任务进入等待、安排确认或执行后**仍能继续接收并编排新任务**，不因单个复合任务长期占用而停止接单

	### 3）N-07.3 Domain 显式队列与等待态协调
	- 每个 Domain Agent 都具备独立、可解释的逻辑任务队列，用于承载 `queued` 子任务
	- `scheduled` 与 `event_triggered` 子任务通过**统一等待订阅与重评估链路**管理
	- 相关队列顺位、等待状态与安排确认结果可在任务详情或事件流中追溯

	### 4）N-07.4 Agent 权限边界与白名单控制
	- 所有 Agent 的工具与外部环境权限**只能通过配置和 Skill/工具白名单授予**
	- **不允许运行时自行提升权限**
	- **未授权能力不得被 Agent 直接调用**

	### 5）N-07.5 ReAct 执行模式与摘要边界
	- Leader / Domain Agent 在进入执行链路后，必须遵循 **ReAct（"推理 + 行动"）模式**
	- 以简短、可控的推理步骤与针对性的工具调用交替推进
	- 系统需保留可审计的**推理摘要**、**行动记录**与**观察结果**
	- 这些摘要属于面向产品、审计与复盘的**压缩事实摘要**，**不等同于也不得替代**模型内部完整原始思维链

	### 6）N-07.6 Agent 超时控制
	- 系统需支持配置 Agent 执行单个步骤的**超时时间**
	- 超时后系统需记录超时事件，并支持自动重试或进入人工介入状态
	- 超时配置需支持按 Agent 类型、任务类型分别设置

	### 7）N-07.7 并发执行与任务优先级
	- 系统需支持 Domain Agent **并发执行**子任务，提升执行效率
	- 系统需支持任务**优先级配置**，高优先级任务可优先执行
	- 需防止优先级反转，确保公平调度

	### 8）N-07.8 外部服务不可用时的降级策略
	- 当外部服务（如大模型 API）不可用时，系统需有明确降级策略
	- 降级策略包括：返回缓存结果、切换备用模型、提示用户稍后重试
	- 降级事件需记录到审计日志

- N-08 多模态输入（优先级：P0，支撑文本 + 图片 / PDF 等资料入口）。可验收口径：用户可在会话或任务创建时上传图片和 PDF，**单个附件大小上限为 50MB**，系统能完成解析并在任务详情或附件详情中展示解析状态和摘要。

- N-09 记忆与知识库（优先级：P0，支撑会话记忆、知识检索与自动学习闭环）。可验收口径：

	### 1）N-09.1 知识库写入与检索命中
	任务结果或资料可显式写入知识库，并能在后续任务中被检索命中；命中记录可在执行报告或结果页中查看。

	### 2）N-09.2 会话短期记忆与 Agent 持久记忆治理
	- 系统至少区分**三类上下文来源**：会话短期记忆、Agent 持久记忆和知识库检索结果
	- 在任务执行前**明确是否加载**相关记忆，避免记忆污染和人格漂移
	- 各类上下文的**加载顺序、覆盖关系和适用范围**可被追踪与解释

	### 3）N-09.3 会话日摘要整理与长期记忆沉淀
	- 系统支持自动学习功能，按日汇总当天会话记录
	- 自动**过滤无用的对话寒暄、重复确认**等干扰内容
	- 将有价值的信息整理为**结构化记忆**写入可追溯的持久记忆
	- 记忆内容**可被人工清理或编辑**

	### 4）N-09.4 等待态冻结快照与释放时上下文策略
	- 对于 `queued`、`scheduled`、`event_triggered` 等等待态任务，系统需明确**入口上下文快照的冻结策略**
	- 定义释放执行时默认采用**冻结快照、最新记忆或受控合并**的规则
	- 该规则必须**稳定、可追踪、可解释**

	### 5）N-09.5 快照失效后的降级与人工介入
	当冻结快照失效或合并后仍无法满足最小执行上下文时，系统必须**触发降级或人工介入**，而不是静默继续执行；相关失效原因、处理动作和最终状态可在任务详情或事件流中追溯。

- N-10 自我学习与经验积累（优先级：P1，支撑任务模板、经验复用和长期优化）。可验收口径：用户可基于历史任务创建任务模板或经验条目，并使用该模板快速发起新任务，模板能被列出和管理；系统自动形成的会话日摘要应可作为长期经验沉淀的一部分，被后续任务复用或人工清理。

- N-11 Skill 管理与懒加载（优先级：P0，支撑工具能力的可管理与按需加载）。可验收口径：Skill 以目录形式被系统识别并出现在管理界面，**未调用 `use_skill` 前工具不可用**，调用后在当前任务上下文内工具变为可用且有调用记录。

- N-12 Prompt 模板与裁剪（优先级：P0，支撑稳定推理和上下文控制）。可验收口径：系统以模板化方式构建 Prompt，支持**版本记录**；当 Prompt 超出窗口限制时，系统必须提供**可解释、可记录的裁剪摘要**，明确本次裁剪影响了哪些上下文类别以及为何触发裁剪；裁剪后若仍无法满足最小执行上下文，应触发**降级策略**或**明确错误码**，而不能静默返回不可信结果。
- N-13 工具与外部环境交互采用沙箱设计（优先级：P0，支撑文件系统、命令执行等基础能力）。可验收口径：

	### 1）工作区定义
	系统定义明确的**工作区（Workspace）概念**，作为 Agent 执行任务的默认可见文件空间，默认为安装目录下的工作区文件夹。

	### 2）权限清单管理模式
	- 工具/Skill 需声明所需权限（**网络访问**、**文件系统读写**、**进程创建**、**动态库加载**等）
	- 根据权限清单与权限策略放行，**未声明权限默认禁止**
	- **禁止提权操作**
	- 工具权限只能通过**配置和 Skill 绑定授予**，不允许 Agent 直接调用未授权命令或访问工作区之外的资源

	### 3）权限策略分类
	系统按操作风险分为三类，默认审批策略如下：

	| 操作类型 | 示例工具 | 默认策略 |
	|---------|---------|---------|
	| 只读操作 | `fs.read_text`、`fs.list_dir`、`fs.stat_path` | allow |
	| 受控写入操作 | `fs.write_text`、`fs.apply_patch` | ask |
	| 命令执行/外部副作用操作 | `shell.exec`、外部系统调用 | deny |

	- 权限裁决采用 **`deny > ask > allow` 优先级原则**：命中更严格策略时必须覆盖较宽松配置
	- 命中 `ask` 的操作**必须要求用户确认后方可继续**
	- 审批结果、审批人和对应任务事件需在审计链路中**可追溯**

	### 4）低置信度入口的特殊处理
	对于来源于 N-01 低置信度安全默认路径的任务，**后续工具调用仍必须完全遵守本条审批策略**，不能因入口模式已被默认确定而跳过权限确认或拒绝裁决。

	### 5）安全防护
	- 系统对所有工具调用参数**先进行安全扫描**，再执行脚本注入过滤
	- 过滤内容包括：
		- **命令注入**：`;`、`|`、`&&`、`$()` 等
		- **路径遍历**：`../`、绝对路径逃逸
		- **环境变量注入**、**特殊字符转义**等
	- 写入操作安全策略：
		- 在权限校验通过的情况下，**允许工具生成可执行文件**（.exe、.sh、.bat、.ps1、.py 等）
		- 但在**执行**这些可执行文件前，需要**再次遵循权限策略和安全检查**
		- 禁止写入包含恶意代码的内容

	### 6）审计与日志
	所有工具调用**必须在工具调用日志中完整记录**，包括：
	- 工具名称
	- 调用参数完整内容
	- 调用结果或返回值
	- 执行开始时间、结束时间
	- 调用成功或失败状态

- N-14 事件流与追踪（优先级：P0，支撑任务时间线、安排确认、ETA 反馈、失败定位与统计分析）。可验收口径：

	### 1）事件流覆盖范围
	父任务和子任务生命周期会产生完整事件流，至少覆盖以下关键阶段：

	| 阶段类型 | 事件示例 |
	|---------|---------|
	| 入口阶段 | `TaskCreated`、入口判定、入口等待 |
	| 编排阶段 | 子任务拆分、安排确认、ETA 估算 |
	| 执行阶段 | 成功释放、步骤执行、结果回传 |
	| 终态阶段 | 完成、失败、取消、超时、人工关闭、人工介入 |

	### 2）任务恢复机制
	- 系统重启后**自动检查并恢复**处于执行中或等待态的中断任务
	- 任务恢复后**继续执行**而非重新开始

	### 3）延迟任务释放追溯
	- 对于 `queued`、`scheduled`、`event_triggered` 三类延迟任务，成功释放时必须能够追溯到**稳定的释放事件序列**
	- 不能仅看到最终进入执行的结果

	### 4）等待异常处理
	对超过配置阈值仍未释放、触发订阅失效、等待条件不再满足或冻结快照失效的任务：
	- 必须产生明确事件和状态变更
	- 用户可在任务详情中查看：**等待原因**、**命中的阈值依据**、**失败步骤**和**错误码**
	- 可基于这些事件生成基础统计报表

	### 5）错误重试机制
	- 对于**可恢复的错误**，系统支持在限定次数内的**自动重试**
	- 在事件流中**清晰标记重试行为和结果**
	- 对于**不可自动恢复**的场景，系统必须进入**明确的人工介入状态**而不是静默停滞
	- 向用户提供：**继续等待**、**重新安排**或**手动关闭**等可执行动作

	### 6）时间线恢复与幂等性
	任务时间线必须支持：
	- 基于稳定 `sequence`、`cursor` 或等价游标的**断线补拉**
	- **重复事件幂等去重**和**顺序恢复**
	- 恢复后的时间线应与后端事实源保持一致
	- 不能因重复投递、连接中断或补拉重放而产生**错误排序**或**重复状态推进**

- N-15 监控告警机制（优先级：P1，支撑系统健康感知、异常提前预警与运营决策）。可验收口径：

	### 1）N-15.1 系统健康指标采集
	- 系统至少采集并持久化以下核心指标：任务成功率、平均耗时、模型调用成功率、平均响应时间、知识检索命中率、附件解析成功率、Agent 执行效率、队列积压深度
	- 指标数据按**小时/天维度聚合存储**，支持历史查询

	### 2）N-15.2 异常检测与预警规则
	- **阈值告警**：如任务失败率超过 X%
	- **趋势告警**：如连续 N 次任务耗时上升
	- **等待异常告警**：如队列积压超过 N 个任务或等待超 X 分钟未释放
	- 告警触发时写入系统事件并在监控面板展示

	### 3）N-15.3 分级告警与通知策略
	- 告警支持分级：**信息**、**警告**、**严重**
	- 不同级别对应不同通知策略
	- P0 阶段默认仅在监控面板展示告警记录，**不强制推送**
	- 用户可配置**告警沉默窗口**避免重复打扰

	### 4）N-15.4 监控面板与运营视图
	系统提供基础监控面板，展示：
	- 核心指标趋势图
	- 当前队列状态
	- 近期失败任务分布和 Top 失败原因
	- 面板数据支持时间范围筛选和导出

	### 5）N-15.5 告警与任务追踪联动
	- 系统告警事件需与任务事件流**打通**
	- 用户可在告警详情中直接跳转查看关联任务的时间线和详情
	- 支持从异常指标追溯到具体失败任务

- N-16 安全与凭证管理（优先级：P0，支撑敏感信息保护与安全审计）。可验收口径：

	### 1）N-16.1 凭证安全管理
	- 系统需支持安全存储 API Keys、数据库密码等敏感凭证
	- 凭证需加密存储，不得明文暴露在配置或日志中
	- 支持凭证的创建、更新、删除和轮换

	### 2）N-16.2 数据脱敏
	- 日志和审计中需对敏感信息（身份证号、手机号、银行卡等）进行**脱敏处理**
	- 脱敏规则需可配置
	- 支持导出报告时自动脱敏

	### 3）N-16.3 审计日志保留
	- 审计日志需支持**长期保留**，至少保留 90 天
	- 支持审计日志导出，便于合规审计
	- 审计日志需包含操作人、操作时间、操作类型、影响范围

	### 4）N-16.4 网络通信安全
	- 系统支持 HTTPS/TLS 加密通信
	- 本地进程间通信需支持加密

- N-17 运维与部署（优先级：P1，支撑系统可运维性与部署便利性）。可验收口径：

	### 1）N-17.1 健康检查接口
	- 系统提供 `/health` 健康检查接口
	- 健康检查包括：服务状态、数据库连接、磁盘空间、队列积压等

	### 2）N-17.2 配置热更新
	- 系统配置修改后**无需重启服务**即可生效
	- 支持配置变更的历史记录和回滚

	### 3）N-17.3 数据备份与恢复
	- 系统支持**自动备份**机制
	- 支持从备份文件恢复数据
	- 备份文件需支持加密存储

	### 4）N-17.4 资源监控
	- 系统采集并展示 CPU、内存、磁盘使用率
	- 资源使用超过阈值时触发告警

- N-10A Agent 自我反思能力（优先级：P1，支撑输出质量自检、错误前置发现与推理闭环）。可验收口径：

	### 1）N-10A.1 输出质量自检
	- Agent 在生成最终输出前，系统自动触发**质量自检流程**
	- 自检维度至少包括：
		- 输出格式是否符合预期
		- 关键字段是否完整
		- 引用依据是否与知识库/附件内容一致
		- 是否存在明显事实错误
	- 自检结果以**结构化摘要**形式记录，不合格时支持**自动重试**或**标记需人工确认**

	### 2）N-10A.2 执行路径复盘
	任务完成后，系统自动生成执行路径复盘摘要，至少包含：
	- 关键决策点
	- 工具调用序列
	- 推理转折点
	- 潜在风险点
	- 该复盘作为任务执行记录的一部分存储，供后续任务优化和经验复用参考

	### 3）N-10A.3 推理一致性校验
	- 对于涉及多轮推理的任务，系统支持对推理链路进行**一致性校验**
	- 检测前后结论是否存在**逻辑矛盾**
	- 发现不一致时在任务详情中**标记**并给出可能原因，辅助用户判断是否需要人工介入

	### 4）N-10A.4 经验提取与主动建议
	- 系统能从历史成功任务中提取可复用的模式（如特定场景下的有效 Prompt、工具组合、参数配置）
	- 在类似新任务入口时向用户或 Agent **主动建议参考**
	- 经验提取结果可被**管理**、**编辑**和**禁用**

## 8. 风险分析

### 8.1 技术风险

| 风险项 | 描述 |
|-------|------|
| Prompt 膨胀 | Prompt 长度膨胀导致模型调用不稳定 |
| 多模态质量 | 多模态附件解析质量不稳定 |
| 权限安全 | 工具权限过大导致安全边界失控 |
| 资源浪费 | Skills 预加载导致上下文窗口和运行时资源浪费 |
| 状态不一致 | 前后端对状态和错误码理解不一致 |
| 并发冲突 | 多任务并发执行时可能产生资源竞争或数据冲突 |
| 服务降级 | 外部服务不可用时降级策略不完善可能导致任务失败 |

### 8.2 入口与等待态风险

| 风险项 | 描述 |
|-------|------|
| 低置信度入口 | 低置信度入口请求若未统一使用正式 `clarification_pending` 或等价状态，可能导致任务列表、事件流和接口契约对"是否已建单"理解不一致 |
| 等待态滞留 | 等待态任务长期滞留却没有明确阈值或异常反馈，导致用户无法判断系统是否仍在有效推进 |
| 触发失效 | 触发订阅失效或等待条件变化未被及时感知，导致任务进入不可见卡死状态 |

### 8.3 编排与交付风险

| 风险项 | 描述 |
|-------|------|
| 任务拆分 | 复杂任务判定不稳定或判定摘要不可解释，导致用户无法理解系统为何拆分任务 |
| blocking 标注 | blocking / non-blocking 标注错误，或"主要目标已达成"的判定摘要不充分，可能导致父任务被错误交付或错误终止 |
| 标注篡改 | blocking / non-blocking 标注若在安排通知或 ETA 发出后仍可被静默改写，可能导致系统以回溯性改标掩盖错误编排或错误交付 |

### 8.4 权限与安全风险

| 风险项 | 描述 |
|-------|------|
| 权限绕过 | 低置信度安全默认路径与工具审批策略衔接不严，可能导致入口模式默认确定后错误绕过 `ask` / `deny` 权限裁决 |
| 边界失效 | 低置信度安全默认路径若未限制后续动作边界，可能在澄清未完成前提前触发规划、拆分或执行 |

### 8.5 结果展示风险

| 风险项 | 描述 |
|-------|------|
| 补充更新 | non-blocking 子任务在父任务交付后晚到成功，若缺少统一补充更新规则，可能导致结果页、通知记录和事件流出现相互矛盾的事实展示 |
| 终态歧义 | 对无最终输出物但已合法终止的任务若未定义统一终态摘要页和统计口径，可能导致结果页、任务终态和报表对"是否已完成交付"给出冲突解释 |
| 阶段混淆 | 阶段性 ETA / 安排通知与最终输出展示边界不清，导致用户将"已安排完成"误解为"已交付完成" |

### 8.6 监控与质量风险

| 风险项 | 描述 |
|-------|------|
| 告警脱节 | 监控告警若与任务事件流未打通，可能导致告警与具体任务脱节，无法追溯异常根因 |
| 自检过度 | Agent 质量自检若过度触发，可能导致任务执行耗时显著增加，需平衡自检深度与执行效率 |
| 校验误报 | 推理一致性校验若误报率过高，可能导致用户对系统信任度下降，需在召回率和准确率之间取得平衡 |
| 历史偏见 | 经验提取若引入历史偏见或过时模式，可能导致新任务被错误引导，需建立经验有效性评估机制 |

### 8.7 安全与运维风险

| 风险项 | 描述 |
|-------|------|
| 凭证泄露 | API Keys、密码等凭证若未妥善管理，可能导致安全风险 |
| 数据泄露 | 敏感信息若未脱敏，可能导致隐私泄露 |
| 磁盘满 | 磁盘空间不足时可能导致任务失败或数据丢失 |
| 配置错误 | 配置修改若未充分测试，可能导致系统不稳定 |

## 9. 需求分析结论

BiosBot 的核心不是"堆叠多少个 Agent 能力点"，而是"能否围绕一个复杂任务形成从入口判定、编排执行、结果交付到知识沉淀的**稳定闭环**"。

### 核心收敛原则

| 原则 | 说明 |
|-----|------|
| **主链路优先** | 任务主链路仍是系统设计的第一优先级，聊天、展示和辅助能力都必须服务于 `N-01`～`N-05` 定义的任务闭环 |
| **结果导向** | 输出物与可交付结果优先于普通对话回复，必须保证任务结果页、阶段性安排通知、最终完成通知和终态摘要页之间**语义一致**，不把"已安排"误当作"已交付" |
| **可观测性** | 过程可观测、可追踪、可审计优先于黑盒执行，确保 `N-14` 事件流能够支撑排障、复盘与验收 |
| **统一服从** | Agent、工具、Prompt、知识库和记忆能力必须统一服从任务主链路和追踪语义 |
| **角色清晰** | Leader/Domain Agent 的价值在于稳定完成**角色边界清晰**的协作、等待态协调、权限边界控制与 ReAct 执行闭环 |
| **受控加载** | 记忆、知识和上下文装配必须坚持受控加载原则，不能在释放执行时被静默重算或覆盖 |

### P0 与 P1 阶段定位

| 阶段 | 定位 |
|-----|------|
| **P0** | 以**任务为中心**，以**交付结果**为目标，以**事件流**和**摘要化观测**为治理手段，以 Agent、记忆、知识、工具和 Prompt 的受控协作为支撑，形成一个可执行、可解释、可验收、可复盘的**复杂任务闭环系统** |
| **P1** | 扩展监控告警（N-15）与 Agent 自我反思（N-10A）能力，形成从**执行到优化**的完整反馈回路 |
