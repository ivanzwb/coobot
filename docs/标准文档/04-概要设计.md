# BiosBot 概要设计

## 1. 文档目的

本文档用于定义 BiosBot 的系统组成、分层结构、模块边界、运行机制和核心链路。
本文档主要支撑 01-需求分析 中的主链路需求 N-01～N-05，以及服务和模块层面的支撑需求 N-06～N-14。
本文档重点回答“模块如何拆分、职责如何归属、链路如何衔接”；具体技术机制、实现策略和风控细节以技术分析文档为准。

## 2. 系统目标

- 建立本地优先的多智能体任务系统。
- 以任务为中心统一会话、附件、知识、工具与结果。
- 让前端、后端、模型、知识库和工具形成单一主链路。

## 3. 系统分层

### 3.1 接入层

- Web 前端。
- HTTP API。
- WebSocket 推送。

职责：

- 请求接收。
- 参数校验。
- 响应封装。
- 实时事件下发。
关联需求：N-01，N-02，N-04。

### 3.2 应用层

- Conversation Service。
- Task Intake Service。
- Task Service。
- Attachment Service。
- Knowledge Base Service。

职责：

- 把外部请求转成内部命令。
- 协调领域服务执行顺序。
- 通过 Task Service 作为任务写入口提交任务状态变更、权限决策结果和执行命令。
关联需求：N-01，N-02，N-03，N-04。

应用层对同一 `taskId` 的写命令负责任务级串行化提交：来自编排、权限确认、步骤执行完成、重试与降级的写操作必须进入同一任务写队列，或在等价的单任务锁 / 版本校验机制下顺序提交，避免并行步骤回交时发生聚合竞争。

应用层还需统一承接任务入口模式：会话创建的任务既可以立即进入执行链路，也可以先进入本地任务队列、定时等待或事件触发等待；一旦满足可执行条件，再统一回交 Task Service 进入后续编排与执行。

其中 `Task Intake Service` 负责在任务创建前完成入口判定：它读取会话输入、附件摘要和基础上下文，产出 `TaskTriggerDecision`，统一给出 `triggerMode`、`scheduledAt?`、`triggerRule?` 与触发依据摘要，然后再调用 Task Service 创建任务。若任务进入等待态，则由 `Task Intake Service` 继续负责登记触发订阅、维护等待中的下一次判断条件，并在条件满足时向 Task Service 提交“释放进入执行链路”的写命令。

`TaskTriggerDecision` 在概要层还必须显式承接等待阈值语义：除入口模式字段外，还需带出 `triggerDecisionSummary`、`waitingThresholdMode`、`waitingThresholdBasisSummary` 和可选的 `nextTriggerCheckAt`，用于解释当前等待模式、阈值来源与下一次判断时机；这些字段在任务创建时即写入任务聚合和查询投影，不能等到异常发生后再临时补算。

对于延迟执行任务，`Task Intake Service` 还需在创建时冻结入口上下文快照：至少包括 `memoryScope`、显式选中的记忆条目、触发判定摘要和创建时可解释的输入摘要。等待态任务在后续被释放执行时默认基于该冻结快照进入记忆范围确认与加载链路，不以释放时刻的最新会话上下文重新覆盖原入口决策，除非任务被显式重新创建或重新发起。

若冻结快照中的显式记忆条目在等待期间被删除、归档、脱敏或因权限变化而不可读，系统也不得静默改用释放时刻的新上下文重算入口决策；`Task Intake Service` 必须按降级规则标记失效条目、保留原冻结摘要，并基于仍可用的冻结快照继续后续链路。若降级后已不满足最小执行上下文要求，则任务需进入明确的失败或待人工处理终态，而不是隐式扩大 `memoryScope` 或重建任务。

运行时等待态协调职责也归 `Task Intake Service` 所有：它必须在服务启动或恢复时重建未完成的等待订阅，驱动定时到期扫描、队列可用槽位检测和事件订阅恢复，并调用 `Trigger Evaluation Service` 完成重评估；`Trigger Source Adapter` 只负责把外部或本地信号标准化为触发事实，不承担后台调度、恢复编排或任务释放所有权。

任务入口模式来源于会话意图解析链路对时间表达、排队意图和事件条件的识别结果，不由前端界面显式选择；前端只负责展示系统识别后的触发模式摘要与等待原因。

### 3.3 编排层

- Task Orchestrator Service。
- Intent Resolver。
- Plan Builder。
- Capability Matcher。
- Execution Scheduler。
- Run Finalizer。

职责：

- 判断是否进入复杂任务模式。
- 生成步骤计划。
- 分配 Agent、Skill、知识检索与工具调用。
- 承担运行时步骤调度、等待恢复和续跑触发。
关联需求：N-02，N-03，N-07.2，N-07.3，N-07.5，N-11，N-12，N-13。

编排层负责产出路由、并行调度、重试与降级等编排结果，并由 Execution Scheduler 在运行时负责步骤发放、并行组控制、等待恢复和续跑触发，但不直接写入任务状态或事件流；所有编排结果和执行回执统一回交 Task Service，由其完成任务状态更新和事件追加。具体判定规则与执行策略以技术分析文档为准。

对于已进入执行链路的 Agent 回合，编排层与 `Agent Execution Service` 的协作模式在概要层固定为 ReAct：Agent 先形成简短推理摘要，再决定是否发起目标 Skill / 工具动作，接着把动作返回的观察结果回灌到当前上下文，再进入下一轮决策；循环终点只能是形成最终答案、进入明确终态或触发人工介入，不能停留在无观察支撑的长篇内部推理。

编排层中的 `Intent Resolver` 仅面向“已进入执行链路的任务”工作，用于判断复杂度、生成 TaskPlan 和选择后续执行路径；任务入口模式判定不归编排层负责，避免把创建前的入口识别与创建后的任务规划混为同一阶段。

对被判定为复杂任务的请求，编排层还必须产出 `complexityDecisionSummary` 作为正式编排事实，并随父任务级路由决策一并回交 Task Service；该摘要用于解释为何进入“Leader 父任务 + Domain 子任务”模式，并成为任务详情页与审计事件的稳定来源。

当任务被识别为需要多 Domain Agent 协作时，编排层还需把用户任务建模为“Leader 父任务 + Domain 子任务”结构：Leader Agent 负责拆分子任务、选择目标 Domain Agent，并生成父任务级编排摘要；每个 Domain 子任务作为独立任务聚合存在，拥有自己的 `taskId`、状态、时间线、入口模式和等待事实。Leader 父任务不直接代替子任务执行，而是负责汇总子任务的排队确认、预计完成时间和最终结果。

### 3.4 领域能力层

- Input Normalization Service。
- Trigger Evaluation Service。
- Trigger Source Adapter。
- Agent Execution Service。
- Knowledge Retrieval Service。
- Skill Invocation Service。
- Output Service。
- Tracking & Event Service。
- Knowledge Ingestion Service。
- Permission Policy Service。

领域能力层中的 `Trigger Source Adapter` 负责接入本地队列变更、定时器到期信号以及系统定义的预定义事件，并将它们标准化为统一的触发事实，再交给 `Trigger Evaluation Service` 判断是否满足释放条件；它不直接持有任务状态，也不直接释放任务。`Trigger Evaluation Service` 负责把时间表达、预定义事件条件和本地队列状态标准化为可执行的触发判断结果，供 `Task Intake Service` 创建等待态任务和后续释放任务使用；它不直接改写任务状态，只输出触发判定结果和触发依据摘要。Permission Policy Service 负责汇总全局、Agent、Skill、工具和目标资源等多层权限策略，并按 `deny > ask > allow` 的最严格优先级输出统一权限决策与判定轨迹；Tracking & Event Service 负责在 Task Service 提交写入时分配单任务内严格递增的 `sequence`、生成事件游标并持久化事件，但不单独成为任务状态写入口。

`Agent Execution Service` 在领域能力层负责承接 ReAct 回合执行：它消费当前步骤上下文与最近观察结果，生成下一轮简短推理摘要，决定是否调用 Skill / 工具，并把动作观察结果标准化回交任务模块与追踪模块；该服务不直接暴露完整原始思维链，只输出后续运行所需的结构化摘要与动作事实。

对等待中的任务，`Trigger Evaluation Service` 还需在每次重评估时产出 `lastTriggerEvaluationSummary`，并复用或刷新 `waitingThresholdMode`、`waitingThresholdBasisSummary`；若判断达到异常阈值，则必须输出标准化的 `waitingAnomalyCode`、`waitingAnomalySummary` 和建议动作，供 Task Service 统一落账为等待异常或人工介入事实。

所有由 `Trigger Source Adapter` 产出的触发事实都必须带有稳定幂等身份：至少包含 `triggerFactId`、`sourceType`、`occurredAt`、`sourceRef` 和可选的 `sourceSignature`。`Task Intake Service` 与 `Trigger Evaluation Service` 必须以该身份完成去重、可信来源校验和重复投递抑制，避免同一触发事实因重复接收而造成多次释放尝试。

### 3.5 基础设施层

- SQLite。
- LanceDB。
- 文件系统。
- LLM Adapter。
- 日志与事件存储。

其中 `LLM Adapter` 是外部模型依赖的唯一隔离层：负责统一模型调用协议、超时控制、失败重试、供应商切换、配额与错误归一化，并向上层暴露稳定的推理接口；应用层、编排层和 Agent 执行链路不直接依赖具体模型厂商 SDK。

## 4. 核心模块关系

### 4.1 会话模块

负责会话、消息和最近任务引用。
关联需求：N-01，N-05。

会话模块还需为后台定时记忆整理提供按日期读取的会话记录视图，供系统在任务主链之外汇总当天会话、过滤无关内容并沉淀持久记忆。

### 4.2 任务模块

负责任务生命周期、步骤、输出物和重试。
关联需求：N-02，N-03，N-04，N-14。

任务模块是任务聚合的唯一写入口：所有任务状态变更、步骤状态推进、权限决策落账和 `ExecutionEvent` 追加都必须通过 Task Service 提交，不允许编排层、Skill 模块或追踪模块直接绕过任务聚合写入任务状态。

任务模块还需拥有面向界面的任务查询投影，统一维护任务详情页和结果页所需的执行视图数据、时间线游标、权限摘要和并行状态；前端页面读取统一来自该查询投影，不直接从输出模块或事件表临时拼装。

任务模块的写一致性边界定义为“任务状态变更 + 事件追加”同提交：同一次任务写操作中，要么同时完成任务聚合更新与事件落库并分配 `sequence`，要么同时失败，不允许只更新状态不落事件，或只追加事件不更新任务状态。

任务模块对同一 `taskId` 的写入需保持顺序一致：若多个并行步骤几乎同时完成，Task Service 必须按任务级串行化顺序依次合并结果，并在必要时基于聚合版本拒绝或重试过期写入。

任务模块还需维护任务入口触发状态，包括立即执行、队列等待、定时等待和事件等待；任务详情页和任务列表需能解释任务为什么尚未执行，以及何时进入下一次可执行判断。

任务模块的任务聚合与查询投影还需显式承接新增解释字段：至少包括 `complexityDecisionSummary`、`lastTriggerEvaluationSummary`、`waitingThresholdMode`、`waitingThresholdBasisSummary`、`waitingAnomalyCode`、`waitingAnomalySummary`、`interventionRequiredReason`、`suggestedActions`、`arrangementStatus`、`arrangementSummary`、`estimatedCompletionAt`、`estimatedDurationMinutes`、`userNotificationStage`、`finalOutputReady` 和 `outputStage`，确保需求、接口、数据库和界面层的对应字段在模块边界内有明确归属。

任务模块还需支持父子任务关系：父任务用于承载 Leader Agent 的拆解、排队确认汇总、预计完成时间和最终汇总结论；子任务用于承载分配给 Domain Agent 的独立执行链路。父任务必须能查询其子任务集合、各子任务所属 Domain Agent、当前排队状态、触发状态和完成状态，并据此生成面对用户的阶段性反馈。

任务模块对入口等待态也必须保留正式事实：任务创建时需落账入口判定结果，任务进入等待态、触发条件变化、条件满足后释放执行时也必须通过 Task Service 追加正式事件，不能只在查询投影中保存当前状态摘要，否则无法支持等待态任务的审计、恢复和复盘。

任务模块还需对入口等待释放建立一次性保障：同一等待订阅只能成功将任务释放进入执行链路一次；重复的事件命中、重复的定时器回调或恢复扫描只能形成幂等重放或被拒绝，不允许同一任务重复进入 `TaskReadyForPlanning`。

当入口等待订阅失效时，任务模块必须与明确终态同提交：`TaskTriggerInvalidated` 不能孤立存在，而必须与 `TaskCancelled`、`TaskTimedOut`、`TaskFailed` 或 `TaskManuallyClosed` 一起落账，避免出现“订阅已失效但任务仍显示为可等待”的伪中间态。

对于 Leader 父任务，任务模块还需支持双阶段对外反馈：第一阶段是在所有 Domain 子任务都已被目标 Agent 接单并完成排队或触发登记后，把父任务推进到“已安排完成”阶段，并写入聚合 ETA；第二阶段是在所有必需子任务完成、失败或终止后，把子任务结果汇总为父任务最终输出，再由 Leader Agent 对用户给出最终通知。Leader 在第一阶段完成后即可继续接收下一轮用户任务，不要求阻塞等待全部子任务执行结束。

因此父任务在概要层必须稳定区分阶段性安排结果与最终交付结果：安排完成阶段更新 `arrangementStatus`、`arrangementSummary`、`estimatedCompletionAt`、`estimatedDurationMinutes`、`userNotificationStage=arranged` 和 `outputStage=arranged_only`；最终汇总完成后再切换为 `userNotificationStage=final_notified`、`finalOutputReady=true` 与最终输出阶段，禁止以 ETA 通知伪装成最终结果。

### 4.3 任务入口模块

负责任务创建前的入口判定、等待订阅登记、触发源接入协调和等待态释放。
关联需求：N-01，N-02，N-14。

任务入口模块由应用层的 `Task Intake Service` 与领域层的 `Trigger Source Adapter`、`Trigger Evaluation Service` 共同组成：前者负责任务创建前的决策与释放命令提交，后两者负责接入触发事实并给出统一判定结果。该模块不生成 `TaskPlan`，也不承担步骤级运行时调度，避免与编排模块和任务模块职责重叠。

在运行期，任务入口模块还需以 `Task Intake Service` 作为唯一等待态协调者：所有等待订阅恢复、定时扫描、队列释放检查和释放命令提交都经由它收口，避免等待态运行逻辑分散到编排层、定时作业或触发源接入层而破坏恢复一致性。

P0 阶段任务队列分为两层：第一层是 Domain Agent 显式拥有的逻辑任务队列，用于承载分配给该 Agent 的 `queued` 子任务，并按该 Agent 队列内的进入顺序稳定 FIFO 排队；第二层是运行时全局执行槽位预算，用于限制已经到达队首且可执行的任务实际进入执行链路的总量。`queuePosition` 默认表示任务在所属 Domain Agent 队列中的顺位，而不是跨全部 Agent 的单一全局顺位。

P0 阶段的可用执行槽位预算在概要层固定为“全局任务执行槽位”模型：由本地运行时配置的最大并发执行任务数减去当前已进入执行链路且尚未结束的任务数得到。`Task Intake Service` 先在每个 Domain Agent 队列内按显式 FIFO 选出候选任务，再基于这一全局预算释放到执行链路；Execution Scheduler 只消费已释放任务，不反向改写 Agent 队列顺序。除已声明的 Domain Agent 显式队列外，不允许再叠加隐藏的 Agent 子队列、模型子配额或任务类型子队列。

### 4.4 附件模块

负责上传、解析、重解析、知识沉淀入口。
关联需求：N-01，N-03，N-05，N-08，N-09.1。

### 4.5 Skill 模块

负责 Skill 管理、懒加载激活、工具注册与调用约束。
关联需求：N-07.1，N-07.4，N-11，N-13。

Skill 模块还需维护与 Skill 绑定的权限策略配置，确保工具激活不等于默认获得高风险动作的执行权限决策。

### 4.6 知识模块

负责知识库资料管理、检索与命中记录。
关联需求：N-05，N-08，N-09.1，N-14。

### 4.7 输出模块

负责生成主输出物、结构化结论、执行报告与引用关系。
关联需求：N-04，N-05，N-14。

输出模块需补充生成结果解释所需的事实性数据，供任务查询投影和复盘链路使用。

输出模块输出主输出物、结构化结论、执行报告和结果解释片段，但不直接拥有页面读模型；结果页中的权限决策摘要、并行状态和时间线展示统一由任务查询投影聚合。

对于父任务双阶段反馈，输出模块还需产出结果阶段事实：当仅完成安排确认时，需生成 `arrangementNoticeSummary` 并把 `outputStage` 标记为 `arranged_only`；当最终汇总完成时，再生成最终主输出物并把 `finalOutputReady` 切换为 `true`。这些字段由输出模块产出事实、由任务模块聚合展示，不允许前端自行推断。

## 5. 核心数据流

```text
用户输入文本 / 附件
  -> 会话服务写入消息
  -> 附件服务登记并解析
  -> 输入标准化服务生成 TaskInput
  -> Task Intake Service 调用 Trigger Evaluation Service 生成 TaskTriggerDecision
  -> 任务服务创建 Leader 父任务并落账入口模式、`triggerDecisionSummary`、`waitingThresholdMode`、`waitingThresholdBasisSummary` 与等待状态
  -> 若父任务处于 queued / scheduled / event_triggered，则由 Task Intake Service 维护父任务等待态并订阅下一次释放条件
  -> Trigger Source Adapter 接入队列变更、定时器到期或预定义事件事实
  -> Trigger Evaluation Service 重评估等待条件并生成 `lastTriggerEvaluationSummary`、释放结论及必要的等待异常事实
  -> 条件满足后，Task Intake Service 以一次性认领语义向 Task Service 提交父任务进入执行链路的释放命令
  -> 记忆选择与记忆加载范围确认
  -> 编排服务生成父任务级 TaskPlan、`complexityDecisionSummary`、拆分 Domain 子任务并产出路由决策
  -> 任务服务创建 Domain 子任务，分别写入 `parentTaskId`、目标 `assignedDomainAgentId`、子任务入口模式与等待事实
  -> 对 `queued` 子任务写入对应 Domain Agent 显式队列；对 `event_triggered` / `scheduled` 子任务登记各自触发订阅
  -> 各 Domain Agent 返回“已入队 / 已登记定时 / 已登记触发”等安排确认
  -> Task Service 聚合全部子任务安排结果，生成父任务级 ETA、`arrangementStatus`、`arrangementSummary` 与阶段性通知状态
  -> Leader Agent 向用户发送“任务已安排好、预计完成时间”通知，并释放前台交互线程
  -> Execution Scheduler 发放步骤并维护等待/续跑
  -> 生成路由决策摘要与并行执行计划
  -> 权限策略合并判定与判定轨迹生成
  -> 各 Domain Agent 的 Agent / Skill / Knowledge / Tool 执行结果分别回交对应子任务 Task Service
  -> 子任务完成后把结果、耗时和失败信息回传父任务聚合链路
  -> 当全部必需子任务完成后，输出模块基于子任务结果生成父任务级 TaskOutput、ExecutionReport、`outputStage`、`finalOutputReady` 和结果解释片段等事实性结果
  -> 输出结果作为父任务最终步骤结果回交 Task Service
  -> Task Service 提交父任务与子任务状态变更、输出引用、`userNotificationStage` 更新与事件追加
  -> Leader Agent 向用户发送最终结果通知
  -> 任务查询投影汇总任务状态、复杂任务判定摘要、等待异常摘要、阈值依据、时间线游标、知识命中、记忆读写、权限摘要、模型调用摘要与输出解释
  -> 前端展示任务详情与结果页
```

概要层的数据流顺序必须满足两条约束：第一，任务入口判定先于任务创建，并以正式事实写入任务聚合，不能由前端直接决定，也不能等到 TaskPlan 生成后再回填；第二，输出属于最终提交前的步骤结果，且先产出事实，再聚合读模型。也就是说，`TaskTriggerDecision` 先由入口判定链路产出并通过 Task Service 写入任务与入口等待事件；输出模块则在任务完成判定前生成主输出物、执行报告和结果解释片段等事实性结果，并将其回交 Task Service 作为最终步骤结果的一部分；随后由 Task Service 在同一任务提交边界内写入任务状态、输出引用和正式时间线事件，任务查询投影再基于这些事实、任务当前状态和正式时间线事件聚合页面字段。不允许结果页先依赖任务查询投影临时拼装输出事实，再反向写回输出模块。

对 `queued`、`scheduled` 和 `event_triggered` 任务，入口上下文快照在任务创建时即被冻结并成为后续释放执行的基线事实；等待期间允许重评估释放条件，但不允许隐式重算创建时的记忆范围和入口解释，避免同一任务因时间推移读到不可审计的不同上下文。若冻结快照中的部分记忆条目失效，系统只能在保留原快照与失效标记的前提下按既定降级规则继续执行，或进入 `TaskInterventionRequired` 等明确终态，不允许静默切换到释放时刻的新记忆集合。

等待态任务的释放必须满足“一次订阅、一次成功释放”约束：等待订阅可被重复检查、重复命中或在恢复过程中被重复扫描，但只有首次成功认领的释放动作可以把任务推进到 `TaskReadyForPlanning`；后续重复释放尝试只能被识别为幂等命中或冲突，不得重复推进任务主状态。对延迟任务的成功释放，最小正式事件序列也必须固定为 `TaskTriggerActivated -> TaskReadyForPlanning -> MemoryScopeConfirmed`；其中 `TaskTriggerActivated` 不得在成功释放场景下省略，`TaskTriggerReevaluated` 仅用于未释放但条件被重新判断的场景。

等待订阅还必须与任务生命周期保持一致：若任务被取消、超时、失败终止或被人工关闭，则对应等待订阅必须同步失效，不得继续接收触发源事件或在后续恢复扫描中重新释放已终止任务；该失效动作必须与 `TaskCancelled`、`TaskTimedOut`、`TaskFailed` 或 `TaskManuallyClosed` 之一同提交写入。

队列释放策略在概要层固定为 FIFO，因此 `queued` 子任务在所属 Domain Agent 队列中的释放顺序必须与该队列内的订阅建立顺序一致；取消子任务后其队列位置应被移除并允许后续同 Agent 子任务顺位前移，但不得改变尚未取消任务之间的相对顺序。P0 阶段的实际执行释放还受全局任务执行槽位预算影响，但不得绕过 Domain Agent 队列顺序。

对于“先安排、后完成”的业务场景，父任务必须支持两次面向用户的正式反馈：第一次反馈发生在全部必需子任务都完成安排确认之后，并携带聚合 ETA；第二次反馈发生在全部必需子任务完成并汇总出父任务结果之后。父任务的 ETA 可以按各 Domain Agent 当前队列位置、历史平均耗时和已知触发时间综合估算，但其估算过程必须形成可追踪摘要。

除任务主链外，系统还存在一条后台定时记忆整理链路：调度器按日读取会话模块中的当天会话记录，交由知识与记忆能力完成无关信息过滤、摘要提炼和持久记忆写入；该链路不直接改写任务时间线，但必须保留可追溯的记忆来源与整理日志。

## 6. 核心事件流

主生命周期事件：

```text
TaskCreated
  -> TaskTriggerResolved
  -> TaskTriggerWaiting / TaskReadyForPlanning
  -> MemoryScopeConfirmed
  -> TaskPlanned
  -> SubTasksCreated / DomainTaskQueued / DomainTaskScheduled / DomainTaskTriggered
  -> TaskArrangementCompleted / TaskEtaEstimated
  -> RouteResolved
  -> StepStarted
  -> TaskCompleted / TaskPartialFailed / TaskFailed / TaskCancelled / TaskTimedOut / TaskManuallyClosed / TaskInterventionRequired
```

条件触发事件：

```text
记忆与上下文：MemoryLoaded / MemoryRead / MemoryWritten
入口等待：TaskTriggerReevaluated / TaskTriggerActivated / TaskTriggerInvalidated
写入控制：TaskWriteQueued / TaskWriteConflictDetected
权限控制：PermissionCheckRequested / PermissionGranted / PermissionDenied
并行与等待：StepParallelQueued / StepWaiting
模型调用：ModelCallStarted / ModelCallCompleted / ModelCallFailed
恢复与降级：TaskAutoRetried / TaskDegraded
```

概要层对外公开的正式事件集以任务时间线与事件恢复链路为准，统一采用 `TaskCreated`、`TaskTriggerResolved`、`TaskTriggerWaiting`、`TaskReadyForPlanning`、`TaskPlanned`、`SubTasksCreated`、`DomainTaskQueued`、`DomainTaskScheduled`、`DomainTaskTriggered`、`TaskArrangementCompleted`、`TaskEtaEstimated`、`TaskWriteQueued`、`TaskWriteConflictDetected`、`StepStarted`、`TaskCompleted`、`TaskPartialFailed`、`TaskFailed`、`TaskCancelled`、`TaskTimedOut`、`TaskManuallyClosed`、`TaskInterventionRequired`、`RouteResolved`、`PermissionCheckRequested`、`PermissionGranted`、`PermissionDenied`、`StepParallelQueued`、`StepWaiting`、`TaskTriggerReevaluated`、`TaskTriggerActivated`、`TaskTriggerInvalidated`、`TaskAutoRetried`、`TaskDegraded`、`MemoryScopeConfirmed`、`MemoryLoaded`、`MemoryRead`、`MemoryWritten`、`ModelCallStarted`、`ModelCallCompleted`、`ModelCallFailed`；附件解析、知识检索、Skill 调用、Agent 执行和输出生成等内部处理动作如需落日志，可作为模块内部事件或审计事实存在，但不替代对外统一的任务时间线事件契约。

主生命周期事件用于表达任务从创建、入口模式判定、等待/释放、规划、安排、执行到完成或终止的稳定阶段；其中 `TaskTriggerResolved` 表示系统已经给出入口判定结果，`TaskTriggerWaiting` 表示任务已进入入口等待态，`TaskReadyForPlanning` 表示任务已满足进入执行链路的前置条件，`MemoryScopeConfirmed` 则是任务进入可规划阶段后的基线事件，即使使用默认记忆范围也必须产生日志事件，确保后续审计、恢复和结果解释可以基于统一起点工作。对多 Agent 场景，`SubTasksCreated` 表示父任务已完成子任务拆分，`DomainTaskQueued`、`DomainTaskScheduled`、`DomainTaskTriggered` 表示子任务已完成目标 Domain Agent 的安排确认并进入对应入口控制状态，`TaskArrangementCompleted` 表示全部必需子任务都已安排完成，`TaskEtaEstimated` 表示父任务已形成可对外展示的聚合预计完成时间。`TaskCancelled`、`TaskTimedOut`、`TaskManuallyClosed` 和 `TaskInterventionRequired` 用于表达未完成执行但已被明确终止或挂起等待人工介入的终态；其中 `TaskInterventionRequired` 专用于冻结快照降级后仍不满足最小执行上下文、且系统不能自动继续执行的场景，不等同于人工已执行关闭动作。对成功释放的延迟任务，`TaskTriggerActivated` 必须作为 `TaskReadyForPlanning` 之前的正式事件出现，用于把释放成功与后续规划起点稳定关联起来。条件触发事件用于表达入口等待重评估、入口条件激活、入口订阅失效、写冲突、权限确认、并行等待、模型调用、记忆读写、自动重试和降级等运行时横切行为，它们可以出现多次或穿插在主生命周期事件之间，但不得违反前述成功释放最小事件序列、子任务安排完成语义与同一 `sequence/cursor` 排序恢复规则。

每日定时记忆整理不属于单任务正式时间线事件的一部分；它作为后台作业运行，其产出应通过持久记忆来源字段和作业日志体现，而不是伪装成某个任务的生命周期阶段。

事件流在概要层统一约束为“单任务内严格递增排序 + 游标恢复”模型：所有事件必须分配稳定 `sequence`，前端和应用层以事件游标恢复历史与增量数据，不以时间戳作为唯一排序依据。

在概要层，事件流追加必须从属于任务写入事务边界：`ExecutionEvent` 不是独立于任务聚合的旁路日志，而是由 Task Service 在同一次提交中驱动 Tracking & Event Service 完成持久化的事实记录。

## 7. 系统运行边界

- 单体部署。
- 本地文件系统。
- 本地数据库和向量索引。
- 通过外部模型 API 获得推理能力。

系统对外部模型 API 的访问统一经由 `LLM Adapter`，并在该层完成超时、重试、熔断、错误归一化和降级出口控制；概要层不允许业务模块直接与具体模型供应商建立耦合。

## 8. 非功能分析

### 8.1 可用性

- 任务执行过程可见。
- 失败后可重试。
- 结果与过程分离展示。
- 路由决策、并行状态、记忆使用和降级处理对用户可见且可理解。

### 8.2 性能

- 长任务采用事件流和分阶段查询。
- Prompt 和输入上下文必须支持裁剪。

### 8.3 安全性

- 工具访问必须经过沙箱。
- 敏感配置最小暴露。
- Skills 和高权限工具必须懒加载和按需激活。
- 写入、编辑、执行和外部系统交互必须经过统一权限策略判断，并按 `deny > ask > allow` 的最严格优先级生成默认执行、待权限确认或拒绝执行等权限决策。

所有 Agent 仅能通过受控的 Service 与工具和外部环境交互，其权限由配置和 Skill 绑定的白名单统一管理，不允许直接绕过服务层访问底层资源或提升自身权限。

## 9. 概要设计结论

概要设计的关键不是“更多模块”，而是“明确边界”。BiosBot 必须保持以下单一原则：

- 一个主中心：Task。
- 一条主链路：输入 -> 入口判定与等待 -> 编排 -> 执行 -> 输出 -> 追踪。
- 一套统一语义：Task、TaskStep、Attachment、TaskOutput、ExecutionEvent。
