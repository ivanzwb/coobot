# BiosBot 技术分析

## 1. 文档目的

本文档用于说明系统在 P0 阶段的技术路线、关键设计决策、实现约束与技术风险控制方法。
本文档主要支撑 01-需求分析 中的 N-01～N-14 需求，重点覆盖主链路（N-01～N-05）及关键支撑能力（N-06～N-14）。
本文档重点回答“为什么这样设计、关键技术机制如何实现、主要技术风险如何控制”；系统分层、模块归属和主链路编排关系以下一篇概要设计文档为准。

## 2. 技术目标

- 用最小复杂度跑通任务闭环。
- 在本地环境稳定支撑多 Agent、知识库、Skills、工具和任务追踪。
- 保持可扩展，但不提前引入平台级复杂度。

## 4. 技术关键点

### 4.1 统一任务模型

系统以 `Task` 为聚合根，所有会话、附件、步骤、输出物、工具调用、知识命中都围绕任务流转。
- 用户侧主交互默认收敛为单一会话窗口：`Conversation` 在技术上用于承载持续上下文、消息归档和最近任务引用，不作为用户显式操作的“新建会话”实体入口；若首次进入或首次发送消息时尚无对应会话记录，可由系统自动初始化默认会话上下文。
- 接入端来源语义应下沉到 `Message`、`Task` 和 `task_events.payload`：系统至少需要稳定记录 `entryPoint`、`originClientId` 或等价字段，用于表达本次输入来自哪个接入端，以及后续进度、结果和权限授权默认应在哪些端显示；`Conversation` 层只保留默认上下文和最近活跃端等聚合信息，不重复承载一套会话创建来源语义。
- `Task` 必须显式携带 `triggerMode`，支持 `immediate`、`queued`、`scheduled`、`event_triggered` 四种入口模式；当触发模式不是立即执行时，还需记录等待条件、计划时间或触发规则摘要，避免任务创建后在系统中变成不可解释的静态记录。
- 对于入口模式判定置信度不足的任务，系统需进入 `clarification_pending` 待澄清状态，等待用户补充信息或确认后继续执行。
- 触发模式判定属于会话意图解析输出：系统根据输入中的时间表达、排队意图和事件条件自动生成 `triggerMode`、`scheduledAt` 和 `triggerRule`，前端不承担显式选择职责；若任务进一步被判定为复杂任务，还需生成可解释的复杂任务判定摘要，至少包含拆分原因、主要目标或关键约束。
- 对于多 Agent 复合任务，系统还需把 `Task` 明确区分为 Leader 父任务和 Domain 子任务：父任务负责拆分、安排确认、ETA 聚合和最终结果汇总；子任务负责承载具体 Domain Agent 的排队、触发、执行和结果回传。父子任务关系必须可持久化、可查询、可追踪。
- 子任务的 `queuePosition` 默认表示其在所属 Domain Agent 显式队列中的顺位；ETA 则由子任务队列顺位、已知触发时间和历史平均执行时长综合估算，不能只依赖单任务的当前状态快照。

##### 4.1.1 入口澄清事件（N-01.5）

| 事件名 | 时机 | 关键字段 |
|--------|------|----------|
| `ClarificationRequested` | 进入澄清待确认状态 | taskId, clarificationFields[], lowConfidenceReason |
| `ClarificationProvided` | 用户补充信息完成澄清 | taskId, providedInputs[], resolutionSummary |
| `ClarificationTimeout` | 澄清超时用户未响应 | taskId, timeoutAt, fallbackDecision |
| `ClarificationClosed` | 用户放弃澄清 | taskId, closeReason |

关联需求：N-01，N-02，N-03，N-04，N-05，N-14。

### 4.2 Prompt 工程

- Prompt 采用 `system / developer / user / context / tool-result` 五层结构。
- Prompt 输出必须映射到 `TaskPlan`、`TaskStepResult`、`TaskOutput`。
- Prompt 必须支持版本化、模板化和可观测性记录。
- Agent 执行 Prompt 必须能稳定约束 ReAct 运行模式：每一轮仅允许生成简短推理摘要、明确的下一步动作意图和对最近观察结果的利用说明，避免脱离工具反馈的长链静态推理。
- Prompt 超长时必须按结构化字段裁剪，禁止字符串硬截断。
关联需求：N-02，N-03，N-04，N-12，N-14。

### 4.3 Skills 与工具体系

- Skills 是工具的上层组织形式。
- Skill 必须懒加载。
- Agent 初始只暴露 `use_skill` 和最小只读能力。
- 通过 `use_skill(skillId)` 动态加载 Skill instructions、tool definitions 和执行器。
关联需求：N-07.1，N-07.4，N-11，N-13。

### 4.4 工具与环境交互

- P0 工具固定为读、写、编辑、执行四类主能力。
- 从权限策略角度，编辑属于写入的一种，统一简化为读、写、执行三类。
- 工具必须运行于工作区沙箱。
- 不允许默认网络访问或系统级高权限。
- 所有工具调用都必须可审计。
关联需求：N-03，N-13，N-14。

Agent 对工具与环境的访问仅通过配置和 Skill 绑定的白名单授予，不允许在运行时动态获取更高权限或直接调用未授权命令、访问工作区以外的资源。

权限控制需采用统一权限策略模型，对读、写、执行（含外部系统交互）分别支持 `allow`、`ask`、`deny` 三种权限决策；其中低风险读操作可采用默认执行权限决策，高风险写入（含编辑）、执行（含外部调用）默认进入权限确认或拒绝执行的权限决策，最终权限决策结果必须进入审计日志和事件流。

### 4.5 多模态处理

- 支持图片 OCR、文本抽取、PDF 解析三条基础链路，单个附件大小上限为 50MB。
- 解析结果统一收敛为 `TaskInput` 和 `Input Slice`。
- 大文件、低质量附件和超长内容必须支持降级和裁剪。
关联需求：N-01，N-03，N-08，N-09.2。

### 4.6 记忆与上下文治理

- 记忆能力分为会话级短期记忆、Agent 级持久记忆和知识库文档三类，不允许混为单一上下文池。
- 任务创建时必须先确认 `memoryScope`，再决定加载哪些历史记忆和知识命中结果。
- 记忆写回必须显式记录来源任务、来源输出和写回原因，避免无边界累积导致记忆污染。
- 除任务内显式写回外，系统还需支持后台定时记忆整理：按日读取当天会话记录，过滤闲聊、重复信息和低价值噪声后，沉淀为 Agent 级持久记忆，避免把原始会话全文直接注入长期记忆池。
- 结果页展示所需的记忆摘要、知识摘要和 Agent 参与摘要，应由后端聚合生成，不由前端临时拼装。
关联需求：N-04，N-05，N-09.1，N-09.2，N-09.3，N-10，N-14。

## 5. 关键技术方案

### 5.1 Prompt 裁剪方案

- 预估上下文窗口与输出保留预算。
- 超长时先裁旧步骤结果，再裁知识命中，再裁长文本输入，再做降级执行。
- 若裁剪仍失败，写入 `PROMPT_OVERFLOW` 并触发降级策略。
关联需求：N-03，N-12，N-14。

### 5.2 Skill 懒加载方案

- 启动阶段只建立 Skill 元数据索引。
- `use_skill` 是唯一激活入口。
- Skill 只在当前 Agent / 当前任务上下文生效。
- 任务结束后释放运行时绑定。
关联需求：N-07.1，N-11。

### 5.3 工具沙箱方案

- 限制工作目录。
- 限制访问路径。
- 命令执行必须超时控制。
- 输出必须支持截断标记和摘要化。
- 写入、编辑、执行和外部系统调用必须在沙箱外再叠加权限策略检查；当权限策略为 `ask` 时，需要先生成权限请求描述，再等待用户进行权限确认后继续执行。
- 所有工具调用参数在进入沙箱前必须经过脚本注入过滤，包括：命令注入过滤（移除或转义 `;`, `|`, `&&`, `$()`, `` ` ``, `$(())` 等Shell元字符）、路径遍历过滤（阻止 `../` 逃逸、验证最终路径在工作目录内）、环境变量注入过滤（禁止 `${VAR}` 或 `$VAR` 形式的未授权变量展开）。
- 沙箱与宿主系统采用权限清单隔离模式：工具/Skill 需在元数据中声明所需权限（网络访问、文件系统读写、进程创建、动态库加载等），沙箱根据权限清单与权限策略放行对应能力，未声明权限默认禁止；禁止提权操作、禁止访问 `/proc`、`/sys` 等系统目录。
- 写入操作安全扫描：**禁止写入可执行文件**（.exe, .sh, .bat, .ps1, .py, .js, .rb 等脚本/二进制文件）、禁止写入包含已知恶意代码模式的内容（如eval调用、动态代码生成、base64编码的可执行payload）；**工具可生成可执行文件（如代码生成场景），但执行前需额外安全检查和权限确认**
关联需求：N-13，N-14。

### 5.4 权限策略与确认机制方案

- 系统为每次高风险动作生成标准化权限请求，至少包含动作类型、发起 Agent、关联 Skill、目标资源、潜在影响和当前权限策略。
- 权限策略检查顺序为：全局默认权限策略 -> Agent 配置 -> Skill 绑定权限策略 -> 工具级权限策略 -> 目标资源限制；判定规则采用“只收紧、不放宽”的优先级合并机制，统一按 `deny > ask > allow` 取最严格结果，下层策略只能在上层结果基础上继续收紧，不能放宽已生效的权限边界。
- 权限策略计算结果需输出标准化判定轨迹，至少记录每一层输入策略、最终权限决策、触发该结果的命中规则和对应资源范围，供后续审计、排障和界面展示使用。
- 当权限决策结果为 `ask` 时，执行链路暂停，向前端返回权限请求；只有收到明确权限确认后才恢复执行，否则写入权限决策为拒绝的事件并结束该动作。
- 当权限决策结果为 `deny` 时，不进入实际工具执行阶段，直接返回权限拒绝错误，并在事件流中记录权限决策原因。
关联需求：N-13，N-14。

### 5.4.1 权限策略设计

#### 5.4.1.1 策略配置结构

策略配置分为五个层级，优先级从低到高为：全局默认 -> Agent配置 -> Skill绑定 -> 工具级 -> 资源级。

每层策略规则可包含以下动作类型：
- 读（read）：读取操作，默认 allow
- 写（write）：写入操作（含编辑），默认 ask
- 执行（execute）：命令执行/外部系统调用，默认 deny

每种动作支持三种决策：allow（允许）、ask（需确认）、deny（拒绝）。

#### 5.4.1.2 策略合并规则

合并算法遵循"只收紧、不放宽"原则：
- 优先级顺序：deny > ask > allow
- 从低优先级层向高优先级层依次合并
- 最终决策取最严格结果
- 记录每一层的决策轨迹供审计使用

#### 5.4.1.3 Ask 模式超时处理

Ask模式需配置超时机制：
- 默认超时时间：5分钟
- 最大超时时间：30分钟
- 超时后可选择自动升级为deny，或保持等待用户确认
- 高风险操作超时时间应短于低风险操作

#### 5.4.1.4 权限策略与界面功能对应

| 权限策略 | 对应功能要求 | 界面展示 |
|----------|--------------|----------|
| allow | 低风险读操作默认执行 | 无需展示 |
| ask | 写入/执行（含外部调用）需确认 | 结果页展示权限决策摘要 |
| deny | 高风险操作拒绝执行 | 任务详情展示拒绝原因 |

##### 5.4.2 权限事件定义

| 事件名 | 时机 | 关键字段 |
|--------|------|----------|
| `PermissionCheckRequested` | 权限检查发起 | taskId, stepId, action, target, policy |
| `PermissionGranted` | 权限已确认 | taskId, stepId, requestId, grantedBy |
| `PermissionDenied` | 权限被拒绝 | taskId, stepId, requestId, deniedBy, reason |

关联需求：N-13，N-14。

### 5.5 并行编排、任务写入冲突与串行化控制方案

- 任务创建入口应统一进入本地任务接入与分发链路：立即执行任务可直接进入编排；排队任务先进入本地任务队列；定时任务在达到 `scheduledAt` 前保持等待；事件触发任务仅在命中预定义触发条件后转入可执行状态。
- 在单一默认会话窗口模型下，前端主链路只需提交消息和任务创建请求，不需要把“新建会话”作为独立用户步骤插入链路；若实现层仍依赖 `conversationId`，应由默认会话上下文自动注入，而不是要求用户先完成会话创建动作。
- 任务创建前应先经过轻量入口判定链路，生成 `TaskTriggerDecision`：它基于会话文本、附件摘要和基础上下文抽取时间表达、排队意图和预定义事件条件，统一产出 `triggerMode`、`scheduledAt?`、`triggerRule?`、`triggerDecisionSummary` 与可选的 `nextTriggerCheckAt?`，然后再由任务模块创建任务；不能先创建任务、后回填入口模式。
- 对于需要拆解的请求，入口判定链路或紧随其后的编排准备阶段还需产出 `complexityDecisionSummary`，用于解释为什么当前任务被视为复杂任务、为何需要拆分为父任务与多个 Domain 子任务；该摘要需进入任务详情或事件流可见范围，不能仅停留在模型内部推理文本。
- 对于复合任务，Leader 父任务进入执行链路后应先完成 Domain 子任务拆分与目标 Agent 分配，再由任务模块为每个子任务生成独立聚合、写入 `parentTaskId` 与 `assignedDomainAgentId`。子任务创建后可分别进入 `queued`、`scheduled` 或 `event_triggered` 等不同入口模式，不要求所有子任务使用同一种入口模式。
- 任务队列、定时调度和事件触发都应由单体内调度器统一协调，不依赖分布式消息中间件；同一任务一旦进入可执行状态，后续仍按统一任务聚合和事件流语义推进。
- Domain Agent 队列必须是显式逻辑队列，而不是仅由全局执行槽位隐式推导：调度器需先在每个 Domain Agent 队列内维护稳定 FIFO 顺位，再结合全局执行槽位预算决定哪些队首子任务可进入执行链路。
- 对于 `queued`、`scheduled` 和 `event_triggered` 三类等待态任务，系统必须持久化等待订阅事实，而不能只依赖进程内存中的队列、定时器或事件监听器；至少需要保存当前条件快照、最近一次评估时间和下一次检查时间，以支持服务重启后的恢复扫描。
- 对于等待时间超过阈值、触发条件失效、冻结快照失效或降级后仍不满足最小执行上下文的任务，系统必须生成标准化等待异常摘要，至少包含异常原因、受阻子任务、最近一次重评估结果、命中的阈值依据和建议动作；等待阈值至少按 `queued`、`scheduled`、`event_triggered` 三类等待模式分类型配置，并形成可对外展示的阈值依据摘要；若不能自动恢复，则需进入明确人工介入状态，而不是继续保持无界等待。
- 服务启动或恢复时，应先重建等待订阅并重新评估所有未释放的入口等待任务：队列任务恢复本地顺位，定时任务按 `scheduledAt / nextTriggerCheckAt` 重建调度，事件触发任务按 `triggerRule` 恢复订阅；恢复过程不得直接跳过正式事件写入。
- 入口等待态本身必须进入正式事件流：任务创建后至少记录 `TaskTriggerResolved`；若任务未立即执行，则继续记录 `TaskTriggerWaiting`；等待条件被重评估时记录 `TaskTriggerReevaluated`；条件满足释放执行时记录 `TaskTriggerActivated` 和 `TaskReadyForPlanning`，确保等待中的任务同样可审计、可恢复、可复盘。
- 当 Leader 父任务拆分出子任务后，系统还需记录 `SubTasksCreated`；每个子任务完成“已入队 / 已登记定时 / 已登记触发”等安排确认后记录 `DomainTaskQueued`、`DomainTaskScheduled` 或 `DomainTaskTriggered`；只有当全部必需子任务都达到安排完成状态后，父任务才可记录 `TaskArrangementCompleted` 并计算 `TaskEtaEstimated`。
- ETA 估算必须形成可追踪摘要：至少要综合 Domain Agent 当前队列顺位、历史平均执行耗时、已知定时点和触发规则，不允许只给出无法解释的静态数字。若部分子任务处于事件触发等待且无法给出确定完成时间，应返回区间估算或“等待外部条件”的摘要，而不是伪精确时间。
- 当父任务仅达到安排完成阶段时，技术上应把 ETA 与安排通知视为阶段性状态，而非最终输出；结果交付链路必须在父任务最终汇总完成后才生成最终输出事实，避免把阶段性通知错误复用为最终结果。
- 当任务合法结束但没有可展示主输出物时，技术契约必须要求结果聚合层生成 `terminalSummary`，并由界面切换到终态摘要页或等价终态摘要区展示；不得依赖临时文案在前端兜底拼装“无输出终态”。
- 当最终交付后收到晚到 non-blocking 成功结果时，系统只允许通过 `supplementalUpdates` 追加补充事实，并以 `supplementalNotificationType` 标记补充通知语义；这类补充通知不得冒充新的最终完成通知。
- 当 blocking / non-blocking 标记在已参与安排、ETA 或交付判定后发生变化时，必须写入 `BlockingDecisionChanged` 事件并触发重评估链路；技术实现不得只静默改写子任务标记后继续沿用旧 ETA 或旧交付结论。
- 编排层在生成 `TaskPlan` 时，需要为可并行步骤分配 `parallelGroupId`，并标记执行模式为串行、并行或等待。
- 对只读知识检索、独立推理和互不依赖的附件解析步骤，可并行执行；涉及共享文件写入、同一输出物修改和同一外部资源提交的步骤必须串行。
- 当检测到共享资源冲突时，不直接失败，而是将后续步骤置为 `waiting`，并记录 `waitingReason` 与预计恢复条件。
- `Agent Execution Service` 驱动单个 Leader / Domain Agent 的执行回合时，必须遵循 ReAct 循环：先产出简短推理摘要，再决定是否调用目标 Skill / 工具，接着把工具返回的观察结果结构化回写到当前执行上下文，随后才能进入下一轮决策或结束；该循环的对外契约是“推理摘要 + 动作记录 + 观察摘要”，而不是暴露完整原始思维链。
- 同一 `taskId` 的状态推进、步骤完成、权限确认、重试和降级写操作，必须先封装为任务写命令并进入单任务顺序消费链路；不允许编排层、工具层或事件消费者绕过任务聚合直接改写任务当前状态。
- 任务聚合必须维护 `aggregateVersion` 作为写入基线；提交写命令时若 `expectedAggregateVersion` 已过期，或同一任务写队列判定当前命令不可安全覆盖最新状态，则拒绝本次提交，统一返回 `TASK_WRITE_CONFLICT`，并记录 `TaskWriteConflictDetected` 事件。

#### 5.5.1 任务版本冲突设计

**版本号生成策略**：
- 采用时间戳 + 自增序列组合
- 时间戳：毫秒级当前时间
- 序列号：同一毫秒内的递增编号
- 版本号比较：先比较时间戳，再比较序列号

**写命令结构**：
- 包含commandId、taskId、expectedVersion、operations、创建时间
- expectedVersion表示提交时期望的版本号

**冲突检测流程**：
1. 提交写命令时携带expectedVersion
2. 读取当前任务的aggregateVersion
3. 比较两者：相等则执行写入并更新版本号；不等则拒绝
4. 拒绝时返回TASK_WRITE_CONFLICT错误

**重试策略**：
- 最多重试3次
- 基础延迟100ms，指数退避
- 最大延迟2秒
- 重试前重新获取最新版本号

**数据库存储设计**：
- aggregateVersion存储为task表的字段（timestamp + sequence合并存储）
- 版本号更新与任务状态变更在同一事务内完成
- 并发写入时使用数据库行锁避免死锁

- 写命令被任务模块接收进入顺序消费链路时，应先记录 `TaskWriteQueued` 事件，再在同一提交边界内完成任务状态变更、查询投影更新和待追加事件持久化，避免并行步骤或重试动作覆盖较新的聚合状态。
- 自动重试只能作用于幂等或可恢复步骤，且重试次数、触发原因和重试后结果必须进入统一事件流。
- Leader 父任务必须支持双阶段通知：第一阶段在 `TaskArrangementCompleted` 与 `TaskEtaEstimated` 之后向用户发出安排完成通知；第二阶段在全部必需子任务完成并汇总结果后向用户发出最终完成通知。两次通知都必须有可追踪事件或状态摘要支撑。

调度器设计要点：

- 批处理大小：每次处理50个订阅

同一订阅只能成功释放一次：
- 释放前检查订阅状态
- 状态非active则跳过

### 5.6 记忆架构与定时整理方案

- 短期记忆挂载在任务执行上下文内，只在当前任务和当前会话范围生效。
- 持久记忆按 Agent 维度存储，写入时必须包含来源任务、摘要和状态，支持后续人工清理和禁用。
- 知识库用于承载可复用资料和结果文档，记忆用于承载偏好、约束和高频上下文，两者检索策略独立。
- 任务启动时先确认记忆范围，再加载相关记忆和知识；任务结束后由输出层统一决定哪些内容进入记忆写回或知识沉淀。
- 定时记忆整理运行在任务主链之外，由后台调度器按日扫描当天会话记录，生成 `DailyConversationDigest`，提炼稳定偏好、持续上下文、附件关键信息和值得长期保留的结论，不把无关寒暄、瞬时噪声或重复表达直接写入持久记忆。
- 定时整理写入的持久记忆必须记录来源日期、来源会话集合、摘要版本和去重键；同一 Agent 在同一日期范围内重复执行整理时，应更新或合并既有摘要，而不是重复生成语义等价的记忆条目。
- 定时记忆整理失败不得阻塞任务执行主链，但必须保留作业日志和失败原因，便于人工补跑与排障。
关联需求：N-05，N-09.1，N-09.2，N-09.3。

### 5.7 事件流实时同步方案

- 所有任务事件统一写入事件表，并通过 WebSocket 推送到前端；HTTP 查询接口负责补拉历史事件和重连恢复。
- 事件对象至少包含 `eventId`、`taskId`、`sequence`、`eventType`、`timestamp`、`summary`、`payload`，必要时带 `stepId`、`agentId`；其中 `sequence` 为单任务内严格递增序号，作为时间线排序、断线补拉和重复事件去重的主依据，`timestamp` 仅用于展示与诊断。
- 若事件涉及单窗口主交互或多端显示范围，`payload` 还需稳定携带 `entryPoint`、`originClientId`、`syncPolicy`、`visibleClientIds`、`displayScope` 或等价字段；前端应据此判断哪些消息、进度、结果和权限授权只在发起端展示，哪些事实已被允许跨端同步。
- WebSocket 推送与 HTTP 补拉都必须基于同一事件游标模型工作，前端以上次已确认的 `sequence` 或 `cursor` 继续拉取后续事件，禁止仅按时间戳范围恢复事件流。
- 事件写入与投递需支持至少一次投递语义，前端和应用层消费端必须按 `eventId` 或 `sequence` 实现幂等处理；同一事件允许重复到达，但不能重复推进任务状态。
- 前端以事件流为准更新任务时间线和步骤状态；若 WebSocket 中断，则根据最后事件游标补拉，避免页面状态与后端执行状态分叉。
- 详细事件定义见第8节统一事件字典。
关联需求：N-02，N-04，N-09.2，N-09.4，N-14。

### 5.8 模型失败归一化与供应商切换方案

- 所有远程模型调用都必须经由统一 `LLM Adapter` 发起，业务模块只依赖统一 `ModelRequest` / `ModelResponse` 和归一化后的调用结果，不直接依赖厂商 SDK 返回的原始错误结构。
- 模型适配层需将供应商超时、限流、服务不可用和响应结构异常分别归一化为 `MODEL_TIMEOUT`、`MODEL_RATE_LIMITED`、`MODEL_PROVIDER_UNAVAILABLE`、`MODEL_RESPONSE_INVALID`，并输出稳定的失败摘要，供任务模块、接口层、数据库和界面使用同一错误语义。
- 模型调用日志至少记录 `modelProvider`、请求摘要、耗时、重试次数、归一化错误码和失败摘要；审计、重试和结果页解释均以归一化结果为准，不以供应商原始错误码作为跨模块契约。
- 供应商切换仅适用于配置中声明可替代关系的模型供应商组合，并且只在超时、限流或供应商不可用等可恢复故障场景触发；切换后需保留原供应商失败事实，并将 `fallbackProvider`、重试来源和最终结果写入统一事件流。
- 若模型返回响应结构不满足协议约束，应优先判定为 `MODEL_RESPONSE_INVALID`；除非存在明确兼容策略，否则不应仅因响应异常无限制切换供应商，避免将提示词、协议或解析错误误判为基础设施故障。
- 模型调用开始、完成和失败都必须进入统一事件流，确保时间线可以解释当前使用的供应商、失败归一化结果以及是否发生过供应商切换。
关联需求：N-03，N-06，N-14。

```
3. 生成配置版本号 (configVersion)
4. 写入配置变更记录表
5. 通知配置刷新（发布-订阅）
6. 各模块监听配置变更事件
7. 刷新内存缓存
8. 返回更新结果
```

##### 配置变更历史

| 字段 | 说明 |
|------|------|
| id | 变更记录ID |
| configType | 配置类型 |
| oldValue | 变更前值（脱敏） |
| newValue | 变更后值（脱敏） |
| changedBy | 变更人 |
| changedAt | 变更时间 |
| reason | 变更原因（可选） |

##### 回滚机制

- 支持按配置版本号回滚
- 回滚操作生成新的变更记录
- 回滚后记录 `rollbackFrom` 关联原始变更
- 保留最近 30 个配置版本

关联需求：N-17.2。

#### 5.10.3 数据备份与恢复方案

系统支持自动备份机制，备份文件加密存储，支持从备份恢复数据。

##### 备份策略

| 备份类型 | 频率 | 保留 | 存储位置 |
|----------|------|------|----------|
| 全量备份 | 每日 | 7天 | 本地加密文件 |
| 增量备份 | 每小时 | 24小时 | 本地加密文件 |
| 配置备份 | 每次变更 | 30次 | 配置文件 |

##### 备份内容

| 数据类型 | 备份方式 | 说明 |
|----------|---------|------|
| SQLite 数据库 | 全量/增量 | 任务、对话、配置 |
| 向量知识库 | 全量 | LanceDB 数据 |
| 工作区文件 | 手动 | 用户可选择 |
| 配置文件 | 每次变更 | JSON 格式 |
| 审计日志 | 归档 | 90天保留 |

##### 加密方案

- 加密算法：AES-256-GCM
- 密钥来源：用户首次启动时生成，保存在系统密钥链中
- 备份文件命名：`backup_{type}_{timestamp}.enc`

##### 恢复流程

```
恢复操作流程：
1. 用户选择恢复点
2. 系统验证备份文件完整性
3. 提示恢复影响（覆盖当前数据）
4. 停止服务
5. 恢复数据库文件
6. 恢复向量库
7. 重新计算统计指标
8. 启动服务
9. 记录恢复日志
```

##### 自动备份调度

- 后台调度器每日凌晨 3:00 执行全量备份
- 每小时执行增量备份
- 备份任务失败时记录日志并告警
- 磁盘空间不足时自动清理过期备份

关联需求：N-17.3。

#### 5.10.4 资源监控方案

系统采集并展示 CPU、内存、磁盘使用率，资源使用超过阈值时触发告警。

##### 采集指标

| 指标 | 采集频率 | 告警阈值 | 说明 |
|------|---------|---------|------|
| CPU 使用率 | 10秒 | > 90% 持续 5分钟 | 进程 CPU |
| 内存使用率 | 10秒 | > 85% 持续 5分钟 | 进程内存 |
| 磁盘使用率 | 60秒 | > 80% | 数据目录所在磁盘 |
| 任务队列深度 | 30秒 | > 1000 | 等待执行任务数 |
| 并发执行数 | 30秒 | > 50 | 同时执行的任务数 |

##### 技术实现

- 使用 Node.js `performance.memory` 获取内存信息
- 使用 `os` 模块获取 CPU 和磁盘信息
- 指标数据写入内存缓存，定时持久化到数据库
- 告警检查独立调度器，周期性检查阈值

##### 告警触发

| 条件 | 告警级别 | 动作 |
|------|---------|------|
| CPU > 90% 持续 5分钟 | 警告 | 记录日志 |
| CPU > 95% 持续 10分钟 | 严重 | 停止接收新任务 |
| 内存 > 85% 持续 5分钟 | 警告 | 记录日志 |
| 内存 > 90% 持续 10分钟 | 严重 | 停止接收新任务 |
| 磁盘 > 80% | 警告 | 通知清理 |
| 磁盘 > 90% | 严重 | 禁止新任务 |

##### 监控面板展示

- 实时 CPU/内存使用率图表
- 磁盘使用率
- 任务队列趋势
- 历史数据查询（支持 7 天）

关联需求：N-17.4。

## 6. 技术风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| Prompt 过长 | 模型调用失败、成本过高 | 结构化裁剪 + 降级执行 |
| Tool 权限过大 | 越权访问文件或执行命令 | 最小权限 + 沙箱 + 错误码 |
| Skill 过多 | Prompt 污染、工具误选 | 懒加载 + Skill 激活门控 |
| 多模态不稳定 | OCR 质量差、结果不可用 | 解析状态 + 置信度 + 重解析 |
| 记忆污染 | 历史偏好混入当前任务、人格漂移 | 记忆范围确认 + 来源记录 + 显式写回 |
| 定时记忆整理失真 | 会话日摘要混入噪声、遗漏关键上下文或重复写入 | 相关性过滤 + 去重键 + 可补跑作业日志 |
| 权限策略失控 | 高风险动作被误执行或频繁误拦截 | allow/ask/deny 分级权限策略 + 权限确认 + 审计事件 |
| 并行冲突 | 多 Agent 同时写同一资源导致状态错乱 | 并行分组 + 冲突检测 + waiting 状态 |
| 任务写入冲突 | 过期 `aggregateVersion` 或乱序写命令覆盖较新任务状态 | 单任务写队列 + 聚合版本校验 + `TASK_WRITE_CONFLICT` |
| 模型供应商不稳定 | 超时、限流、供应商不可用导致任务执行中断或错误扩散 | 失败归一化 + 有界重试 + 供应商切换 |
| 状态不一致 | 前后端显示不一致 | 统一 DTO、统一错误码、统一事件流 |

## 7. 技术分析结论

P0 技术路线的核心不是追求先进架构，而是保证以下四件事稳定成立：

- 能稳定创建和执行任务。
- 能稳定接入 Skills、Prompt 和工具能力。
- 能稳定展示过程与结果。
- 能在失败、降级和边界场景下保持可解释性。
