# BiosBot 功能要求

## 1. 文档目的

本文档用于定义 BiosBot P0 至 P2 阶段的功能要求，明确系统必须提供的业务能力、功能边界、优先级和验收口径。
本文档主要支撑 01-需求分析 中的需求落地，并为技术分析、概要设计、详细设计、接口设计、界面设计和测试设计提供功能基线。
本文档重点回答“系统必须具备哪些功能、哪些功能进入当前阶段、每项功能达到什么程度才算完成”。

## 2. 产品范围

| 定位 | 说明 |
|-----|------|
| 平台类型 | 本地优先的多智能体协作平台 |
| 输入类型 | 文本、图片、文档等 |
| 核心闭环 | Leader Agent + Domain Agent + Skills + 知识库 + 任务追踪 |

### P0 核心目标

用户能够**上传文本和附件**，发起复杂任务，看到系统如何**拆解与执行**，并获得**可复用、可解释**的最终结果。

## 3. 角色与使用对象

| 角色 | 关注点 |
|-----|------|
| 个人开发者与 AI 爱好者 | 本地部署、Agent 配置、Skills、工具与实验能力 |
| 专业知识工作者 | 资料分析、结构化输出、引用依据和过程可复核 |
| 小团队负责人 | 任务复盘、失败定位、知识沉淀和流程复用 |

#### 4.2.3 Leader Agent 任务编排

| 编号 | 功能描述 | 来源 |
|-----|---------|-----|
| N-07.2.1 | Leader Agent 负责任务意图理解、拆解与编排，选择合适的 Domain Agent 或工具链，并对多轮执行结果进行汇总、重试与对用户输出 | N-02, N-03, N-07.2 |
| N-07.2.2 | Leader Agent 对复杂任务的拆解必须基于会话意图解析结果，且系统需记录并展示复杂任务判定摘要，至少说明拆分原因、主要目标或关键约束 | N-02, N-14 |
| N-07.2.3 | Leader Agent 支持把一个复合任务拆解为多个 Domain 子任务，并为每个子任务分配目标 Domain Agent、入口模式和执行摘要 | N-02, N-03, N-07.2 |
| N-07.2.4 | Leader Agent 支持在编排阶段为每个子任务标注 blocking 或 non-blocking 属性，并展示该子任务是否属于父任务最终交付所必需的执行项及其判定依据 | N-02, N-03, N-14 |
| N-07.2.5 | Leader Agent 支持在全部必需子任务完成安排确认后，先向用户反馈"任务已安排好"及聚合 ETA；在全部必需子任务完成后，再向用户反馈最终结果 | N-02, N-03, N-07.2, N-14 |

#### 4.2.4 ReAct 执行模式

| 编号 | 功能描述 | 来源 |
|-----|---------|-----|
| N-07.5.1 | Leader Agent 与 Domain Agent 在实际执行时需遵循 ReAct（"推理 + 行动"）模式，以简短推理步骤、针对性工具调用和观察反馈交替推进；每轮观察结果都必须能够影响下一轮决策，直到形成最终结果、明确失败或人工介入结论 | N-07.5, N-12, N-13, N-14 |
| N-07.5.2 | 与 N-07 相关的推理展示能力只允许输出 `reasoningSummary`、`observationSummary`、`actionSummary` 或等价压缩摘要，用于界面展示、审计与复盘；不允许把模型内部完整原始思维链作为接口返回、页面展示或长期存储内容 | N-07.5, N-12, N-14 |

#### 4.2.5 Domain Agent 能力

| 编号 | 功能描述 | 来源 |

| N-07.3.2 | Domain Agent 支持独立承接 `queued`、`scheduled`、`event_triggered` 三类子任务入口模式，并向 Leader 返回"已入队 / 已登记定时 / 已登记触发"等安排确认 | N-02, N-03, N-07.3, N-14 |

#### 4.2.6 多 Agent 协作

| 编号 | 功能描述 | 来源 |
|-----|---------|-----|
| N-07.2.6 | 同一任务中允许 Leader Agent 调度多个 Domain Agent 协同工作，但 Domain Agent 之间不得直接相互调用，只能通过 Leader Agent 进行协作 | N-02, N-03, N-07.2 |
| N-07.2.7 | 支持记录并展示 Leader Agent 的路由决策摘要，包括所选 Domain Agent、Skill 或工具链及其主要原因 | N-02, N-07.2, N-14 |
| N-07.2.8 | 支持记录并展示父任务与子任务关系、子任务所属 Domain Agent、排队顺位、触发状态、安排状态和聚合 ETA | N-02, N-03, N-07.2, N-07.3, N-14 |
| N-07.2.9 | 当同一任务涉及多个 Domain Agent 时，支持对无冲突步骤并行调度；涉及共享资源写入时，系统需自动串行化处理或提示等待 | N-02, N-03, N-14 |

### 4.8 会话与任务能力

|-----|---------|-----|
| N-01.1 | 会话输入与 Task 创建：支持基于文本和附件创建复杂任务；输入消息写入默认会话上下文后，任务可在当前会话窗口中被持续追踪，并可通过任务 ID、自然语言或快捷指令查询状态、查看详情或触发常用操作；任务列表和详情页仅作为补充查询入口 | N-01, N-02 |
| N-01.2 | 入口模式自动识别与判定摘要：支持系统根据会话文本和附件上下文自动识别立即执行、进入任务队列、定时执行或事件触发执行等任务触发模式；支持从会话内容中提取定时任务的计划执行时间或事件触发任务的预定义触发条件，并在任务列表和详情页展示对应的等待状态摘要 | N-01, N-14 |
| N-01.3 | 默认单一会话窗口主交互：支持把文本输入、多模态输入、进度显示、结果输出、权限授权和常用任务操作统一收敛到每个接入端的同一默认会话窗口完成；常用任务操作至少包括查看详情、查看结果摘要、重试、取消、补充澄清、处理权限确认和执行等待态建议动作；不向用户暴露多会话创建、切换和会话列表管理主流程；任务详情页和结果页作为补充查看、跳转和复盘入口，不替代主交互面 | N-01, N-04, N-13, N-14 |
| N-01.4 | 多端显示范围与同步策略：支持不同任务入口和可扩展接入端接入同一会话主交互模型；P0 至少支持 Web 端，后续可扩展到桌面端、移动端或其他接入端而不改变主交互语义；每个接入端默认各自承载同一主窗口模型，消息在多端间按策略同步，默认不同步，相关用户输入、多模态输入、进度显示、结果输出和权限授权信息默认只在发起该次用户输入的接入端展示，显式启用同步后才跨端分发 | N-01, N-04, N-13, N-14 |
| N-01.5 | 低置信度入口澄清与安全默认边界：当系统对入口模式判定置信度不足，或缺少定时 / 触发执行所需关键条件时，支持先创建一个可在任务列表和详情页中查询到的待澄清 / 待确认任务，再进入明确可观测的澄清、确认或安全默认处理路径，并记录判定失败原因、补救动作和最终采用的入口模式；低置信度入口创建的待澄清 / 待确认任务必须进入正式 `clarification_pending` 或等价状态；用户补充信息、系统完成确认或采用安全默认路径后，必须在原 Task 上完成状态推进与入口模式确认，不得通过重新创建第二个 Task 替代原单；若用户放弃继续澄清，原 Task 必须进入明确终态而不是无限停留 | N-01, N-14 |
| N-01.6 | 会话内任务操作与快捷指令：支持用户在同一会话窗口通过自然语言或快捷指令对指定任务执行常用操作；至少覆盖查看任务详情、查看结果摘要、重试任务、取消任务、补充澄清、处理权限确认和执行等待态建议动作；系统需在消息流中返回结构化操作回显，并对高风险状态变更继续执行权限校验、审计记录与原 Task 状态推进 | N-01, N-02, N-13, N-14 |

#### 4.8.2 N-02 从任务创建到步骤编排

| 编号 | 功能描述 | 来源 |
|-----|---------|-----|
| N-02.1 | 复杂任务判定与可解释摘要：支持基于会话意图和附件上下文判定复杂任务，并展示复杂任务判定摘要，至少说明拆分原因、主要目标或关键约束 | N-02, N-14 |
| N-02.2 | 父子任务编排与 Domain 分配：支持对 `queued`、`scheduled`、`event_triggered` 等非立即执行复合任务先创建父任务与等待登记信息；仅在等待条件满足并完成 `TaskTriggerActivated`、`TaskReadyForPlanning`、`MemoryScopeConfirmed` 后，才生成正式父任务级 `TaskPlan` 并进入执行编排；支持对于复合任务自动创建 Leader 父任务与多个 Domain 子任务，并在任务详情中展示父子关系、目标 Agent、安排状态和 ETA 摘要；支持 Domain 子任务分别进入所属 Agent 队列或触发等待态，且队列顺位按 Agent 维度展示，不与其他 Agent 混淆 | N-02, N-03, N-07.2, N-07.3, N-14 |
| N-02.3 | blocking / non-blocking 标注与改标治理：支持对 blocking / non-blocking 子任务标记形成可审计冻结事实；当该标记已用于生成安排通知、ETA 或降级交付判定后，后续若发生调整，系统必须记录明确变更事件、变更原因和影响范围，并将原安排结论或 ETA 标记为需重评估，禁止通过静默改标回溯性修正既有交付结论 | N-02, N-14 |
| N-02.4 | 安排完成与 ETA 生成：支持父任务在全部 blocking 子任务完成安排确认后先进入"已安排完成"阶段，并向用户反馈预计完成时间；支持对等待中的定时任务、事件触发任务和排队任务记录等待异常原因、最近一次重评估结果、命中的阈值依据和建议动作；当任务无法继续自动推进时，系统需把父任务或子任务推进到明确的人工介入状态，而不是保持静默等待 | N-01, N-02, N-03, N-14 |
| N-02.5 | 降级交付判定摘要：支持当仅 non-blocking 子任务失败、超时或被降级跳过时，父任务仍可进入可解释的部分完成或降级交付路径，并在结果页明确标注未满足的非关键子任务差异；父任务进入部分完成或降级交付时，必须产出结构化降级交付判定摘要，至少包含已完成的 blocking 子任务、被忽略或失败的 non-blocking 子任务、对最终结果的影响范围以及本次仍可交付的理由，并在结果页、通知或事件流中可追溯 | N-02, N-03, N-04, N-14 |

## 7. 验收标准

### P0 验收标准

| 序号 | 验收项 |
|-----|-------|
| 1 | 用户能完成模型配置并通过连通性测试；配置保存后，后续新发起的模型调用按最新配置生效，且用户能看到当前生效的模型配置与可用状态 |
| 2 | 用户能在默认单一会话窗口中上传附件并发起复杂任务，无需先显式新建会话 |
| 3 | 用户能在单一会话窗口完成输入、查看进度、接收结果和处理权限授权；任务详情页与结果页只作为补充查看入口 |
| 4 | 系统能把任务拆解为不少于规划、执行、汇总三级阶段 |
| 5 | 用户能查看步骤、耗时、Agent、失败信息和工具摘要 |
| 6 | 用户能查看父任务与子任务关系、各子任务所属 Domain Agent、排队或触发状态，以及聚合 ETA |
| 7 | 用户能查看 Agent 执行过程中形成的关键推理摘要、工具调用记录和观察结果摘要，并理解这些观察如何影响后续动作或最终结论 |
| 8 | 用户能查看复杂任务判定摘要，理解系统为什么把当前请求拆分为父任务和多个子任务 |
| 9 | 用户能查看入口模式判定失败或低置信度时的澄清 / 确认 / 安全默认处理结果，理解系统为何没有直接执行 |
| 10 | 用户能查看低置信度入口任务进入正式 `clarification_pending` 或等价状态，并在原 Task 上完成澄清、确认或安全默认后的状态推进；放弃澄清时，原 Task 会进入明确终态而不是生成新单或长期挂起 |
| 11 | 结果页能稳定展示主输出物、结构化结论、引用依据和执行摘要 |
| 12 | 对不存在最终输出物但已合法终止的任务，系统仍能提供可重访的终态摘要页或终态结果区，并与真正完成最终交付的结果页明确区分 |
| 13 | 用户能在任务详情或结果页看清主要路由决策、Agent 参与情况，以及自动重试或降级执行记录 |
| 14 | 系统能先反馈"任务已安排好"和 ETA，再在全部必需子任务完成后反馈最终结果 |
| 15 | 系统能区分 blocking 与 non-blocking 子任务，并在非关键子任务失败时给出可解释的部分完成或降级交付结果 |
| 16 | 系统在 blocking / non-blocking 标记已用于安排通知、ETA 或交付判定后，不会通过静默改标回溯性修正既有结论；若发生调整，用户能看到明确变更记录和重评估结果 |
| 17 | 系统在部分完成或降级交付时能展示结构化降级交付判定摘要，并在 non-blocking 子任务晚到成功时仅以补充更新方式呈现，不静默改写既有最终输出 |
| 18 | 系统对晚到 non-blocking 成功结果采用统一通知策略，不会把补充更新伪装成第二次"最终完成"通知 |
| 19 | 当等待中的任务长时间未释放、触发条件失效或冻结快照失效时，系统能展示等待异常原因、命中的阈值依据，并在不能自动恢复时进入明确人工介入状态 |
| 20 | 用户能查看本次任务使用和写入的关键记忆条目，并对记忆加载范围进行控制 |
| 21 | 系统能对写入、编辑、执行（含外部系统交互)按权限策略进行默认执行、触发权限确认或拒绝执行处理，并保留完整审计记录 |
| 22 | 系统能在事件连接中断后基于 `sequence` 或 `cursor` 恢复完整时间线，不因重复投递产生重复状态推进 |
| 23 | 在默认配置下，任务相关输入、进度、结果和权限授权只在发起端展示；启用同步策略后，相关消息才能跨端同步且不改变原时间线事实顺序 |
| 24 | 系统能在解析失败、局部失败、Prompt 超长、Skill 未激活等场景下给出可执行反馈 |

### P1 验收标准

> 监控告警与 Agent 自我反思能力验收以以下结果为准：

| 序号 | 验收项 |
|-----|-------|
| 1 | 系统能采集并展示任务成功率、平均耗时、模型调用成功率、平均响应时间、知识检索命中率、附件解析成功率等核心指标，指标数据可按时间范围筛选 |
| 2 | 系统能在任务失败率、队列积压、等待超时等场景下自动触发告警，告警记录可在监控面板查看 |
| 3 | 告警支持分级，用户可配置告警沉默窗口 |
| 4 | 监控面板展示核心指标趋势图、队列状态、失败任务分布，用户可从告警跳转至关联任务详情 |
| 5 | Agent 在生成最终输出前能自动触发质量自检，自检结果记录在任务详情中，不合格时支持自动重试或标记 |
| 6 | 任务完成后自动生成执行路径复盘摘要，包含关键决策点、工具调用序列、潜在风险点 |
| 7 | 系统能对推理链路进行一致性校验，发现不一致时在任务详情中标记并给出可能原因 |
| 8 | 系统能从历史成功任务中提取可复用模式，并在类似新任务入口时主动建议 |

## 8. 功能要求结论

P0 阶段功能要求的核心不是覆盖更多能力点，而是围绕任务主链路建立可执行、可追踪、可解释的产品闭环。
