# BiosBot 需求分析

## 1. 文档目的
本文档基于《BiosBot 需求分析文档》，将业务需求转化为具体的系统功能定义。重点明确**“知识库作为 Agent 专属资产”**的架构设计，并精确规范**“单 Agent 单任务”**的并发执行模型，确保开发团队准确实现知识的隔离性、专属性、可追溯性以及资源可控的任务调度。

### 1.1 产品范围
BiosBot 是一个运行在用户本地设备上的多智能体工作台。核心功能包括：
- **资源管理**：本地模型、Prompt 模板、Skills、工具集的配置。
- **Agent 编排**：可视化构建 Domain Agent，**为每个 Agent 绑定专属知识库**。
- **任务执行**：复杂任务的自动拆解、ReAct 循环执行、定时触发。
- **知识管理**：**Agent 维度的**文件解析、向量索引、版本冲突处理与隔离检索。
- **监控与运维**：全流程日志追踪、异常干预、资源监控、数据备份。

### 1.2 用户角色
- **定制者 (Customizer)**：主要用户，负责配置模型、编写 Prompt、安装 Skill、**为 Agent 挂载专属知识库**。
- **观察者 (Observer)**：查看任务执行过程、日志和结果的用户。## 2. 功能模块概览

## 3. 详细功能要求

### F01. 系统基础设置 (System Settings)

#### F01.1 本地工作目录配置 (Local Workspace Configuration)
- **功能描述**：管理系统所有数据（模型缓存、知识库、日志、配置、备份）的存储根目录。支持用户自定义指定，也支持系统默认自动创建。
- **处理逻辑**：
  1. **启动检测**：
     - 应用启动时，检查配置文件 `config.json` 中是否记录了 `workspace_path`。
  2. **场景 A：用户已指定过路径**
     - 直接使用该路径。
     - 校验路径是否存在及读写权限。若无效（如磁盘被移除），弹出错误提示并引导重新选择。
  3. **场景 B：首次启动 / 未指定路径**
     - **默认策略**：系统自动获取当前操作系统登录用户的**家目录 (Home Directory)**。
     - **默认子目录**：在家目录下创建名为 `BiosBot_Workspace` 的文件夹（例如：`C:\Users\Alice\BiosBot_Workspace` 或 `/home/alice/BiosBot_Workspace`）。
     - **自动初始化**：若该目录不存在，系统自动创建，并初始化标准子目录结构 (`/models`, `/knowledge`, `/logs`, `/config`, `/backup`)。
     - **静默进入**：初始化完成后，直接进入主界面，**不强制弹窗打断用户**（除非遇到权限错误）。
  4. **场景 C：用户主动修改路径**
     - 用户在“设置页”点击“更改工作目录”。
     - 弹出系统文件选择器供用户指定新路径。
     - **数据迁移向导**：若新路径为空，提示用户是否将旧数据迁移至新位置。
- **验收标准**：
  - 首次安装启动后，能在用户家目录下找到 `BiosBot_Workspace` 文件夹且包含完整子结构。
  - 用户可在设置中随时查看和修改当前路径。
  - 若家目录无写权限（极罕见情况），必须弹出阻塞性错误提示，强制要求用户选择其他有权限的目录。

#### F01.2 数据自动备份与恢复
- **功能描述**：每日自动备份关键配置文件和元数据，支持手动全量导出/导入。
- **处理逻辑**：
  1. 每日凌晨（默认 02:00）将 `config` 和部分 `metadata` 打包加密存入 `/backup` 目录，保留最近 7 份。
  2. 提供“导出工作空间”功能，生成包含配置和知识库索引的压缩包。
  3. 提供“导入工作空间”功能，校验完整性后恢复数据。
- **验收标准**：备份文件不占用过大空间；导入后系统能正常启动且配置无损。

### F02. 模型与资源中心 (Model Hub)

#### F02.1 模型接入与管理
- **功能描述**：支持添加本地模型文件 (GGUF 等) 或配置外部 API (OpenAI/Ollama/OpenRouter)。
- **输入**：文件路径 或 API Key + Base URL。
- **处理逻辑**：
  1. **本地模型**：解析文件头获取参数量、架构信息，生成模型卡片。
  2. **API 模型**：发送轻量测试请求验证连通性。
  3. **状态监控**：实时显示模型状态（就绪/加载中/错误）。
- **验收标准**：
  - 10 秒内完成连接测试并反馈结果。
  - 模型卡片清晰展示名称、参数量、上下文窗口大小。

#### F02.2 异构模型协同配置
- **功能描述**：允许为不同 Agent 分配不同模型，系统自动适配指令格式。
- **处理逻辑**：在 Agent 配置时，下拉列表仅展示“就绪”状态的模型；系统内部维护模型能力标签（如模态信息）。

### F03. 能力组件库 (Capability Library)

#### F03.1 Prompt 模板管理
- **功能描述**：创建、编辑、预览和管理 Prompt 模板。
- **功能点**：
  - **编辑器**：支持变量插值（如 `{{user_input}}`, `{{context}}`）。
  - **实时预览**：输入测试变量，右侧实时渲染最终 Prompt 效果。
  - **标签体系**：为模板打标（如“代码”、“写作”、“分析”），便于检索。
- **验收标准**：保存后立即可在 Agent 配置中被引用；预览无延迟。

#### F03.2 本地 Skill 管理
- **功能描述**：浏览、安装、配置和升级 Skill（技能包）。
- **功能点**：
  - **安装/卸载**：从上传本地文件包进行安装，支持卸载/禁用。
  - **参数配置**：安装后可配置 Skill 特有参数（如果有）。
  - **版本控制**：无版本控制，安装新的会替换老的。
- **验收标准**：当Skill正在被使用, 安装升级 Skill 应该失败, 提示用户Skill 被Agent使用的信息。

#### F03.3 基础工具集权限配置
- **功能描述**：管理系统内置工具（文件读写、网络请求、Shell 命令）的可用性。
- **功能点**：
  - 提供至少 5 种基础工具开关。
  - 支持细粒度权限（如：允许读文件但禁止写文件；允许访问特定域名）。
- **验收标准**：禁用某工具后，Agent 调用该工具时立即被拦截并报错“权限不足”。

### F04. Agent 工厂 (Agent Factory)

#### F04.1 Domain Agent 构建向导
- **功能描述**：可视化组装专业 Agent，**核心步骤包含“挂载专属知识库”**。
- **输入**：名称、描述、选定模型、Prompt 模板、已安装 Skills、**专属知识库集合**。
- **处理逻辑**：
  1. 分步引导用户选择组件。
  2. **知识库绑定步骤**：
     - 展示“新建知识库”或“从现有文件添加”选项。
     - 用户在此处上传的文件，**物理存储和向量索引将标记为该 Agent ID 独有**。
     - 支持多选已上传至该 Agent 名下的文件集合。
  3. **完整性校验**：只有当 模型、Prompt、Skill（可选）、知识库（可选）均配置有效时，才允许“创建”。
  4. **状态看板**：创建后生成 Agent 卡片，显示 `模型✓ | Prompt✓ | Skill✓ | 知识库 (N 个文件) ✓` 状态。
- **验收标准**：
  - 未绑定知识库的 Agent，在执行需要检索的任务时应提示“缺少知识上下文”或调用通用知识（若配置允许）。
  - 卡片清晰展示绑定的文件数量及总容量。

#### F04.2 Agent 详情页 - 知识库管理
- **功能描述**：在已创建的 Agent 详情页中，提供独立的“知识库”Tab 页，管理该 Agent 的专属资料。
- **功能点**：
  - **文件列表**：仅展示绑定给当前 Agent 的文件。
  - **增删改**：支持直接上传新文件（自动解析并索引到当前 Agent）、删除文件（释放当前 Agent 的索引，可选是否物理删除文件）。
  - **版本冲突**：针对当前 Agent 的同名文件上传，执行版本覆盖或历史保留逻辑。
- **验收标准**：在 Agent A 中上传的文件，绝不出现在 Agent B 的知识库列表中，除非用户手动再次上传/关联。

#### F04.3 Leader Agent 默认配置
- **功能描述**：系统预置不可删除的 Leader Agent，负责任务拆解与调度。
- **约束**：用户不可删除 Leader，但和其他的 Domain Agent 一样，可调整其使用的模型和全局 Prompt 策略, 安装 Skills、配置专属知识库。

### F05. 任务执行引擎 (Task Engine)

#### F05.1 任务分发与 Agent 占用逻辑
- **功能描述**：定义 Leader Agent 与 Domain Agent 的任务处理并发规则。
- **核心规则**：
  1. **单任务约束**：**任何 Agent（含 Leader）在同一时刻只能处于一个活跃任务的执行上下文中。**
  2. **Leader 的“调度即释放”机制**：
     - 当 Leader 接收到新任务时，它进入 `RUNNING` 状态进行意图分析和任务拆解。
     - 一旦拆解完成并生成子任务指令，Leader **立即调用系统调度接口**将子任务分发给对应的 Domain Agent。
     - **关键点**：分发动作完成后，Leader **立即标记自身为 `IDLE` (空闲)**，即使子任务尚未开始执行或正在执行中。
     - **结果**：Leader 可以立即接收并处理用户的下一个新任务，实现“高吞吐调度”。
  3. **Domain Agent 的串行执行机制**：
     - Domain Agent 接收子任务后，进入 `RUNNING` 状态。
     - 在该子任务结束（成功/失败/终止）前，该 Domain Agent **无法接收或处理任何其他任务**。
     - 若此时有新任务分配给该 Domain Agent，新任务必须进入 **该 Agent 的专属等待队列 (`Agent_Queue`)**，状态标记为 `QUEUED`。

#### F05.2 任务排队与资源竞争
- **功能描述**：处理多个任务请求同一个忙碌 Agent 的场景。
- **处理逻辑**：
  1. **Leader 侧排队**：若 Leader 正在拆解任务（极短时间），新任务在入口层短暂排队（`WAITING_FOR_LEADER`）。
  2. **Domain Agent 侧排队**：
     - 系统维护每个 Domain Agent 的独立 FIFO 队列。
     - 当 Domain Agent 完成当前任务变为 `IDLE` 时，自动从队列头部拉取下一个任务执行。
     - 用户可在界面上查看某 Agent 的“待办任务队列”长度。
  3. **优先级插队（可选 P1）**：支持用户将特定紧急任务在队列中置顶。

#### F05.3 ReAct 循环可视化
- **功能描述**：实时展示任务执行的“思考 - 行动 - 观察”过程。
- **界面要求**：
  - **时间轴视图**：按时间顺序展示每个步骤。
  - **分段显示**：
    - `💡 思考`：展示模型推理内容（如“我需要先读取文件 A”）。
    - `🛠️ 行动`：展示调用的工具及参数（如 `read_file(path='A.pdf')`）。
    - `👀 观察`：展示工具返回结果摘要。
  - **交互控制**：每个步骤旁提供“暂停”按钮；任务总控栏提供“强制终止”按钮。
- **界面展示**：
  - **Leader 视图**：显示“当前正在拆解任务 X”，完成后迅速切换为“等待新任务”或“正在拆解任务 Y”。**不显示** 子任务的详细执行过程（那是 Domain Agent 的事）。
  - **Domain Agent 视图**：显示“正在执行任务 X (步骤 3/10)”。若队列中有任务，显示“当前忙碌，队列中还有 N 个任务等待”。
  - **全局任务流**：清晰展示 任务 A -> Leader(拆解) -> 子任务 A1(Domain Agent 1 执行) ... 同时 任务 B -> Leader(拆解)。

#### F05.4 定时任务调度
- **功能描述**：支持 Cron 表达式或自然语言（“每周一上午 9 点”）设置任务。
- **处理逻辑**：
  1. 任务加入调度队列，状态为 `Scheduled`。
  2. 到达触发时间，Leader 非阻塞地启动任务实例。
  3. **资源排队**：若此时内存不足，任务进入 `Queued_Waiting_Resource` 状态，轮询检测资源，可用后自动执行。
- **验收标准**：时间到达后自动触发；资源不足时不报错失败，而是排队等待。

#### F05.5 异常介入与超时处理
- **功能描述**：处理执行中的卡顿、授权等待和错误。
- **处理逻辑**：
  - **授权超时**：若 Agent 请求用户授权（Ask）超过 10 分钟（可配）无响应，自动标记为“授权超时 - 拒绝”，释放资源并通知用户。
  - **等待态异常**：子任务阻塞超过阈值，标记为“异常”，提供 [重试] [调整参数] [关闭] 选项。
  - **强制终止**：用户点击终止，立即终止任务，生成包含已完成步骤的终止报告。

#### F05.6 强制终止与资源释放
- **逻辑补充**：
  - 终止某个 Domain Agent 正在执行的任务，该 Agent 立即变为 `IDLE`，并自动从队列中拉取下一个任务（若有）。
  - 终止 Leader 正在拆解的任务，Leader 立即变为 `IDLE`。

### F06. Agent 专属知识库 (Agent-Specific Knowledge Base)

#### F06.1 知识库隔离架构
- **功能描述**：系统底层采用 `Agent_ID + File_ID` 的双重索引机制。
- **处理逻辑**：
  1. **物理存储**：文件可物理复用（节省磁盘），但**向量数据（Embeddings）必须按 Agent_ID 隔离存储**或在元数据中严格标记 `owner_agent_id`。
  2. **检索边界**：检索请求必须携带 `agent_id` 过滤器，严禁跨 Agent 检索。
  3. **权限继承**：知识库的访问权限完全继承自 Agent 的配置。若 Agent 被禁用，其知识库不可被检索。

#### F06.2 专属文件解析与上传
- **功能描述**：用户在 Agent 配置页或详情页上传文件。
- **处理逻辑**：
  1. 用户上传文件（单文件 ≤ 100MB）。
  2. 系统自动识别类型：扫描件调用 OCR Skill，文本直接提取。
  3. **索引绑定**：解析后的向量数据直接写入该 Agent 的专属向量空间（或带标签的分区）。
  4. **质量校验**：标准 PDF 提取完整度需 > 95%，否则在该 Agent 的知识库中标记“低质量”，不影响其他 Agent。
  5. **元数据生成**：自动生成标题（可编辑）和概述，并记录 `created_for_agent: [Agent_Name]`。

#### F06.3 版本冲突处理 (Agent 维度)
- **功能描述**：处理**同一 Agent 下**的同名文件上传。
- **处理逻辑**：
  1. 检测到当前 Agent 库中存在同名文件时，弹窗提示：[覆盖旧版本] 。
  2. **隔离性说明**：Agent A 中的 `report.pdf (v2)` 与 Agent B 中的 `report.pdf (v1)` 互不影响，视为完全不同的知识资产。
- **验收标准**：检索时，Agent A 只能查到自己的 v2，绝不可能误读到 Agent B 的 v1。

#### F06.4 带溯源的 RAG 检索
- **功能描述**：Agent 调用知识库时，返回内容必须携带来源标注。
- **输出格式**：`内容摘要... [依据：{Agent_Name} 知识库 -> 文件名.pdf, 第 P3 页, 第 2 条]`。
- **安全约束**：若 Agent 尝试通过 Prompt 注入攻击获取其他 Agent 的知识库内容，系统应在检索层拦截并返回空结果。

#### F06.5 知识库生命周期管理
- **功能描述**：处理 Agent 删除时的知识库去向。
- **处理逻辑**：
  1. 用户删除 Agent 时，弹出确认框：“是否同时删除该 Agent 的专属知识库文件？”
  2. **选项**：
     - `是`：物理删除文件及向量索引，释放空间。
     - `否`：保留文件至“孤儿知识库”回收站，允许后续挂载给其他 Agent。
- **验收标准**：删除操作严格执行用户选择，无残留垃圾数据或误删。

### F07. 监控与审计台 (Monitor & Audit)

#### F07.1 全链路任务追踪
- **功能描述**：以树状结构展示任务及其子任务的执行详情。
- **功能点**：
  - 每个节点标注 `parent_task_id`。
  - 展示每个子任务所使用的 Agent 名称、模型版本、耗时。
  - 支持按任务 ID 搜索和过滤。
- **功能点补充**：
  - 日志中需记录任务所使用的 **Agent ID** 及其 **绑定的知识库版本快照**。
  - 若发生知识检索错误，能定位到具体是哪个 Agent 的哪个文件解析失败。

#### F07.2 资源实时监控
- **功能描述**：仪表盘展示 CPU、内存使用率。
- **监控指标**：
  - **Agent 状态矩阵**：实时展示每个 Agent 的状态 (`IDLE`, `RUNNING`, `QUEUED_TASKS: N`)。
  - **队列深度**：监控每个 Domain Agent 的积压任务数。若某 Agent 队列长期 > 5，提示用户“该领域 Agent 可能成为瓶颈，建议复制新增实例”。
  - **Leader 吞吐量**：统计 Leader 每分钟成功拆解并分发的任务数（体现其非阻塞特性）。
- **处理逻辑**：
  - 当单一任务占用资源超过阈值（如内存 90%），触发告警。
  - 若新任务导致资源溢出，自动触发 F05.3 的排队机制。

#### F07.3 审计日志导出
- **功能描述**：导出任务执行日志、配置变更历史。
- **安全要求**：导出前自动脱敏敏感信息（如 API Key、个人隐私数据）。

### F08. 统一交互窗口 (Unified Interaction Window) **[新增/重构]**

#### F08.1 唯一输入区域
- **功能描述**：界面底部提供唯一的输入框，作为用户与系统交互的唯一入口。
- **功能点**：
  - **文本输入**：支持多行自然语言输入。
  - **附件上传**：支持拖拽或点击上传文件（文档、图片、代码等）和粘贴图片。
    - 限制：单文件 ≤ 100MB，支持常见格式。
    - 预览：上传后在输入框上方显示文件缩略图/名称列表，支持移除。
  - **发送控制**：发送按钮、停止生成按钮（当 Agent 正在回复时显示）。
  - **快捷键**：Enter 发送，Shift+Enter 换行。

#### F08.2 流式输出与过程可视化
- **功能描述**：在主区域实时展示 Leader 及 Domain Agents 的执行过程。
- **展示逻辑**：
  - **Leader 思考**：显示“正在分析意图...”、“正在拆解任务...”。
  - **任务分发**：以时间轴或卡片形式展示：“[Leader] 分配任务给 [Agent A]: 读取文件...”、“[Agent A] 执行中...”。
  - **流式回复**：Agent 的最终结论或中间回复以打字机效果呈现。
  - **ReAct 细节**：默认折叠显示详细的思考过程（Thought/Action/Observation），用户可点击展开查看具体步骤、工具调用参数及原始日志。
  - **引用溯源**：若涉及知识库，回复中自动嵌入可点击的引用标记，点击高亮来源文件。

#### F08.3 历史记录浏览（扁平化）
- **功能描述**：由于没有“会话”概念，历史记录表现为**单一的、连续的时间流**。
- **功能点**：
  - **无限滚动**：向上滚动加载更早的对话记录。
  - **定位跳转**：支持搜索关键词，快速定位到历史某条消息。
  - **上下文切断**：用户可手动插入“--- 新话题 ---”标记（或通过命令），提示 Leader 在此处逻辑上切断前后强关联，但物理记录仍保留在同一列表中。
  - **导出**：提供“导出当前所有记录”功能，生成单个 Markdown/Txt 文件。

#### F08.4 附件管理与回顾
- **功能描述**：在对话流中，用户上传的附件以卡片形式展示。
- **功能点**：
  - 点击附件卡片可预览内容或下载。
  - 后续对话中提及该文件时，Leader 能自动关联到上传时的文件上下文。

## 4. 非功能性要求 (NFR)

### 4.1 性能要求
- **响应速度**：
  - 界面操作（保存、切换）响应时间 < 200ms。
  - 模型连接测试 < 10s。
  - ReAct 日志流式输出延迟 < 1s。
- **检索隔离开销**：增加 Agent ID 过滤条件后，向量检索延迟增加不超过 50ms。
- **Leader 延迟**：Leader 完成任务分发并恢复 `IDLE` 状态的耗时应 < 500ms（不含模型推理时间，仅指调度逻辑）。
- **资源限制**：
  - 单个文件解析上限 100MB。
  - 内存占用达到 95% 时触发保护机制（排队或降级）。

### 4.2 可靠性与安全性
- **数据持久化**：意外断电后，正在进行的任务状态可恢复或优雅失败，不损坏数据库文件。
- **本地隐私**：严禁在未授权情况下将用户文档、API Key 上传至任何外部服务器（除用户配置的 LLM API 端点外）。
- **权限隔离**：Agent 只能在指定的工作目录内读写文件，禁止访问系统敏感目录（如 `/etc`, `C:\Windows`）。
- **逻辑隔离**：必须在代码层和数据库查询层双重验证 `agent_id`，防止越权访问其他 Agent 的私有知识。
- **数据残留**：删除 Agent 并选择“删除知识库”后，必须确保向量数据库和文件系统中无任何该 Agent 的数据残留。
- **死锁预防**：若任务 A 的子任务需要 Agent X，而 Agent X 正在处理任务 B 的子任务（该子任务又依赖任务 A 的结果），系统需检测此类循环依赖并报错，避免无限排队。
- **队列持久化**：Agent 的任务排队信息需存入本地数据库，防止软件崩溃后队列丢失。

### 4.3 可维护性
- **日志规范**：所有模块需输出结构化日志（JSON 格式），包含 TraceID，便于故障排查。
- **配置热更**：修改 Prompt 或 Skill 配置后，无需重启应用，新建任务即刻生效。
- **迁移性**：支持将 Agent A 的部分知识库文件“迁移”或“复制”给 Agent B，操作需在 UI 上显式进行，后台重新建立索引关联。

### 4.4 兼容性与路径规范
- **跨平台路径处理**：
  - Windows: 默认 `%USERPROFILE%\BiosBot_Workspace`
  - macOS/Linux: 默认 `$HOME/BiosBot_Workspace`
- **特殊字符处理**：若系统用户名包含特殊字符或非 ASCII 字符，需确保路径编码正确，避免中文/空格导致文件读写失败。
- **权限最小化**：默认创建的目录权限应仅限当前用户读写，防止其他本地用户窥探数据。

## 5. 数据字典与状态机

### 5.1 Agent 状态定义
| 状态码 | 状态名称 | 描述 | 转换条件 |
| :--- | :--- | :--- | :--- |
| `IDLE` | 空闲 | 无任务执行，可立即接收新任务 | 任务完成 / 任务终止 / 分发完成 (Leader) |
| `RUNNING` | 执行中 | 正在处理唯一的一个活跃任务 | 接收新任务 (从 IDLE) |
| `BUSY_WITH_QUEUE` | 忙碌且有排队 | 正在执行任务，且专属队列中有等待任务 | 运行中且有新任务分配进来 |

### 5.2 任务状态定义
| 状态码 | 状态名称 | 描述 |
| :--- | :--- | :--- |
| `PARSING` | 解析中 | Leader 正在分析用户输入和附件 |
| `DISPATCHING` | 分发中 | Leader 正在拆解并分配子任务 |
| `QUEUED` | 排队中 | 子任务等待 Domain Agent 空闲 |
| `RUNNING` | 执行中 | Domain Agent 正在执行 ReAct 循环 |
| `AGGREGATING` | 汇总中 | Leader 正在收集结果并生成最终回复 |
| `COMPLETED` | 已完成 | 任务结束，结果已展示 |
| `EXCEPTION` | 异常 | 执行出错 |
| `TERMINATED` | 已终止 | 用户强制停止 |

### 5.3 知识库关联关系表 (简化)
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `kb_entry_id` | UUID | 知识库条目唯一标识 |
| `agent_id` | UUID | **所属 Agent ID (外键，强关联)** |
| `file_path` | String | 物理文件路径 (可共享) |
| `vector_partition` | String | 向量数据库分区名 (格式：`vec_{agent_id}`) |
| `version_tag` | String | 版本号 (v1, v2...) |
| `status` | Enum | 解析中/就绪/错误/归档 |

### 5.4 任务流转逻辑
1. **User** 在 **唯一窗口** 输入：“分析这个财报 PDF 并总结风险” + [上传 file.pdf]。
2. **Leader** (`PARSING`)：读取 file.pdf 元数据，识别意图为“财务分析”。
3. **Leader** (`DISPATCHING`)：
   - 拆解任务 1：提取 PDF 文本 -> 指派给 **OCR Agent**。
   - 拆解任务 2：分析文本风险 -> 指派给 **Finance Agent** (需等待任务 1 完成)。
4. **Leader** 变 `IDLE`。
5. **OCR Agent** (`RUNNING`)：执行提取，返回文本。
6. **Finance Agent** (`RUNNING`)：读取文本 + 调用其专属知识库（若有），生成风险分析。
7. **Leader** (`AGGREGATING`)：收集结果，生成最终总结回复。
8. **System**：在 **唯一窗口** 流式输出最终总结，并附带 ReAct 过程折叠面板。
9. **User** 继续追问：“针对第二点风险，有什么建议？” -> **Leader** 结合上文历史继续处理。

### 5.5 全局对话流数据结构 (Global Chat Flow)
| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `msg_id` | UUID | 消息唯一标识 |
| `timestamp` | Timestamp | 发送时间 |
| `role` | Enum | `user`, `assistant` (Leader/Domain), `system` |
| `content` | Text | 文本内容 |
| `attachments` | JSON | 附件列表 `{file_id, name, path, type}` |
| `related_task_id` | UUID | 关联的任务 ID（若此消息由任务触发） |
| `meta` | JSON | 元数据（ReAct 步骤、引用源、错误信息等） |
