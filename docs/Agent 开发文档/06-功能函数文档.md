# BiosBot 功能函数文档

## 1. 文档目的

本文档统一定义 BiosBot P0 阶段的功能函数契约，覆盖以下内容：

- 核心服务的职责与函数签名
- 对外 API 路径、请求 DTO、响应 DTO
- 任务事件流与权限交互载荷
- 错误码与字段映射约束

本文档是编码实现、接口联调与测试用例设计的函数契约真相源。若 05 更新，06 必须同步更新，不允许继续沿用旧版多会话、旧记忆表或未定稿扩展模型。

## 2. 设计基线

### 2.1 架构约束

- 单用户、单会话窗口：系统不暴露多会话 CRUD；对话历史统一落在 `session_memory`。
- 任务中心化：任务聚合根为 `tasks`，执行轨迹为 `task_logs`。
- 双层记忆：活跃历史与 STM 落在 `session_memory`，LTM 元数据落在 `long_term_memory`，向量内容落在 LanceDB。
- Agent 隔离：知识库、长期记忆、工具权限按 `agent_id` 隔离。
- API 基线路径统一为 `/api/v1/*`。

### 2.2 术语口径

- Chat History / STM：全局唯一的活跃历史流，对应 `session_memory`。
- LTM：长期记忆，对应 `long_term_memory` + LanceDB 向量记录。
- Clarification：Leader 因低置信度、缺参或无可用 Agent 而向用户发起的澄清流程。
- ReAct 日志：任务执行中的 `THOUGHT`、`ACTION`、`OBSERVATION` 摘要，对应 `task_logs`。
- ASK 授权：安全沙箱命中 `ASK` 策略后，等待用户对高风险操作作出明确决策。

### 2.3 状态枚举基线

`tasks.status` 允许值必须与 05、07 保持一致：

- `WAITING_FOR_LEADER` - 用户输入后，任务在此等待 Leader 处理（若 Leader 忙则排队）
- `CLARIFICATION_PENDING`
- `PARSING`
- `DISPATCHING`
- `QUEUED`
- `QUEUED_WAITING_RESOURCE`
- `RUNNING`
- `AGGREGATING`
- `COMPLETED`
- `EXCEPTION`
- `TERMINATED`

`task_logs.step_type` 允许值：

- `THOUGHT`
- `ACTION`
- `OBSERVATION`

`agent_tool_permissions.policy` 允许值：

- `DENY`
- `ASK`
- `ALLOW`

## 3. 核心服务函数契约

### 3.1 任务调度核心

#### 3.1.1 `TaskOrchestrator`

| 函数 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `createTask(input: TaskInput)` | 用户输入、附件、可选澄清数据 | `Promise<Task>` | 创建根任务；若需澄清则置为 `CLARIFICATION_PENDING` |
| `dispatchSubtasks(taskId: UUID, dag: DAG)` | 根任务 ID、任务 DAG | `Promise<void>` | 按 DAG 拆解并分发子任务 |
| `handleTimeout()` | 无 | `void` | 周期扫描超时任务并执行异常收敛 |
| `terminateTask(taskId: UUID)` | 任务 ID | `Promise<void>` | 强制终止任务，释放资源并写审计 |

#### 3.1.2 任务函数约束

- `createTask` 必须先落库 `tasks`，再进入内存队列或事件链路。
- 若入口判定需要澄清，函数不得直接创建运行态 Domain 子任务。
- `terminateTask` 必须把任务收敛到 `TERMINATED`，并清理相关运行句柄。

## 5. 请求 DTO 契约

### 5.1 任务请求 DTO

#### 5.1.1 `CreateTaskRequestDto`

```ts
interface CreateTaskRequestDto {
  content: string;
  attachments?: Array<{
    name: string;
    path?: string;
    mimeType?: string;
    size?: number;
  }>;
  clarificationData?: Record<string, unknown>;
}
```

约束：

- `content` 为必填，不能为空字符串。
- 低置信度、缺失关键参数或无可用 Agent 时，服务端必须返回 `CLARIFICATION_PENDING`。
- 创建成功后必须生成一条 `tasks` 记录；必要时同步写入首条 `session_memory` 用户消息。

#### 5.1.2 `RetryAfterClarificationRequestDto`

```ts
interface RetryAfterClarificationRequestDto {
  clarificationData?: Record<string, unknown>;
  newAgentConfigured?: boolean;
}
```

约束：

- `newAgentConfigured=true` 时，Leader 必须刷新 Agent 能力视图。
- 仅允许在 `CLARIFICATION_PENDING` 状态下调用。

#### 5.1.3 `ClarifyTaskRequestDto`

```ts
interface ClarifyTaskRequestDto {
  clarificationData: Record<string, unknown>;
  message?: string;
}
```

约束：

- `clarificationData` 为必填。
- 调用成功后，任务状态从 `CLARIFICATION_PENDING` 重置为 `PARSING` 或 `DISPATCHING`。

## 7. 事件流契约

### 7.1 任务事件订阅

- 订阅入口：`GET /api/v1/tasks/:id/events`
- 传输方式：WebSocket 或 SSE
- 事件必须按时间顺序发送，客户端按接收顺序渲染

### 7.2 事件 DTO

```ts
type TaskEventDto =
  | {
      type: 'STEP_UPDATE';
      data: {
        stepType: 'THOUGHT' | 'ACTION' | 'OBSERVATION';
        content: string;
        timestamp: number;
      };
    }
  | {
      type: 'STATUS_CHANGE';
      data: {
        status:
          | 'PENDING'
          | 'WAITING_FOR_LEADER'
          | 'CLARIFICATION_PENDING'
          | 'PARSING'
          | 'DISPATCHING'
          | 'QUEUED'
          | 'QUEUED_WAITING_RESOURCE'
          | 'RUNNING'
          | 'AGGREGATING'
          | 'COMPLETED'
          | 'EXCEPTION'
          | 'TERMINATED';
      };
    }
  | {
      type: 'AUTH_REQUEST';
      data: {
        authId: string;
        tool: string;
        args: Record<string, unknown>;
        expiresAt?: string;
      };
    }
  | {
      type: 'CLARIFICATION_REQUEST';
      data: {
        taskId: string;
        questions: string[];
      };
    };
```

### 7.3 事件语义约束

- `STEP_UPDATE` 只承载可审计摘要，不得透传完整原始思维链。
- `STATUS_CHANGE` 必须使用数据库状态枚举，不得混用临时前端状态名。
- `AUTH_REQUEST` 必须与 `authPendingMap` 中的挂起请求一一对应。
- `CLARIFICATION_REQUEST` 仅在任务进入正式澄清流程时触发。

- 工具路径、域名、命令等参数不合法时返回 `TOOL_PARAMETER_INVALID`，不得降级为通用 500。

## 9. 映射关系与实现规则

### 9.1 API 到持久化映射

| API / 函数 | 主要落表 | 说明 |
|-----------|----------|------|
| `POST /api/v1/tasks` | `tasks`、`session_memory` | 创建任务并记录用户输入 |
| `GET /api/v1/tasks/:id/events` | `tasks`、`task_logs` | 回放状态与 ReAct 摘要 |
| `POST /api/v1/tasks/:id/clarify` | `tasks` | 更新任务输入并重启规划 |
| `POST /api/v1/tasks/:id/terminate` | `tasks`、`audit_logs` | 强制终止并写审计 |
| `POST /api/v1/agents/:id/knowledge/upload` | `knowledge_files` | 文件元数据落库，正文与向量落 FS/LanceDB |
| `GET /api/v1/memory/ltm` | `long_term_memory` | 元数据查询，语义检索联动 LanceDB |
| `GET /api/v1/memory/history` | `session_memory` | 仅返回活跃历史 |
| `POST /api/v1/chat` | `session_memory`、`tasks` | 追加消息并触发任务 |
| `PUT /api/v1/agents/:id/tools/permissions` | `agent_tool_permissions` | 更新工具策略 |
| `POST /api/v1/auth/decision` | `agent_tool_permissions`、`audit_logs` | 可选持久化策略并写审计 |
| `GET/POST/PUT/DELETE /api/v1/scheduler/jobs` | `scheduled_jobs` | 定时任务 CRUD |
| `GET /api/v1/scheduler/jobs/:id/logs` | `job_execution_logs` | 任务执行历史查询 |
| `GET /api/v1/scheduler/status` | 内存状态 | 调度器运行时状态 |
| `POST /api/v1/scheduler/jobs/:id/trigger` | `scheduled_jobs`、`job_execution_logs` | 手动触发定时任务 |

### 9.2 实现规则

- 不再定义 `conversation`、`messages`、`memory_entries` 等旧模型接口。
- 聊天、历史、记忆接口必须直接使用 `session_memory` 与 `long_term_memory` 字段语义。
- 删除 LTM 时必须先把 `long_term_memory.is_active` 置为 `0`，再清理向量索引。
- 知识上传成功后必须生成 `knowledge_files` 记录；解析状态通过 `status` 字段表达。
- 任务事件与任务状态字段必须共享同一套枚举词表。
