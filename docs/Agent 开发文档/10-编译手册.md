# BiosBot 编译手册

## 1. 文档目的

本文档用于说明 BiosBot 在 P0 阶段的工程落地基线、开发环境准备、构建方式、运行方式、初始化步骤、联调验证与常见问题排查方法。

本文档主要支撑 01-需求分析、02-功能要求、03-技术分析、04-概要设计、05-详细设计、06-功能函数文档、07-数据库文档 与 08-界面文档 的工程实施。

本文档重点回答以下问题：

- P0 阶段系统应以什么工程形态落地。
- 本地开发和联调前需要准备哪些环境与目录。
- 前后端如何编译、启动、初始化、通过命令行快速部署和验证。
- 部署、健康检查、备份和监控在 P0 与 P1 的边界如何划分。

## 2. 适用范围与边界

### 2.1 适用范围

本文档适用于 BiosBot P0 阶段的本地优先单体实现，覆盖以下内容：

- Web 前端开发与构建。
- Node.js 单体后端开发与构建。
- SQLite、LanceDB、文件工作区、Skill 目录的初始化。
- 本地运行、联调、冒烟验证与最小发布校验。

### 2.2 P0 边界

P0 阶段的工程与部署边界如下：

- 单体架构，不拆分微服务。
- 本地优先，不构建复杂云端控制平面。
- 默认单用户或小团队，不构建多租户体系。
- 基础运行依赖为 SQLite、LanceDB、本地文件系统与外部 LLM Provider API。

### 2.3 当前仓库状态说明

当前工作区以标准文档为主，尚未提交可直接执行的正式源码目录与 `package.json` 脚本。因此：

- 本手册定义的是 P0 阶段正式工程的编译与运行基线。
- 文中的命令、脚本名、目录结构、数据目录和启动顺序作为正式工程的固定实施基线。
- 后续工程实现按本文定义的目录和脚本落地，不再另行定义第二套编译与启动规范。

## 4. 工程落地基线

### 4.1 目标技术栈

| 层次 | 技术基线 | 说明 |
|------|----------|------|
| 前端 | TypeScript + React + Vite + Zustand | Web 主交互界面 |
| 后端 | TypeScript + Node.js + Express | 单体服务，承载 HTTP API 与 WebSocket |
| 数据库 | SQLite + Drizzle ORM | 结构化数据持久化 |
| 向量库 | LanceDB | 本地嵌入式知识检索 |
| 文件存储 | 本地文件系统 | 附件、工作区、输出物 |
| 模型接入 | LLM Adapter | 统一外部模型调用入口 |
| 日志 | pino | 运行日志与审计日志 |
| 配置 | node-config | 分层配置管理 |

### 4.2 目标运行形态

工程形态收敛为“前端开发服务 + 后端单体服务 + 本地数据目录”：

- 浏览器访问 Web UI。
- Node.js 后端提供 HTTP API、WebSocket 与静态资源服务。
- 所有任务、事件、附件、知识库和工作区文件在本地保存。
- 模型调用统一经由 LLM Adapter 访问外部 Provider。

```text
│   ├── uploads/

├── data/

│   ├── knowledge.lancedb/
│   └── logs/

```

生产部署态目录如下：

```text
biosbot/
├── config/
├── dist/
│   ├── server/
│   └── web/
├── skills/
├── workspace/
├── data/
└── package.json
```

说明：

- 开发态按前后端分目录维护。
- 部署态收敛为单一 Node.js 运行单元。
- `dist/server/` 存放后端运行产物，`dist/web/` 存放前端静态产物。
- `skills/`、`workspace/`、`data/` 必须为显式目录，不得混入源码目录。

### 5.4 运行资源估算

以下为不同并发级别下的运行环境估算，供部署规划参考：

| 并发任务数 | 建议 CPU 核心 | 建议内存 | 建议磁盘 IOPS | 说明 |
|-----------|---------------|----------|---------------|------|
| 1 | 2 | 8GB | 普通 SSD | 最小可用基线 |
| 3（默认） | 4 | 16GB | 普通 SSD | 标准开发与联调 |
| 3 + 向量检索高频 | 4+ | 16GB+ | 高性能 SSD | LanceDB 索引超 100 万向量时 I/O 压力增大 |

说明：

- 每个并发任务最大占用 512MB 内存，叠加 Node.js 进程基础开销约 300MB。
- 模型调用为外部网络 I/O，对本机 CPU 无显著负载，但需保证到 LLM Provider 的网络延迟稳定。
- 磁盘空间需预留 `data/logs/` 日志增长、`workspace/outputs/` 输出物累积和 `data/knowledge.lancedb/` 向量索引膨胀。
- 以上为估算值，实际以压测结果为准。

## 7. 初始化顺序

系统首次启动或本地重建环境时，应严格按以下顺序初始化：

1. 基础设施层：加载配置，检查目录，初始化数据库连接与文件系统。
2. 知识与记忆模块：初始化持久记忆与 LanceDB。
3. Skill 模块：扫描 `skills/` 目录，仅建立元数据索引。
4. 模型适配模块：加载模型供应商配置并验证最小可用性。
5. 任务模块：恢复活跃等待订阅，检查未完成任务状态。
6. 会话模块：初始化默认会话上下文。
7. 调度器：启动等待订阅扫描器与后台任务。

关键约束：

- `use_skill(skillId)` 之前，Skill 工具不得暴露为可调用状态。
- 延迟任务恢复时必须恢复等待订阅，不得直接跳过到 `TaskReadyForPlanning`。
- 等待任务成功释放时，最小事件顺序必须满足 `TaskTriggerActivated -> TaskReadyForPlanning -> MemoryScopeConfirmed`。

### 7.1 启动失败分级与降级策略

以下定义各初始化步骤失败时的处理策略（来源：05-详细设计）：

| 初始化步骤 | 失败处理 | 是否致命 | 说明 |
|-----------|---------|---------|------|
| 数据库连接 | 终止启动，返回非零退出码 | 是 | SQLite 不可用则全部功能不可用 |
| LanceDB 路径不可访问 | 记录日志，继续启动 | 否（降级） | 知识检索不可用，其他功能正常 |
| Skill 目录缺失 | 创建空目录或使用默认值 | 否 | 无 Skill 可用但不影响主链路 |
| 模型连通性失败 | 记录日志，健康检查返回 `degraded` | 否（降级） | 无法执行任务但系统可访问 |
| 等待订阅恢复失败 | 记录失败，启动调度器重评估 | 否 | 调度器会在后续扫描中重新拾回 |
| 配置文件缺失 | 使用 `default.json` 基线值 | 否 | 仅 `SQLITE_PATH` 缺失时致命 |

启动降级约束：

- 只有数据库连接失败为致命错误，其余均允许降级运行。
- 降级状态必须反映在 `/health` 端点响应中，`status` 返回 `degraded` 而非 `healthy`。
- 降级组件恢复后应自动升级健康状态，无需重启。

## 8. 脚本约定

以下脚本名为正式工程固定脚本名。正式工程必须按以下脚本名提供能力，不再使用其他替代命名。

### 8.1 后端脚本

| 脚本 | 说明 |
|------|------|
| `npm run dev` | 启动后端开发服务 |
| `npm run build` | 编译 TypeScript 到 `dist/` |
| `npm run start` | 启动生产构建产物 |
| `npm run lint` | 执行后端静态检查 |
| `npm run test` | 执行后端单元/集成测试 |
| `npm run db:migrate` | 执行数据库迁移 |
| `npm run db:seed` | 初始化基础数据；无种子数据时执行空初始化并返回成功 |
| `npm run healthcheck` | 本地自检，验证数据库/路径/模型配置 |

后端脚本执行结果固定要求如下：

- `npm run dev` 启动 HTTP API、WebSocket、等待订阅恢复和调度器。
- `npm run build` 生成 `backend/dist/` 运行产物。
- `npm run start` 仅运行 `backend/dist/` 中的构建产物，不再触发编译。
- `npm run db:migrate` 仅执行迁移，不隐式写入测试数据。
- `npm run db:seed` 仅写入基础初始化数据，不执行迁移。
- `npm run healthcheck` 失败时返回非零退出码。

### 8.2 前端脚本

| 脚本 | 说明 |
|------|------|
| `npm run dev` | 启动 Vite 开发服务 |
| `npm run build` | 编译前端生产包 |
| `npm run preview` | 本地预览生产包 |
| `npm run lint` | 执行前端静态检查 |
| `npm run test` | 执行前端测试 |

前端脚本执行结果固定要求如下：

- `npm run dev` 启动 Web 开发服务，并可联通后端 API 与 WebSocket。
- `npm run build` 生成 `web/dist/` 静态产物。
- `npm run preview` 仅预览 `web/dist/`，不重新构建。

### 8.3 根级脚本

| 脚本 | 说明 |
|------|------|
| `npm run dev` | 并行启动前后端开发服务 |
| `npm run build` | 顺序构建后端与前端 |
| `npm run check` | 执行 lint + test + build |
| `npm run start` | 启动发布包中的生产服务 |
| `npm run deploy:local` | 一条命令完成本地单机部署初始化与启动 |
| `npm run deploy:prod` | 一条命令完成发布目录部署校验与启动 |

根级脚本执行顺序固定如下：

- `npm run dev` 先检查目录与配置，再并行启动 `backend` 与 `web`。
- `npm run build` 先构建后端，再构建前端。
- `npm run check` 先执行 lint，再执行 test，最后执行 build。
- `npm run start` 从发布目录根启动 `dist/server/` 中的后端服务，并加载同级 `config/`、`skills/`、`workspace/` 与 `data/`。
- `npm run deploy:local` 默认串行执行目录检查、依赖安装、配置模板校验、数据库初始化、前后端构建、服务启动与健康检查；命令失败必须返回非零退出码。
- `npm run deploy:prod` 默认串行执行发布目录校验、构建产物收敛、配置校验、数据库初始化、生产服务启动与健康检查；禁止要求用户手工逐条补跑核心步骤。

### 8.4 命令行快速部署约定

正式工程必须提供统一的命令行快速部署入口，作为本地试运行和发布部署的标准路径。

标准命令如下：


迁移与种子数据要求：

必须持久化的核心表（来源：07-数据库文档）：

| 表名 | 用途 | 初始化要求 |
|------|------|----------|
| `tasks` | 任务主聚合根 | 迁移建表 |
| `task_steps` | 任务步骤 | 迁移建表 |
| `task_events` | 事件流回放源 | 迁移建表，启动前必须存在 |
| `task_trigger_subscriptions` | 等待订阅，启动恢复依赖 | 迁移建表 |

| `task_outputs` | 任务输出物 | 迁移建表 |
| `permission_requests` | 权限请求与决策 | 迁移建表 |
| `model_call_logs` | 模型调用日志 | 迁移建表 |
| `memory_entries` | 持久记忆 | 迁移建表 |
| `memory_consolidation_jobs` | 记忆整理作业 | 迁移建表 |
| `knowledge_base_documents` | 知识库文档 | 迁移建表 |

幂等性保证（来源：07-数据库文档）：

- `memory_entries` 使用 `dedup_key` 字段保证同一 Agent、同一日期范围的定时整理幂等去重。
- `model_call_logs` 使用 `request_fingerprint` 请求指纹保证幂等标识。
- `task_trigger_subscriptions` 使用 `release_token` 保证同一等待订阅只能成功释放一次。

- 确认以上全部表已建立，可通过 `sqlite3 data/database.sqlite ".tables"` 检查。
- 确认种子数据写入后，`agents` 和 `skills` 表有基础记录。
- 确认主键、外键和关键联合索引已建立，尤其是：`task_id + sequence` 唯一索引、`task_id + cursor` 唯一索引、`permission_request_id + confirmation_version` 联合索引、`agent_id + summary_date + dedup_key` 唯一索引。

目录要求：

- SQLite 文件必须创建在 `data/` 下。
- 日志目录必须创建在 `data/logs/` 下。
- 向量库目录必须创建在 `data/knowledge.lancedb/` 下。

#### 10.1.1 WebSocket 实时事件服务

WebSocket 是前端接收任务事件增量推送的主通道，合约如下（来源：06-功能函数文档、03-技术分析）：

连接地址：

```
ws://<host>:<PORT>/ws
```

订阅格式：

```json
{
  "action": "subscribe",
  "topics": ["tasks/<taskId>/events", "permissions"]
}
```

事件推送格式：

```json
{
  "eventType": "TaskStatusChanged",
  "taskId": "task_001",
  "sequence": 15,
  "timestamp": 1710576600000,
  "summary": "人类可读摘要",
  "payload": { "previousStatus": "planning", "currentStatus": "running" }
}
```

可订阅事件类型：

| 事件类型 | 说明 |
|---------|------|
| `TaskStatusChanged` | 任务状态变更 |
| `TaskStepChanged` | 步骤状态变更 |
| `PermissionRequested` | 权限请求 |
| `OutputGenerated` | 输出生成 |
| `MemoryUpdated` | 记忆更新 |
| `ErrorOccurred` | 错误发生 |

事件流约束（来源：03-技术分析 §5.7）：

- 事件写入与投递采用至少一次投递语义，前端必须按 `eventId` 或 `sequence` 实现幂等处理。
- WebSocket 推送与 HTTP 补拉都基于同一事件游标模型，前端以上次已确认的 `sequence` 或 `cursor` 继续拉取。
- 禁止仅按时间戳恢复事件流；`sequence` 为单任务内严格递增序号，`timestamp` 仅用于展示与诊断。
- 同一事件允许重复到达，但不得重复推进任务状态。
- WebSocket 断连后前端通过 HTTP 补拉接口恢复事件流，不得仅依赖重连重播。

WebSocket 安全与网络约束：

- 启用 HTTPS 时，WebSocket 必须同步升级为 `wss://`，不得在 TLS 已启用场景回退为明文 `ws://`。
- WebSocket 连接默认采用 Token 鉴权；生产模式下不得允许匿名长连接订阅任务事件。
- 服务端与客户端需维持 30 秒心跳周期，用于识别僵尸连接与触发断线恢复。
- P0 默认不开放对外网监听；未显式配置反向代理或安全边界前，HTTP/HTTPS/WebSocket 均按本机访问口径运行。

### 11.1.1 前端静态产物托管

生产模式下，前端静态产物由后端 Node.js 服务直接托管，不依赖额外的 Nginx 或反向代理：

- 后端通过 Express 静态中间件加载 `dist/web/`，对外提供前端页面。
- SPA 路由回退：所有未命中 `/api`、`/health`、`/ws` 的请求均回退到 `dist/web/index.html`，由前端路由接管。
- API 路由与静态资源不得冲突：`/api/*`、`/health/*`、`/ws` 的路径由后端处理，其余路径由前端处理。
- `web/public/` 目录下的静态资源（如 `favicon.ico`）在构建时复制到 `web/dist/`，后端托管时直接可访问。
- `dist/web/` 目录必须为只读，后端不得在运行时写入此目录。

- 配置目录、`skills/`、`workspace/` 与 `data/` 与运行进程同机可访问。
- 日志输出到控制台和 `data/logs/`。

### 11.4 数据迁移与环境切换

从开发环境切换到生产环境时，数据目录处理规则如下：

| 目录/文件 | 处理方式 | 说明 |
|---------|---------|------|
| `data/database.sqlite` | 重新初始化 | 生产环境不应携带开发态测试数据 |
| `data/knowledge.lancedb/` | 重新初始化 | 向量索引需重建 |
| `data/logs/` | 清空 | 生产日志从零开始 |
| `workspace/` | 保留目录结构，清空内容 | 目录必须存在 |
| `skills/` | 完整复制 | Skill 定义与环境无关 |
| `config/` | 替换为生产配置 | `local.json` 必须重新配置 |

生产初始化流程：

```bash
cd biosbot
npm run db:migrate
npm run db:seed
npm run start
```

约束：

- 禁止将开发态 SQLite 文件直接复制到生产环境。
- 生产环境必须执行完整的 `db:migrate` + `db:seed` 流程。
- 若需保留开发数据（如知识库内容），应通过导出/导入接口操作，不得直接复制数据文件。

## 12. 启动后最小验证

### 12.1 健康检查

接口契约中已定义：

- `GET /health`（整体健康状态）
- `GET /health/ready`（就绪检查，确认服务可接受请求）
- `GET /health/live`（存活检查，确认进程未死锁）

边界说明：

- 01/02 文档将完整运维与部署能力定义为 P1。
- 06 文档已预留健康检查接口契约。
- 因此 P0 提供最小健康检查能力；完整监控、告警、备份与配置回滚在 P1 扩展。

最小健康检查至少覆盖：

- 服务进程状态。
- SQLite 连接可用性。
- 数据目录与磁盘空间可用性。
- 调度器运行状态。
- 活跃等待订阅恢复状态。
- 等待订阅积压或执行队列积压状态。

#### 12.1.1 健康检查响应模型

`GET /health` 响应结构如下（来源：03-技术分析 §5.10.1、06-功能函数文档 §2.12）：

```json
{
  "status": "healthy | degraded | unhealthy",
  "timestamp": 1710576600000,
  "checks": {
    "service":   { "status": "ok", "message": null },
    "database":  { "status": "ok", "message": null },
    "disk":      { "status": "ok", "message": "可用空间: 5.2GB" },
    "queue":     { "status": "ok", "message": "积压任务: 3" },
    "scheduler": { "status": "ok", "message": null }
  }
}
```

各检查项超时与阈值：

| 检查项 | 检查内容 | 超时 | 阈值 | 失败影响 |
|--------|---------|------|------|--------|
| `service` | 进程正常运行 | 100ms | — | unhealthy |
| `database` | SQLite 连接可用 | 500ms | — | unhealthy |
| `disk` | 可用空间足够 | 200ms | > 100MB | degraded |
| `queue` | 等待任务积压可控 | 200ms | < 1000 | degraded |
| `scheduler` | 调度器运行正常 | 100ms | — | degraded |

状态判定规则：

- 全部 `ok` → `healthy`。
- `service` 或 `database` 失败 → `unhealthy`（服务不可用）。
- 其他检查项失败 → `degraded`（服务可用但能力受限）。

#### 12.1.2 重启恢复与一致性验证

启动后除健康检查外，还应验证服务重启后的任务恢复行为：

- 对于 `waiting` 状态任务，系统重启后必须基于 `task_trigger_subscriptions` 重建等待订阅，并重新计算 `nextCheckAt`，不得依赖内存态定时器残留。
- 对于重启前处于 `running` 的任务，系统不得静默标记为成功；必须进入明确恢复路径，例如恢复到最近稳定步骤、进入 `intervention_required`，或记录等价恢复动作。
- 恢复过程中不得跳过正式事件写入；至少要能在时间线中看到重启恢复事实，例如 `SystemRestartDetected` 或等价恢复事件摘要。
- 重启后不得出现任务状态仍为 `running`、但没有活跃步骤执行与无后续恢复动作的僵尸状态。

### 12.2 主链路冒烟验证

启动后至少执行以下冒烟用例：

| 验证项 | 通过标准 |
|--------|----------|
| 会话输入 | 默认会话窗口可创建任务 |
| 入口模式识别 | 能识别 `immediate`、`queued`、`scheduled`、`event_triggered` |
| 低置信度澄清 | 能进入 `clarification_pending` |
| 等待订阅恢复 | 重启后等待任务不丢失 |
| 时间线排序 | 事件按 `sequence` 单调递增 |
| 权限确认 | 高风险动作可进入 `ask` 或 `deny` 路径 |
| 结果交付 | 能区分安排通知、最终结果、终态摘要、补充更新 |

### 12.3 关键结果字段校验

联调时应重点检查以下字段是否可稳定返回：

- 任务详情：`triggerMode`、`aggregateVersion`、`permissionStatus`、`lastEventSequence`。
- 等待态任务：`waitingThresholdMode`、`waitingThresholdBasisSummary`、`lastTriggerEvaluationSummary`。
- 结果页：`outputStage`、`finalOutputReady`、`degradedDeliverySummary`、`supplementalUpdates`、`terminalSummary`。
- 时间线：`sequence`、`cursor`、`eventType`、`summary`、`payload`。

### 12.4 关键错误码与恢复策略校验

联调和发布前还应重点校验以下错误归一化与恢复行为：

| 场景 | 预期错误码/结果 | 运行要求 |
|------|----------------|----------|
| 任务写入基于过期版本提交 | `TASK_WRITE_CONFLICT` | 不得覆盖 `tasks` 当前聚合状态 |
| 权限确认基于过期事件序号或版本提交 | `PERMISSION_CONFIRMATION_CONFLICT` | 不得写入新的权限决策日志或通过/拒绝事件 |
| 模型调用超时 | `MODEL_TIMEOUT` | 可按最多 3 次、基础延迟 100ms、指数退避、最大 2 秒策略有限重试 |
| 模型限流 | `MODEL_RATE_LIMITED` | 可触发退避重试或受控供应商切换 |
| 模型供应商不可用 | `MODEL_PROVIDER_UNAVAILABLE` | 仅在声明可替代关系时允许切换到 `fallbackProvider` |
| 模型响应结构异常 | `MODEL_RESPONSE_INVALID` | 不得无限制切换供应商掩盖协议错误 |

校验约束：

- 模型失败日志与事件需记录 `modelProvider`、`normalizedErrorCode`、`retryAttempt`、`fallbackProvider?` 和失败摘要。
- 供应商切换只允许发生在超时、限流或供应商不可用等可恢复故障场景。
- 写入冲突和权限确认冲突必须以接口错误显式暴露，不能用最后写入覆盖前一状态。

## 14. 安全与运行约束

### 14.1 权限与工具边界

- 读、写、执行权限统一受 `allow / ask / deny` 控制。
- 高风险写入、编辑、执行或外部调用默认不得直接放行。
- 未激活 Skill 的工具不可见、不可调用。
- 工具参数进入沙箱前必须经过命令注入、路径遍历和环境变量注入过滤。
- `ask` 权限确认默认超时为 5 分钟，最大超时为 30 分钟；默认超时行为为保持等待，可按策略配置升级为 `deny`。

### 14.2 密钥与敏感信息

- `LLM_API_KEY` 等敏感配置不得提交到仓库。
- 日志打印必须脱敏。
- P0 保证本地配置文件隔离；P1 扩展为加密存储、轮换与回滚。

### 14.3 安全发布检查清单

发布前必须逐项确认以下安全基线（来源：02-功能要求 N-16、01-需求分析 §8.7）：

| 检查项 | 验证方法 | 对应用例 |
|--------|---------|--------|
| API Key 未入库 | `git log --all -p -- '*.json' '*.env*' \| grep -i api_key` | T-N-16.1-01 |
| 日志脱敏生效 | 检查 `data/logs/` 中无明文密钥、令牌 | T-N-16.2-01 |
| 审计日志可写 | 确认 `audit_logs` 表存在且可插入 | T-N-16.3-01 |
| HTTPS 配置 | 启用 `HTTPS_PORT` 后确认 TLS 可用，明文回退被拒绝 | T-N-16.4-01 |
| 权限超时配置 | 确认默认 5min、最大 30min、超时行为符合基线 | T-N-13-06 |
| 沙箱拦截 | 模拟命令注入和路径遍历参数，确认被拦截 | T-N-13-07 |
| 未声明权限工具 | 确认未注册权限的工具默认禁止调用 | T-N-13-08 |

### 14.4 事件一致性

- 任务状态更新与事件追加必须在同一提交边界内完成。
- 前端恢复与补拉必须以 `sequence` 或 `cursor` 为准，不能仅以时间戳排序。
- 不允许结果页先拼接临时事实再反写输出模块。

## 15. 常见问题排查

### 15.1 后端无法启动

检查顺序：

1. Node.js 版本是否满足要求。
2. 配置文件是否缺失 `SQLITE_PATH`、`LANCEDB_PATH`、`LLM_BASE_URL`、`LLM_API_KEY`。
3. `data/`、`workspace/`、`skills/` 目录是否存在。
4. SQLite 文件是否被占用或无写权限。
5. 模型连通性是否失败。

### 15.2 前端无法联通后端

检查顺序：

1. Vite 代理地址是否正确。
2. 后端端口是否为 `3000` 或已同步更新。
3. WebSocket 地址是否与 HTTP 基址一致。
4. CORS 或本地代理配置是否缺失。

### 15.3 等待态任务重启后丢失

重点检查：

- `task_trigger_subscriptions` 是否落库。
- 启动恢复流程是否在调度器启动前执行。
- 是否错误地把内存定时器当作唯一等待状态来源。

### 15.4 时间线乱序或重复

重点检查：

- 是否使用 `sequence` 而不是时间戳排序。
- 是否存在重复释放导致多次写入 `TaskTriggerActivated`。
- 是否绕过 Task Service 直接写事件表。

### 15.5 Skill 可见但无法执行

重点检查：

1. Skill 目录是否同时包含 `SKILL.md` 与 `scripts/`。
2. 是否只完成了元数据扫描但未调用 `use_skill(skillId)`。
3. 工具是否命中权限确认或被权限策略拒绝。
4. 脚本运行环境是否存在。

### 15.6 结果页字段不完整

重点检查：

- 输出模块是否回交了 `outputStage`、`terminalSummary`、`supplementalUpdates`。
- 查询投影是否直接依赖事实结果聚合，而不是临时前端拼装。
- 合法终止但无最终输出时，是否错误遗漏 `terminalSummary`。

### 15.7 WebSocket 连接失败或事件丢失

检查顺序：

1. WebSocket 端点路径是否为 `/ws`，且与后端监听路径一致。
2. Vite 开发代理是否配置了 `ws: true`。
3. 前端重连后是否通过 HTTP 补拉接口恢复事件流，而不是仅依赖重连重播。
4. 是否按 `sequence` 而非时间戳去重与排序。
5. 事件消费端是否实现了幂等处理（同一事件可能重复到达）。

### 15.8 健康检查返回 degraded

按 `checks` 各子项定位问题：

| 检查项失败 | 排查方向 |
|-----------|----------|
| `disk` | 磁盘空间低于 100MB，清理 `data/logs/` 或 `workspace/cache/` |
| `queue` | 等待任务积压超过 1000，检查订阅释放逻辑或扩大扫描批量 |
| `scheduler` | 调度器线程挂死，检查死循环或超时 |
| `database` | SQLite 锁定或损坏，检查并发写入或文件权限 |
| `service` | 服务进程异常，检查日志中的未捕获异常 |

### 15.9 任务写入冲突或权限确认冲突

检查顺序：

1. 是否存在基于过期 `aggregateVersion` 提交的重试、取消或步骤完成写命令。
2. `task_write_commands` 中是否出现 `status=rejected` 且 `conflict_code=TASK_WRITE_CONFLICT`。
3. 是否存在基于过期 `expectedSequence` 或 `confirmationVersion` 的权限确认请求。
4. `permission_decision_logs` 是否错误写入了本应被拒绝的并发确认结果。
5. 接口层是否把冲突显式返回为 `TASK_WRITE_CONFLICT` 或 `PERMISSION_CONFIRMATION_CONFLICT`，而不是静默吞掉。

### 15.10 模型调用频繁超时或错误码不一致

检查顺序：

1. 所有模型调用是否都经由统一 `LLM Adapter`，而不是绕过适配层直接访问供应商 SDK。
2. `model_call_logs` 中是否把供应商错误统一归一化为 `MODEL_TIMEOUT`、`MODEL_RATE_LIMITED`、`MODEL_PROVIDER_UNAVAILABLE`、`MODEL_RESPONSE_INVALID`。
3. 可恢复故障是否按最多 3 次、基础延迟 100ms、指数退避、最大 2 秒策略执行有限重试。
4. 供应商切换是否仅发生在声明可替代关系的组合上，并写入 `fallbackProvider` 与重试来源。
5. 若错误属于响应结构异常，是否被错误地当作基础设施故障无限切换供应商。

## 16. P0 与 P1 实施分界

### 16.1 P0 必须完成

- 本地前后端可编译、可启动、可联调。
- SQLite、LanceDB、工作区、Skill 目录可初始化。
- 可完成默认会话输入、任务创建、任务详情、结果查看主链路。
- 可完成等待订阅恢复、事件序列补拉和最小健康检查。

### 16.2 P1 再实现

- 完整 `/health` 体系与更丰富的健康检查项。
- 配置热更新、变更历史与回滚。
- 自动备份、恢复与加密备份文件。
- 资源监控、告警面板与运行指标留存（N-15：任务成功率、平均耗时、模型调用成功率、队列积压等指标按小时/天维度聚合）。
- 任务模板与经验积累（N-10：基于历史任务创建可复用模板）。
- 完整运维部署能力（N-17）。
- 异常检测与预警规则（N-15.2：阈值告警、趋势告警、等待异常告警）。

## 17. 编译手册结论

本手册定义了 BiosBot P0 阶段正式工程的固定编译、运行与发布规范，而不是对当前文档仓库现状的脚本摘录。后续工程实现必须按本文落地，并满足以下底线：

- 架构仍为本地优先单体实现。
- 运行依赖仍为 SQLite、LanceDB、本地文件系统与统一 LLM Adapter。
- 启动顺序、等待订阅恢复、事件一致性和结果交付语义必须与标准文档保持一致。
- 编译、运行、联调、排障和发布前检查必须可被重复执行并可审计。
- 健康检查响应模型、WebSocket 事件流契约和数据库迁移幂等性必须满足本文定义的基线。
- 前端静态产物托管、SPA 路由回退和 API 路径隔离必须在生产构建中正确实现。
