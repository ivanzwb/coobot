# BiosBot 编译手册

## 1. 文档目的

本文档基于最新的 [05-详细设计]、[06-功能函数文档]、[07-数据库文档]、[08-界面文档] 与当前仓库实际结构，定义 BiosBot P0 阶段的工程准备、构建、启动、初始化、发布和部署验证方法。

本文档重点回答以下问题：

- 当前仓库应如何完成前后端构建与本地部署
- 运行时目录、配置文件、数据库和向量库应如何初始化
- 开发态、构建态和单机发布态的启动方式分别是什么
- 部署后应如何做健康检查、冒烟验证和常见故障排查

本文档只描述当前 P0 已定稿且当前仓库已具备实现基础的单机部署闭环，不再沿用旧版多会话路由、等待订阅式旧部署叙事或不存在的根级统一脚本。

## 2. 部署基线

### 2.1 架构基线

- 单用户、单会话窗口
- 单体后端服务 + Web 前端静态资源
- SQLite 存储结构化数据
- LanceDB 存储向量索引
- 本地文件系统存储知识文件、运行日志、工作区文件和输出物
- WebSocket 用于任务事件实时推送，HTTP 用于配置、任务、知识、记忆和审计接口

### 2.2 当前仓库基线

当前仓库已经包含可执行的前后端工程：

- `backend/`：Node.js + TypeScript 后端
- `web/`：React + Vite 前端
- `skills/`：本地 Skill 包目录
- `scripts/dev.sh`：Bash 启动辅助脚本
- `docs/Agent 开发文档/`：设计与交付文档

与旧版手册不同，当前仓库不是“仅有文档、尚无源码”的状态；编译与部署说明必须以现有 `backend/package.json`、`web/package.json`、`backend/config/default.json` 和当前目录结构为准。

### 2.3 P0 部署边界

P0 仅承诺以下工程能力：

- 本机开发运行
- 本机构建产物生成
- 单机手工发布
- 数据目录、工作目录、日志目录和知识目录初始化
- 健康检查与基础冒烟验证

P0 不承诺以下能力：

- 多节点集群部署
- 自动化灰度发布
- 容器编排平台接入
- 完整运维控制台
- 根级一键跨平台发布编排器

## 3. 目录与运行时结构

### 3.1 当前源码目录

```text
coobot/
├── backend/
│   ├── config/
│   ├── data/
│   ├── dist/
│   ├── e2e/
│   ├── scripts/
│   ├── src/
│   ├── tests/
│   └── workspace/
├── web/
│   ├── dist/
│   ├── public/
│   ├── src/
│   └── tests/
├── skills/
├── docs/
├── data/
├── workspace/
└── scripts/
```

说明：

- 当前后端默认从 `backend/` 目录启动，因此默认运行时数据路径相对 `backend/` 解析。
- 当前默认数据库位于 `backend/data/biosbot.db`。
- 当前默认向量库位于 `backend/data/vectordb`。
- 当前默认后端工作目录位于 `backend/workspace`。
- 当前默认 Skill 根目录配置为 `../skills`，即仓库根级 `skills/`。

### 3.2 当前构建产物目录

- 后端构建产物：`backend/dist/`
- 前端构建产物：`web/dist/`

说明：

- `backend/dist/` 由 `npm run build` 生成，入口为 `dist/index.js`
- `web/dist/` 由 `npm run build` 生成，为 Vite 静态产物
- 当前仓库未提供根级 `dist/server` / `dist/web` 收敛脚本，因此文档不得把该形态写成当前已实现事实

### 3.3 单机发布建议目录

若需要做 P0 单机发布，建议采用以下目录收敛方式：

```text
biosbot-release/
├── backend/
│   ├── dist/
│   ├── config/
│   ├── package.json
│   ├── package-lock.json
│   ├── node_modules/
│   ├── data/
│   └── workspace/
├── web/
│   └── dist/
└── skills/
```

说明：

- 后端进程仍以 `backend/` 作为工作目录运行
- 前端静态资源可由独立静态服务器托管，也可在后续版本接入后端统一托管
- 若需要迁移数据目录和工作目录，可通过配置文件修改相对路径

## 5. 配置说明

### 5.1 后端配置文件

当前后端使用 `config` 包读取配置，默认配置文件位于：

```text
backend/config/default.json
```

关键配置项如下：

| 配置项 | 说明 |
|------|------|
| `server.port` | HTTP 服务端口 |
| `server.host` | 监听地址 |
| `database.path` | SQLite 文件路径 |
| `vectorDb.path` | LanceDB 路径 |
| `workspace.path` | 后端工作区路径 |
| `skills.path` | Skill 根目录 |
| `llm.provider` | 模型提供方 |
| `llm.baseUrl` | 模型 API 地址 |
| `llm.apiKey` | 模型密钥 |
| `scheduler.*` | 调度扫描参数 |
| `execution.*` | 并发、步骤超时等执行参数 |
| `attachment.*` | 附件大小、切片和 OCR 参数 |

### 5.2 本地覆盖配置

建议在以下文件中维护本机私有配置：

```text
backend/config/local.json
```

示例：

```json
{
  "server": {
    "port": 3002,
    "host": "localhost"
  },
  "database": {
    "path": "./data/biosbot.db"
  },
  "vectorDb": {
    "path": "./data/vectordb"
  },
  "workspace": {
    "path": "./workspace"
  },
  "skills": {
    "path": "../skills"
  },
  "llm": {
    "provider": "openai",
    "apiKey": "<YOUR_API_KEY>",
    "baseUrl": "https://api.openai.com/v1",
    "defaultModel": "gpt-4"
  }
}
```

配置约束：

- 密钥不得写入 Git 仓库
- 数据路径和工作路径必须指向可写目录
- 若修改后端端口，前端代理配置必须同步更新

### 5.3 前端开发代理配置

当前前端开发代理定义见 [vite.config.ts](c:/Projects/coobot/web/vite.config.ts)：

- `/api` 代理到 `http://localhost:3002`
- `/ws` 代理到 `ws://localhost:3002`

前端开发代理约束：

- 代理端口必须与后端 `server.port` 保持一致
- WebSocket 路径必须保持为 `/ws`
- 若修改后端监听地址或端口，前端开发配置必须同步修改

## 6. 初始化流程

### 6.1 首次初始化顺序

首次搭建时，按以下顺序执行：

1. 检查 Node.js 与 npm 版本
2. 安装后端依赖
3. 安装前端依赖
4. 准备 `backend/config/local.json`
5. 初始化数据库与基础数据
6. 启动后端服务
7. 启动前端服务
8. 做健康检查与主链路冒烟验证

### 6.2 安装依赖

后端：

```bash
cd backend
npm install
```

前端：

```bash
cd web
npm install
```

### 6.3 初始化数据库

当前仓库数据库初始化以 `drizzle-kit push` 和种子脚本为准：

```bash
cd backend
npm run db:push
npm run db:seed
```

可选演示数据：

```bash
cd backend
npm run db:seed:demo
```

说明：

- `npm run db:push` 根据当前 Drizzle schema 将结构同步到 SQLite
- `npm run db:seed` 初始化基础数据
- `npm run db:seed:demo` 用于生成演示数据，不属于生产必需步骤
- 一次性迁移脚本按需单独执行，不属于每次初始化必跑步骤

### 6.4 目录初始化要求

以下目录必须在启动前存在，若不存在需创建：

- `backend/data/`
- `backend/data/vectordb/`
- `backend/workspace/`
- `backend/workspace/uploads/`
- `backend/workspace/outputs/`
- `backend/workspace/cache/`
- `skills/`

说明：

- `backend/src/db/init.ts` 会处理部分初始化逻辑，但部署脚本仍应保证父目录存在
- SQLite 文件会在首次连接和初始化阶段创建
- LanceDB 目录会在首次写入阶段创建

## 7. 开发环境运行

### 7.1 后端开发运行

```bash
cd backend
npm run dev
```

当前后端开发命令由 [backend/package.json](c:/Projects/coobot/backend/package.json) 中的 `tsx watch src/index.ts` 提供。

后端启动后应完成以下动作：

- 读取配置
- 创建 HTTP 服务
- 初始化 WebSocket 服务
- 启动调度器 `schedulerService`
- 启动每日记忆整理定时任务

依据 [index.ts](c:/Projects/coobot/backend/src/index.ts)，后端默认输出：

- `BiosBot server running on http://localhost:3002`
- `WebSocket server running on ws://localhost:3002/ws`

### 7.2 前端开发运行

```bash
cd web
npm run dev
```

当前前端开发命令由 [web/package.json](c:/Projects/coobot/web/package.json) 中的 `vite` 提供。

前端启动后应满足：

- 浏览器可访问 `http://localhost:5173`
- `/chat` 页面可加载
- API 代理可访问后端
- WebSocket 可连接 `/ws?clientId=...&entryPoint=web`

### 7.3 Bash 一体化开发脚本

当前仓库提供 [scripts/dev.sh](c:/Projects/coobot/scripts/dev.sh)，可用于 Bash、Git Bash 或 WSL 场景：

```bash
./scripts/dev.sh dev
```

其他支持命令：

- `./scripts/dev.sh install`
- `./scripts/dev.sh init`
- `./scripts/dev.sh build`
- `./scripts/dev.sh dev:backend`
- `./scripts/dev.sh dev:web`
- `./scripts/dev.sh start`

限制说明：

- 该脚本是 Bash 脚本，不应写成 Windows PowerShell 原生命令基线
- 当前仓库未提供等价的根级 PowerShell 部署脚本
- Windows 用户若不使用 Git Bash / WSL，应按本手册中的分步命令执行

## 9. 健康检查与联调验证

### 9.1 健康检查接口

当前后端已实现以下接口：

- `GET /health`
- `GET /health/ready`

基于 [routes/index.ts](c:/Projects/coobot/backend/src/routes/index.ts)，检查逻辑如下：

- `/health` 返回整体状态和组件级诊断
- `/health/ready` 检查数据库、LLM、调度器的就绪情况

### 9.2 健康检查命令

后端启动后可执行：

```bash
curl http://localhost:3002/health
curl http://localhost:3002/health/ready
```

预期：

- `/health` 返回 `healthy`、`degraded` 或 `unhealthy`
- `/health/ready` 在数据库和调度器正常时返回 200

### 9.3 WebSocket 连通性检查

前端任务实时流使用 WebSocket，URL 生成逻辑见 [taskRealtime.ts](c:/Projects/coobot/web/src/realtime/taskRealtime.ts)：

```text
ws://localhost:5173/ws?clientId=<clientId>&entryPoint=web
```

通过 Vite 代理后，后端实际接收路径为：

```text
ws://localhost:3002/ws?clientId=<clientId>&entryPoint=web
```

约束：

- 浏览器端通过查询参数传递 `clientId` 和 `entryPoint`
- 不依赖自定义 WebSocket Header 传递客户端上下文
- 若站点使用 HTTPS，则前端应自动升级为 `wss://`

### 9.4 最小冒烟验证

部署后至少完成以下验证：

1. 打开 `/chat`
2. 发送一条普通任务消息
3. 确认任务创建成功
4. 确认 WebSocket 收到任务事件
5. 调用 `/health` 与 `/health/ready`
6. 访问 Agent、Skills、Knowledge、Memory、Monitor、Settings 页面

## 11. 常见问题排查

### 11.1 后端启动失败

排查顺序：

1. 检查 Node.js 版本
2. 检查 `backend/config/local.json` 是否存在非法 JSON
3. 检查 `backend/data/` 是否可写
4. 检查 SQLite 文件路径是否可创建
5. 检查 LLM 配置是否完整

### 11.2 `/health/ready` 返回 503

常见原因：

- 数据库不可用
- 调度器未启动
- LLM 检测失败但数据库和调度器正常时，`ready` 仍可能为 `true`，需同时结合 `/health` 观察降级状态

### 11.3 前端可打开但无法联通后端

检查项：

- 后端是否监听 `localhost:3002`
- Vite 代理是否仍指向 `http://localhost:3002`
- 浏览器控制台是否存在 `/api` 或 `/ws` 代理错误

### 11.4 WebSocket 无法建立

检查项：

- 后端是否已调用 `setupWebSocket(server)`
- 连接路径是否为 `/ws`
- URL 是否包含 `clientId`
- 若站点为 HTTPS，前端是否使用 `wss://`

### 11.5 数据初始化后仍无基础数据

检查项：

- 是否执行了 `npm run db:seed`
- 是否错误地只执行了 `npm run build`
- 是否使用了错误的数据库路径，导致读取了空库
