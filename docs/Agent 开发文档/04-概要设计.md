# BiosBot 概要设计
## 1. 文档目的
本文档基于《BiosBot 需求分析》、《BiosBot 功能要求》及《BiosBot 技术分析》，将业务需求与技术选型转化为具体的系统架构设计。旨在明确系统模块划分、接口定义、数据流向及关键业务流程，指导后续的详细设计与编码实现。

## 2. 设计原则
- **本地优先 (Local-First)**：所有核心数据（模型、知识库、记忆、日志）默认存储于用户本地，除用户配置的 LLM API 外，无数据外传。
- **单体模块化 (Modular Monolith)**：P0 阶段采用单体架构，内部通过事件总线解耦，降低部署复杂度。
- **任务为中心 (Task-Centric)**：系统设计围绕“任务全链路闭环”展开，从入口澄清到结果交付再到知识沉淀。
- **隔离与安全 (Isolation & Security)**：严格实施 Agent 维度的知识库/记忆隔离，工具执行遵循最小权限原则。

## 3. 系统总体架构
### 3.1 逻辑架构分层
- 系统自下而上分为四层：
  - **基础设施层 (Infrastructure Layer)**
    - 存储引擎：SQLite (关系型数据), LanceDB (向量数据), Local File System (文件/模型)。
    - 运行时环境：Node.js + TypeScript (后端逻辑), Electron + React 或 Tauri (桌面壳/前端渲染)。
    - 外部依赖：Ollama/LLM API, OCR 引擎。
  - **核心服务层 (Core Services Layer)**
    - 任务调度中心 (Task Orchestrator)：负责任务拆解、DAG 生成、子任务分发、状态机管理。
    - Agent 运行时 (Agent Runtime)：管理 Leader/Domain Agent 实例，维护 ReAct 循环，处理上下文窗口。
    - 记忆引擎 (Memory Engine)：管理 STM (滑动窗口) 与 LTM (向量检索/事实提取)。
    - 知识库引擎 (Knowledge Engine)：负责文件解析、分块、向量化、隔离检索。
    - 安全沙箱 (Security Sandbox)：工具调用拦截、权限裁决 (DENY>ASK>ALLOW)、路径校验。
    - 事件总线 (Event Bus)
    内部消息队列，用于模块间解耦通信（如：任务状态变更、审计日志写入、UI 实时推送）。
  - **交互表现层 (Presentation Layer)**
    - 统一对话窗口：唯一输入出口，流式展示 ReAct 过程。
    - 配置工作台：Agent 工厂、Skill 市场、Prompt 模板、系统设置。
    - 监控审计台：任务甘特图、资源监控、日志查询。
### 3.2 物理部署架构
- **客户端应用**：
  - P0（功能验证阶段）：基于 React Web 客户端，与 Node.js 后端服务通信。
  - P1（可选演进）：支持替换为 Tauri 等更轻量桌面壳，但对外 API 与模块划分保持不变。
- **数据目录**：{UserHome}/BiosBot_Workspace，包含 config, models, knowledge, logs, backup 子目录。
- **网络边界**：仅允许 outbound 连接至用户配置的 LLM API 端点，禁止其他外网访问（除非 Skill 显式授权）。

## 4. 核心模块设计
本节从概要层面给出与《02-功能要求》中 F01～F09 以及《03-技术分析》中 4.1～4.5 对应的核心模块设计视图，用于指导后续详细设计与实现。

### 4.0 系统基础设置与配置中心 (F01 对应)
- **职责**：承载工作目录、环境自检、备份/恢复等系统级配置能力，对应 02-功能要求 中 F01 及 03-技术分析 中 5.1/5.3 的存储与目录结构。
- **组成**：
  - 配置管理器：读取/写入 config/settings.json，暴露统一的 ConfigService 给其他模块使用。
  - 工作目录管理器：负责初始化 {UserHome}/BiosBot_Workspace 目录结构（config/models/knowledge/logs/backup），并在用户修改路径时迁移或重建目录。
  - 备份与恢复子模块：按计划任务触发，将关键配置与 Metadata 打包到 backup/ 目录，提供导入/导出接口。
- **与其他模块关系**：
  - 为任务调度中心、Agent 运行时、知识库引擎、记忆引擎提供统一的 workspace_path、数据库/向量库连接等基础依赖。
  - 与事件总线集成，在配置变更或备份完成时发布事件，供监控台与 UI 订阅。
### 4.1 任务调度中心 (Task Orchestrator)
- **职责**：作为系统的“大脑”，负责任务的生命周期管理。
- **入口门控**：
  - 接收用户输入，运行意图识别。
  - 若置信度 < 阈值，状态置为 CLARIFICATION_PENDING，暂停后续流程，触发 UI 澄清交互。
- **拆解与分发**：
- Leader Agent 分析意图，生成子任务 DAG。
  - 非阻塞机制：分发完成后，Leader 立即释放资源 (IDLE)，子任务进入对应 Domain Agent 的 FIFO 队列。
- **状态机管理**：
  - 维护全局任务状态表 (tasks 表)。
  - 监听子任务完成事件，触发聚合 (AGGREGATING) 或异常处理。
  - 超时守卫：轮询 RUNNING/QUEUED 任务，超时自动标记 EXCEPTION。
### 4.2 Agent 运行时 (Agent Runtime)
- **职责**：执行具体任务逻辑，维护 ReAct 循环。
- **实例管理**：
  - Leader Agent：单例，系统预置，不可删除。
  - Domain Agent：按需实例化，支持多版本配置（不同模型/Prompt/Skill）。
- **并发控制**：
  - 每个 Agent 维护独立 Task Queue。
  - 同一时刻仅处理一个任务 (RUNNING)，其余排队 (QUEUED)。
- **ReAct 执行器**：
  - 循环：Thought -> Action (调用 Tool/RAG) -> Observation。
  - 上下文注入：动态组装 System Prompt + STM 历史 + LTM 检索结果 + RAG 片段。
  - 流式输出：每一步骤通过 Event Bus 推送至 UI。
### 4.3 模型与资源中心 / 能力组件库 (Model Hub & Capability Library)
- **职责**：承载 F02/F03 约定的模型接入与管理、Prompt 模板管理、Skill 管理与基础工具权限配置，对应 03-技术分析 中 3.3（模型适配与推理引擎）和 4.4（工具执行与安全沙箱）。
- **模型与资源中心 (Model Hub)**：
  - 模型注册表：维护本地模型与远程 API 模型清单（名称、上下文窗口、模态能力），对外暴露统一的 ILlmProvider 接口实现。
  - 连接测试与健康检查：提供 10 秒内完成的连通性测试接口，结合 8.2 的自检机制，对接 UI 的“模型状态面板”。
  - 模型分配策略：在 Agent 配置中引用模型注册表条目，实现“不同 Agent 绑定不同模型”的异构模型协同配置。
- **能力组件库 (Capability Library)**：
  - Prompt 模板子模块：管理模板元数据与版本，支持在 Agent 配置与任务调度中心中按 ID 引用，并结合 03-技术分析 中的上下文管理策略进行裁剪与拼装。
  - Skill 管理子模块：维护已安装 Skill 清单、版本与参数，向 Agent 运行时暴露可用工具集合。
  - 基础工具权限配置：与安全沙箱共享同一权限配置表，实现 F03.3 所述的工具开关与白名单控制。

### 4.4 记忆引擎 (Memory Engine)
- **职责**：管理短期与长期记忆，支撑个性化与连续性。
- **短期记忆 (STM)**：
  - 存储：SQLite session_memory 表（同时作为全局对话流存储）。
  - 策略：采用“滑动窗口 + 归档”组合算法：前台在上下文组装阶段基于 Token 预算裁剪低重要性消息，后台归档任务按时间窗口与最小条数阈值将较早历史汇总为摘要写入 LTM，并在 session_memory 中标记 `is_archived/ltm_ref_id`，与 03-技术分析 4.5.2。
- **长期记忆 (LTM)**：
  - 存储：SQLite (元数据) + LanceDB (向量)。
  - 写入管道：异步后台进程，提取 Fact/Preference，经用户确认（可选）后入库。
  - 检索增强：每次回复前，基于 Query 进行混合检索 (Vector + Keyword)，按时间衰减加权排序。
  - 隔离性：LTM 数据强制绑定 agent_id，检索时自动过滤。
### 4.5 知识库引擎 (Knowledge Engine)
- **职责**：管理 Agent 专属知识资产。
- **隔离架构**：
  - 物理目录：/knowledge/{agent_id}/。
  - 向量分区：LanceDB Collection 命名 vec_{agent_id} 或 Metadata 强过滤。
- **处理流水线**：
  - 上传 -> 类型识别 -> 解析 (PDF/OCR/Txt) -> 分块 (Chunking) -> 向量化 (Embedding) -> 入库。
  - 异步执行，不阻塞 UI。
- **溯源机制**：
  - Chunk Metadata 包含 {file_name, page_num, offset}。
  - RAG 返回结果携带溯源标签，UI 支持点击高亮原文。
### 4.6 监控与审计台 (Monitor & Audit)
- **职责**：对应 F07，基于 03-技术分析 中 6.1/6.2 的可靠性与断点恢复策略，提供任务全链路追踪、资源监控与审计日志导出能力。
- **任务追踪子模块**：
  - 订阅任务调度中心与 Agent 运行时在 Event Bus 上发布的状态变更事件，将任务树（父子任务关系）与时间线写入可查询存储（如 task_logs/审计表）。
  - 向前端暴露“树状视图 + 甘特图”数据结构，支撑 02-功能要求 中 F07.1 的展示需求。
- **资源监控子模块**：
  - 周期性采集 CPU/内存/磁盘等资源指标，并结合 03-技术分析 中的资源配额策略，触发阈值告警事件。
  - 为 UI 提供 Agent 状态矩阵与队列深度等指标，支撑 F07.2。
- **审计日志子模块**：
  - 聚合安全沙箱、权限裁决器与系统关键操作的审计事件，支持按时间/任务/Agent 过滤与导出，满足 F07.3。

### 4.7 统一交互窗口与前端框架 (Unified Interaction Window & Frontend)
- **职责**：对应 F08，作为唯一交互入口与结果展示层，承载任务流、ReAct 流、水位监控与记忆管理入口。
- **结构设计**：
  - 左侧导航：对话、Agent、Skills、Prompt、监控台、系统设置等入口，对应 02-功能要求 中附录的 UI 原型关键点。
  - 中部主区域：单一对话流窗口，展示用户消息、Leader/Domain Agent 回复、任务执行卡片与 ReAct 折叠详情。
  - 右侧边栏：任务详情与资源监控视图，与监控与审计台模块联动。
- **统一输入区**：
  - 底部输入框 + 附件上传控件，封装为 ChatInput 组件，内部通过 WebSocket/HTTP 调用任务管理接口（7.1）。
  - 支持 F08.1～F08.4 要求的快捷键、多附件预览与历史浏览入口。
- **与后端交互**：
  - 使用 WebSocket/SSE 订阅任务状态、ReAct 步骤与资源监控数据流，映射到前端状态管理（如 Zustand）。

### 4.8 安全沙箱 (Security Sandbox)
- **职责**：保障本地执行安全。
- **权限裁决器**：
  - 拦截所有 Tool 调用请求。
  - 查询 agent_tool_permissions 表。
  - 执行 DENY (直接拒绝), ASK (挂起任务等待 UI 回调), ALLOW (放行)。
- **文件系统沙箱**：
  - 校验所有文件路径是否以 workspace_path 开头。
  - 黑名单目录拦截 (/etc, C:\Windows 等)。
  - 审计日志：所有裁决结果、文件操作、API 调用均记录至 audit_logs。

## 5. 数据设计
### 5.1 数据库 schema (SQLite)
- **agents**: id, name, model_config_json, prompt_template_id, status, created_at
- **tasks**: id, parent_id, agent_id, status, input_payload, output_summary, error_msg, created_at, updated_at
- **task_logs**: id, task_id, step_type (thought/action/observation), content, timestamp
- **knowledge_files**: id, agent_id, file_path, status (processing/ready/error), version_tag
- **session_memory**: id, session_id, role, content, summary, token_count
- **long_term_memory**: id, agent_id, category, key, value, embedding_id, confidence, access_count
- **audit_logs**: id, event_type, actor (agent/user), details_json, timestamp
### 5.2 向量数据库 (LanceDB)
- **Collections**: 按 agent_id 动态创建或逻辑分区。
- **Schema**: vector (float[]), text (string), metadata (json: {file_id, page, agent_id})
## 5. 数据设计
本节从概要层面对关系数据库、向量数据库和文件系统结构进行设计，承接《02-功能要求》第 6 章“数据字典与状态机”和《03-技术分析》第 5 章的数据层方案。

### 5.1 关系数据库 schema (SQLite)
- **agents**（Agent 定义表）：`id, name, model_config_json, prompt_template_id, status, created_at`
  - 对应 02-功能要求 6.1 中的 Agent 状态，`status` 字段取值包括 `IDLE`/`RUNNING`/`BUSY_WITH_QUEUE`，用于驱动监控面板与队列控制。
- **tasks**（任务主表）：`id, parent_task_id, assigned_agent_id, status, input_payload, output_summary, error_msg, created_at, updated_at`
  - 对应 02-功能要求 6.2 中的任务状态机，`status` 字段覆盖 `CLARIFICATION_PENDING`、`PARSING`、`DISPATCHING`、`QUEUED`、`RUNNING`、`AGGREGATING`、`COMPLETED`、`EXCEPTION`、`TERMINATED` 等枚举值。
  - `parent_task_id` 支撑任务树追踪，与 F07.1 的全链路任务追踪一致。
- **task_logs**（任务步骤与 ReAct 日志）：`id, task_id, step_type (thought/action/observation), content, timestamp`
  - 对应 F05.3、F07.1，记录 ReAct 步骤与时间线，为前端的时间轴视图和任务详情页提供数据来源。
- **knowledge_files**（知识库文件表）：`id, agent_id, file_path, vector_partition, version_tag, status`
  - 对应 02-功能要求 6.3 中的知识库关联关系表，`vector_partition` 字段与向量库分区（如 `vec_{agent_id}`）保持一致。
- **session_memory**（全局对话流 + 短期记忆 STM 表）：`id, session_id, role, content, attachments_json, related_task_id, meta_json, summary, token_count, created_at, is_archived, ltm_ref_id`
  - 与 03-技术分析 4.5.2 的滑动窗口算法配合，`token_count` 用于控制每个会话的上下文窗口大小；同时作为唯一的全局 Chat History 存储，通过 `is_archived`、`related_task_id` 等字段区分 STM 视图与完整历史对话流。
- **long_term_memory**（长期记忆 LTM 元数据表）：`id, agent_id, category, key, value, embedding_id, confidence, access_count, last_accessed, is_active`
  - 对应 02-功能要求 F09 与 03-技术分析 4.5.3/4.5.4 中的记忆模型，支撑按偏好/事实/项目等类别管理长期记忆。
- **audit_logs**（审计日志表）：`id, event_type, actor (agent/user), details_json, timestamp`
  - 对应 F07.3 与 03-技术分析 4.4.1/7.2，记录权限裁决与关键安全事件，支撑审计导出。

### 5.2 向量数据库 (LanceDB 等)
- **Collections/Partitions 设计**：
  - 按 Agent 维度划分 Collection 或 Partition，例如 `vec_{agent_id}`，与 knowledge_files/long_term_memory 中的 `agent_id` 和 `vector_partition` 保持一致，实现知识库与记忆的物理/逻辑隔离。
- **Schema 概要**：
  - 字段包括 `embedding`（向量）、`text`（原文或摘要）、`metadata`（JSON：`{file_id, page, agent_id, chunk_id, source_type}`）。
  - 与 03-技术分析 4.2.2/4.5.3 中的文件切片与记忆向量化流程相对应。

## 6. 关键业务流程设计
本节从概要层面对核心业务链路进行描述，对应《02-功能要求》第 5～6 章中的任务流转逻辑与 UI 流程，以及《03-技术分析》4.1～4.5 的实现方案。

### 6.1 任务执行全流程（对应 F05 / 任务状态机）
- **输入**：用户在统一窗口输入指令 + 附件。
- **澄清**：若低置信度，进入 CLARIFICATION_PENDING，用户补充后继续。
- **拆解**：Leader Agent 解析意图，生成子任务列表。
- **分发**：Leader 将子任务推入对应 Domain Agent 队列，自身变 IDLE。
- **执行**：
  - Domain Agent 出队任务，状态 RUNNING。
  - 加载专属知识库 (RAG) 与记忆 (STM/LTM)。
  - 执行 ReAct 循环，每步推送日志至 UI。
  - 遇到高风险工具 -> 触发 ASK -> 用户授权 -> 继续。
- **聚合**：子任务完成，Leader 收集结果，生成最终回复。
- **沉淀**：异步提取关键 Fact 写入 LTM，归档会话至 STM，对应 F09 的记忆写入管道。

### 6.2 知识库上传与索引流程
- **上传**：用户在 Agent 详情页拖拽文件。
- **预检**：检查文件大小 (<100MB)，同名文件版本冲突提示。
- **解析**：后台线程调用解析器 (PDF/OCR)。
- **向量化**：调用 Embedding 模型生成向量。
- **入库**：
  - 文件存入 /knowledge/{agent_id}/。
  - 向量存入 LanceDB (带 agent_id 标签)。
  - 元数据写入 SQLite knowledge_files。
- **通知**：UI 更新文件状态为“就绪”，并同步任务详情页/知识库管理页的文件状态。

### 6.3 权限裁决流程
- **请求**：Agent 发起 call_tool(name='write_file', args={...})。
- **拦截**：Sandbox 拦截请求，查询权限配置。
- **决策**：
  - DENY: 抛出异常，任务进入 EXCEPTION。
  - ALLOW: 直接执行工具代码。
  - ASK:
    - 挂起当前协程。
    - 发送 AUTH_REQUEST 事件至 UI，启动 10min 倒计时。
  - 分支 A (用户同意): 恢复协程，执行工具。
  - 分支 B (用户拒绝/超时): 终止任务，释放资源，记录审计日志。

### 6.4 会话与记忆管理流程（对应 F08 / F09）
- **对话流记录**：所有用户与 Agent 消息写入 session_memory 表，并通过 session_id/related_task_id 关联任务；该表既承载全局对话流（Chat History），又作为 STM 的物理存储。
- **STM 更新**：每轮对话结束，更新对应的 session_memory 记录（或插入新记录），维护 summary/token_count 等字段，执行滑动窗口裁剪，保证满足上下文窗口约束。
- **LTM 提取与确认**：后台记忆提取器根据对话内容生成候选偏好/事实；高置信度项可直接入库，低置信度或敏感信息通过记忆管理界面弹出确认。
- **记忆检索注入**：新任务或新一轮对话开始前，按当前 Query 检索 STM/LTM，将结果注入模型上下文，支撑“继续昨天那个项目”等场景。

## 7. 接口设计 (Internal API)
本节给出内部 API 的概要分层设计，用于支撑 F01～F09 中各模块的实现，并与《03-技术分析》中 3.x/4.x 组件的职责保持一致。所有接口以 `/api/` 为前缀，对外通过 HTTP/JSON 与 WebSocket/SSE 暴露。

### 7.1 任务管理接口（对应 F05/F07/F08）
- `POST /api/tasks`：创建新任务（输入文本、附件、触发模式等），由任务调度中心接管，返回任务 ID。
- `GET /api/tasks/{id}`：获取任务详情（基本信息、当前状态、绑定 Agent、最近步骤摘要）。
- `GET /api/tasks/{id}/events` (WebSocket/SSE)：订阅任务 ReAct 步骤与状态流，用于对话流和任务详情页实时展示。
- `POST /api/tasks/{id}/terminate`：强制终止任务（对应 F05.6），触发资源释放与终止报告生成。
- `POST /api/tasks/{id}/clarify`：提交入口澄清信息（对应 F05.0），驱动任务从 `CLARIFICATION_PENDING` 进入可执行链路。

### 7.2 Agent / 模型 / 能力管理接口（对应 F02/F03/F04）
  - `GET /api/agents`：获取 Agent 列表及运行状态（队列长度、当前任务等）。
  - `POST /api/agents`：创建/更新 Agent 配置（模型、Prompt、Skills、知识库绑定等）。
  - `DELETE /api/agents/{id}`：删除 Agent（含可选的知识库/记忆清理策略）。
  - `GET /api/models`：列出已注册的本地模型与 API 模型及其能力信息。
  - `POST /api/models`：注册新模型（本地路径或 API 信息）。
  - `POST /api/models/{id}/test`：触发模型连通性与性能测试（对应 F02.1 的 10 秒连接测试）。
  - `GET /api/prompts` / `POST /api/prompts`：列出与创建/更新 Prompt 模板，供 Agent 引用。
  - `GET /api/skills`：列出已安装 Skill 及其参数/权限配置。
  - `POST /api/v1/skills/preview` / `POST /api/v1/skills/import` / `DELETE /api/v1/skills/:id`：执行 Skill 安装预读取、正式导入与卸载。
  - `POST /api/agents/:id/skills`：为特定 Agent 绑定/解绑 Skill，并更新权限策略（对应 F03.2/F03.3）。

### 7.3 知识库管理接口（对应 F04/F06）
- `POST /api/agents/{id}/knowledge/upload`：上传知识文件，触发后台解析与向量化（对应 6.2 流程）。
- `GET /api/agents/{id}/knowledge/files`：列出当前 Agent 的知识库文件列表及解析状态。
- `DELETE /api/agents/{id}/knowledge/files/{fileId}`：删除或解除绑定指定文件（配合“物理删除/仅解绑”选项）。
- `POST /api/agents/{id}/knowledge/files/{fileId}/reindex`：重新解析并重建向量索引（对应 F06.5 的“重新解析”）。

### 7.4 记忆与会话管理接口（对应 F08/F09）
- LTM 管理：
  - `GET /api/memory/ltm`：查询长期记忆（支持关键词/向量搜索、按类别/Agent 过滤）。
  - `DELETE /api/memory/ltm/{id}`：删除特定记忆条目，满足用户“遗忘”指令。
  - `PUT /api/memory/settings`：更新记忆策略（如 STM 保留轮数、自动提取开关等）。
- 会话与对话流：
  - `GET /api/conversations` / `GET /api/conversations/{id}`：按逻辑会话维度浏览历史对话（对应 F08.3）。
  - `GET /api/conversations/{id}/messages`：分页获取对话消息及关联任务/引用元数据。

### 7.5 监控与审计接口（对应 F07）
- `GET /api/metrics/agents`：获取 Agent 状态矩阵与队列深度，用于监控台展示。
- `GET /api/metrics/resources`：获取 CPU/内存/磁盘等资源使用情况，配合资源阈值监控。
- `GET /api/tasks/{id}/timeline`：按时间线返回任务及其子任务的执行轨迹（树形/甘特图数据）。
- `GET /api/audit/logs`：按时间范围、任务、Agent 查询审计日志；
  - 支持导出为文件，满足 F07.3 的审计日志导出要求。

### 7.6 系统与配置接口（对应 F01）
- `GET /api/system/config` / `PUT /api/system/config`：读取与更新工作目录、资源阈值等全局配置。
- `POST /api/system/backup` / `POST /api/system/restore`：触发一次性备份/恢复流程，与 4.0 的配置中心与备份子模块配合。
- `GET /api/system/health`：返回关键依赖（数据库、向量库、模型服务等）的健康状态，用于启动自检与运行期健康检查。

## 8. 非功能性设计
本节承接《02-功能要求》第 5 章 NFR 与《03-技术分析》第 6～7 章，从概要层面对性能、可靠性、安全性、可维护性与兼容性进行设计约束。

### 8.1 性能设计
- **流式响应**：ReAct 步骤采用 Server-Sent Events (SSE) 或 WebSocket 推送，确保 UI 无卡顿（对应 02-功能要求 5.1 中“流式输出延迟 < 1s”）。
- **调度延迟**：任务调度中心与 Agent 运行时的实现需保证 Leader 在完成子任务分发后 500ms 内恢复为 `IDLE` 状态（不含模型推理时间），与 02-功能要求的 Leader 延迟指标一致。
- **检索与记忆延迟**：增加 agent_id 过滤后的向量检索延迟、STM/LTM 读写引入的额外耗时应控制在 200ms 以内，超出部分需在监控台中可观测并支持调优。
- **资源配额**：内存使用超过 90% 时，自动暂停新任务入队或将任务标记为 `QUEUED_WAITING_RESOURCE`，结合 03-技术分析 6.1 的资源配额策略执行排队或降级。

### 8.2 可靠性与容错
- **状态持久化与恢复**：任务、队列和记忆相关表的写入需采用事务，保证崩溃后状态一致性；启动自检流程应扫描非终态任务并按心跳/时间阈值恢复或标记为异常，对应 03-技术分析 6.2。
- **数据备份**：每日定时备份 Config 与 Metadata 至 /backup，保证在工作目录损坏或迁移时可恢复基础配置与索引。
- **队列持久化**：Domain Agent 的任务队列状态需持久化在 SQLite 中，满足 02-功能要求 5.2 中“队列持久化”的约束。
- **死锁预防**：在任务调度与依赖管理中检测循环依赖，避免任务 A/B 互相等待导致无限排队。
- **自检机制**：启动时检查 Ollama 服务、Node.js 版本、关键 npm 依赖包、向量库/数据库驱动等环境依赖，提供修复向导。

### 8.3 安全与隐私
- **数据加密**：API Key 等敏感信息使用 OS 密钥环或等价机制加密存储，防止本地泄露。
- **路径与权限沙箱**：硬编码校验文件操作路径前缀，限制在 workspace 目录内，并按照 02-功能要求 5.2 的“权限最小化”原则控制本地目录权限。
- **Agent/知识/记忆隔离**：在数据库查询与向量检索层双重注入 `agent_id` 过滤条件，保证不同 Agent 间的知识库与长期记忆不会越权访问。
- **审计追踪**：所有关键操作（包括权限裁决、任务终止、知识库删除等）留痕至 audit_logs，支持脱敏导出。

### 8.4 可维护性
- **结构化日志**：系统各模块输出 JSON 结构化日志，并包含 TraceID/TaskID/AgentID 等关键字段，便于跨模块排查问题（对应 02-功能要求 5.3）。
- **配置热更新**：Prompt 模板、Skill 配置、模型与权限策略的变更应在新任务创建时立即生效，无需重启应用，与 02-功能要求的配置热更约束保持一致。
- **知识与记忆迁移**：支持在 UI 层对 Agent 知识库文件进行迁移或复制，后台重建索引与记忆关联，以适应 Agent 角色变更。

### 8.5 兼容性与路径规范
- **跨平台路径**：统一通过 Node.js `path` 与 `fs` 模块处理 Windows/macOS/Linux 的路径差异，默认工作目录分别位于 `%USERPROFILE%\BiosBot_Workspace` 与 `$HOME/BiosBot_Workspace`。
- **特殊字符处理**：对包含中文、空格或特殊字符的用户名和路径进行正确编码与解码，避免文件读写失败。
- **权限边界**：默认仅授予当前登录用户对工作目录的读写权限，防止其他本地用户窥探数据，满足 02-功能要求 5.4 的安全边界约束。

## 9. 演进规划
- P0 (当前)：单体架构，本地优先，单用户，核心任务闭环。
- P1 (未来)：支持多用户协作 (云端同步)，Skill 热更新市场，更复杂的 DAG 编排可视化。
