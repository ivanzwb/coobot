# BiosBot 需求分析

## 1. 文档目的

本文档用于分析 BiosBot 的业务背景、用户需求、场景约束、优先级与需求来源，为后续方案设计提供依据。
本文档主要支撑 02-功能要求、03-技术分析、04-概要设计、05-详细设计 以及后续接口、数据库、界面和测试文档的编制。
本文档重点回答“为什么要做、谁来使用、核心场景是什么、需求优先级如何划分”。

## 2. 背景分析

当前通用大模型产品主要解决"单轮问答"问题，但在以下场景中存在明显不足：

| 不足场景 | 问题描述 |
|---------|---------|
| 复杂任务 | 需要拆解、执行和复盘，单轮输出不可追踪 |
| 业务资料 | 大量存在于图片、PDF、扫描件等非纯文本形式 |
| 私有知识 | 个人用户需要使用私有知识，但不希望把全部资料交给外部平台 |
| 多 Agent 组合 | 不同任务需要不同 Agent 和工具组合，单一模型无法稳定覆盖全部路径 |

**BiosBot 需求由此产生：**

- 需要一个**本地优先**、**可配置**、**可追踪**的多智能体工作台
- 需要让多模态输入、有学习能力，有知识库、Skills、工具和任务系统真正协同工作

## 3. 用户需求分析

### 3.1 私人助理 + 专业任务

| 类别 | 内容 |
|-----|------|
| **核心诉求** | 既能像私人助理一样长期协助处理待办、资料整理、提醒跟进和结果复用，也能支撑专业资料密集型任务的理解、提炼、比对和结构化交付，并在单用户长期使用过程中形成稳定的交付标准、过程复盘和经验沉淀 |
| **核心痛点** | 个人场景下缺少可持续复用的上下文与记忆，导致每次都要重新描述背景；专业资料场景下预处理、跨文档比对和人工归纳成本高，结果常缺少可核验依据与稳定表达；同时任务流程、权限边界和交付标准若缺少统一治理，也会导致失败后难以定位是模型、工具还是知识链路问题 |

## 4. 场景分析

| 场景 | 描述 |
|-----|------|
| **4.1 私人助理 + 专业任务处理** | 系统既作为个人长期助理承接待办、资料整理和提醒跟进，也作为专业任务工作台承接资料分析、统一交付和复盘沉淀 |
| **4.2 资料分析** | 用户输入文本问题并上传图片/PDF，系统先做解析，再做分析，再生成结构化结论和草稿 |
| **4.3 工程协作** | 用户要求 Agent 阅读项目文件、修改文本文件、执行脚本并返回结果摘要 |
| **4.4 失败复盘** | 用户查看任务时间线、工具调用和错误摘要，定位失败步骤并重新执行 |
| **4.5 多 Agent 安排** | 用户输入复合任务后，Leader Agent 先拆分出多个分配给不同 Domain Agent 的子任务 |
| **4.6 多 Agent 汇总交付** | 各 Domain Agent 在各自子任务完成后把结果回传给 Leader 父任务 |
| **4.7 等待态异常与人工介入** | 当复合任务中的定时任务、事件触发任务或排队任务在各自配置的等待阈值内未能进入执行链路时 |

### 4.1 私人助理 + 专业任务处理（详述）

- 在个人使用方式下，系统应能像私人助理一样持续承接**待办整理**、**资料归档**、**提醒跟进**、**结果复用**和**上下文延续**，减少用户重复描述背景的成本
- 在专业任务方式下，系统应能围绕图片、扫描件、年报、纪要等资料完成理解、提炼、比对和结构化输出，并保留清晰的**依据来源**与**过程摘要**
- 在单用户模式下，系统应能支撑任务拆解、过程追踪、权限边界控制和统一交付标准，使个人协助能力与专业任务交付能力落在同一任务闭环中
- 该场景要求系统既能处理高频轻量的个人协助，也能处理高要求、可审计的专业任务，并把高价值结果、知识和经验持续沉淀为长期资产

### 4.5 多 Agent 安排（详述）

- Leader Agent 在编排阶段需为每个子任务标注其是否属于父任务交付所必需的 **blocking** 子任务或可降级的 **non-blocking** 子任务，并在任务详情和事件流中保留该判定摘要
- 系统在全部 blocking 子任务完成安排确认后，先向用户反馈"任务已安排好"和聚合预计完成时间
- Leader Agent 随后继续接收新任务，不阻塞等待当前复合任务执行结束
- 若部分子任务因等待事件长期未命中、队列长时间未释放、触发订阅失效或冻结快照失效而无法继续推进，系统必须把父任务推进到明确可观测的**等待异常**或**人工介入**状态

### 4.6 多 Agent 汇总交付（详述）

- 若仅 non-blocking 子任务失败或被降级跳过，系统应在最终结果中**明确标记该差异**，而不是把该任务误判为主链路失败

### 4.7 等待态异常与人工介入（详述）

- 系统不能保持无反馈等待，必须记录：**等待异常原因**、**当前受阻子任务**、**最近一次重评估结果**、**命中的阈值依据**和**建议动作**
- 该阈值至少支持按 `queued`、`scheduled`、`event_triggered` 三类任务分别配置
- 若系统通过降级仍不能满足最小执行上下文，则需进入**人工介入状态**
- 允许用户在界面上查看原因，选择**继续等待**、**重新安排**或**手动关闭**

## 5. 需求拆解

本章列出的需求以 `N-xx` 编号，供功能要求、技术分析和测试用例文档引用。

### 5.1 业务主链路需求

- N-01 从会话输入到任务创建（优先级：P0，对应“从会话到任务”的入口链路）。由以下二级子需求组成：
	- N-01.1 会话输入与任务创建。可验收口径：用户可以在会话窗口发送消息并成功创建至少 1 个任务，且该任务能在当前会话窗口中被持续追踪，实时显示任务和其子任务的状态和关键信息，如对应的Domain Agent，有需要时，可以通过任务 ID、自然语言或快捷指令查询之前任务的状态、查看详情或触发常用操作；任务列表和详情页仅作为补充查询入口。

	- N-01.2 入口模式自动识别与判定摘要。可验收口径：系统需根据会话输入自动识别任务入口模式，并在创建后为用户展示结构化入口判定摘要；该摘要至少包含入口模式、触发该判定的主要依据、等待场景下未立即执行的原因，以及用户下一步可执行动作。

	- N-01.3 默认单一会话窗口主交互。可验收口径：用户侧只保留**一个默认会话窗口**作为主交互面，不提供多会话创建、切换、会话列表管理等主流程逻辑；文本输入、多模态输入、进度反馈、结果输出、权限授权和常用任务操作默认都在该窗口内完成。这里的常用任务操作至少包括查看详情、查看结果摘要、重试、取消、补充澄清、处理权限确认和触发等待态建议动作；任务详情页和结果页仅作为补充查看与复盘入口。
	- N-01.4 单一会话窗口展示与确认前置规则。可验收口径：系统需在默认会话窗口内提供统一的任务事实展示与确认前置规则。用户在会话窗口需可完整查看任务状态、事件流、结果终态和权限裁决结果；涉及权限确认、执行确认或人工介入处置时，界面须先展示最小事实摘要（至少包含任务标识、当前状态、触发原因、影响范围和待确认动作）后再允许操作。本子需求仅定义展示与确认规则，不重复定义交互主入口。

	- N-01.5 低置信度入口澄清与安全默认边界。可验收口径：
	  - 当系统对入口模式判定置信度不足时，应先创建待澄清/待确认任务，置于 `clarification_pending` 状态
	  - 禁止静默猜测后直接执行，必须记录判定失败原因、补救动作和最终入口模式
	  - 用户补充信息或采用安全默认路径后，必须在原任务上完成状态推进
	  - 不得绕过 N-13 定义的 `deny > ask > allow` 权限裁决
	  - 待澄清/安全默认未完成前，不得进入实际执行链路
	- N-01.6 会话内任务操作与快捷指令。可验收口径：
	  - 用户可在同一会话窗口中通过自然语言或快捷指令对指定任务执行常用操作，至少包括查看任务详情、查看结果摘要、重试任务、取消任务、补充澄清、处理权限确认和执行等待态建议动作
	  - 系统需在会话消息流中返回结构化操作回显：对于查询类操作返回详情摘要，对于控制类操作返回执行结果、状态变化、影响范围和下一步建议
	  - 该能力属于 N-01.3 定义的会话主交互操作能力，不要求用户跳转到独立任务详情页才能获取关键信息或完成主链路中的常用操作
	  - 涉及高风险写入、执行或状态变更的会话内操作，仍必须遵守权限裁决、审计记录和原任务状态推进规则
- N-02 从任务创建到步骤编排（优先级：P0，对应 Leader Agent 对任务的拆解、父子任务建立与规划）。由以下二级子需求组成：
	- N-02.1 复杂任务判定与判定摘要
	  对于经会话意图解析判定为复杂任务的请求，系统必须输出复杂任务判定结果；该结果至少包含拆分原因、主要目标和关键约束，并在任务详情视图或会话内详情回显中可见、在事件流或审计链路中可追溯。
	- N-02.2 父子任务编排与 Domain 分配
	  系统生成 Leader 父任务与 Domain 子任务编排结果；该结果至少包含 1 个父任务、对应子任务列表、每个子任务的目标 Domain Agent、执行顺序或并发关系，以及当前任务状态。对于已进入执行链路的父任务，系统需基于用户输入和当前可用 Domain Agent 信息生成首轮 TaskPlan，并在任务详情视图或会话内详情回显中展示上述信息；对于 `queued`、`scheduled`、`event_triggered` 等等待态父任务，允许先完成任务创建、等待登记和后续释放，在释放进入执行链路后再生成正式 TaskPlan。首轮子任务执行回传观察结果后，Leader Agent 还需执行统一的父任务目标满足性判定；该判定是后续重编排、首版最终交付、降级交付与补充更新的唯一上游口径，不再额外定义独立的“主要目标已达成”平行语义。若目标未满足且仍具备继续推进条件，系统必须基于原始用户输入、当前已获得的观察结果和剩余可用 Domain Agent 能力再次生成后续子任务编排，而不是把首轮规划视为一次性静态结果。
	- N-02.3 blocking / non-blocking 标注与改标治理
	  Leader Agent 在编排阶段必须为每个子任务标注其是否属于父任务交付所必需的 blocking 子任务或可降级的 non-blocking 子任务，并在任务详情视图或会话内详情回显中展示该标记及其判定依据；该标记一旦被用于生成安排通知、ETA 或降级交付判定，就必须作为可审计事实被冻结；后续若因人工介入或系统纠正而调整 blocking / non-blocking 归类，必须记录明确变更事件、变更原因和影响范围，并同步使原安排结论或 ETA 进入需重评估状态，禁止通过静默改标来回溯性修正既有交付结论。判定口径上，凡是缺失后会导致父任务首版结果不成立、不可信、不可用，或会推翻首版结论的子任务，必须标记为 blocking；凡是可以在首版结果交付后以补充更新方式追加、且不会推翻首版结论的增强性、补充性、可延后子任务，才可标记为 non-blocking。安全、权限、合规、关键校验、回滚保障和最终汇总所必需的子任务，不得标记为 non-blocking；当系统无法稳定判断某子任务是否影响主目标成立时，默认按 blocking 处理。Leader 可给出首轮 blocking / non-blocking 判定，但系统必须保留最终采纳权：若标注违反上述强约束或与父任务交付依赖冲突，系统不得直接采纳该标注，必须进入重编排、拒绝执行或人工介入路径。
	- N-02.4 安排完成与 ETA 生成：父任务在全部 blocking 子任务完成安排确认后，才能形成安排完成状态与 ETA 摘要；该摘要至少包含当前已安排的 blocking 子任务、预计完成时间、估算依据和当前阶段标记，并且必须可在任务详情或事件流中追溯，且不能把安排完成误标为最终交付完成。
- N-03 从步骤执行到输出物生成（优先级：P0，对应 Domain Agent 子任务执行、结果回传与父任务汇总）。由以下二级子需求组成：
	- N-03.1 子任务执行记录与结果回传
	  每个已执行的子任务在系统中都有执行记录，包括成功/失败、耗时、调用的 Agent/Skill/工具；子任务完成后，其结果必须能够在父任务详情、结果摘要或事件流中被查看和追溯。对于复合任务，Leader Agent 必须能够消费这些子任务观察结果，判断当前父任务目标是否已经满足；若尚未满足，则在原父任务上下文内继续触发下一轮子任务编排、分配和执行，而不是要求用户重新发起一条新任务。

	- N-03.2 blocking 完成后的最终输出生成
	  在全部 blocking 子任务完成后，父任务才具备进入最终输出生成阶段的必要条件；是否生成至少 1 个可被结果页展示的最终输出，仍必须以后续统一的父任务目标满足性判定结果为准。换言之，blocking 子任务全部完成并不自动等同于父任务已满足最终交付条件；若统一判定结果仍为未满足，则系统必须继续重编排、补充执行、降级处理或进入人工介入路径，而不是直接产出最终输出。
- N-04 从输出物生成到结果交付（优先级：P0，对应结果页展示与交付物生成）。由以下二级子需求组成：
	- N-04.1 最终输出结果页
	  对于已完成且存在最终输出的任务，结果页必须展示主输出物和结构化结论，并提供稳定的结果访问入口以便用户重复打开；重复访问时应看到与该任务当前最终状态一致的结果内容。
	- N-04.2 安排完成但未最终交付的阶段性结果页：对于仅完成安排确认但尚未形成最终输出的父任务，结果页可以提前访问并展示阶段性 ETA 与安排通知，但必须明确标记“尚未完成最终交付”，且不得把该通知视为最终交付结果。
	- N-04.3 无最终输出的终态摘要交付：对于因用户放弃澄清、人工介入后手动关闭、不可恢复失败或等价原因而合法终止且不存在最终输出物的任务，系统仍必须提供可重访的结果或终态摘要页，明确展示关闭 / 终止原因、最后有效状态、是否存在阶段性安排信息以及后续可执行动作，并将其与“已有最终输出的结果交付完成”严格区分；只有存在最终输出物时，才视为结果交付完成，但不存在最终输出物的合法终态任务仍必须具备可审计、可复访的终态交付摘要。
	- N-04.4 non-blocking 差异、降级交付与补充更新规则：系统必须统一定义 non-blocking 子任务对父任务交付的影响规则。若 non-blocking 子任务失败、超时或被降级跳过，但统一的父任务目标满足性判定结果已允许首版交付，则允许父任务进入降级交付或部分完成路径，而不是一律判定为主链路失败；该判定必须输出结构化摘要，至少包含已完成的 blocking 子任务、被忽略或失败的 non-blocking 子任务、对最终结果的影响范围以及本次仍可交付的理由，并在结果页或事件流中可追溯。对于这类 non-blocking 差异，系统需在最终输出或执行摘要中明确标记其影响范围，而不得推翻父任务主链路结果；若父任务完成最终交付后仍有 non-blocking 子任务晚到返回成功结果，系统必须将其写入补充事件或执行摘要更新，并明确标注为交付后的补充信息，不得静默覆盖既有最终输出、不得隐式重开父任务主链路，也不得在无明确标识的情况下把补充结果伪装成初始最终交付内容。对于该类补充更新，系统必须定义统一通知策略：要么明确仅在结果页 / 任务详情中被动展示而不再次发送“最终完成”通知，要么发送独立的“补充更新”通知并与首次最终完成通知严格区分，禁止以第二次通知伪装成首次最终交付。
- N-05 从结果交付到知识沉淀（优先级：P0，对应任务复盘与长期资产沉淀）。由以下二级子需求组成：
	- N-05.1 结果加入知识库：用户可以从结果页将输出显式加入知识库，并在知识库页查看对应文档。

### 5.2 关键支撑需求

本节对 N-06、N-07、N-08、N-09、N-11、N-12、N-13、N-14、N-15、N-16、N-17 等关键支撑需求做二级拆解，以提升关键支撑能力的追踪精度。

- N-06 模型配置（优先级：P0，支撑多 Agent 与多模态能力正常使用）。可验收口径：

	### N-06.1 模型配置与连通性校验
	用户可通过设置页完成多个模型配置并通过连通性测试；并且用户可在界面上明确看到当前生效的模型配置与可用状态。配置保存后，可以在 Agent 配置页面选择使用，Agent 配置页面可以配置模型调优的参数，覆盖选择模型里的全局配置，后续新发起的模型调用能够按最新配置生效。

	### N-06.2 任务级执行配置快照
	除"后续新发起调用"外，系统还需定义任务级执行配置快照规则并可追溯：
	- 父任务进入执行链路时冻结本次执行配置快照（至少包含模型标识、关键参数与生效版本）
	- `queued`、`scheduled`、`event_triggered` 等等待态任务释放执行以及自动重试默认继承该快照；若策略允许改用最新配置，必须记录切换原因、配置差异与生效时点
	- 人工重试或重编排可按明确策略选择“沿用快照”或“使用最新配置”，并在事件流或审计链路中记录最终采用口径

- N-07 Agent 管理（优先级：P0，支撑 Leader / Domain Agent 的创建、配置、队列与调度）。可验收口径：

	### N-07.1 Agent 生命周期与基础配置管理
	用户可在界面上创建、编辑、启停 Domain Agent；保存后的 Agent 配置可用于后续新发起任务并通过连通性校验；系统首次启动时自动初始化默认 Leader Agent，配置为系统内置角色，具备完整的任务路由、拆解、编排与汇总能力，**Leader Agent 始终存在且不可删除**。

	### N-07.2 Leader / Domain 角色边界与协作约束
	- Leader Agent 负责**复合任务的路由、拆解、编排与汇总**
	- Domain Agent 负责**各自领域内的执行**
	- Domain Agent 之间**不得绕过 Leader** 直接相互调用
	- Leader 在父任务进入等待、安排确认或执行后**仍能继续接收并编排新任务**，不因单个复合任务长期占用而停止接单

	### N-07.3 Domain 显式队列与等待态协调
	- 每个 Domain Agent 都具备独立的逻辑任务队列，用于承载 `queued` 子任务；用户可在任务详情或事件流中查看当前队列归属与排队状态
	- `scheduled` 与 `event_triggered` 子任务必须纳入统一的等待态管理规则；用户可查看等待条件、最近一次评估结果和当前是否满足释放条件
	- 相关队列顺位、等待状态与安排确认结果可在任务详情或事件流中追溯

	### N-07.4 Agent 权限边界与白名单控制
	- 所有 Agent 的工具与外部环境权限**只能通过配置和 Skill/工具白名单授予**
	- **不允许运行时自行提升权限**
	- **未授权能力不得被 Agent 直接调用**

	### N-07.5 ReAct 执行模式与摘要边界
	- Leader / Domain Agent 在进入执行链路后，必须遵循 **ReAct（"推理 + 行动"）模式**
	- 以简短、可控的推理步骤与针对性的工具调用交替推进
	- 对 Leader Agent 而言，ReAct 回合不仅用于选择下一步动作，还必须承担**父任务目标满足性判定与增量重编排**：Leader 需要基于用户输入、当前可用 Domain Agent 信息和最新观察结果执行统一的目标满足性判定；若目标未满足，则继续生成下一轮 Domain 子任务并推进执行，直至目标满足、判定无法继续自动推进或进入人工介入/终态路径
	- 系统需保留可审计的**推理摘要**、**行动记录**与**观察结果**
	- 这些摘要属于面向产品、审计与复盘的**压缩事实摘要**，**不等同于也不得替代**模型内部完整原始思维链

	### N-07.6 Agent 超时控制
	- 系统需支持配置 Agent 执行单个步骤的**超时时间**
	- 超时后系统需记录超时事件，并支持自动重试或进入人工介入状态
	- 超时配置需支持按 Agent 类型、任务类型分别设置

	### N-07.7 并发执行与任务优先级
	- 系统需支持 Domain Agent 根据 Leader Agent 的编排，**并发执行**或者**顺序执行**子任务，提升执行效率
	- 顺序执行时，由 Leader Agent 传递 Domain Agent 之前的依赖数据，确保任务按顺序执行
	- 系统需支持 Leader 在编排结果中给出任务优先级建议，并由 Domain Agent 在自身显式队列中消费该优先级，确保高优先级任务可优先执行

	### N-07.8 外部服务不可用时的降级策略
	- 当外部服务（如大模型 API）不可用时，系统需有明确降级策略
	- 降级策略包括：返回缓存结果、切换备用模型、提示用户稍后重试
	- 降级事件需记录到审计日志
	- 降级策略与 N-06 任务级执行配置快照的关系需遵循统一优先级：默认优先沿用任务快照；仅在服务不可用且满足预设降级条件时，才允许覆盖快照切换备用模型。发生覆盖时必须记录触发原因、原配置与新配置差异、生效时点、影响范围和回滚策略，并在事件流与审计链路中可追溯。

- N-08 多模态输入（优先级：P0，支撑文本 + 图片 / PDF 等资料入口）。可验收口径：

	### N-08.1 附件上传与解析
	用户可在会话或任务创建时上传图片和 PDF，**单个附件大小上限为 50MB**，系统能完成解析并在任务详情或附件详情中展示解析状态和摘要。

- N-09 记忆与知识库（优先级：P0，支撑会话记忆、知识检索与长期记忆治理）。可验收口径：

	### N-09.1 知识库写入与检索命中
	任务结果或资料可显式写入知识库，并能在后续任务中被检索命中；命中记录可在执行报告或结果页中查看。

	### N-09.2 会话短期记忆与 Agent 持久记忆治理
	- 系统至少区分**三类上下文来源**：会话短期记忆、Agent 持久记忆和知识库检索结果
	- 在任务执行前**明确是否加载**相关记忆，避免记忆污染和人格漂移
	- 各类上下文的**加载顺序、覆盖关系和适用范围**可被追踪与解释

	### N-09.3 会话日摘要整理与长期记忆沉淀
	- 系统支持自动学习功能，按日汇总当天会话记录
	- 自动**过滤无用的对话寒暄、重复确认**等干扰内容
	- 将有价值的信息整理为**结构化记忆**写入可追溯的持久记忆
	- 记忆内容**可被人工清理或编辑**
	- 该类沉淀属于后续检索、加载与上下文装配使用的**记忆资产**，不直接等同于任务模板、经验条目或反思结论；如需升级为可复用经验资产，应进入 N-10 的治理链路

	### N-09.4 等待态冻结快照与释放时上下文策略
	- 对于 `queued`、`scheduled`、`event_triggered` 等等待态任务，系统需明确**入口上下文快照的冻结策略**
	- 定义释放执行时默认采用**冻结快照、最新记忆或受控合并**的规则
	- 该规则必须在任务详情或事件流中明确展示本次采用的上下文策略、对应依据和生效时点

	### N-09.5 快照失效后的降级与人工介入
	当冻结快照失效或合并后仍无法满足最小执行上下文时，系统必须**触发降级或人工介入**，而不是静默继续执行；相关失效原因、处理动作和最终状态可在任务详情或事件流中追溯。

- N-10 自我学习与经验积累（优先级：P1，支撑任务模板、经验复用和长期优化）。可验收口径：用户可基于历史任务和 N-09 形成的记忆沉淀创建任务模板或经验条目，并使用这些显式经验资产快速发起新任务；模板和经验条目需可被列出、管理、编辑、禁用和复用。N-10 关注的是可被长期治理和显式复用的**经验资产**，不直接承载原始会话记忆、知识库内容或单次任务反思记录。

- N-11 Skill 管理与按需使用（优先级：P0，支撑工具能力的可管理与按需使用）。可验收口径：
	### N-11.1 Skill 管理
	- Skill 以统一目录形式被系统识别并出现在管理界面，用户能够查看 Skill 的名称、功能描述、权限需求、当前启用状态和当前生效版本。
	- Skill 需要先配置到对应 Agent，后续才能在任务中被该 Agent 使用；同一 Skill 可被多个 Agent 配置，但每个 Agent 的配置关系、激活记录和调用记录需独立可追溯。
	- 系统需支持 Skill 的**懒加载**：执行前只加载 Skill 元信息（如名称、功能描述、权限需求和版本信息），实际调用时才加载具体实现和依赖资源。
	- **未调用 `use_skill` 前工具不可用**；调用后工具仅在当前任务上下文内变为可用，且需要保留激活记录、调用记录和对应任务关联。
	- 未安装、未启用、未绑定到当前 Agent 或未通过权限校验的 Skill，不得进入可调用状态。
	### N-11.2 Skill 安装与卸载
	- 用户可通过管理界面安装新的 Skill 包；安装成功后，该 Skill 可被系统识别并进入可配置列表。
	- 安装 Skill 需进行安全扫描和权限校验，禁止安装包含恶意代码、签名校验失败或权限需求超出策略边界的 Skill；对于不合规的 Skill 包，系统应拒绝安装并返回明确原因。
	- 用户可通过管理界面卸载已安装的 Skill 包；卸载前系统需提示卸载确认信息，至少包含当前版本、被哪些 Agent 使用以及卸载影响范围。
	- 对于已被执行链路引用的 Skill，系统必须拒绝卸载并返回明确原因，避免正在执行的任务上下文被破坏。
	- 卸载成功后，该 Skill 包及其相关资源应从系统中完全移除，并且不再出现在 Agent 的可配置列表中，也不得继续被新任务激活。
	- 系统必须记录 Skill 的安装与卸载操作，至少包含操作主体（当前用户）、操作时间、目标 Skill、处理结果和影响 Agent 范围，并在审计链路中可追溯。
	### N-11.3 Skill 版本治理与变更记录
	- 系统需对每个已安装 Skill 维护明确的版本信息、当前生效版本和版本变更记录；P0 阶段要求版本可识别、可切换、可追溯，不要求支持同一 Skill 的复杂灰度发布或多版本并行运行。
	- 用户可通过管理界面执行 Skill 版本更新；版本更新应作为独立治理动作处理，而不是与首次安装混为同一语义。
	- 若目标 Skill 当前版本已被执行链路引用，系统必须拒绝更新或切换版本，并返回明确原因；未被执行链路引用时，更新前需完成兼容性校验、权限差异提示和影响范围说明。
	- 系统必须记录 Skill 的版本更新和版本切换信息，至少包含操作主体（当前用户）、操作时间、目标 Skill、版本变化、处理结果和影响 Agent 范围，并在审计链路中可追溯。

- N-12 Prompt 模板与裁剪（优先级：P0，支撑 Agent 角色边界、Prompt 结构一致和上下文控制）。可验收口径：

	### N-12.1 Prompt 模板
	- 系统以模板化方式构建 Prompt，支持**版本记录**；模板仅定义结构与插槽规则，不直接固化某个 Agent 的具体角色文案。
	- 模板结构至少包含：
	   - **系统角色**：定义 Agent 的角色定位、行为规范和能力边界
	   - **任务描述**：明确当前任务的目标、输入和输出要求
	   - **短期记忆**：提供与当前任务相关的会话上下文和历史信息
	   - **持久记忆**：提供与当前任务相关的长期记忆和知识库信息
	   - **工具使用说明**：提供可用工具的功能描述、调用方式和权限要求
	- 每个 Agent 必须支持独立维护 Prompt 配置（至少包含角色定位、行为规范、能力边界等内容），并在渲染时按模板插槽注入；不同 Agent 的 Prompt 内容不可被全局模板隐式覆盖。
	- 模板内容的修改需保留变更记录，并在任务详情或事件流中可追溯该任务实际使用的模板结构版本、Agent 配置版本和对应变更说明。
	- 当 Agent 将绑定模板切换到新版本时，系统必须自动执行迁移流程（结构兼容校验、插槽映射补全、冲突标记与回退策略）；迁移失败时不得静默生效，必须返回明确错误并保留原生效版本。

	### N-12.2 Prompt 裁剪与降级
	- 当 Prompt 超出窗口限制时，系统必须提供可记录的裁剪摘要，明确本次裁剪影响了哪些上下文类别、裁剪原因和剩余保留内容；裁剪后若仍无法满足最小执行上下文，应触发**降级策略**或返回**明确错误码**，而不能静默返回不可信结果。

- N-13 工具与外部环境交互采用沙箱设计（优先级：P0，支撑文件系统、命令执行等基础能力）。可验收口径：

	### N-13.1 工作区定义
	系统定义明确的**工作区（Workspace）概念**，作为 Agent 执行任务的默认可见文件空间，默认为安装目录下的工作区文件夹。

	### N-13.2 权限清单管理模式
	- 工具/Skill 需声明所需权限（**网络访问**、**文件系统读写**、**进程创建**、**动态库加载**等）
	- 根据权限清单与权限策略放行，**未声明权限默认禁止**
	- **禁止提权操作**
	- 工具权限只能通过**配置和 Skill 绑定授予**，不允许 Agent 直接调用未授权命令或访问工作区之外的资源

	### N-13.3 权限策略分类
	系统按操作风险分为三类，默认审批策略如下：

	| 操作类型 | 示例工具 | 默认策略 |
	|---------|---------|---------|
	| 只读操作 | `fs.read_text`、`fs.list_dir`、`fs.stat_path` | allow |
	| 受控写入操作 | `fs.write_text`、`fs.apply_patch` | ask |
	| 命令执行/外部副作用操作 | `shell.exec`、外部系统调用 | deny |

	- 权限裁决采用 **`deny > ask > allow` 优先级原则**：命中更严格策略时必须覆盖较宽松配置
	- 命中 `ask` 的操作**必须要求用户确认后方可继续**
	- 审批结果、审批主体（当前用户）和对应任务事件需在审计链路中可追溯。

	### N-13.4 低置信度入口的特殊处理
	对于来源于 N-01 低置信度安全默认路径的任务，**后续工具调用仍必须完全遵守本条审批策略**，不能因入口模式已被默认确定而跳过权限确认或拒绝裁决。

	### N-13.5 安全防护
	- 系统对所有工具调用参数**先进行安全扫描**，再执行脚本注入过滤
	- 过滤内容包括：
		- **命令注入**：`;`、`|`、`&&`、`$()` 等
		- **路径遍历**：`../`、绝对路径逃逸
		- **环境变量注入**、**特殊字符转义**等
	- 写入操作安全策略：
		- 在权限校验通过的情况下，**允许工具生成可执行文件**（.exe、.sh、.bat、.ps1、.py 等）
		- 但在**执行**这些可执行文件前，需要**再次遵循权限策略和安全检查**
		- 禁止写入包含恶意代码的内容

	### N-13.6 审计与日志
	所有工具调用**必须在工具调用日志中完整记录**，包括：
	- 工具名称
	- 调用参数完整内容
	- 调用结果或返回值
	- 执行开始时间、结束时间
	- 调用成功或失败状态

- N-14 事件流与追踪（优先级：P0，支撑任务时间线、安排确认、ETA 反馈、失败定位与统计分析）。可验收口径：

	### N-14.1 事件流覆盖范围
	父任务和子任务生命周期会产生完整事件流，至少覆盖以下关键阶段：

	| 阶段类型 | 事件示例 |
	|---------|---------|
	| 入口阶段 | `TaskCreated`、入口判定、入口等待 |
	| 编排阶段 | 子任务拆分、安排确认、ETA 估算 |
	| 执行阶段 | 成功释放、步骤执行、结果回传 |
	| 终态阶段 | 完成、失败、取消、超时、人工关闭、人工介入 |

	### N-14.2 任务恢复机制
	- 系统重启后**自动检查并恢复**处于执行中或等待态的中断任务
	- **检查点定义**：每个任务执行步骤完成后自动生成检查点，记录任务状态、已执行的工具调用、输入输出摘要和决策摘要；检查点需持久化存储，支持从最近检查点恢复
	- **验证标准**：恢复时需验证检查点完整性（无数据损坏）、幂等性（重执行无副作用）、时效性（快照未过期），仅当三项均满足时方可自动恢复
	- **恢复范围**：从检查点恢复后继续执行后续步骤，已执行的工具调用结果可复用，但需重新执行当前步骤的决策逻辑
	- 无法证明安全时必须进入人工确认、补偿或受控重试路径，并在事件流中记录恢复判定依据

	### N-14.3 延迟任务释放追溯
	- 对于 `queued`、`scheduled`、`event_triggered` 三类延迟任务，成功释放时必须能够查看完整释放事件序列，至少包括等待登记、触发条件满足或队列释放、进入执行链路三个阶段
	- 不能仅看到最终进入执行的结果

	### N-14.4 等待异常处理
	对超过配置阈值仍未释放、触发订阅失效、等待条件不再满足或冻结快照失效的任务：
	- 必须产生明确事件和状态变更
	- 用户可在任务详情中查看：**等待原因**、**命中的阈值依据**、**失败步骤**和**错误码**
	- 可基于这些事件生成基础统计报表

	### N-14.5 错误重试机制
	- 对于**可恢复的错误**，系统支持在限定次数内的**自动重试**
	- 在事件流中**清晰标记重试行为和结果**
	- 对于**不可自动恢复**的场景，系统必须进入**明确的人工介入状态**而不是静默停滞
	- 向用户提供：**继续等待**、**重新安排**或**手动关闭**等可执行动作
	- 动重试需同时满足 N-14.2 定义的恢复安全前提；对于无法证明幂等或存在外部副作用风险的步骤，不得直接自动重试，必须进入人工确认、补偿或受控重试路径。

	### N-14.6 时间线恢复与幂等性
	任务时间线必须支持：
	- 在页面刷新、连接中断后恢复或重新进入任务详情时，用户仍能继续看到完整时间线
	- 时间线中的事件不得出现重复展示、错误排序或状态回退
	- 恢复后的时间线展示结果应与系统实际任务状态和事件事实保持一致

- N-15 监控告警机制（优先级：P1，支撑系统健康感知、异常提前预警与运营决策）。可验收口径：

	### N-15.1 系统健康指标采集
	- 系统至少采集并持久化以下核心指标：任务成功率、平均耗时、模型调用成功率、平均响应时间、知识检索命中率、附件解析成功率、Agent 执行效率、队列积压深度
	- 指标数据按**小时/天维度聚合存储**，支持历史查询

	### N-15.2 异常检测与预警规则
	- **阈值告警**：如任务失败率超过 X%
	- **趋势告警**：如连续 N 次任务耗时上升
	- **等待异常告警**：如队列积压超过 N 个任务或等待超 X 分钟未释放
	- 告警触发时写入系统事件并在监控面板展示

	### N-15.3 分级告警与通知策略
	- 告警支持分级：**信息**、**警告**、**严重**
	- 不同级别对应不同通知策略
	- P0 阶段默认仅在监控面板展示告警记录，**不强制推送**
	- 用户可配置**告警沉默窗口**避免重复打扰

	### N-15.4 监控面板与运营视图
	系统提供基础监控面板，展示：
	- 核心指标趋势图
	- 当前队列状态
	- 近期失败任务分布和 Top 失败原因
	- 面板数据支持时间范围筛选和导出

	### N-15.5 告警与任务追踪联动
	- 系统告警事件需与任务事件流**打通**
	- 用户可在告警详情中直接跳转查看关联任务的时间线和详情
	- 支持从异常指标追溯到具体失败任务

- N-16 安全与凭证管理（优先级：P0，支撑敏感信息保护与安全审计）。可验收口径：

	### N-16.1 凭证安全管理
	- 系统需支持安全存储 API Keys、数据库密码等敏感凭证
	- 凭证需加密存储，不得明文暴露在配置或日志中
	- 支持凭证的创建、更新、删除和轮换

	### N-16.2 数据脱敏
	- 日志和审计中需对敏感信息（身份证号、手机号、银行卡等）进行**脱敏处理**
	- 脱敏规则需可配置
	- 支持导出报告时自动脱敏

	### N-16.3 网络通信安全
	- 系统支持 HTTPS/TLS 加密通信

- N-17 运维与部署（优先级：P1，支撑系统可运维性与部署便利性）。可验收口径：

	### N-17.1 健康检查接口
	- 系统提供可访问的健康检查入口，供用户或运维确认服务是否可用
	- 健康检查包括：服务状态、数据库连接、磁盘空间、队列积压等

	### N-17.2 数据备份与恢复
	- 系统支持**自动备份**机制
	- 支持从备份文件恢复数据
	- 备份文件需支持加密存储

	### N-17.3 资源监控
	- 系统采集并展示 CPU、内存、磁盘使用率
	- 资源使用超过阈值时触发告警

## 8. 风险分析

### 8.1 技术风险

| 风险项 | 描述 |
|-------|------|
| Prompt 膨胀 | 当 Prompt 拼装后超出模型窗口限制、裁剪频率过高或关键上下文被反复截断时，可能导致模型返回错误码、结果缺字段或前后轮输出依据不一致 |
| 多模态质量 | 当图片/PDF 解析失败率升高、OCR 结果缺页漏段或版面识别错误时，可能导致后续摘要、比对和结构化结论引用错误原文 |
| 权限安全 | 当高风险工具被授予过宽权限、Agent 可访问超出工作区范围的资源，或 `ask` / `deny` 操作被错误放行为 `allow` 时，可能导致未授权写入、命令执行或外部副作用发生 |
| 资源浪费 | 当大量 Skill 在任务开始前被预先加载、无关上下文被提前注入，或长时间等待态任务持续占用运行时资源时，可能导致上下文窗口被挤占、任务启动变慢或系统资源持续升高 |
| 状态不一致 | 当前后端对同一任务状态、错误码或终态语义映射不一致时，可能导致列表、详情、结果页和事件流展示互相冲突，进而引发错误操作或错误统计 |
| 并发冲突 | 当多个任务并发访问同一文件、共享工具实例、同一 Agent 资源池或同一外部服务配额时，可能导致写入覆盖、队列阻塞、执行顺序错乱或任务结果相互污染 |
| 服务降级 | 当外部模型、OCR、检索或其他依赖服务不可用，而系统未能在阈值内切换备用路径、返回缓存结果或给出明确失败反馈时，可能导致任务长时间挂起、直接失败或返回缺少依据的降级结果 |

### 8.2 入口与等待态风险

| 风险项 | 描述 |
|-------|------|
| 低置信度入口 | 当低置信度入口请求未统一进入 `clarification_pending` 或等价状态，或在澄清完成前已被部分系统视为可执行任务时，可能导致任务列表、事件流和接口契约对"是否已建单"理解不一致 |
| 等待态滞留 | 当 `queued`、`scheduled` 或 `event_triggered` 任务超过配置阈值仍未释放，且在一段时间内没有新的等待事件、没有最近一次重评估结果或没有异常状态推进时，可能导致任务长期停留在等待态而用户无法判断系统是否仍在有效推进 |
| 触发失效 | 当事件订阅已失效、定时触发器未注册成功、等待条件已变化但系统未重新评估，或触发回调长期未写入事件流时，可能导致任务进入不可见卡死状态 |

### 8.3 编排与交付风险

| 风险项 | 描述 |
|-------|------|
| 任务拆分 | 当相似输入在短时间内被判为不同入口模式或不同拆分结果，或判定摘要缺少拆分原因、主要目标、关键约束等必要字段时，用户将无法判断系统为何拆分任务，也难以复核编排是否正确 |
| blocking 标注 | 当 blocking / non-blocking 标注与真实交付依赖不一致，或目标满足性判定摘要缺少已完成 blocking 子任务、影响范围和可交付理由等必要信息时，可能导致父任务被错误交付或错误终止 |
| 标注篡改 | 当 blocking / non-blocking 标注在安排通知或 ETA 发出后仍可被静默修改，且没有生成改标事件、影响范围说明或重评估标记时，可能导致系统以回溯性改标掩盖错误编排或错误交付 |

### 8.4 权限与安全风险

| 风险项 | 描述 |
|-------|------|
| 权限绕过 | 当低置信度安全默认路径在状态推进后未继续套用 `deny > ask > allow` 规则，或会话内快捷操作绕开审批检查时，可能导致入口模式默认确定后错误绕过 `ask` / `deny` 权限裁决 |
| 边界失效 | 当低置信度安全默认路径未限制后续动作边界，且系统在澄清未完成前已触发规划、拆分、工具调用或状态推进时，可能导致任务越过应有的确认门槛直接进入执行链路 |

### 8.5 结果展示风险

| 风险项 | 描述 |
|-------|------|
| 补充更新 | 当 non-blocking 子任务在父任务交付后晚到成功，而系统未统一标记为补充更新、重复发送最终完成通知或直接覆盖原输出时，可能导致结果页、通知记录和事件流出现相互矛盾的事实展示 |
| 终态歧义 | 当无最终输出物但已合法终止的任务缺少统一终态摘要页、关闭原因展示或统计口径时，可能导致结果页、任务终态和报表对"是否已完成交付"给出冲突解释 |
| 阶段混淆 | 当阶段性 ETA / 安排通知与最终输出共用相同展示样式、状态文案或通知类型时，可能导致用户将"已安排完成"误解为"已交付完成" |

### 8.6 监控与质量风险

| 风险项 | 描述 |
|-------|------|
| 告警脱节 | 当监控告警未关联任务 ID、会话上下文或事件流定位信息时，可能导致告警与具体任务脱节，无法追溯异常根因 |
| 自检过度 | 当质量自检在低风险任务上重复触发、重试阈值过高或自检链路本身调用过多模型与工具时，可能导致任务执行耗时显著增加并挤占主链路资源 |
| 校验误报 | 当推理一致性校验在引用充足、结论正确的任务上频繁标记异常，或误把表达差异判为逻辑冲突时，可能导致用户对系统信任度下降并增加不必要的人工复核 |
| 历史偏见 | 当经验提取长期复用过时 Prompt、旧工具组合或不再适用的成功模式，而缺少有效期、禁用或人工修正机制时，可能导致新任务被错误引导 |

### 8.7 安全与运维风险

| 风险项 | 描述 |
|-------|------|
| 凭证泄露 | 当 API Keys、数据库密码等凭证以明文出现在配置文件、日志、导出内容或错误信息中，或轮换后旧凭证仍可继续使用时，可能导致外部服务被未授权访问 |
| 数据泄露 | 当日志、审计记录、结果导出或知识库内容未按规则脱敏，且包含身份证号、手机号、银行卡号等敏感信息时，可能导致隐私数据外泄 |
| 磁盘满 | 当附件缓存、日志、数据库或备份文件持续增长且未触发清理、压缩或容量告警时，可能导致任务写入失败、服务异常或数据丢失 |
| 配置错误 | 当模型、Agent、Skill 或权限配置修改后未经过连通性校验和最小回归验证就投入使用时，可能导致新任务调用失败、权限异常或结果与当前配置不一致 |

## 9. 需求分析结论

BiosBot 的核心不是"堆叠多少个 Agent 能力点"，而是"能否围绕一个复杂任务形成从入口判定、编排执行、结果交付到知识沉淀的**稳定闭环**"。

### 核心收敛原则

| 原则 | 说明 |
|-----|------|
| **主链路优先** | 任务主链路仍是系统设计的第一优先级，聊天、展示和辅助能力都必须服务于 `N-01`～`N-05` 定义的任务闭环 |
| **结果导向** | 输出物与可交付结果优先于普通对话回复，必须保证任务结果页、阶段性安排通知、最终完成通知和终态摘要页之间**语义一致**，不把"已安排"误当作"已交付" |
| **可观测性** | 过程可观测、可追踪、可审计优先于黑盒执行，确保 `N-14` 事件流能够支撑排障、复盘与验收 |
| **统一服从** | Agent、工具、Prompt、知识库和记忆能力必须统一服从任务主链路和追踪语义 |
| **角色清晰** | Leader/Domain Agent 的价值在于稳定完成**角色边界清晰**的协作、等待态协调、权限边界控制与 ReAct 执行闭环 |
| **受控加载** | 记忆、知识和上下文装配必须坚持受控加载原则，不能在释放执行时被静默重算或覆盖 |

### P0 与 P1 阶段定位

| 阶段 | 定位 |
|-----|------|
| **P0** | 以**任务为中心**，以**交付结果**为目标，以**事件流**和**摘要化观测**为治理手段，以 Agent、记忆、知识、工具和 Prompt 的受控协作为支撑，形成一个可执行、可解释、可验收、可复盘的**复杂任务闭环系统** |
| **P1** | 扩展监控告警（N-15）与经验复用（N-10）能力，形成从**执行到优化**的完整反馈回路 |
