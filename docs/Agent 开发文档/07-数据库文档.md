# BiosBot 数据库文档

## 1. 文档目的

本文档定义 BiosBot P0 阶段的持久化实体、表结构、关系、索引和状态约束。
本文档主要支撑 05-详细设计 中的模块落地，并为接口实现、事件存储、查询优化和数据维护提供持久化依据。
本文档重点回答“哪些信息需要落库、以什么表和字段保存、如何建立关系与索引”；这里的实体和字段均以持久化为目的，不展开模块内部运行时对象和处理流程。

## 2. 设计原则

- 任务是主聚合根。
- 会话历史、任务历史、知识库资料三者边界明确。
- 主表承载当前聚合状态，明细表承载过程明细，事件表承载回放事实源。

## 6. 核心表结构

### 6.1 conversations

本表主要承接默认单一会话窗口上下文与最近活跃接入端事实，不承担显式“新建会话来源”或多会话切换主流程语义；在单用户场景下默认只维护唯一上下文记录；来源：N-01.3、N-01.4、N-14。

- `id`：主键。
- `title`：会话标题。
- `status`：会话状态。
- `last_active_client_id`：可选，最近一次写入本会话主交互消息的接入端标识。
- `latest_task_id`：最近任务引用。
- `created_at` / `updated_at` / `last_message_at`。

### 6.2 messages

本表主要承接默认会话窗口中的输入消息、来源端标识与跨端显示范围事实，保证消息展示与同步策略可按发起端稳定追踪；来源：N-01.1、N-01.4、N-14。

- `id`：主键。
- `conversation_id`：外键。
- `task_id`：可选外键。
- `entry_point`：消息来源入口。
- `origin_client_id`：可选，消息发起端标识。
- `sync_policy`：消息创建时生效的同步策略。
- `visible_client_ids`：可选，JSON 数组；该消息允许显示的接入端范围。
- `role`：消息角色。
- `content`：消息正文。
- `created_at`：创建时间。

### 6.3 attachments

- `id`：主键。
- `conversation_id`：外键。
- `task_id`：可选外键。
- `message_id`：可选外键。
- `file_name` / `mime_type` / `file_size` / `page_count`。
- `upload_status` / `parse_status` / `parse_method`。
- `parse_summary` / `parse_confidence`。
- `persisted_to_knowledge_base`。

### 6.4 tasks

本表是任务主链路的核心落库实体，承接入口判定、等待态重评估、父子任务安排、双阶段通知、降级交付、补充更新、多端显示范围与权限/写队列控制等状态事实；来源：N-01.1、N-01.2、N-01.4、N-01.5、N-02.1、N-02.2、N-02.4、N-02.5、N-03.1、N-03.2、N-03.3、N-04.1、N-04.2、N-04.3、N-04.4、N-07.2、N-07.3、N-07.5、N-09.2、N-09.4、N-09.5、N-13、N-14。

- `id`：主键。
- `conversation_id`：外键。
- `status`：任务状态。
- `trigger_mode`：任务入口模式，如 `immediate`、`queued`、`scheduled`、`event_triggered`。
- `trigger_status`：触发状态，如 `ready`、`queued`、`waiting_schedule`、`waiting_event`、`triggered`。
- `scheduled_at`：可选，定时任务的计划执行时间。
- `trigger_rule`：可选，事件触发规则摘要，JSON 对象。
- `trigger_decision_summary`：任务入口判定摘要，用于解释系统为何将该任务识别为当前触发模式。
- `complexity_decision_summary`：可选，复杂任务判定摘要，用于解释系统为何将当前请求识别为需要拆分为父任务与多个 Domain 子任务。
- `clarification_required_fields`：可选，JSON 数组；当任务因低置信度入口判定进入正式待澄清状态时，记录仍需用户补充或系统确认的关键字段。
- `clarification_resolution_summary`：可选，低置信度入口在原 Task 上完成澄清、采用安全默认路径或完成确认后的处理摘要。
- `clarification_closed_reason`：可选，若用户放弃继续澄清或系统将待澄清任务关闭，则记录关闭原因。
- `next_trigger_check_at`：可选，下一次入口等待态重评估时间。
- `intake_input_summary`：任务创建时冻结的入口输入摘要，用于保证延迟任务在释放执行时仍可解释原始入口语义。
- `entry_point`：任务对应输入来源入口。
- `origin_client_id`：任务对应主交互发起端标识。
- `sync_policy`：任务相关输入、进度、结果和权限授权的默认跨端同步策略。
- `visible_client_ids`：可选，JSON 数组；任务相关消息默认允许显示的接入端范围。
- `display_scope`：任务当前展示范围语义，如 `origin_only`、`synced_clients`、`all_clients`。
- `queue_position`：可选，任务在本地等待队列中的顺位快照。
- `parent_task_id`：可选，父任务 ID；用于把 Domain 子任务关联回 Leader 父任务。
- `assigned_domain_agent_id`：可选，当前子任务所属 Domain Agent。
- `arrangement_status`：可选，父任务或子任务的安排状态；允许值和语义以 05-详细设计 的状态枚举词表基线为准。该字段只表达安排链路状态，不承载最终通知阶段。
- `arrangement_summary`：可选，父任务对全部子任务排队/触发安排的聚合摘要。
- `estimated_completion_at`：可选，预计完成时间。
- `estimated_duration_minutes`：可选，预计总耗时分钟数。
- `user_notification_stage`：可选，面向用户的通知阶段；允许值和语义以 05-详细设计 的状态枚举词表基线为准。
- `waiting_anomaly_code`：可选，等待异常码，如 `WAIT_TIMEOUT`、`TRIGGER_INVALIDATED`、`SNAPSHOT_INVALIDATED`。
- `waiting_anomaly_summary`：可选，等待异常摘要，用于向界面和接口解释当前受阻原因。
- `last_trigger_evaluation_summary`：可选，最近一次入口重评估摘要。
- `waiting_threshold_mode`：可选，当前命中的等待阈值所属模式，如 `queued`、`scheduled`、`event_triggered`。
- `waiting_threshold_basis_summary`：可选，当前命中的等待阈值依据摘要，用于说明阈值来源、命中条件和当前判断结果。
- `intervention_required_reason`：可选，进入人工介入状态的直接原因。
- `current_step_id`：当前步骤。
- `assigned_leader_agent_id`：Leader Agent。
- `route_decision_summary`：路由决策摘要。
- `current_reasoning_summary`：可选，当前或最近一轮 Agent 执行回合的简短推理摘要；仅用于展示和审计，不保存完整原始思维链。
- `last_observation_summary`：可选，最近一次被后续决策消费的工具/检索/环境观察摘要。
- `next_action_summary`：可选，基于最近观察结果形成的下一步动作意图摘要。
- `plan_revision`：默认 1；表示当前父任务已落库的最新编排版本号。Leader 基于新的观察结果完成一次有效增量重编排时递增，用于稳定表达同一用户请求下的多轮编排事实。
- `replan_count`：默认 0；记录当前父任务已发生的增量重编排次数。
- `latest_plan_summary`：可选，最近一次正式 TaskPlan 或增量重编排后的计划摘要。
- `goal_satisfaction_status`：可选，最近一次目标满足性判断结果，如 `unknown`、`unsatisfied`、`satisfied`、`manual_intervention_required`。
- `goal_satisfaction_summary`：可选，最近一次目标满足性判断摘要，用于说明当前为何继续重编排、结束执行或转入人工介入。
- `selected_agent_ids`：本次任务选中的 Agent 列表，JSON 数组。
- `selected_skill_ids`：本次任务激活的 Skill 列表，JSON 数组。
- `memory_scope`：记忆加载范围，如 `conversation`、`history`、`selected`。
- `selected_memory_entry_ids_snapshot`：可选，创建时冻结的显式记忆条目 ID 集合，JSON 数组。
- `invalid_memory_entry_ids`：可选，释放执行前判定为失效的冻结记忆条目 ID 集合，JSON 数组。
- `snapshot_resolution`：可选，冻结快照失效处理结果，如 `unchanged`、`degraded_continue`、`manual_intervention_required`、`terminated`。
- `memory_load_summary`：任务启动时的记忆加载摘要。
- `memory_write_summary`：任务结束后的记忆写回摘要。
- `aggregate_version`：任务聚合版本号，用于任务级乐观锁和过期写入校验。
- `pending_write_command_count`：当前待消费任务写命令数量。
- `last_accepted_write_command_id`：最近一次被任务聚合接受的写命令 ID。
- `last_event_sequence`：当前任务已落库的最后事件序号，用于时间线排序和增量恢复。
- `last_event_cursor`：当前任务最近一次事件游标快照，用于查询接口和订阅恢复，并用于生成接口层的 `nextEventCursor`。
- `permission_status`：当前任务权限状态，如 `not_required`、`pending`、`approved`、`denied`。
- `permission_summary`：任务内权限决策摘要。
- `permission_decision_trace`：任务级权限判定摘要，记录最近一次高风险动作的权限判定轨迹摘要。
- `error_code` / `error_message`。
- `retryable`。
- `retry_count`：整任务累计重试次数。
- `last_retry_source`：最近一次重试来源，如 `auto`、`manual`。
- `degrade_reason` / `degrade_action`：最近一次降级原因与处理动作。
- `model_provider`：最近一次成功或失败的模型提供方标识。
- `model_failure_code`：最近一次归一化后的模型失败码。
- `model_failure_summary`：最近一次模型失败摘要。
- `created_at` / `started_at` / `completed_at` / `total_duration_ms`。

### 6.5 task_steps

本表需承接步骤级 ReAct 执行摘要，用于表达某一步内的简短推理、实际动作与工具观察结果，而不是保存完整原始思维链；来源：N-03.1、N-07.5、N-12、N-13、N-14。

- `id`：主键。
- `task_id`：外键。
- `attachment_id`：可选外键。
- `agent_id`：执行 Agent。
- `type` / `name` / `status`。
- `parallel_group_id`：并行步骤组 ID。
- `execution_mode`：执行模式，如 `serial`、`parallel`、`waiting`。
- `waiting_reason`：进入等待状态的原因。
- `reasoning_summary`：可选，该步骤当前或最后一轮的简短推理摘要。
- `action_summary`：可选，该步骤执行的动作摘要，如调用的 Skill、工具或检索动作。
- `observation_summary`：可选，该步骤获得并被后续使用的观察结果摘要。
- `observation_source_type`：可选，观察来源类型，如 `tool_result`、`knowledge_hit`、`model_output`。
- `observation_ref`：可选，观察来源引用，如工具调用日志 ID、知识命中文档 ID 或事件 ID。
- `input_summary` / `output_summary` / `error_message`。
- `retry_count` / `started_at` / `completed_at` / `duration_ms`。

### 6.6 task_outputs

- `id`：主键。
- `task_id`：外键。
- `type` / `title` / `content` / `format`。
- `is_primary`。
- `output_stage`：输出阶段；允许值和语义以 05-详细设计 的状态枚举词表基线为准。`arranged_only` 仅表示阶段性安排通知，不表示最终交付完成。
- `agent_execution_summary`：Agent 参与摘要，JSON 对象。
- `knowledge_hit_summary`：知识命中摘要，JSON 对象。
- `memory_changes`：本次任务新增或更新的记忆条目摘要，JSON 数组。
- `model_call_summary`：模型调用汇总，JSON 对象。
- `reasoning_summary`：可选，结果页可展示的关键推理摘要；用于解释输出如何形成，但不承载完整原始思维链。
- `observation_summary`：可选，结果页可展示的关键观察摘要；用于解释哪些工具观察、检索结果或环境反馈支撑了当前输出。
- `react_loop_summary`：可选，JSON 对象；用于聚合记录关键 ReAct 回合中“推理 -> 动作 -> 观察 -> 决策变化”的压缩摘要。
- `normalized_failure_codes`：本次任务出现的归一化失败码列表，JSON 数组。
- `arrangement_notice_summary`：可选，阶段性 ETA / 安排通知摘要，JSON 对象。
- `degraded_delivery_summary`：可选，结构化降级交付判定摘要，JSON 对象；至少包含 `completed_blocking_task_ids`、`failed_or_skipped_non_blocking_task_ids`、`impact_summary`、`delivery_reason`。
- `supplemental_updates`：可选，交付后补充更新记录，JSON 数组；用于保存晚到 non-blocking 成功结果的到达时间、来源子任务、补充说明和影响范围。该字段只追加补充事实，不覆盖原最终输出内容。
- `supplemental_notification_type`：可选，补充更新通知类型，文本；用于标记本次补充更新对应的通知语义，如结果补充、影响说明或仅时间线更新。该字段只用于描述补充更新通知，不得复用为新的最终完成通知。
- `terminal_summary`：可选，终态摘要，JSON 对象；当任务合法结束但不存在可展示主输出物时必须写入，至少包含 `terminal_reason`、`completion_boundary`、`no_output_reason`、`recommended_next_actions`。
- `available_actions`：结果页可执行动作，如加入知识库、加入记忆，JSON 数组。
- `created_at`。

### 6.7 knowledge_base_documents

- `id`：主键。
- `agent_id`：所属 Agent。
- `source_attachment_id`：来源附件。
- `title` / `content` / `source_type`。
- `created_at` / `updated_at`。

### 6.8 memory_entries

- `id`：主键。
- `agent_id`：所属 Agent。
- `conversation_id`：可选外键，用于标记会话级来源。
- `source_task_id`：可选外键，用于标记来源任务。
- `source_job_id`：可选外键，用于标记来源整理作业。
- `source_type`：来源类型，如 `manual`、`task_output`、`attachment_digest`、`daily_consolidation`。
- `scope`：记忆范围，如 `conversation`、`history`、`manual`。
- `summary_date`：可选，若为定时会话整理生成的记忆，则记录摘要对应日期。
- `source_conversation_ids`：可选，JSON 数组，记录本条持久记忆关联的会话 ID 集合。
- `dedup_key`：可选，用于同一 Agent、同一日期范围的定时整理幂等去重。
- `summary_version`：可选，定时整理摘要版本号，用于补跑后的来源追踪。
- `title` / `content` / `summary`。
- `status`：状态，如 `active`、`archived`、`deleted`。
- `created_at` / `updated_at` / `last_used_at`。

### 6.9 task_memory_links

- `id`：主键。
- `task_id`：外键。
- `memory_entry_id`：外键。
- `link_type`：关联类型，如 `loaded`、`written`、`updated`。
- `reason`：关联原因或摘要。
- `created_at`。

### 6.10 task_write_commands

- `id`：主键。
- `task_id`：外键。
- `command_type`：写命令类型，如 `step_completed`、`permission_resolved`、`retry_requested`、`degrade_applied`。
- `expected_aggregate_version`：提交该写命令时声明的任务聚合版本。
- `status`：写命令状态，如 `queued`、`processing`、`applied`、`rejected`。
- `conflict_code`：冲突错误码，如 `TASK_WRITE_CONFLICT`。
- `payload`：JSON 对象，保存写命令原始载荷；当写命令用于等待异常处理或人工介入决策时，还需支持 `waitingAnomalyCode`、`waitingAnomalySummary`、`interventionRequiredReason`、`suggestedActions`。
- `created_at` / `processed_at`。

### 6.11 permission_requests

- `id`：主键。
- `task_id`：外键。
- `step_id`：可选外键。
- `agent_id`：可选外键。
- `skill_id`：可选字段。
- `action_type`：动作类型，如 `write`、`edit`、`exec`、`external_call`。
- `target`：目标资源或目标系统标识。
- `permission_mode`：权限策略模式，如 `allow`、`ask`、`deny`。
- `status`：请求状态，如 `pending`、`approved`、`rejected`、`expired`。
- `reason`：触发权限请求的原因说明。
- `decision_source`：当前权限决策来源，如 `system_policy`、`user_confirmation`。
- `decision_trace_summary`：面向接口和界面的权限判定轨迹摘要。
- `decision_trace_payload`：JSON 对象，记录全局、Agent、Skill、工具、目标资源各层权限策略输入与最终权限决策。
- `confirmation_version`：权限确认版本号，用于并发确认时的乐观锁控制。
- `expected_sequence`：创建权限请求时前端或调用方所基于的事件序号快照。
- `resolved_sequence`：该权限请求形成最终权限决策时对应写入的事件序号。
- `requested_at` / `resolved_at`。

当 `expected_sequence` 与任务当前最新事件序号不一致，或 `confirmation_version` 校验失败时，不写入新的权限决策日志，并向接口层返回 `PERMISSION_CONFIRMATION_CONFLICT`。

### 6.12 task_trigger_subscriptions

- `id`：主键。
- `task_id`：外键。
- `trigger_mode`：入口等待类型，如 `queued`、`scheduled`、`event_triggered`。
- `status`：订阅状态，如 `active`、`triggered`、`invalidated`、`cancelled`、`expired`。
- `subscription_key`：触发订阅标识，用于关联本地队列槽位、定时器句柄或预定义事件条件。
- `condition_snapshot`：JSON 对象，记录当前等待条件快照。
- `threshold_config_snapshot`：可选，JSON 对象，记录该等待订阅命中的阈值配置快照。
- `queue_release_policy`：可选，队列释放策略；P0 固定为 `fifo`。
- `slot_budget_scope`：可选，槽位预算范围；P0 固定为 `global_runtime_slots`。

## 9. 数据约束

- `tasks` 只维护当前任务状态，不保存完整步骤明细。
- `tasks.aggregate_version` 是任务聚合写入的乐观锁基线；所有进入 `task_write_commands` 的写命令都必须声明预期版本或在等价队列顺序下被消费。
- `task_events` 是时间线和实时推送的事实源。
- `task_outputs` 是结果交付唯一来源，不从消息表反推最终结果。
- 知识库资料生命周期独立于会话和任务。
- `memory_entries` 为 Agent 级持久记忆主表，是否在某次任务中被加载或写入，由 `task_memory_links` 记录。
- `memory_entries.source_type = daily_consolidation` 的记录表示后台定时整理生成的会话日摘要，必须携带 `summary_date`、`source_conversation_ids` 和稳定 `dedup_key`，避免重复运行时生成语义等价的多份长期记忆。
- `memory_entries.source_job_id` 用于把一条持久记忆回溯到具体整理作业；查询日摘要来源时，应优先返回 `source_job_id`、`summary_version` 与 `source_conversation_ids` 组合，而不是依赖非结构化摘要文本反推来源。
- `task_write_commands` 用于记录任务级串行化写命令及其冲突结果，任何因过期聚合版本被拒绝的写入都不得直接覆盖 `tasks` 当前状态。
- `task_trigger_subscriptions` 用于记录排队、定时和事件触发任务的等待条件与释放情况；等待态任务不得只依赖内存定时器或临时订阅状态维持可恢复性。
- `tasks.intake_input_summary`、`tasks.memory_scope` 与 `tasks.selected_memory_entry_ids_snapshot` 共同构成延迟任务的入口上下文冻结快照；任务进入等待态后，释放执行时必须默认基于该快照继续执行，不得隐式改写为释放时刻的最新会话上下文。
- 当 `tasks.status = clarification_pending` 时，必须同时通过 `tasks.clarification_required_fields`、`tasks.trigger_decision_summary` 或等价字段表达待澄清原因；后续补充信息、默认确认或关闭动作都必须更新原 `tasks.id` 对应记录，不得通过重建第二条任务记录替代原单。
- `tasks.complexity_decision_summary` 用于沉淀复杂任务判定事实；若系统创建了 Leader 父任务和 Domain 子任务，则该字段不得为空或仅保留不可解释的模型原始文本。
- 若冻结快照中的显式记忆条目在释放前失效，必须把失效条目写入 `tasks.invalid_memory_entry_ids`，并以 `tasks.snapshot_resolution` 明确记录处理结果；系统不得静默切换到释放时刻的新记忆集合重算入口决策。
- 当 `tasks.snapshot_resolution = manual_intervention_required` 时，任务必须进入 `TaskInterventionRequired` 终态并停止继续释放，不得被映射为 `TaskManuallyClosed` 或普通失败。
- 当任务因等待时间过长、触发条件失效或快照失效而不能继续自动推进时，必须写入 `tasks.waiting_anomaly_code`、`tasks.waiting_anomaly_summary`、`tasks.last_trigger_evaluation_summary`、`tasks.waiting_threshold_mode`、`tasks.waiting_threshold_basis_summary`，并在需要人工介入时同时写入 `tasks.intervention_required_reason`；系统不得仅依赖临时内存日志表达等待异常。
- `task_trigger_subscriptions.release_token` 用于保证同一等待订阅只能成功释放一次；若重复命中相同条件，只能视为幂等重放或冲突，不得再次推进任务主状态。
- `queued` 任务在 P0 阶段按 FIFO 释放，因此 `queue_position` 必须与订阅建立顺序一致；取消队列任务后允许后续任务顺位前移，但不得改变未取消任务之间的相对顺序。其释放判断只允许基于 `slot_budget_scope = global_runtime_slots` 的全局预算，不得额外引入未声明的子队列。
- 对分配给 Domain Agent 的 `queued` 子任务，`queue_position` 必须解释为该任务在所属 Agent 显式队列中的顺位；父任务的 ETA 由各子任务所在 Agent 队列顺位、已知触发时间和历史平均执行时长共同估算。
- 父任务与子任务必须通过 `tasks.parent_task_id` 建立关系；父任务只有在全部必需子任务都达到 `queued_confirmed`、`trigger_registered` 或等价安排完成状态后，才能写入 `arrangement_status=arranged_completed` 并发出 `TaskArrangementCompleted`。
- `task_outputs.output_stage = arranged_only` 的记录只用于表达阶段性安排通知或 ETA 摘要，不得被标记为最终主输出；只有 `output_stage = final` 的输出才允许作为结果页最终交付物和知识沉淀来源。
- 若父任务以部分完成或降级交付方式结束，则必须把可审计的结构化判定写入 `task_outputs.degraded_delivery_summary`，不得只在 `tasks.degrade_reason` 或非结构化摘要中保留“主要目标已达成”的结论。
- 若父任务在 `task_outputs.output_stage = final` 后收到晚到 non-blocking 成功结果，则只能把补充事实追加到 `task_outputs.supplemental_updates` 和 / 或 `task_events.payload.supplementalUpdate`，不得覆盖原 `task_outputs.content`、不得回写成第二条首版最终输出，也不得改变首次最终交付对应的时间线事实。
- 若 `task_outputs.supplemental_updates` 非空且本次补充结果对用户产生再次通知，则必须同时写入 `task_outputs.supplemental_notification_type`；系统不得只通过非结构化消息文本推断补充更新通知语义。
- 若任务合法结束但没有主输出物，则必须写入 `task_outputs.terminal_summary`；结果页或终态摘要页不得依赖临时拼装文案替代持久化终态摘要。
- 若 blocking / non-blocking 标记在已参与安排、ETA 计算或交付判定后发生变化，必须通过 `task_events` 追加 `BlockingDecisionChanged` 事件，并在载荷中记录 `blockingDecisionChangeSummary`；系统不得静默覆盖子任务标记后仅保留最终状态。
- `task_trigger_subscriptions` 在任务取消、超时、失败终止或人工关闭后必须转为 `invalidated`、`cancelled` 或 `expired`，不得继续参与后续触发评估。
- `TaskTriggerInvalidated` 不得单独落账；当订阅失效时，必须与任务终态变更同提交，并在 `invalidation_reason` 中明确标记 `task_cancelled`、`task_timeout`、`task_failed` 或 `manual_closed` 等原因。
- 成功释放的延迟任务必须按 `TaskTriggerActivated -> TaskReadyForPlanning -> MemoryScopeConfirmed` 顺序落账；`TaskTriggerReevaluated` 只用于未释放的重评估，不得替代成功释放事件。
- `Task Intake Service` 是等待态协调的唯一写入驱动者；数据库层不得依赖内存态队列扫描器或临时定时器句柄作为唯一恢复依据，所有等待态恢复所需事实必须可由 `tasks` 与 `task_trigger_subscriptions` 重建。
- `memory_consolidation_jobs` 用于记录定时整理和手动补跑作业；同一 Agent、同一日期范围在已有运行中作业时，不得再接受第二个并发补跑请求，接口层统一返回 `MEMORY_CONSOLIDATION_ALREADY_RUNNING`。
- `permission_requests` 用于记录待权限确认和已完成权限决策的高风险动作，最终权限决策轨迹由 `permission_decision_logs` 保存，并通过 `confirmation_version` 控制并发确认覆盖。
- `model_call_logs` 用于沉淀模型调用事实和失败归一化结果，业务模块不直接依赖供应商原始错误码进行持久化。
- `task_events.sequence` 是任务内唯一稳定排序依据，`cursor` 是对外恢复协议字段；两者必须与 `tasks.last_event_sequence`、`tasks.last_event_cursor` 保持一致更新。
- 因过期 `expected_sequence` 或过期 `confirmation_version` 触发的权限确认冲突，不得写入新的权限决策日志或权限通过/拒绝事件，接口层统一返回 `PERMISSION_CONFIRMATION_CONFLICT`。
- 因过期 `aggregate_version` 或任务写队列冲突触发的任务写入失败，不得直接覆盖任务聚合当前状态，接口层统一返回 `TASK_WRITE_CONFLICT`。

`task_events` 事件载荷统一采用以下结构：

事件表及其 `payload` 统一承接任务时间线、入口等待与澄清、父子任务安排、双阶段交付、补充更新通知、多端显示范围和写入冲突等可审计事实；来源：N-01.2、N-01.4、N-01.5、N-02.2、N-02.4、N-02.5、N-03.1、N-03.2、N-03.3、N-04.2、N-04.3、N-04.4、N-07.2、N-07.3、N-07.5、N-09.2、N-09.4、N-09.5、N-13、N-14。

- `id`：事件 ID。
- `task_id`：所属任务。
- `sequence`：单任务内严格递增序号。
- `cursor`：事件游标。
- `event_type`：事件类型，如 `TaskCreated`、`TaskTriggerResolved`、`TaskTriggerWaiting`、`TaskReadyForPlanning`、`TaskPlanned`、`SubTasksCreated`、`DomainTaskQueued`、`DomainTaskScheduled`、`DomainTaskTriggered`、`TaskArrangementCompleted`、`TaskEtaEstimated`、`GoalSatisfactionEvaluated`、`TaskReplanned`、`TaskWriteQueued`、`TaskWriteConflictDetected`、`BlockingDecisionChanged`、`StepStarted`、`TaskCompleted`、`TaskPartialFailed`、`TaskFailed`、`TaskCancelled`、`TaskTimedOut`、`TaskManuallyClosed`、`TaskInterventionRequired`、`RouteResolved`、`PermissionCheckRequested`、`PermissionGranted`、`PermissionDenied`、`StepParallelQueued`、`StepWaiting`、`TaskTriggerReevaluated`、`TaskTriggerActivated`、`TaskTriggerInvalidated`、`TaskAutoRetried`、`TaskDegraded`、`MemoryScopeConfirmed`、`MemoryLoaded`、`MemoryRead`、`MemoryWritten`、`ModelCallStarted`、`ModelCallCompleted`、`ModelCallFailed`。
- `step_id`：可选，关联步骤。
- `agent_id`：可选，关联 Agent。
- `status`：事件状态。
- `summary`：供时间线直接展示的摘要文本。
- `payload`：JSON 对象，按事件类型承载 `decisionReason`、`selectedAgents`、`parallelGroupId`、`waitingReason`、`memoryScope`、`memoryEntryIds`、`triggerMode`、`triggerStatus`、`triggerDecisionSummary`、`complexityDecisionSummary`、`nextTriggerCheckAt`、`triggerRule`、`clarificationRequiredFields`、`clarificationResolutionSummary`、`clarificationClosedReason`、`subscriptionId`、`releaseToken`、`triggerFactId`、`triggerSourceType`、`triggerSourceRef`、`triggerSourceSignature`、`invalidationReason`、`lastTriggerEvaluationSummary`、`waitingAnomalyCode`、`waitingAnomalySummary`、`waitingThresholdMode`、`waitingThresholdBasisSummary`、`interventionRequiredReason`、`suggestedActions`、`permissionMode`、`permissionRequestId`、`decisionSource`、`decisionTraceSummary`、`aggregateVersion`、`expectedAggregateVersion`、`writeCommandId`、`conflictCode`、`modelProvider`、`normalizedErrorCode`、`retryAttempt`、`fallbackProvider`、`target`、`retrySource`、`degradeReason`、`degradeAction`、`degradedDeliverySummary`、`supplementalUpdate`、`supplementalNotificationType`、`blockingDecisionChangeSummary`、`terminalSummary`、`reasoningSummary`、`nextActionSummary`、`observationSummary`、`observationSourceType`、`observationRef`、`usedObservationIds`、`planRevision`、`replanCount`、`latestPlanSummary`、`goalSatisfactionStatus`、`goalSatisfactionSummary`、`replanReason` 等字段，用于表达入口等待、正式待澄清状态、复杂任务判定、一次性释放、触发事实身份、命中的阈值依据、等待异常、订阅失效、权限策略、权限请求、写队列冲突、降级交付判定、补充更新通知、blocking 改标事实、模型失败归一化，以及 ReAct 回合中的简短推理、观察摘要、目标满足性判断与增量重编排事实等上下文信息。
- `payload`：JSON 对象，在单窗口主交互和多端显示场景下还需承载 `entryPoint`、`originClientId`、`syncPolicy`、`visibleClientIds`、`displayScope` 等字段，用于表达该事件默认在哪些接入端显示，以及是否仅在发起端展示输入、进度、结果和权限授权信息。
- `payload`：JSON 对象，在冻结快照校验或延迟任务释放场景下还需承载 `intakeInputSummary`、`selectedMemoryIdsSnapshot`、`invalidMemoryEntryIds`、`snapshotResolution`、`minimumContextSatisfied` 等字段，用于表达入口快照、失效条目与降级处理结果。
- `payload`：JSON 对象，在父子任务和安排确认场景下还需承载 `parentTaskId`、`childTaskId`、`assignedDomainAgentId`、`arrangementStatus`、`arrangementSummary`、`queuePositionInAgent`、`estimatedCompletionAt`、`estimatedDurationMinutes`、`userNotificationStage`、`finalOutputReady`、`outputStage`、`arrangementNoticeSummary` 等字段，用于表达 Domain 子任务安排确认、父任务 ETA 聚合、双阶段用户通知以及阶段性通知与最终交付的区别。
- `payload`：JSON 对象，在 `GoalSatisfactionEvaluated`、`TaskReplanned` 或等价父任务增量编排事件中还需承载 `planRevision`、`replanCount`、`goalSatisfactionStatus`、`goalSatisfactionSummary`、`replanReason`、`usedObservationIds`、`latestPlanSummary`，用于保证“基于哪些观察得出当前目标未满足并触发下一轮编排”可被稳定追溯；若 `goalSatisfactionStatus=satisfied`，则不得再生成新的 `TaskReplanned` 事件。
- `created_at`：事件创建时间。

## 10. 数据库文档结论

数据库设计必须服务于三个目标：

- 支撑任务执行。
- 支撑过程追踪。
- 支撑结果复盘与长期沉淀。
