# BiosBot 数据库文档

## 1. 文档目的

本文档对 BiosBot P0 阶段的数据库设计进行统一说明，明确：

- SQLite 物理表结构与字段语义
- 核心实体之间的关系与约束
- 向量数据库 LanceDB 的配套设计
- 文件系统中的持久化落点

本文件是后续迁移脚本、DAO 层实现、接口 DTO 映射和测试用例设计的数据库真相源。若 05 更新了表结构或字段定义，07 必须同步更新，不允许继续沿用旧版 `conversations`、`messages`、`memory_entries` 等模型。

## 2. 设计基线

### 2.1 总体约束

- 单用户、单会话窗口：系统不以 `conversation_id` 或 `session_id` 作为主分片键，所有对话统一落在全局唯一的 `session_memory` 中。
- 任务触发方式：通过 `tasks.trigger_mode` 字段区分（`immediate`/`scheduled`/`event_triggered`）。
- 本地优先：SQLite 存储元数据与日志，LanceDB 存储向量，文件系统存储知识文件、模型与备份。
- 单体模块化：任务、记忆、知识库、Agent 配置与审计共享同一套数据库实例。
- Agent 隔离：知识库、长期记忆、工具权限等均按 `agent_id` 进行隔离。

### 2.2 核心设计原则

- 单一真相源：
  - 对话历史与 STM 唯一落点是 `session_memory`
  - LTM 元数据唯一落点是 `long_term_memory`
  - 任务聚合根是 `tasks`
  - 执行轨迹唯一落点是 `task_logs`
  - Agent 配置与能力画像唯一落点是 `agents`（Prompt 三件套、`capability_status` 等），内置工具集合由代码 `ToolHub` 统一提供、不落库；Skill 绑定真相源为 `agent_skills` + `skills`
  - 审计日志唯一落点是 `audit_logs`
  - 定时任务配置与执行日志唯一落点是 `scheduled_jobs` 与 `job_execution_logs`
  - 知识文件元数据唯一落点是 `knowledge_files`
  - 模型配置唯一落点是 `model_configs`
  - Skill 元数据唯一落点是 `skills` 与 `agent_skills`
  - Agent Prompt 配置落点是 `agents` 表的 `role_prompt`、`behavior_rules`、`capability_boundary` 字段
  - 工具权限唯一落点是 `agent_tool_permissions`
- 软删除优先：长期记忆通过 `is_active` 失效，避免破坏溯源与审计链路。
- 查询驱动建索引：索引围绕状态、时间、Agent、任务和归档扫描路径设计。

## 3. 物理模型总览

### 3.1 表清单

#### 3.1.1 控制与配置层

- `agents`：Agent 基础配置、Prompt 与能力调度状态（心跳）；内置工具不在本表存储
- `agent_tool_permissions`：Agent 对基础工具的权限策略
- `skills`：本地 Skill 包元数据
- `agent_skills`：Agent 与 Skill 的绑定关系
- `model_configs`：模型配置与状态

#### 3.1.2 任务执行层

- `tasks`：任务主表，承载生命周期状态
- `task_logs`：ReAct 执行步骤日志

#### 3.1.3 知识与记忆层

- `knowledge_files`：知识文件元数据
- `session_memory`：全局 Chat History + STM
- `long_term_memory`：长期记忆元数据

#### 3.1.4 审计层

- `audit_logs`：高风险操作与权限裁决日志

#### 3.1.5 调度层

- `scheduled_jobs`：定时任务配置
- `job_execution_logs`：定时任务执行历史

### 3.2 关系概览

```text
agents                  1 --- n   agent_skills
skills                  1 --- n   agent_skills
agents                  1 --- n   agent_tool_permissions
agents                  1 --- n   knowledge_files
agents                  1 --- n   long_term_memory

tasks                   1 --- n   task_logs
tasks                   0 --- n   session_memory

session_memory          0 --- n   long_term_memory
  通过 session_memory.ltm_ref_id -> long_term_memory.id 形成归档关联

audit_logs              n --- 0..1 tasks
audit_logs              n --- 0..1 agents

scheduled_jobs          1 --- n   job_execution_logs
scheduled_jobs          0 --- 1    tasks
  job_execution_logs.triggered_task_id -> tasks.id 形成任务关联
```

说明：

- `session_memory` 是全局唯一消息流，不再存在独立会话主表。
- `long_term_memory` 用于承接历史归档摘要与显式保存的事实、偏好、项目知识。
- `tasks` 是任务执行主线，`task_logs` 记录精简 ReAct 轨迹，不存放完整思维链。

## 4. SQLite 表结构设计

以下字段、取值范围与索引建议均以 05 的 DDL 为准。

### 4.0 `model_configs`

用途：存储模型配置信息，支持多 Agent 复用同一模型配置。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `name` | TEXT | 是 | 模型显示名称 |
| `provider` | TEXT | 是 | `openai`、`openrouter` |
| `model_name` | TEXT | 是 | 模型名称（如 gpt-4o、qwen-plus） |
| `base_url` | TEXT | 否 | API 端点地址 |
| `api_key` | TEXT | 否 | API 密钥 |
| `context_window` | INTEGER | 否 | 上下文窗口大小，默认 4096 |
| `temperature` | REAL | 否 | 模型默认温度，不填使用模型默认值 |
| `status` | TEXT | 否 | `ready`、`loading`、`error`、`offline` |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_model_configs_status (status)`

说明：
  - `temperature` 字段放在模型配置层，表示该模型的默认温度设置。

### 4.1 `agents`

用途：保存 Leader Agent 与 Domain Agent 的基础配置。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `name` | TEXT | 是 | Agent 名称 |
| `type` | TEXT | 是 | `LEADER` 或 `DOMAIN` |
| `model_config_id` | TEXT | 是 | 关联 `model_configs.id` |
| `temperature` | REAL | 否 | 不填代表使用模型默认温度 |
| `status` | TEXT | 否 | `IDLE`、`RUNNING`、`BUSY_WITH_QUEUE`、`OFFLINE` 等 |
| `role_prompt` | TEXT | 否 | 角色定位 Prompt |
| `behavior_rules` | TEXT | 否 | 行为规范 |
| `capability_boundary` | TEXT | 否 | 能力边界 |
| `last_capability_heartbeat` | DATETIME | 否 | 能力通道最近心跳时间 |
| `capability_status` | TEXT | 否 | `ONLINE`、`BUSY`、`OFFLINE`（调度可用性视角） |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_agents_status (status)`
- `idx_agents_capability_status (capability_status)`（建议）

说明：
  `temperature` 字段放在 Agent 层，表示该 Agent 维度的默认温度设置，用于覆盖模型配置的默认值。
  
  **温度优先级**：`agents.temperature` > `model_configs.temperature` > 硬编码默认值 (0.3)

  **Skill 名称列表**：不存储在 `agents` 表；通过 `agent_skills` 连接 `skills` 查询。历史库可通过迁移脚本将原 `agent_capabilities` 合并入本表并删除旧表。内置工具名列表由服务端 `ToolHub.listBuiltinTools()` 决定（不含 `skill:*`）；Skill 工具仅在运行时按需暴露给大模型，见设计文档 2.2.8 / 2.2.9。

### 4.2 `skills`

用途：保存本地 Skill 包的安装元数据。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID 或包名 |
| `name` | TEXT | 是 | Skill 名称 |
| `description` | TEXT | 是 | Skill 描述，来源于 `SKILL.md.description` |
| `version` | TEXT | 否 | Skill 版本 |
| `author` | TEXT | 否 | Skill 作者，来源于 `metadata.author` |
| `runtime_language` | TEXT | 否 | 声明的运行时语言：`javascript` / `python`；为空表示未声明运行时 |
| `detected_language` | TEXT | 否 | 服务端扫描安装包后识别到的语言结果 |
| `install_mode` | TEXT | 是 | 安装模式：`copy_only` / `managed` |
| `root_dir` | TEXT | 是 | Skill 在本地文件系统中的根目录 |
| `entrypoint` | TEXT | 否 | 入口脚本或命令；`copy_only` 模式可为空 |
| `config_schema_json` | TEXT | 否 | 参数 Schema |
| `tool_manifest_json` | TEXT | 否 | 解析后的工具清单与权限声明 |
| `compatibility` | TEXT | 否 | 环境兼容性说明 |
| `installed_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `enabled` | BOOLEAN | 否 | 默认 `1` |

约束：

- `install_mode` 只能取 `copy_only` 或 `managed`。
- 当 `runtime_language` 为空时，表示该 Skill 未声明运行时语言，系统只保留目录与元数据，不执行依赖安装与生命周期脚本。
- `entrypoint` 在 `managed` 模式下通常应有值，在 `copy_only` 模式下允许为空。

### 4.3 `agent_skills`

用途：保存 Agent 与 Skill 的绑定关系及其局部配置。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `agent_id` | TEXT | 是 | 主键一部分，关联 `agents.id` |
| `skill_id` | TEXT | 是 | 主键一部分，关联 `skills.id` |
| `config_json` | TEXT | 否 | Agent 维度的 Skill 配置 |

主键：

- `PRIMARY KEY (agent_id, skill_id)`

索引：

- `idx_agent_skills_agent (agent_id)`

### 4.4 `agent_tool_permissions`

用途：保存 Agent 对内置基础工具的显式策略。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `agent_id` | TEXT | 是 | 主键一部分，关联 `agents.id` |
| `tool_name` | TEXT | 是 | 主键一部分 |
| `policy` | TEXT | 是 | `DENY`、`ASK`、`ALLOW` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

主键：

- `PRIMARY KEY (agent_id, tool_name)`

索引：

- `idx_permissions_agent (agent_id)`

说明：

- **内置七类工具**（`read_file`、`write_file`、`edit_file`、`list_directory`、`exec_shell`、`http_request`、`system_info`）的默认策略由服务端代码统一维护；新建 Agent 或库初始化**不会**向本表批量插入这些默认值。
- 仅当策略被用户/API **修改为与代码默认值不同**，或授权流程选择「持久化」且结果异于默认时，才写入或更新对应行；若改回与代码默认一致，应删除该行（由实现层 `persistAgentToolPolicy` 处理）。
- **Skill 工具**（`skill:*`）等在绑定 Agent 时仍可写入策略行；运行时仍以「表中有记录则优先，否则走各工具类型的代码默认」为准（与 `SecuritySandbox` 一致）。

### 4.5 `tasks`

用途：任务聚合根，保存任务树、执行状态、输入输出摘要与心跳信息。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `parent_task_id` | TEXT | 否 | 父任务 ID，可为空 |
| `root_task_id` | TEXT | 是 | 根任务 ID，用于全局追踪 |
| `assigned_agent_id` | TEXT | 是 | 关联 `agents.id` |
| `status` | TEXT | 是 | `PENDING`、`WAITING_FOR_LEADER`、`CLARIFICATION_PENDING`、`PARSING`、`DISPATCHING`、`QUEUED`、`QUEUED_WAITING_RESOURCE`、`RUNNING`、`AGGREGATING`、`COMPLETED`、`EXCEPTION`、`TERMINATED` |
| `trigger_mode` | TEXT | 否 | 任务触发方式：`immediate`（手动）、`scheduled`（定时）、`event_triggered`（事件触发） |
| `input_payload` | TEXT | 否 | 输入 JSON |
| `output_summary` | TEXT | 否 | 输出摘要 |
| `error_msg` | TEXT | 否 | 错误消息 |
| `retry_count` | INTEGER | 否 | 默认 `0` |
| `heartbeat` | DATETIME | 否 | 最近心跳时间 |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `started_at` | DATETIME | 否 | 开始时间 |
| `finished_at` | DATETIME | 否 | 结束时间 |

索引：

- `idx_tasks_status_agent (status, assigned_agent_id)`
- `idx_tasks_heartbeat (heartbeat)`
- `idx_tasks_trigger_mode (trigger_mode)`

说明：

- 本表以 05 当前状态机为准。
- 等待、澄清、异常、聚合等阶段均通过 `status` 与日志组合表达。
- 任务触发方式通过 `trigger_mode` 字段区分，不再使用独立的 `SCHEDULED` 状态。

### 4.6 `task_logs`

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | 是 | 主键，自增 |
| `task_id` | TEXT | 是 | 关联 `tasks.id` |
| `step_index` | INTEGER | 是 | 任务内步骤序号 |
| `step_type` | TEXT | 是 | `THOUGHT`、`ACTION`、`OBSERVATION` |
| `content` | TEXT | 是 | 当前步骤内容 |
| `tool_name` | TEXT | 否 | 若为工具调用则记录工具名 |
| `tool_args_json` | TEXT | 否 | 工具参数 JSON |
| `timestamp` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_logs_task (task_id)`

- 该表保存可审计摘要，不作为完整思维链持久化载体。

### 4.7 `knowledge_files`

用途：保存知识文件与解析结果的元数据。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `agent_id` | TEXT | 是 | 关联 `agents.id`，按 Agent 强隔离 |
| `file_name` | TEXT | 是 | 原始文件名 |

| `status` | TEXT | 否 | `PROCESSING`、`READY`、`ERROR` |

| `meta_info_json` | TEXT | 否 | 页数、切块数、OCR 置信度等信息 |

### 4.8 `session_memory`

用途：P0 阶段唯一的对话历史与 STM 表。所有 user、assistant、system 消息统一写入本表。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | 是 | 主键，自增 |
| `role` | TEXT | 否 | `user`、`assistant`、`system` |
| `content` | TEXT | 是 | 消息正文 |
| `attachments_json` | TEXT | 否 | 附件元数据 JSON |
| `related_task_id` | TEXT | 否 | 关联任务 ID |
| `meta_json` | TEXT | 否 | ReAct 摘要、引用和 UI 扩展元数据 |
| `summary` | TEXT | 否 | 归档摘要或实时压缩结果 |
| `token_count` | INTEGER | 是 | token 数 |
| `importance` | REAL | 否 | 默认 `0.5` |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `is_archived` | BOOLEAN | 否 | 默认 `0`，`1` 表示已归档到 LTM |
| `ltm_ref_id` | TEXT | 否 | 关联的 `long_term_memory.id` |

索引：

- `idx_history_active (is_archived, created_at)`
- `idx_history_search (is_archived, created_at)`

说明：

- `session_memory` 同时承担完整历史对话流与活跃 STM 视图。
- 前台上下文组装主要读取 `is_archived = 0` 的活跃历史。
- 后台归档任务会将符合条件的历史摘要写入 `long_term_memory`，再回填 `ltm_ref_id` 并标记 `is_archived = 1`。

### 4.9 `long_term_memory`

用途：保存长期记忆元数据，向量内容由 LanceDB 承载。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `agent_id` | TEXT | 是 | 关联 `agents.id`，按 Agent 强隔离 |
| `category` | TEXT | 否 | `preference`、`fact`、`project`、`summary` |
| `key` | TEXT | 是 | 逻辑键 |
| `value` | TEXT | 是 | 记忆正文 |
| `embedding_id` | TEXT | 是 | LanceDB 中的向量记录 ID |
| `confidence` | REAL | 否 | 置信度 |
| `access_count` | INTEGER | 否 | 默认 `0` |
| `last_accessed` | DATETIME | 否 | 最近访问时间 |
| `is_active` | BOOLEAN | 否 | 默认 `1` |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_ltm_agent (agent_id, is_active)`
- `idx_ltm_key (key)`

说明：

- 历史归档生成的会话摘要在本表中使用 `category = 'summary'`。
- 向量检索以 `embedding_id` 与 LanceDB 中的记录对应。
- 软删除通过 `is_active = 0` 实现，并联动清理向量索引。

### 4.10 `audit_logs`

用途：记录权限裁决、工具调用、安全阻断等可审计事件。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | 是 | 主键，自增 |
| `event_type` | TEXT | 是 | 事件类型 |
| `actor_id` | TEXT | 是 | 行为主体，Agent ID 或 `USER` |
| `task_id` | TEXT | 否 | 关联任务 ID |
| `details_json` | TEXT | 否 | 已脱敏详情 JSON |
| `result` | TEXT | 否 | `ALLOW`、`DENY`、`TIMEOUT` 等 |
| `timestamp` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

说明：

- 本表是权限决策和敏感事件的持久化审计基线。

### 4.11 `scheduled_jobs`

用途：存储用户的定时任务配置。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | TEXT | 是 | 主键，UUID |
| `name` | TEXT | 是 | 任务名称 |
| `description` | TEXT | 否 | 任务描述 |
| `cron_expression` | TEXT | 是 | 标准 Cron 表达式 |
| `timezone` | TEXT | 否 | 默认 `UTC` |
| `task_template_json` | TEXT | 是 | 任务模板 JSON，包含 prompt、targetAgentId、attachments 等 |
| `enabled` | BOOLEAN | 否 | 默认 `1` |
| `concurrency_policy` | TEXT | 否 | 默认 `FORBID`，取值 `FORBID` / `ALLOW` / `REPLACE` |
| `last_run_at` | DATETIME | 否 | 上次执行时间 |
| `next_run_at` | DATETIME | 是 | 下次执行时间，用于快速索引和扫描 |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |
| `updated_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_jobs_next_run (next_run_at)`
- `idx_jobs_enabled (enabled)`

说明：

- `task_template_json` 示例：`{ "prompt": "生成每日销售报表", "target_agent_id": "agent_data_01", "attachments": [], "clarification_data": null }`
- `concurrency_policy` 语义：
  - `FORBID`：禁止并发执行，若上次任务未完成则跳过
  - `ALLOW`：允许并发执行
  - `REPLACE`：若上次任务未完成则取消并重新执行

### 4.12 `job_execution_logs`

用途：存储定时任务的执行历史记录，用于审计和前端展示。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | 是 | 主键，自增 |
| `job_id` | TEXT | 是 | 关联 `scheduled_jobs.id` |
| `scheduled_time` | DATETIME | 是 | 计划执行时间 |
| `actual_start_time` | DATETIME | 否 | 实际开始时间 |
| `actual_end_time` | DATETIME | 否 | 实际结束时间 |
| `triggered_task_id` | TEXT | 否 | 关联的 `tasks.id`，若创建任务失败则为 null |
| `status` | TEXT | 是 | `SUCCESS`、`FAILED`、`SKIPPED`、`MISSED` |
| `error_message` | TEXT | 否 | 错误信息 |
| `retry_count` | INTEGER | 否 | 默认 `0` |
| `created_at` | DATETIME | 否 | 默认 `CURRENT_TIMESTAMP` |

索引：

- `idx_logs_job (job_id)`
- `idx_logs_status (status)`

说明：

- `SKIPPED` 表示因并发策略（FORBID）而跳过的执行
- `MISSED` 表示系统未能及时触发（如服务宕机期间）

## 5. 向量数据库设计

### 5.1 存储位置

- 本地嵌入式 LanceDB，目录位于 `{workspace}/lancedb`

### 5.2 表设计策略

- 可选方案一：按 Agent 建分表，例如 `vec_{agent_id}`
- 推荐方案：使用统一表 `global_vectors`，通过 `agent_id` 与 `source_type` 过滤

推荐统一表的原因：

- 备份与恢复更简单
- 减少分表数量带来的管理开销
- 与 `long_term_memory.embedding_id` 的引用关系更稳定

### 5.3 向量记录 Schema

```ts
{
  vector: number[768],
  text: string,
  metadata: {
    agent_id: string,
    source_type: 'knowledge' | 'ltm',
    file_id?: string,
    page_num?: number,
    chunk_id: string
  }
}
```

### 5.4 检索约束

- 查询时必须注入 `agent_id` 过滤，避免跨 Agent 泄漏
- 若来源为长期记忆，`source_type` 应为 `ltm`
- 若来源为知识文件，`source_type` 应为 `knowledge`
- 可使用 IVF_PQ 等索引加速检索

## 7. 生命周期与一致性规则

### 7.1 会话归档

- 所有新消息先写入 `session_memory`，默认 `is_archived = 0`
- 后台归档任务扫描 24 小时前且超过阈值的未归档记录
- 归档时生成摘要，写入 `long_term_memory`
- 归档成功后，原 `session_memory` 记录更新：
  - `is_archived = 1`
  - `ltm_ref_id = long_term_memory.id`
  - `summary = 归档摘要`

### 7.2 任务一致性

- `tasks` 与 `task_logs` 的写入应保持原子性
- 任务入队前必须先持久化 `tasks.status = QUEUED`
- 应用重启后通过扫描非终态任务恢复执行队列

### 7.3 记忆一致性

- `long_term_memory` 与 LanceDB 记录应一一对应，关系由 `embedding_id` 维护
- 删除 LTM 时先标记 `is_active = 0`，再异步删除向量记录
- 归档操作应保证 `session_memory` 状态更新与 LTM 写入处于同一事务或具备补偿机制

### 7.4 审计一致性

- 权限拒绝、授权超时、工具执行结果都必须写入 `audit_logs`
- `details_json` 必须在入库前完成脱敏

### 7.5 定时任务调度一致性

- 调度器启动时扫描 `enabled=true` 的任务并加载到内存
- 每次触发后必须计算并更新 `next_run_at`
- 执行日志必须与任务执行记录保持关联
- 手动触发不更新 `next_run_at`，仅记录执行历史
- 删除任务时应同时清理相关执行历史（或按策略保留）

## 8. 索引与性能建议

- `tasks`：优先保证状态查询、Agent 调度查询和心跳扫描性能
- `task_logs`：以 `task_id` 为主索引，支撑时间线回放
- `session_memory`：重点优化活跃历史读取与归档扫描
- `long_term_memory`：仅承担元数据过滤，语义检索主要由 LanceDB 负责
- `knowledge_files`：按 `agent_id + status` 检索，支撑知识库列表与解析进度查询

在 P0 单用户场景下，SQLite 足以承载元数据查询压力；性能瓶颈主要会出现在向量检索与大文件解析，而不是关系表扫描。
