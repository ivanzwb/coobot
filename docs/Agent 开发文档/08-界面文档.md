# BiosBot 界面文档

## 1. 文档目的

本文档统一定义 BiosBot P0 阶段 Web 端的信息架构、页面职责、组件拆分、页面状态和关键交互。

本文档重点回答以下问题：

- 页面如何组织
- 用户如何完成主链路操作
- 各页面如何消费 API、事件流与任务状态
- 前端展示态如何与后端任务状态、记忆模型、权限模型保持一致

本文档是前端页面实现、交互验收和界面测试用例的真相源。若 05 或 06 更新，08 必须同步更新，不允许继续沿用旧版多会话、检查点恢复、Prompt 独立管理页等未在当前设计中定稿的页面模型。

## 2. 设计基线

### 2.1 核心约束

- 单用户、单会话窗口：用户进入系统后直接落在统一的 ChatWindow，不提供多会话列表、新建会话和切换会话主流程。
- 主链路闭环优先：文本输入、多模态输入、进度反馈、权限授权和结果回显优先在会话页闭环完成。
- 任务中心化：任务详情页和结果页是补充查看入口，不替代会话页的主交互职责。
- 记忆分层：活跃历史来自 `session_memory`，长期记忆来自 `long_term_memory`，界面不得再引入旧 `conversation/messages/memory_entries` 概念。
- Agent 隔离：知识库、长期记忆和工具权限相关界面均以 `agent_id` 为隔离边界。
- API 基线路径统一为 `/api/v1/*`。

### 2.2 页面基线

P0 Web 端围绕以下核心页面组织：

- 会话页 `ChatWindow`
- Agent 工厂页 `AgentFactory`
- 监控审计台 `MonitorDashboard`
- 记忆管理页 `MemoryDashboard`
- 设置页

补充页面：

- 任务详情页
- 结果页
- 技能管理页
- 知识库页
- 定时任务管理页

全局浮层：

- `AuthModal`
- 全局错误提示
- 导出、删除、补跑确认框

### 2.3 前端术语口径

- 会话页：单一主交互窗口，对应 05 中的 `ChatWindow`。
- ReAct 详情面板：展示 `THOUGHT`、`ACTION`、`OBSERVATION` 的摘要时间轴，不展示完整原始思维链。
- 已安排通知：父任务已完成拆解与分派，但尚未形成最终交付物的阶段性通知。
- 最终完成通知：任务已完成最终可交付结果的通知。
- 补充更新：最终交付后晚到的非阻塞结果更新，不覆盖原最终输出。

## 3. 信息架构与路由设计

### 3.1 信息架构

```text
Web App
├── ChatWindow
├── Task Detail
├── Result View
├── AgentFactory
├── Skills
├── Knowledge Base
├── MemoryDashboard
├── MonitorDashboard
└── Settings
```

### 3.2 路由设计

```text
/
|-- / /chat                # 默认首页与会话窗口（`/` 与 `/chat` 均指向 ChatWindow）
|-- /tasks/:taskId         # 任务详情页
|-- /results/:taskId       # 结果页（可选；可与任务详情合并）
|-- /agents                # Agent 工厂页
|-- /skills                # Skills 管理页
|-- /knowledge             # 知识库页（可按 Agent 在页内筛选；与 `/knowledge/:agentId` 二选一）
|-- /memory                # 记忆管理页
|-- /scheduler             # 定时任务管理页
|-- /scheduler/:jobId      # 定时任务详情页（可选；可在列表内展开历史代替）
|-- /monitor               # 监控审计台
|-- /settings              # 设置页
```

路由约束：

- `/` 与 `/chat` 均为会话首页（等价）。
- 任务详情页和结果页必须可从会话页中的任务卡片或结果卡片跳转进入。

## 4. 布局总览

| 页面 | 布局类型 | 主要区域 |
|------|---------|---------|
| 会话页 | 单列主内容 + 顶部状态区 | 消息流、输入区、ReAct 详情、任务状态卡片 |
| 任务详情页 | 顶部摘要 + 主内容双区 | 任务概要、时间线、步骤、附件、人工介入 |
| 结果页 | 顶部横幅 + 内容区 | 结果摘要、输出物、引用、权限与记忆摘要 |
| Agent 工厂页 | 双栏或抽屉编辑布局 | Agent 列表、基础配置、Skill 配置、知识库 |
| Skills 管理页 | 列表 + 导入区 | Skill 列表、预读取确认、导入结果与 item 级错误 |
| 知识库页 | 列表布局 | 文档列表、上传入口、解析状态 |
| 记忆管理页 | 列表 + 详情抽屉 | LTM 列表、来源链路、会话上下文 |
| 定时任务管理页 | 列表 + 抽屉编辑布局 | 任务列表、执行历史、任务配置、运行状态 |
| 监控审计台 | 仪表盘布局 | 指标卡片、图表、Agent 矩阵、日志表格 |
| 设置页 | 分段表单布局 | 工作目录、模型配置、系统阈值、健康检查 |

## 5. 页面职责与组件拆分

### 5.1 会话页 `ChatWindow`

职责：

- 作为默认唯一主交互面，承载文本输入、附件上传、任务创建、进度反馈、结果回显和权限确认。
- 从 `session_memory` 分页加载消息流，并保持时间顺序连续展示。
- 用户发送消息后调用 `POST /api/v1/chat` 或 `POST /api/v1/tasks` 触发任务执行。
- 订阅 `GET /api/v1/tasks/:id/events` 返回的事件流，并实时更新任务状态与步骤摘要。
- 收到 `AUTH_REQUEST` 事件时弹出 `AuthModal`，由用户提交 `/api/v1/auth/decision`。
- 当任务处于 `CLARIFICATION_PENDING` 时，在消息流内展示澄清问题卡片并允许用户直接补充。
- 支持历史浏览、关键字搜索和导出当前记录。

核心组件：

- `MessageList`
- `UserBubble`
- `AgentBubble`
- `TaskCard`
- `ChatInput`
- `FileUploader`
- `MentionPicker`
- `ReActDetailPanel`
- `AuthModal`

交互规则：

- 主链路必须优先在会话页完成，不要求用户跳转到任务详情页才能查看实时进度。
- 消息流按时间倒序分页加载历史，但渲染时保持阅读顺序一致。
- 任务卡片必须区分三类状态：澄清中、执行中、已完成。
- 权限确认必须中断当前高风险动作，并提供允许、拒绝、持久化策略选择。

页面草图：

```text
┌──────────────────────────────────────────────────────────────┐
│ Header: 当前任务状态 / 全局连接状态                           │
├──────────────────────────────────────────────────────────────┤
│ MessageList                                                  │
│  ├─ UserBubble                                               │
│  ├─ AgentBubble                                              │
│  ├─ TaskCard (状态、ETA、查看详情)                            │
│  └─ ClarificationCard / AuthRequestCard                      │
├──────────────────────────────────────────────────────────────┤
│ ReActDetailPanel（可折叠）                                   │
├──────────────────────────────────────────────────────────────┤
│ ChatInput + FileUploader + Send                              │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 任务详情页

职责：

- 补充展示会话页中的任务卡片详情，承接复盘、排障与深度查看需求。
- 展示任务基本信息：状态、创建时间、开始时间、完成时间、错误摘要。
- 展示任务树结构：父任务、子任务、分派 Agent、当前队列和 ETA。
- 展示 `task_logs` 驱动的步骤时间线。
- 展示附件、知识命中、人工介入原因和权限决策记录。
- 提供重试与终止操作入口。

核心组件：

- `TaskOverviewSection`
- `TaskHierarchySection`
- `TaskTimelineSection`
- `TaskStepsSection`
- `TaskAttachmentSection`
- `TaskInterventionPanel`
- `TaskActionBar`

交互规则：

- 页面状态必须直接映射 `tasks.status`，不引入另一套独立业务状态枚举。
- `THOUGHT`、`ACTION`、`OBSERVATION` 只展示压缩摘要，不展示完整原始链路。
- 当任务为父任务时，必须清楚区分“已安排完成”与“最终交付完成”。

├──────────────────────────────────────────────────────────────┤
│ TaskStepsSection                                             │

职责：

- 展示最终输出物和结构化结果摘要。
- 展示引用来源、知识命中、关键记忆读写摘要。
- 展示权限决策摘要和结果相关操作。
- 当任务只有阶段性安排结果而无最终输出时，展示终态摘要而不是伪装成标准结果页。
- 当存在补充更新时，以独立区块展示晚到结果和影响说明。

核心组件：

- `ResultStageBanner`
- `ResultSummarySection`
- `ResultOutputSection`
- `ResultReferenceSection`
- `ResultMemorySection`
- `ResultPermissionSection`
- `ResultActionSection`
- `SupplementalUpdateSection`

交互规则：

- “已安排”阶段必须明确标注为非最终输出。
- 补充更新不得覆盖原最终输出主体。
- 无最终输出但合法终止的任务必须有稳定可访问的终态摘要视图。

### 5.4 Agent 工厂页 `AgentFactory`

职责：

- 管理 Domain Agent 的创建、编辑、停用、启用和删除。
- 展示 Agent 的模型、Prompt 模板、Skill 绑定和知识库概览。
- 修改配置后，新任务自动生效，无需重启。

Tab 结构：

- `BasicSettings`
- `SkillConfig`
- `KnowledgeBase`

核心组件：

- `AgentList`
- `AgentEditorDrawer`
- `BasicSettingsForm`
- `SkillConfigPanel`
- `KnowledgeBasePanel`

交互规则：

- Leader Agent 由系统初始化，不在页面中创建。
- 创建和编辑时，模型字段来源于 `/api/v1/models`。
- 删除 Agent 前必须校验是否存在 `RUNNING` 或 `QUEUED` 任务。
- `deleteKnowledge=true|false` 的删除策略必须在确认框中明确展示。

### 5.5 Skills 管理页

职责：

- 展示已安装 Skill 列表及启用状态。
- 对本地 Skill zip 安装包执行预读取、确认导入与覆盖更新。
- 卸载 Skill，处理卸载失败提示。
- 展示 Skill 的版本、`installMode`、运行时状态和“检测到语言”标签。

核心组件：

- `SkillList`
- `SkillImportDialog`
- `SkillPreviewCard`
- `SkillImportResultBanner`
- `SkillActivationToggle`
- `SkillItemError`

交互规则：

- 导入入口必须先调用 `POST /api/v1/skills/preview` 执行安装预读取，再由用户确认调用 `POST /api/v1/skills/import`。
- Skills 列表项的描述字段固定读取 `SKILL.md.description`，不得回退为文件首行或截断首段文本。
- 列表标题区应展示 `version`、`installMode` 和“检测到语言”标签；运行时校验明细不得直接在 item 内容区展开。
- 再次导入相同 `skill-id` 必须按“覆盖更新”交互处理，页面不提供独立“更新”按钮。
- `copy_only` 与 `managed` 必须作为可见状态语义被稳定展示，避免用户把未声明运行时语言的 Skill 误解为安装失败。
- 正在被 Agent 使用的 Skill 禁止卸载。
- 卸载失败提示必须锚定到对应 Skill item，不得只显示全局错误；再次发起卸载前需先清空旧错误。
- 导入校验失败时，界面只展示即时失败结果，不生成“校验失败”的残留列表项。

页面草图：

```text
┌──────────────────────────────────────────────────────────────┐
│ Header: Skills / 导入 zip / 搜索                             │
├───────────────────────────────┬──────────────────────────────┤
│ SkillList                     │ SkillImportDialog            │
│ ┌───────────────────────────┐ │ - 选择 zip                   │
│ │ SkillItem                 │ │ - 调用 preview              │
│ │ name / description        │ │ - 展示 SkillPreviewCard     │
│ │ [version] [installMode]   │ │ - 确认 import               │
│ │ [检测到语言] [启用状态]   │ ├──────────────────────────────┤
│ │ 卸载按钮 / item 错误      │ │ SkillImportResultBanner     │
│ └───────────────────────────┘ │ - 导入成功 / 失败 / 回滚说明 │
└───────────────────────────────┴──────────────────────────────┘
```

### 5.8 定时任务管理页 `SchedulerDashboard`

职责：

- 展示定时任务列表及启用状态。

- 展示任务执行历史和运行状态。
- 展示调度器总体状态（运行中任务数、可用槽位）。

核心组件：

- `SchedulerStatusPanel`
- `ScheduledJobList`
- `ScheduledJobEditorDrawer`
- `JobExecutionLogTable`
- `JobTriggerButton`
- `JobEnableToggle`

- 任务列表必须展示下次执行时间、启用状态和并发策略。

## 6. 关键交互流程

### 6.1 会话输入到任务执行

```text
用户输入消息
-> ChatInput 提交
-> 调用 POST /api/v1/chat 或 POST /api/v1/tasks
-> 在消息流中插入用户消息气泡
-> 创建任务卡片
-> 订阅 GET /api/v1/tasks/:id/events
-> 根据事件流更新任务状态、步骤和结果回显
```

### 6.2 ASK 授权流程

```text
任务执行命中 ASK
-> 后端推送 AUTH_REQUEST
-> 前端打开 AuthModal
-> 用户选择允许/拒绝/持久化策略
-> 调用 POST /api/v1/auth/decision
-> 关闭模态框并更新任务卡片状态
```

### 6.3 澄清流程

```text
Leader 判定低置信度或缺少关键信息
-> 任务进入 CLARIFICATION_PENDING
-> 消息流插入澄清卡片
-> 用户补充信息
-> 调用 POST /api/v1/tasks/:id/clarify
-> 任务回到 PARSING 并继续规划
```

### 6.4 记忆来源追溯

```text
用户在 MemoryDashboard 点击“查看来源”
-> 打开 MemorySourceDrawer
-> 拉取长期记忆详情与关联上下文
-> 在 SessionHistoryPanel 展示来源消息片段
-> 支持跳回 ChatWindow 的对应时间位置
```

## 7. 页面状态设计

### 7.1 会话页状态

- `bootstrapping`：初始化消息流和默认页面资源。
- `idle`：可输入、无发送中的请求。
- `sending`：消息已提交，等待接口响应。
- `streaming`：正在接收任务事件流。
- `clarification_pending`：存在待用户补充的澄清任务。
- `auth_pending`：存在待用户确认的高风险操作。
- `error`：发送、订阅或渲染链路出现错误。

### 7.2 任务详情页状态

- `loading`
- `ready`
- `refreshing`
- `retry_submitting`
- `terminate_submitting`
- `error`

### 7.3 结果页状态

- `loading`
- `ready`
- `arranged_only`
- `completed`
- `terminated_with_summary`
- `supplemental_update_available`
- `error`

### 7.4 管理类页面状态

- `empty`
- `loading`
- `ready`
- `submitting`
- `item_action_failed`
- `error`

状态约束：

- 展示态可以细化交互，但不能改变后端状态事实。
- 任务卡片展示文案必须可映射回真实 `tasks.status`。
- 当存在权限确认或澄清时，页面必须优先展示阻塞原因。

## 8. 前端数据模型与状态管理

### 8.1 Store 拆分建议

- `chat.store`
  - 消息流
  - 当前输入态
  - 当前任务卡片态
  - ReAct 面板开关
- `task.store`
  - 任务详情缓存
  - 时间线缓存
  - 任务动作提交态
- `agent.store`
  - Agent 列表
  - 编辑表单
  - 知识库与 Skill 绑定摘要
- `memory.store`
  - LTM 列表
  - 搜索条件
  - 来源抽屉状态
- `settings.store`
  - 模型配置
  - 工作目录
  - 健康检查结果
- `ui.store`
  - 全局弹窗
  - Toast
  - 全局错误态

### 8.2 前端字段映射原则

- 会话消息主数据源是 `session_memory` 的 API 返回值。
- 长期记忆主数据源是 `long_term_memory` 的 API 返回值。
- 任务步骤主数据源是 `task_logs` 的 API 或事件流投影。
- 审计与权限确认主数据源分别来自 `/api/v1/audit/logs` 和 `/api/v1/auth/decision` 相关链路。

## 9. 视觉与交互约束

### 9.1 通用约束

- 列表页需要明确的加载态、空态和错误态。
- 高风险操作必须二次确认。
- 上传、删除、终止、卸载等不可逆动作必须展示影响范围。
- 列表项状态统一通过标签或徽章表达，不使用隐式文案。

### 9.2 消息与结果展示约束

- 消息气泡应区分 `user`、`assistant`、`system` 三类角色。
- ReAct 内容只展示摘要，不展示完整原始推理链。
- 结果页必须区分最终结果、阶段性安排和补充更新。
- 权限确认卡片与普通系统提示卡片需要明显不同的风险样式。

### 9.3 组件命名约束

- 页面专属样式必须使用页面前缀，避免复用过于泛化的类名。
- 列表和弹窗组件命名应与页面职责直接对应，避免沿用旧版 `conversation-*` 多会话语义。
