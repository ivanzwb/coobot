# BiosBot 测试用例文档

## 1. 文档目的

本文档定义 BiosBot P0 阶段的测试范围、测试用例和验收口径。

本文档重点回答以下问题：

- 当前 P0 主链路需要验证哪些能力
- 每类能力的前置条件、步骤和预期结果是什么
- 测试如何覆盖接口、数据库、界面和运行时一致性

本文档只保留当前设计已定稿的测试对象，不再继续沿用已废弃交互模型、历史遗留恢复机制或未在当前基线中定稿的扩展测试域。

## 2. 测试基线

### 2.1 业务基线

- 单用户、单会话窗口
- 任务中心化执行模型
- `session_memory` 作为唯一活跃历史与 STM 存储
- `long_term_memory` 作为唯一 LTM 元数据存储
- `tasks` / `task_logs` 作为任务与执行轨迹真相源
- `agent_tool_permissions` 驱动 `DENY` / `ASK` / `ALLOW` 权限决策

### 2.2 接口基线

P0 测试仅覆盖以下主接口族：

- `/api/v1/tasks`
- `/api/v1/tasks/:id/events`
- `/api/v1/tasks/:id/clarify`
- `/api/v1/tasks/:id/retry-after-clarification`
- `/api/v1/tasks/:id/terminate`
- `/api/v1/chat`
- `/api/v1/chat?limit=50`
- `/api/v1/chat/export`
- `/api/v1/memory/ltm`
- `/api/v1/memory/ltm/:id`
- `/api/v1/memory/history`
- `/api/v1/agents`
- `/api/v1/agents/:id/knowledge/upload`
- `/api/v1/agents/:id/skills`
- `/api/v1/agents/:id/tools/permissions`
- `/api/v1/agents/capabilities`
- `/api/v1/skills`
- `/api/v1/skills/preview`
- `/api/v1/skills/import`
- `/api/v1/skills/:id`
- `/api/v1/models`
- `/api/v1/models/:id/test`
- `/api/v1/metrics/resources`
- `/api/v1/audit/logs`
- `/api/v1/auth/decision`
- `/api/v1/scheduler/jobs`
- `/api/v1/scheduler/jobs/:id/logs`
- `/api/v1/scheduler/status`
- `/api/v1/scheduler/jobs/:id/trigger`

### 2.3 测试分层

- 单元测试：验证函数级逻辑、状态机和安全校验
- 集成测试：验证接口、数据库、向量库和文件系统协同
- 端到端测试：验证会话页主链路、会话内授权卡片、结果页和管理页交互
- 回归测试：验证需求修订后主链路与关键异常路径未回退

### 2.4 状态流转基线

`tasks.status` 的测试覆盖必须至少包含以下状态与关键流转：

- `CLARIFICATION_PENDING -> PARSING`
- `WAITING_FOR_LEADER -> PARSING`
- `QUEUED -> RUNNING -> COMPLETED`
- `RUNNING -> EXCEPTION`
- `RUNNING -> TERMINATED`
- `QUEUED_WAITING_RESOURCE -> QUEUED`

## 3. 测试范围

### 3.1 覆盖范围

- 工程初始化、启动、健康检查
- 会话输入到任务创建
- 任务状态流转与事件一致性
- 澄清流程
- Leader 拆解与 Domain Agent 执行
- ReAct 步骤日志
- 权限裁决与 ASK 授权
- 记忆归档、LTM 检索与删除
- 知识文件上传、解析与检索
- Agent、Skill、模型配置管理
- 结果页、监控台、记忆管理页主交互
- 服务重启后的任务恢复和等待态恢复

### 3.2 不在 P0 覆盖范围内

- 历史会话列表、切换与删除
- 历史恢复页及其配套接口
- 模板治理与迁移链路
- 已废弃接口族
- 未在 05、06、08 中定稿的扩展事件或中间态

## 4. 工程与环境测试

### 用例 T-ENG-01：工程目录初始化符合运行时基线

- 前置条件：全新工作区。
- 步骤：
  1. 创建 `backend`、`web`、`skills`、`workspace`、`data` 目录。
  2. 创建 `workspace/uploads`、`workspace/outputs`、`workspace/cache`、`data/logs`。
  3. 准备配置文件与数据库目录。
- 预期结果：
  - 目录结构与当前运行时基线一致。
  - 工作目录、数据目录和日志目录相互独立。

### 用例 T-ENG-02：依赖安装与数据库初始化顺序正确

- 前置条件：后端、前端包定义完整。
- 步骤：
  1. 在 `backend` 执行依赖安装。
  2. 在 `web` 执行依赖安装。
  3. 执行数据库迁移与种子初始化。
- 预期结果：
  - 后端、前端依赖均可正常安装。
  - SQLite、LanceDB、日志目录被正确初始化。
  - 迁移与种子执行顺序稳定，不互相覆盖。

### 用例 T-ENG-03：开发启动与健康检查可用

- 前置条件：目录与依赖已就绪。
- 步骤：
  1. 启动后端开发服务。
  2. 调用 `GET /health`、`GET /health/ready`、`GET /health/live`。
  3. 启动前端开发服务并访问 `/chat`。
- 预期结果：
  - 健康检查接口可用。
  - `/health/ready` 在关键依赖缺失时返回失败状态。
  - 会话页能够加载并与后端联通。

### 用例 T-ENG-04：生产构建与发布包结构一致

- 前置条件：开发环境可正常运行。
- 步骤：
  1. 构建后端产物。
  2. 构建前端产物。
  3. 检查发布目录中的服务、静态资源、配置、工作目录和数据目录。
- 预期结果：
  - 构建成功。
  - 发布目录具备完整运行时结构。

## 5. 任务主链路测试

### 用例 T-TASK-01：会话页创建任务成功

- 前置条件：系统已启动，至少存在一个可用模型和一个在线 Domain Agent。
- 步骤：
  1. 打开 `/chat`。
  2. 输入任务文本。
  3. 点击发送。
- 预期结果：
  - 用户消息写入 `session_memory`。
  - 成功创建 `tasks` 记录。
  - 前端出现任务卡片并进入 `PARSING` 或 `CLARIFICATION_PENDING`。

### 用例 T-TASK-02：澄清流程在原任务上闭环完成

- 前置条件：准备一个缺少关键参数的输入样本。
- 步骤：
  1. 在会话页提交缺少关键参数的请求。
  2. 观察任务是否进入 `CLARIFICATION_PENDING`。
  3. 在澄清卡片中补充信息。
  4. 调用 `/api/v1/tasks/:id/clarify`。
- 预期结果：
  - 同一任务进入澄清链路，而不是新建第二个任务。
  - 补充信息后任务重新进入 `PARSING`。
  - 时间线或事件流可见澄清请求与澄清完成事实。

### 用例 T-TASK-03：Leader 拆解与 Domain Agent 执行链路正常

- 前置条件：存在至少两个在线 Domain Agent，能力互补。
- 步骤：
  1. 提交一个需要拆解的复合任务。
  2. 检查父任务和子任务的创建情况。
  3. 观察会话页、任务详情页和事件流中的执行过程。
- 预期结果：
  - 根任务经历 `WAITING_FOR_LEADER`、`PARSING` 与 `DISPATCHING` 状态。
  - Leader 完成规划并分发子任务。
  - 子任务进入对应 Agent 队列执行。
  - 父任务最终进入 `AGGREGATING` 后收敛为 `COMPLETED` 或 `EXCEPTION`。

### 用例 T-TASK-04：任务终止接口可中断执行

- 前置条件：存在一个 `RUNNING` 状态任务。
- 步骤：
  1. 调用 `/api/v1/tasks/:id/terminate`。
  2. 刷新任务详情和事件流。
- 预期结果：
  - 任务进入 `TERMINATED`。
  - 后续不再产生新的步骤事件。
  - 审计日志记录终止动作。

### 用例 T-TASK-05：任务事件流与状态变更一致

- 前置条件：存在一个从创建到完成可完整运行的任务样本。
- 步骤：
  1. 订阅 `/api/v1/tasks/:id/events`。
  2. 记录 `STEP_UPDATE`、`STATUS_CHANGE`、`AUTH_REQUEST`、`CLARIFICATION_REQUEST` 事件。
  3. 对照任务详情页与数据库状态。
- 预期结果：
  - 事件顺序与任务执行顺序一致。
  - `STATUS_CHANGE` 使用真实任务状态枚举。
  - `STEP_UPDATE` 中的步骤类型只包含 `THOUGHT`、`ACTION`、`OBSERVATION`。

### 用例 T-TASK-06：任务状态机覆盖当前设计的关键流转

- 前置条件：准备可分别触发澄清、排队执行、异常、终止和等待资源的样本任务。
- 步骤：
  1. 触发低置信度任务并验证 `CLARIFICATION_PENDING -> PARSING`。
  2. 触发需要 Leader 规划的复合任务并验证 `WAITING_FOR_LEADER -> PARSING -> DISPATCHING`。
  3. 触发普通执行任务并验证 `QUEUED -> RUNNING -> COMPLETED`。
  4. 触发工具异常或模型失败并验证 `RUNNING -> EXCEPTION`。
  5. 在执行中调用终止接口并验证 `RUNNING -> TERMINATED`。
  6. 模拟 Agent 离线并验证 `QUEUED_WAITING_RESOURCE -> QUEUED`。
- 预期结果：
  - 每条流转均符合 05 与 06 定义的任务状态基线。
  - 不出现旧版状态值或未定义中间态。
  - 任务详情页、事件流和数据库状态三者一致。

## 6. 记忆与上下文测试

### 用例 T-MEM-01：活跃历史写入与读取正常

- 前置条件：系统已启动。
- 步骤：
  1. 连续发送多条会话消息。
  2. 调用 `/api/v1/memory/history?limit=50`。
  3. 查询 `session_memory`。
- 预期结果：
  - 消息按时间顺序写入 `session_memory`。
  - 历史接口返回的字段与数据库一致。
  - 默认读取的是活跃历史视图。

### 用例 T-MEM-02：历史归档正确写入长期记忆

- 前置条件：存在超过阈值且超过时间窗口的未归档历史。
- 步骤：
  1. 触发归档任务。
  2. 查询 `session_memory` 和 `long_term_memory`。
  3. 检查对应 LanceDB 记录。
- 预期结果：
  - 归档后旧历史被标记 `is_archived=1`。
  - 归档摘要写入 `long_term_memory`。
  - `ltm_ref_id` 与长期记忆记录形成关联。

### 用例 T-MEM-03：长期记忆检索与删除正常

- 前置条件：存在至少一条可检索的长期记忆。
- 步骤：
  1. 调用 `/api/v1/memory/ltm` 进行搜索。
  2. 删除一条长期记忆。
  3. 重新搜索并检查数据库状态。
- 预期结果：
  - 搜索可返回按 Agent 隔离的 LTM 结果。
  - 删除后 `is_active=0`。
  - 删除动作同步影响向量检索结果。

### 用例 T-MEM-04：会话历史查询与聊天记录接口保持一致

- 前置条件：系统中已存在最近 50 条以内的会话记录。
- 步骤：
  1. 调用 `GET /api/v1/chat?limit=50`。
  2. 调用 `GET /api/v1/memory/history?limit=50`。
  3. 对比时间顺序、角色、内容和附件占位符。
- 预期结果：
  - 两个接口返回的基础消息事实一致。
  - `GET /api/v1/chat` 作为只读查询接口，不创建新任务、不修改历史状态。
  - 历史记录展示与 ChatWindow 的加载结果一致。

### 用例 T-MEM-05：上下文组装在 LTM 不可用时可降级

- 前置条件：准备 LTM 或 LanceDB 不可用场景。
- 步骤：
  1. 模拟 LanceDB 不可用。
  2. 提交一个需要上下文检索的任务。
- 预期结果：
  - 系统降级为活跃历史或关键词匹配。
  - 会话主链路不被阻塞。
  - 监控台或日志中记录降级事实。

## 7. 知识库测试

### 用例 T-KB-01：知识文件上传与解析状态正确

- 前置条件：存在可用 Agent。
- 步骤：
  1. 调用 `/api/v1/agents/:id/knowledge/upload` 上传文件。
  2. 检查知识库页状态变化。
  3. 查询 `knowledge_files`。
- 预期结果：
  - 文件被存入 Agent 专属知识目录。
  - `knowledge_files.status` 经历 `PROCESSING -> READY` 或 `ERROR`。
  - 元数据与物理文件路径一致。

### 用例 T-KB-02：知识检索严格按 Agent 隔离

- 前置条件：Agent A 和 Agent B 各自拥有不同知识文件。
- 步骤：
  1. 使用 Agent A 发起知识检索。
  2. 使用 Agent B 发起相同检索。
  3. 对比命中结果。
- 预期结果：
  - 检索结果仅来自各自 Agent 的知识范围。
  - 不发生跨 Agent 泄漏。

## 9. 定时任务调度测试

### 用例 T-SCH-01：定时任务创建与持久化

- 前置条件：系统已启动。
- 步骤：
  1. 调用 `POST /api/v1/scheduler/jobs` 创建定时任务。
  2. 查询 `scheduled_jobs` 表。
- 预期结果：
  - 任务创建成功并持久化到数据库。
  - `next_run_at` 根据 Cron 表达式正确计算。
  - `enabled` 默认为 `true`。

### 用例 T-SCH-02：定时任务按计划触发执行

- 前置条件：存在一个已创建且启用的定时任务。
- 步骤：
  1. 等待任务到达 `next_run_at`。
  2. 观察调度器触发任务执行。
  3. 检查 `job_execution_logs` 和 `tasks` 表。
- 预期结果：
  - 调度器正确扫描到期的任务。
  - 创建对应的任务记录到 `tasks` 表。
  - 记录执行历史到 `job_execution_logs`。

### 用例 T-SCH-03：手动触发定时任务

- 前置条件：存在一个已创建的定时任务。
- 步骤：
  1. 调用 `POST /api/v1/scheduler/jobs/:id/trigger`。
  2. 检查 `job_execution_logs` 和 `tasks` 表。
- 预期结果：
  - 手动触发成功创建任务。
  - 执行日志记录触发时间，但不更新 `next_run_at`。
  - 任务正常进入执行流程。

### 用例 T-SCH-04：定时任务启用/禁用切换

- 前置条件：存在一个已创建的定时任务。
- 步骤：
  1. 调用 `POST /api/v1/scheduler/jobs/:id/disable`。
  2. 等待任务计划触发时间。
  3. 检查任务是否未被触发。
  4. 调用 `POST /api/v1/scheduler/jobs/:id/enable`。
  5. 再次等待触发时间。
- 预期结果：
  - 禁用后任务不再自动触发。
  - 启用后任务恢复正常触发。

### 用例 T-SCH-05：定时任务编辑后正确计算下次执行时间

- 前置条件：存在一个已创建的定时任务。
- 步骤：
  1. 获取当前 `next_run_at`。
  2. 调用 `PUT /api/v1/scheduler/jobs/:id` 修改 Cron 表达式。
  3. 检查新的 `next_run_at`。
- 预期结果：
  - Cron 表达式更新成功。
  - `next_run_at` 根据新表达式重新计算。

### 用例 T-SCH-06：定时任务删除

- 前置条件：存在一个已创建的定时任务。
- 步骤：
  1. 调用 `DELETE /api/v1/scheduler/jobs/:id`。
  2. 查询 `scheduled_jobs` 表。
- 预期结果：
  - 任务从数据库中删除。
  - 调度器内存中不再持有该任务引用。

### 用例 T-SCH-07：并发策略 FORBID 正确跳过执行

- 前置条件：存在一个 `concurrency_policy=FORBID` 的定时任务，上次任务仍在运行。
- 步骤：
  1. 任务 A 开始执行且未完成。
  2. 等待任务 A 的下一次计划触发时间。
  3. 检查执行日志。
- 预期结果：
  - 新任务被跳过，记录 `status=SKIPPED`。
  - 不创建新的任务记录。

### 用例 T-SCH-08：调度器状态查询正确反映运行状态

- 前置条件：调度器已启动。
- 步骤：
  1. 调用 `GET /api/v1/scheduler/status`。
- 预期结果：
  - 返回 `running=true`。
  - 返回当前执行槽位使用情况和总槽位数。

### 用例 T-SCH-09：定时任务执行历史查询

- 前置条件：存在执行历史的定时任务。
- 步骤：
  1. 调用 `GET /api/v1/scheduler/jobs/:id/logs`。
  2. 可选按状态和时间筛选。
- 预期结果：
  - 返回该任务的所有执行历史。
  - 支持按 `SUCCESS`、`FAILED`、`SKIPPED`、`MISSED` 状态筛选。

### 用例 T-SCH-10：Cron 表达式格式校验

- 前置条件：系统已启动。
- 步骤：
  1. 尝试创建包含无效 Cron 表达式的定时任务。
- 预期结果：
  - 返回错误提示。
  - 不创建无效任务。

## 8. 权限与安全测试

### 用例 T-SEC-01：DENY 策略直接阻断工具执行

- 前置条件：某 Agent 对目标工具策略为 `DENY`。
- 步骤：
  1. 让任务触发该工具调用。
  2. 检查事件流、任务状态和审计日志。
- 预期结果：
  - 工具执行被阻断。
  - 审计日志记录 `DENY` 结果。
  - 任务进入明确异常路径，而不是静默继续。

### 用例 T-SEC-02：ASK 授权允许后恢复执行

- 前置条件：某 Agent 对目标工具策略为 `ASK`。
- 步骤：
  1. 发起高风险任务。
  2. 收到 `AUTH_REQUEST`。
  3. 用户选择允许。
  4. 调用 `/api/v1/auth/decision`。
- 预期结果：
  - 会话页消息流内出现授权卡片并展示工具名与参数摘要。
  - 授权后任务继续执行。
  - 审计日志记录 `User Granted`。

### 用例 T-SEC-03：ASK 超时自动拒绝

- 前置条件：目标工具策略为 `ASK`，超时时间可配置。
- 步骤：
  1. 触发高风险工具调用。
  2. 不做任何人工确认，等待超时。
- 预期结果：
  - 授权请求自动失效。
  - 任务进入拒绝或终止路径。
  - 审计日志记录超时原因。

### 用例 T-SEC-04：路径穿越与非法参数被执行前拦截

- 前置条件：系统已启动。
- 步骤：
  1. 构造包含 `..` 的文件路径参数。
  2. 构造非法域名或非法命令参数。
