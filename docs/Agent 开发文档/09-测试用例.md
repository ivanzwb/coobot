# BiosBot 测试用例清单

## 1. 文档目的

本文档用于定义 BiosBot P0 阶段的测试用例清单，验证主要业务链路和关键支撑能力。
本文档主要支撑 01-需求分析、02-功能要求、03-技术分析、04-概要设计、05-详细设计、06-功能函数文档、07-数据库文档、08-界面文档 和 10-编译手册 的验收与联调。
本文档重点回答“每条核心需求如何验证、测试步骤是什么、预期结果是什么”。

本文档按 01-需求分析 中的需求编号 `N-01`～`N-14`、`N-16` 组织测试用例，并对 P1 阶段的 `N-10` 保留前向兼容验收用例，用于验证 P0 阶段的主要业务链路、安全基线与关键支撑能力。

## P0 工程基线与编译手册联调验证

### 工程安装、初始化、启动与发布验证

**用例 T-ENG-01：首次环境搭建满足编译手册目录基线**（验证依据：10-编译手册第 4、9 章）
- 前置条件：工作区为全新工程目录或已清理到仅保留文档的状态。
- 步骤：
  1. 按编译手册创建 `backend`、`web`、`skills`、`workspace`、`data` 目录。
  2. 创建 `workspace/uploads`、`workspace/outputs`、`workspace/cache` 和 `data/logs` 子目录。
  3. 在 `backend/config` 下准备 `default.json`、`development.json`、`production.json` 和本机私有配置文件。
  4. 检查 `skills/<skill-id>/SKILL.md` 与 `skills/<skill-id>/scripts/` 的目录结构约束。
- 预期结果：
  - 工程目录结构与编译手册中的开发态目录基线一致。
  - `skills/`、`workspace/`、`data/` 为独立显式目录，未与源码目录混放。
  - 缺少 `SKILL.md` 的目录不会被视为合法 Skill。

**用例 T-ENG-02：依赖安装、数据库迁移与种子初始化顺序正确**（验证依据：10-编译手册第 8、9 章）
- 前置条件：`backend/package.json` 与 `web/package.json` 已具备正式脚本定义。
- 步骤：
  1. 进入 `backend` 执行 `npm install`。
  2. 进入 `web` 执行 `npm install`。
  3. 返回 `backend` 依次执行 `npm run db:migrate` 与 `npm run db:seed`。
  4. 检查 `data/database.sqlite`、`data/knowledge.lancedb/` 和 `data/logs/` 是否已建立。
- 预期结果：
  - 依赖安装顺序满足“先后端，后前端”的固定约束。
  - `npm run db:migrate` 仅执行迁移，不隐式写入测试数据。
  - `npm run db:seed` 仅写入基础初始化数据；无种子数据时也返回成功。
  - 如需联调 UI，可额外执行 `npm run db:seed:demo` 写入可重复刷新的演示数据。
  - SQLite、向量库和日志目录均位于编译手册定义的数据目录下。

**用例 T-ENG-03：后端开发启动满足恢复与健康检查口径**（验证依据：10-编译手册第 7、8、10、12 章）
- 前置条件：基础目录、模型配置和数据库迁移已完成。
- 步骤：
  1. 进入 `backend` 执行 `npm run dev`。
  2. 观察启动日志中的端口、数据库连接、等待订阅恢复和调度器初始化信息。
  3. 调用 `/health/live`，并检查后端是否可建立 WebSocket 通道。
  4. 若存在历史等待态任务，检查其是否在启动后被恢复为活跃等待订阅。
- 预期结果：
  - `npm run dev` 同时启动 HTTP API、WebSocket、等待订阅恢复和调度器。
  - 后端启动完成后可返回有效的 `/health/live` 响应。
  - 等待订阅恢复发生在调度器正式运行前，不会出现重复释放或丢失等待态任务。

**用例 T-ENG-04：前端开发启动满足主路由与联调口径**（验证依据：10-编译手册第 8、10 章）
- 前置条件：后端开发服务已运行。
- 步骤：
  1. 进入 `web` 执行 `npm run dev`。
  2. 访问 `/chat`、`/tasks/:taskId`、`/results/:taskId` 等主页面路由。
  3. 在前端发起一个会触发 API 请求和时间线订阅的任务。
  4. 检查 Vite 代理、HTTP 请求和 WebSocket 是否都能连通后端。
- 预期结果：
  - `npm run dev` 成功启动 Web 开发服务。
  - 会话页首屏可渲染，主页面路由可正常访问。
  - API 代理与 WebSocket 联调正常，前端可收到任务时间线增量事件。

**用例 T-ENG-05：本地联调顺序与主链路验证一致**（验证依据：10-编译手册第 10、12 章）
- 前置条件：前后端开发服务均已启动。
- 步骤：
  1. 按编译手册定义的顺序启动后端与前端开发服务。
  2. 在默认会话窗口输入文本并上传附件，发起一个任务。
  3. 观察任务在会话页、任务详情页和结果页中的表现。
  4. 再创建一个 `queued`、`scheduled` 或 `event_triggered` 任务，验证等待态与释放链路。
- 预期结果：
  - 本地联调顺序与编译手册一致，不依赖额外未定义脚本。
  - 主链路至少覆盖会话输入、任务创建、时间线展示、结果交付四个环节。
  - 非立即执行任务满足 `TaskTriggerResolved -> TaskTriggerWaiting -> TaskTriggerActivated -> TaskReadyForPlanning -> MemoryScopeConfirmed` 的释放口径。

**用例 T-ENG-06：生产构建产物与发布包结构一致**（验证依据：10-编译手册第 4、8、11 章）
- 前置条件：前后端源码目录完整，构建脚本可执行。
- 步骤：
  1. 进入 `backend` 执行 `npm run build`。
  2. 进入 `web` 执行 `npm run build`。
  3. 将 `backend/dist/` 收敛到发布目录 `biosbot/dist/server/`。
  4. 将 `web/dist/` 收敛到发布目录 `biosbot/dist/web/`，并检查发布目录中的 `config/`、`skills/`、`workspace/`、`data/`、`package.json`。
- 预期结果：
  - 后端构建产物固定输出到 `backend/dist/`。
  - 前端构建产物固定输出到 `web/dist/`。
  - 发布包目录固定收敛为 `biosbot/dist/server/` 与 `biosbot/dist/web/` 两层产物结构。
  - 发布目录中包含编译手册要求的全部运行时目录和文件。

**用例 T-ENG-07：发布包启动与最小健康检查一致**（验证依据：10-编译手册第 8、11、12 章）
- 前置条件：已生成完整的 `biosbot/` 发布目录。
- 步骤：
  1. 进入 `biosbot` 执行根级 `npm run start`。
  2. 调用 `GET /health`、`GET /health/ready`、`GET /health/live`。
  3. 检查运行进程是否能访问同级 `config/`、`skills/`、`workspace/` 与 `data/`。
  4. 检查日志是否同时输出到控制台和 `data/logs/`。
- 预期结果：
  - 根级 `npm run start` 从 `dist/server/` 启动生产服务，而不是回退到开发目录执行。
  - 前端静态资源从 `dist/web/` 提供或可被反向代理稳定访问。
  - 最小健康检查至少覆盖服务状态、数据库连接、磁盘空间、调度器状态、等待订阅恢复状态以及等待订阅或执行队列积压状态。

**用例 T-ENG-08：本地健康检查脚本失败时返回非零退出码**（验证依据：10-编译手册第 8、12 章）
- 前置条件：后端已提供 `npm run healthcheck` 脚本。
- 步骤：
  1. 在配置正确时执行一次 `npm run healthcheck`。
  2. 人为破坏一个关键依赖，例如移除 `SQLITE_PATH`、篡改 `LANCEDB_PATH` 或提供不可用模型地址。
  3. 再次执行 `npm run healthcheck`，记录退出码和输出。
- 预期结果：
  - 正常配置下，`npm run healthcheck` 返回成功状态。
  - 关键依赖不可用时，脚本返回非零退出码，并明确指出失败项。

**用例 T-ENG-09：健康检查接口返回组件级诊断与积压状态**（验证依据：04-概要设计 4.6，06-功能函数 2.12，02-功能要求 N-17.1.1）
- 前置条件：后端服务已启动；SQLite、LanceDB 和文件系统目录可用。
- 步骤：
  1. 分别调用 `GET /health`、`GET /health/ready`、`GET /health/live`。
  2. 检查每个响应体中是否包含结构化的组件级状态字段，如服务状态、数据库连接、磁盘空间、调度器状态、等待订阅恢复状态、等待订阅或执行队列积压数量。
  3. 人为使 SQLite 数据库文件不可读或移除 LanceDB 目录，再次调用健康检查。
  4. 创建若干等待态任务使队列积压后，再调用 `GET /health` 检查积压字段。
- 预期结果：
  - `GET /health` 返回汇总状态与各组件独立诊断结果，不是仅返回单一 `ok/fail` 值。
  - 组件级状态至少覆盖数据库连接、磁盘空间、调度器状态和队列积压深度。
  - 当关键组件不可用时，`GET /health/ready` 返回非 200 状态码，且响应体明确指出失败组件。
  - 队列积压数量字段在存在等待态任务时返回大于 0 的值，与实际积压一致。

**用例 T-ENG-10：模块初始化顺序与等待订阅恢复时序正确**（验证依据：05-详细设计 4.5，04-概要设计 4.6）
- 前置条件：后端服务未运行；数据库中存在若干历史等待态任务的活跃订阅记录。
- 步骤：
  1. 启动后端服务，收集完整启动日志。
  2. 在日志中定位各模块初始化的顺序标记，检查是否满足：基础设施层 → 知识/记忆模块 → Skill 索引 → 模型适配 → 任务模块（含等待订阅恢复） → 会话模块 → 调度器启动。
  3. 检查等待订阅恢复是否发生在调度器正式启动定时扫描之前。
  4. 在调度器扫描启动后，检查历史等待态任务是否已被恢复为活跃订阅。
- 预期结果：
  - 启动日志中各模块初始化顺序与 05-详细设计定义的 7 步启动序列一致。
  - 任务模块的等待订阅恢复在调度器正式运行前完成，不会出现调度器先扫描空订阅池、随后才恢复历史订阅导致的漏释放。
  - 历史等待态任务在启动完成后可在订阅列表中查到，且 `status=active`、`nextCheckAt` 已更新。
  - 若某模块初始化失败，后续依赖模块不会继续启动，健康检查接口返回对应失败组件。

**用例 T-ENG-11：健康检查组件级细粒度诊断与阈值合规验证**（验证依据：04-概要设计 5.10.1，06-功能函数 2.12，03-技术分析 5.10.2）
- 前置条件：后端服务已启动；SQLite、LanceDB、文件系统目录和调度器均可用；健康检查接口支持组件级细粒度诊断。
- 步骤：
  1. 调用 `GET /health`，检查响应体中每个组件的诊断结构是否包含独立状态、度量值和阈值字段。
  2. 检查 SQLite 组件是否返回可用空间度量，并验证其是否满足最小阈值（如 > 100MB）。
  3. 检查 LanceDB 组件是否返回连接池实际状态（如活跃连接数、空闲连接数），而不是硬编码 `ok`。
  4. 分别检查 `workspace/uploads`、`workspace/outputs`、`workspace/cache` 三个子目录的可写性诊断，而不是仅检查根 `workspace/` 目录。
  5. 检查调度器组件是否返回执行线程池状态（如活跃线程数、空闲线程数、队列深度）。
  6. 检查等待订阅注册表大小是否与持久化的 `task_trigger_subscriptions` 中 `status=active` 的记录数一致。
  7. 人为缩减 SQLite 可用空间至阈值以下，再次调用 `GET /health`。
- 预期结果：
  - 每个组件的诊断结构包含独立的 `status`、`metric`（度量值）和 `threshold`（阈值）字段，而不是仅返回全局 `ok/fail`。
  - SQLite 可用空间在正常时 `status=healthy`，低于阈值时 `status=degraded` 或 `status=unhealthy`，且返回实际可用空间值和阈值定义。
  - LanceDB 连接池返回实际连接池度量，而不是在任意状态下都返回固定值。
  - 文件系统可写性检查覆盖三个运行时子目录，任一子目录不可写时对应组件标记为 `unhealthy`。
  - 调度器线程池返回可解释的活跃与空闲线程比例，而不是仅标记 `ok`。
  - 等待订阅注册表大小与数据库持久化记录一致，不出现内存态注册表与持久化订阅数不匹配。
  - 组件诊断字段口径与 06-功能函数文档健康检查接口定义和 07-数据库文档中 `system_health_snapshots`（若存在）一致。

**用例 T-ENG-12：系统重启恢复运行中与等待中任务一致性**（验证依据：02-功能要求 N-14.6，05-详细设计 4.1 状态机）
- 前置条件：后端服务运行中；存在至少一个 `running` 状态的任务（含正在执行的步骤）和一个 `waiting` 状态的任务；数据库中保存了对应的任务状态和步骤状态。
- 步骤：
  1. 记录 `running` 任务的当前步骤列表、执行进度和最新 `aggregateVersion`。
  2. 记录 `waiting` 任务的等待订阅状态和 `nextCheckAt`。
  3. 强制终止后端进程（模拟非正常重启），再重新启动后端服务。
  4. 启动完成后，查询两个任务的详情、步骤状态和时间线。
  5. 对 `running` 任务检查是否按恢复策略继续执行或进入明确的中断恢复状态。
  6. 触发 `waiting` 任务的释放条件，检查释放链路是否正常。
- 预期结果：
  - `running` 任务在重启后不会被丢弃或静默标记为成功，而是进入明确的恢复路径（如重试中断步骤、标记为 `intervention_required` 或恢复到最近已完成步骤继续执行）。
  - `running` 任务的已完成步骤结果不会因重启而丢失，`aggregateVersion` 保持一致。
  - `waiting` 任务的等待订阅在重启后被正确恢复，释放条件命中后可正常进入执行链路。
  - 时间线中记录 `SystemRestartDetected` 或等价恢复事件，标明中断原因和恢复动作。
  - 不会出现任务在重启后同时处于 `running` 且无活跃步骤执行的"僵尸"状态。

**用例 T-ENG-13：主页面在各响应式断点下布局与组件适配符合界面基线**（验证依据：08-界面文档 12.1、12.2）
- 前置条件：前端页面可正常访问；浏览器支持设备模拟；会话页、任务详情页和结果页均存在可观察数据。
- 步骤：
  1. 分别以 `375px`、`640px`、`820px`、`1080px`、`1366px` 宽度打开 `/chat`、`/tasks/:taskId` 和 `/results/:taskId`。
  2. 检查 `xs` 断点下页面是否退化为单列布局，侧边栏是否默认隐藏并通过显式操作展开。
  3. 检查 `sm`、`md`、`lg`、`xl` 断点下页面是否按文档定义切换为紧凑双列、标准双列、宽双列和三列布局。
  4. 在移动端宽度下检查消息流、任务卡片和操作按钮是否切换到简化展示样式。
- 预期结果：
  - 页面在 `xs`、`sm`、`md`、`lg`、`xl` 五档断点下都能稳定渲染，不出现主操作区遮挡、横向溢出或关键按钮不可达。
  - 侧边栏、消息流、任务卡片和操作按钮的布局变化符合 08-界面文档 `12.1`、`12.2` 的断点与组件适配规则。
  - 同一任务在桌面端与移动端展示的信息密度不同，但核心状态、结果和操作入口保持语义一致。

**用例 T-ENG-14：键盘导航与屏幕阅读器语义满足无障碍基线**（验证依据：08-界面文档 14.1、14.2、14.3）
- 前置条件：前端页面可正常访问；至少存在会话输入框、任务卡片、权限确认弹窗和结果页操作按钮；浏览器可检查无障碍树。
- 步骤：
  1. 使用 `Tab` / `Shift+Tab` 依次遍历会话页、任务详情页和结果页中的主要交互元素，记录焦点顺序与焦点样式。
  2. 触发权限确认弹窗或抽屉，检查焦点是否被限制在当前模态区域内，关闭后是否返回触发源。
  3. 检查图片、图标按钮、表单标签和状态区域是否提供 `alt`、可访问名称、显式标签或等价 ARIA 属性。
  4. 在任务进度更新或结果加载完成时，检查动态内容是否提供可被辅助技术感知的实时通知机制。
- 预期结果：
  - 所有关键交互元素都可通过键盘访问，Tab 顺序符合页面逻辑流程，焦点状态清晰可见。
  - 模态弹窗或抽屉打开后形成有效焦点陷阱，关闭后焦点回到原始触发元素。
  - 图片、表单和图标按钮具备稳定的无障碍语义，不依赖纯视觉上下文推断用途。
  - 动态内容更新可被辅助技术感知，且页面不依赖颜色作为唯一的信息传达方式。

### N-01.2 入口模式自动识别与判定摘要

**用例 T-N-01.2-01：系统可自动识别队列、定时或事件触发模式**（验证需求：N-01.2）
- 前置条件：系统已启动，会话输入解析链路支持识别排队意图、时间表达和预定义事件条件。
- 步骤：
  1. 在会话页输入一条明确表达“进入任务队列”意图的消息并创建任务。
  2. 再输入一条包含未来执行时间的消息并创建任务。
  3. 再输入一条包含预定义事件触发条件的消息并创建任务。
  4. 打开任务列表和对应任务详情页。
- 预期结果：
  - 三种任务都能成功创建，并展示不同的 `triggerMode`，且其值由会话意图解析结果自动填入。
  - 排队任务显示队列等待状态或队列位置；定时任务显示计划执行时间；事件触发任务显示等待中的触发条件摘要。
  - 若输入中缺少定时或触发执行所需关键条件，系统不会静默猜测后直接执行，而是进入明确的澄清、确认或安全默认处理路径，并记录最终采用的入口模式与原因。
  - 对于低置信度场景，系统会先创建一个可在任务列表和详情页中查询到的待澄清 / 待确认任务，而不是在澄清完成前完全不建单。
  - 对于非立即执行任务，时间线中至少可见 `TaskTriggerResolved` 与 `TaskTriggerWaiting`；当等待条件满足后，还应按顺序出现 `TaskTriggerActivated`、`TaskReadyForPlanning` 与 `MemoryScopeConfirmed`，且同一任务不会因重复事件命中重复进入可规划状态。
  - 若任务为延迟执行模式，任务详情中可查看冻结的入口摘要与记忆范围说明，且在进入执行前不会被当前会话的新上下文静默覆盖。
  - 当满足对应条件后，任务才进入后续编排与执行链路。

**用例 T-N-01.2-02：入口等待态具备可恢复的事件与订阅事实**（验证需求：N-01.2，N-14）
- 前置条件：系统已启动，数据库已开启任务事件与入口等待订阅持久化。
- 步骤：
  1. 创建一个未来时间执行的定时任务。
  2. 在任务尚未触发前检查任务时间线与任务入口等待订阅记录。
  3. 模拟服务重启后再次查询该任务详情与等待订阅记录。
  4. 等待任务达到释放条件并继续观察时间线。
- 预期结果：
  - 任务创建后可查询到入口等待订阅事实，且包含当前 `triggerMode`、条件快照和下一次检查时间。
  - 服务重启后等待态任务不会丢失，仍能恢复等待中的触发条件与下一次评估时间。
  - 排队任务的释放判断仅受全局任务执行槽位预算影响；在相同预算下，先进入等待队列的任务必须先被释放，不存在隐式 Agent 子队列或模型子配额打乱 FIFO 顺序。
  - 等待态恢复、定时扫描和释放进入执行链路都由统一的任务入口协调链路完成，而不是依赖单次内存态定时器；恢复后不会出现职责漂移导致的重复释放。
  - 任务释放进入执行链路时，时间线按顺序出现 `TaskTriggerActivated`、`TaskReadyForPlanning`、`MemoryScopeConfirmed` 等事件，且重复恢复扫描不会再次写入第二组释放事件。

**用例 T-N-01.2-03：等待订阅失效后不得再次释放任务**（验证需求：N-01.2，N-14）
- 前置条件：系统已启动，存在一个处于 `scheduled` 或 `event_triggered` 等待态的任务。
- 步骤：
  1. 在任务尚未释放前取消该任务，或将其标记为超时/人工关闭。
  2. 检查任务入口等待订阅状态和任务时间线。
  3. 再次模拟原触发条件命中或恢复扫描执行。
- 预期结果：
  - 等待订阅被置为失效状态，并记录对应失效原因。
  - 时间线出现 `TaskTriggerInvalidated`，并与 `TaskCancelled`、`TaskTimedOut`、`TaskFailed` 或 `TaskManuallyClosed` 之一共同构成可见终态事实。
  - 后续即使原触发条件再次命中，也不会再出现新的 `TaskTriggerActivated` 或 `TaskReadyForPlanning`。

**用例 T-N-01.2-04：入口模式推断过程中间态在会话页可观测**（验证需求：N-01.2，N-14）
- 前置条件：系统已启动；会话页支持展示入口模式推断过程的中间状态（如 `trigger_inferring`、`trigger_inferred`）。
- 步骤：
  1. 在会话页输入一条包含时间表达的任务请求并发送。
  2. 观察发送后、任务创建前的短暂中间态，确认会话页是否展示"正在分析入口模式"或等价推断中指示。
  3. 等待推断完成后，确认会话页是否切换到"已识别触发模式"并展示识别结果摘要。
  4. 再输入一条明确的立即执行任务，对比中间态展示差异。
- 预期结果：
  - 任务提交后、入口判定完成前，会话页进入可观测的推断中状态，用户不会面对无反馈的空白等待。
  - 推断完成后，会话页展示识别结果摘要（如"已识别为定时任务"或"已识别为立即执行"），并在随后进入任务创建流程。
  - 对于立即执行任务，推断中状态可极短，但仍应有瞬态指示而不是直接跳到任务创建完成。
  - 推断失败或低置信度时，中间态应切换到澄清提示而不是静默消失。

**用例 T-N-01.2-05：入口模式显式确认接口与触发模式变更**（验证需求：N-01.2，N-14）
- 前置条件：系统已启动；会话意图解析链路支持自动推断入口模式；`POST /api/tasks/:id/confirm-trigger` 接口已部署。
- 步骤：
  1. 在会话页输入一条包含模糊时间表达的任务请求，使系统自动推断出 `triggerMode=scheduled`。
  2. 在任务创建后、任务进入执行或等待链路前，调用 `POST /api/tasks/:id/confirm-trigger`，将 `triggerMode` 显式确认为 `immediate`（覆盖系统推断结果）。
  3. 查看任务详情页和时间线中的 `triggerMode`、`triggerDecisionSummary` 和确认事件。
  4. 再创建一个相同类型的任务，这次调用确认接口保留系统推断结果不变，验证幂等确认行为。
  5. 对已进入执行链路或已处于终态的任务调用确认接口，验证拒绝行为。
- 预期结果：
  - 确认接口可在系统推断完成后、任务进入执行链路前变更或锁定最终 `triggerMode`。
  - 变更后任务详情中的 `triggerMode` 和 `triggerDecisionSummary` 反映用户确认后的最终结果，而不是系统原始推断值。
  - 时间线出现 `TriggerModeConfirmed` 或等价确认事件，载荷包含确认前推断值、确认后最终值和确认来源。
  - 保留系统推断结果的确认调用按幂等处理，不产生重复的确认事件。
  - 对已进入执行链路或已处于终态的任务调用确认接口，系统返回状态前置条件不满足错误，不改变既有任务状态。

### N-01.4 单一会话窗口展示与确认前置规则

**用例 T-N-01.4-01：默认仅发起端显示且显式同步后才跨端分发**（验证需求：N-01.4，N-04，N-13，N-14）
- 前置条件：系统已启动；至少存在两个已登录接入端，其中一个为本次输入发起端；系统支持可配置的多端同步策略。
- 步骤：
  1. 在发起端的同一会话窗口中输入文本并上传 1 个多模态附件，发起任务。
  2. 在未开启同步策略的情况下，观察另一接入端是否自动出现同一轮输入相关的进度、结果和权限授权信息。
  3. 显式开启同步策略后，再发起一次任务并观察另一接入端的展示情况。
- 预期结果：
  - 在默认配置下，发起端之外的其他接入端不会自动展示该次输入对应的进度、结果和权限授权信息。
  - 启用同步策略后，其他接入端才会收到相关消息，且展示顺序与原任务时间线事实保持一致，不产生乱序或重复推进。

**用例 T-N-01.4-02：多端同步载荷保留来源端与显示范围字段语义**（验证需求：N-01.4，N-13，N-14）
- 前置条件：系统已启动；至少存在两个接入端；会话详情、任务详情、消息接口和时间线接口会返回 `lastActiveClientId`、`entryPoint`、`originClientId`、`syncPolicy`、`visibleClientIds`、`displayScope` 或等价字段。
- 步骤：
  1. 在接入端 A 发起一次默认 `origin_only` 的任务，分别查询会话详情、消息、任务详情和时间线接口返回。
  2. 在接入端 B 检查该任务相关输入、进度、结果和权限确认是否可见。
  3. 显式启用同步策略后，再在接入端 A 发起一次任务，并再次分别查询上述接口返回。
  4. 对比接入端 A 与接入端 B 可见的数据范围、字段值和时间线顺序。
- 预期结果：
  - 默认 `origin_only` 场景下，消息、任务和事件载荷中的来源端与显示范围字段保持一致，非可见接入端不会收到不应展示的任务事实。
  - 显式启用同步后，接入端 B 才能看到与其显示范围匹配的输入、进度、结果和权限确认信息。
  - 同一条跨端同步事实在不同接口中的 `originClientId`、`syncPolicy`、`visibleClientIds`、`displayScope` 语义一致，不出现消息、任务、事件三处口径不一致。
  - 会话详情中的 `lastActiveClientId` 与最近一次真实发起输入的 `originClientId` 保持一致，不会脱离消息/任务事实单独漂移。
  - 跨端同步后的时间线顺序仍以原任务事件顺序为准，不因接入端差异产生乱序、重复或越权可见问题。

**用例 T-N-01.4-03：多端并发处理同一权限确认时只允许一次生效**（验证需求：N-01.4，N-13，N-14）
- 前置条件：系统已启动；至少存在接入端 A、B；同一任务产生一个待权限确认或待确认执行动作；接口或事件返回 `interactionDedupKey`、`interactionDecisionStatus`、`factSummary` 或等价字段。
- 步骤：
  1. 在接入端 A 和接入端 B 几乎同时打开同一权限确认面板。
  2. 先在接入端 A 提交确认，再在接入端 B 提交对同一事实的确认或拒绝。
  3. 观察两个接入端的页面反馈、任务详情和时间线。
  4. 查询任务详情接口和事件表，检查是否只保留一个最终有效处理结果。
- 预期结果：
  - 系统只接受最先生效的有效交互结果，后续对同一事实的重复裁决被拒绝，不会再次推进任务状态。
  - 接入端 B 在结果同步后只能看到“已由其他接入端处理”或等价提示，而不是继续保留可点击确认按钮。
  - `interactionDedupKey`、`interactionDecisionStatus`、`resolvedByClientId`、`resolvedAt` 或等价字段在接口和事件中保持一致。
  - 权限策略裁决语义与并发交互去重语义可区分，不会出现后到交互覆盖既有最终结果的情况。

**用例 T-N-01.4-04：非原始接入端处理前必须展示最小事实摘要**（验证需求：N-01.4，N-01.6，N-13，N-14）
- 前置条件：同一任务在接入端 A 触发高风险执行确认、权限确认或人工介入处置；接入端 B 可查看该任务；页面或接口返回 `factSummary` 或等价字段。
- 步骤：
  1. 在接入端 B 打开该待处理交互。
  2. 检查界面是否先展示任务标识、当前状态、触发原因、影响范围和待确认动作。
  3. 尝试在摘要未完整渲染前直接提交确认。
  4. 再在摘要完整展示后提交确认或取消。
- 预期结果：
  - 非原始接入端必须先看到统一的最小事实摘要，界面才允许进入确认提交。
  - 若摘要缺失或字段不完整，界面不得直接放行确认动作。
  - `factSummary` 在任务详情、事件载荷和界面展示中的核心语义一致，不出现“接口有值但页面未展示”的断层。

### N-01.5 低置信度入口澄清与安全默认边界

**用例 T-N-01.5-01：入口模式低置信度时进入澄清或安全默认路径**（验证需求：N-01.5，N-14）
- 前置条件：系统已启动；入口模式识别链路支持输出置信度或缺参原因。
- 步骤：
  1. 输入一条同时包含模糊时间表达和模糊触发条件的任务请求，例如“晚点有结果了再帮我处理一下”。
  2. 创建任务并查看任务详情页、时间线和可能出现的澄清提示。
  3. 分别补充一次明确信息，或选择系统提供的安全默认处理方式。
- 预期结果：
  - 系统识别当前请求不足以可靠判定 `scheduled` 或 `event_triggered`，不会直接放入执行链路。
  - 系统会先创建一个待澄清 / 待确认任务，并在任务列表和详情页中可见其当前状态；该状态必须为正式 `clarification_pending` 或等价状态，而不是仅靠临时提示文案表示。
  - 任务详情页或时间线中可查看低置信度原因、缺失条件和本次补救动作。
  - 用户补充关键信息后，系统才写入最终 `triggerMode`；若采用安全默认路径，也能看到最终采用模式及其原因。
  - 无论用户补充信息还是采用安全默认路径，系统都必须在原 Task 上完成状态推进与入口模式确认，不会额外创建第二个 Task 替代原单。
  - 若用户明确放弃继续澄清，原 Task 会进入明确终态，而不是无限停留在待确认状态。

**用例 T-N-01.5-02：冻结记忆快照失效时按降级规则处理**（验证需求：N-01.5，N-09.5，N-14）
- 前置条件：系统已启动；存在一个 `scheduled` 或 `event_triggered` 任务，且创建时显式选择了至少两条记忆条目。
- 步骤：
  1. 创建延迟执行任务，记录任务详情中的冻结记忆条目快照。
  2. 在任务释放前删除、归档或撤销其中一条记忆条目的读取权限。
  3. 等待任务达到释放条件并观察任务详情、时间线和最终状态。
- 预期结果：
  - 系统不会静默改用释放时刻的新会话上下文或新的记忆集合重算入口决策。
  - 任务详情或相关事件中可看到失效记忆条目标记，以及 `snapshotResolution` 或等价降级处理结果。
  - 若剩余冻结快照仍满足最小执行上下文，任务按降级策略继续执行；若不满足，则任务进入 `TaskInterventionRequired` 或等价明确人工介入终态，而不是被隐式放行、映射为人工关闭，或继续进入执行链路。

**用例 T-N-01.5-03：待澄清超时与用户放弃澄清后任务进入明确终态**（验证需求：N-01.5，N-04.3，N-14）
- 前置条件：系统已启动；入口模式判定支持低置信度路径；系统已配置澄清超时时间。
- 步骤：
  1. 发起一条低置信度入口请求，使系统创建 `clarification_pending` 状态任务。
  2. 在澄清超时之前不提交任何补充信息，等待任务超过配置的澄清超时时间。
  3. 查看任务详情、时间线和终态摘要。
  4. 再次发起一条低置信度入口请求并创建待澄清任务，这次显式执行“放弃澄清”操作。
  5. 查看第二个任务的最终状态、时间线和终态摘要。
- 预期结果：
  - 超时场景：任务时间线出现 `ClarificationTimeout` 事件，任务从 `clarification_pending` 进入明确终态（如 `closed` 或 `cancelled`），不会无限停留在待澄清状态。
  - 放弃澄清场景：任务时间线出现 `ClarificationClosed` 事件，记录 `closeReason`，任务进入明确终态。
  - 两种场景下，任务的终态摘要页或终态结果区可重访，展示关闭原因、最后有效状态和后续可执行动作。
  - 两种场景下，任务均不会在超时或放弃后被静默重开或隐式进入执行链路。
  - `clarification_closed_reason` 字段被正确写入，内容可追溯。

**用例 T-N-01.5-04：澄清接口契约与状态推进一致**（验证需求：N-01.5，N-14）
- 前置条件：系统已启动；存在一个处于 `clarification_pending` 状态的任务；澄清接口 `POST /api/tasks/:id/clarify` 已部署。
- 步骤：
  1. 调用 `POST /api/tasks/:id/clarify`，携带 `providedInputs`（补充的关键字段）和 `resolutionSummary`。
  2. 查询任务详情，检查 `triggerMode`、`clarificationResolutionSummary`、`clarificationRequiredFields` 字段。
  3. 查询任务时间线，检查是否出现 `ClarificationProvided` 事件。
  4. 对同一任务再次调用澄清接口，验证重复澄清的处理方式。
  5. 对一个非 `clarification_pending` 状态的任务调用澄清接口，验证拒绝行为。
- 预期结果：
  - 首次澄清后，任务从 `clarification_pending` 推进到已解析的 `triggerMode`，`clarificationResolutionSummary` 非空。
  - 时间线出现 `ClarificationProvided` 事件，载荷包含补充字段和解析摘要。
  - 对已完成澄清的任务重复调用，系统返回"任务已不在待澄清状态"错误或幂等处理。
  - 对非 `clarification_pending` 任务调用，系统返回状态前置条件不满足错误。
  - 澄清后的任务可继续进入后续编排链路，不会停滞。

### N-02.1 复杂任务判定与可解释摘要

### N-02.2 父子任务编排与 Domain 分配

**用例 T-N-02.2-01：复杂任务被拆解为父任务、子任务和多步骤计划**（验证需求：N-02.2）
- 前置条件：模型配置正常；存在可路由的 Domain Agent。
- 步骤：
  1. 在会话页发起包含多个子需求的复杂任务。
  2. 打开任务详情页，查看时间线与步骤列表。
- 预期结果：
  - Leader Agent 会把复杂任务拆成不少于 2 个 Domain 子任务；对于立即执行任务，应直接生成包含不少于 3 个步骤的父任务级 TaskPlan（如规划、子任务安排、汇总）。
  - 对于 `queued`、`scheduled`、`event_triggered` 等非立即执行复杂任务，父任务可先完成创建与等待登记而不立即生成正式 TaskPlan；当等待条件满足后，时间线需按顺序出现 `TaskTriggerActivated`、`TaskReadyForPlanning`、`MemoryScopeConfirmed`，随后才生成正式父任务级 TaskPlan。
  - 每个 Domain 子任务都带有独立 `taskId`、目标 Domain Agent、入口模式和父任务关联关系。
  - 每个步骤都带有清晰的描述和执行状态（pending/running/done）。
  - 任务详情页可看到 Leader Agent 的主要路由决策摘要；当存在可并行步骤时，界面可区分并行执行与等待中的步骤。

**用例 T-N-02.2-02：Domain Agent 子任务支持独立队列和触发任务**（验证需求：N-02.2，N-07.3，N-14）
- 前置条件：系统已启动；存在一个支持显式队列的 Domain Agent 和一个支持事件触发的 Domain Agent。
- 步骤：
  1. 发起一个复合任务，使其中一个子任务被分配为 `queued`，另一个子任务被分配为 `event_triggered`。
  2. 打开父任务和两个子任务详情页。
  3. 分别观察队列顺位、触发条件和后续释放执行过程。
- 预期结果：
  - `queued` 子任务显示其在所属 Domain Agent 队列中的 `queuePosition`，且该顺位不与其他 Agent 的队列混淆。
  - `event_triggered` 子任务显示其触发条件和等待中的触发状态。
  - `event_triggered` 子任务通过统一等待订阅与重评估链路进入可执行状态，不依赖独立“触发任务池”；同一触发事实重复命中不会重复释放任务。
  - 父任务的安排摘要能够同时汇总“一个已入队、一个已登记触发”的安排结果，并生成统一 ETA 或 ETA 摘要说明。

**用例 T-N-02.2-03：首轮观察不足时 Leader 在原父任务内触发增量重编排**（验证需求：N-02.2，N-03.1，N-07.5，N-14）
- 前置条件：系统已启动；至少存在两个可用 Domain Agent；任务详情接口和时间线接口可查看父任务编排版本、目标满足性摘要和子任务变化。
- 步骤：
  1. 发起一个复杂任务，使首轮至少生成 2 个 Domain 子任务。
  2. 让其中一个子任务返回“信息不足”或“需补充外部观察”类结果，另一个子任务正常完成。
  3. 观察父任务时间线、详情摘要和子任务列表，确认 Leader 是否在原父任务上下文中继续编排下一轮子任务，而不是要求用户重新创建任务。
  4. 记录第二轮编排后新增的子任务、计划摘要和目标满足性判断结果。
- 预期结果：
  - 父任务在首轮子任务观察返回后出现明确的目标满足性判断结果，能够说明当前为什么“尚未满足”。
  - 时间线出现 `GoalSatisfactionEvaluated` 与 `TaskReplanned` 或等价事件，且事件载荷可追溯本轮使用的观察摘要、`planRevision` / `replanCount` 和重编排原因。
  - 系统在原父任务下生成新的 Domain 子任务或新的父任务级步骤，不要求用户重新发起新的顶层任务。
  - 父任务详情中的计划摘要、下一步动作摘要和子任务列表能反映第二轮编排结果，且原首轮记录不被无痕覆盖。

### N-02.3 blocking / non-blocking 标注与改标治理

**用例 T-N-02.3-01：blocking / non-blocking 标记不得被静默改写**（验证需求：N-02.3，N-14）
- 前置条件：存在一个父任务，其子任务 blocking / non-blocking 标记已被用于生成安排通知、ETA 或降级交付判定。
- 步骤：
  1. 记录当前父任务详情中的子任务标记、安排通知内容、ETA 和相关时间线事件。
  2. 模拟系统纠正或人工介入，尝试把一个已参与交付判定的 blocking 子任务改标为 non-blocking，或反向改标。
  3. 重新查看父任务详情、安排摘要、ETA 展示和时间线。
- 预期结果：
  - 系统不会静默改写既有 blocking / non-blocking 标记对应的历史结论。
  - 若发生标记调整，时间线中会出现明确变更事件、变更原因和影响范围。
  - 原安排通知、ETA 或交付判定会进入需重评估状态，用户可以看到更新后的解释，而不是看到被无痕覆盖的旧结论。

**用例 T-N-02.3-02：blocking 改标后触发安排与 ETA 重评估**（验证需求：N-02.3，N-02.4，N-14）
- 前置条件：存在一个父任务，已完成 `TaskArrangementCompleted` 和 `TaskEtaEstimated`；其子任务 blocking/non-blocking 标记已参与安排通知和 ETA 生成。
- 步骤：
  1. 记录当前父任务的 `arrangementSummary`、`estimatedCompletionAt` 和已发送的安排通知内容。
  2. 通过人工介入或系统纠正，将一个参与安排判定的 blocking 子任务改标为 non-blocking。
  3. 查看父任务时间线、安排摘要、ETA 和任务详情页展示。
  4. 再将该子任务改回 blocking，检查是否再次触发重评估。
- 预期结果：
  - 改标后时间线出现 `BlockingDecisionChanged` 事件，载荷包含变更原因、变更前后标记和影响的安排结论范围。
  - 改标触发 ETA 重新计算，`estimatedCompletionAt` 或 `estimatedDurationMinutes` 发生变化或进入 `reassessment_required`。
  - `arrangementSummary` 更新为反映新标记结构的内容，原安排结论不被无痕覆盖。
  - 任务详情页进入重评估提示状态，用户可看到变更摘要和影响说明。
  - 再次改回 blocking 时，同样触发新一轮重评估并记录事件。

### N-02.4 安排完成与 ETA 生成

**用例 T-N-02.4-01：Leader 在全部子任务安排完成后先反馈 ETA**（验证需求：N-02.4，N-07.2，N-14）
- 前置条件：系统已启动；至少存在两个可用 Domain Agent，且二者都支持显式队列或等待订阅型安排。
- 步骤：
  1. 发起一个需要两个 Domain Agent 分别执行的复合任务。
  2. 观察父任务时间线、子任务列表和面向用户的通知。
  3. 等待两个子任务都完成“已入队 / 已登记定时 / 已登记触发”等安排确认状态。
- 预期结果：
  - 父任务时间线出现 `SubTasksCreated`，并为每个子任务出现 `DomainTaskQueued`、`DomainTaskScheduled` 或 `DomainTaskTriggered` 中的对应事件。
  - 当全部 blocking 子任务都完成安排后，父任务出现 `TaskArrangementCompleted` 与 `TaskEtaEstimated`。
  - 用户先收到“任务已安排好”的反馈，其中包含聚合 ETA；Leader Agent 在此后无需阻塞等待子任务实际完成。

**用例 T-N-02.4-02：等待异常任务进入人工介入状态**（验证需求：N-02.4，N-14）
- 前置条件：系统已启动；存在一个父任务，其子任务之一处于 `scheduled`、`event_triggered` 或 `queued` 等待态。
- 步骤：
  1. 让该子任务持续超过预期等待阈值，或模拟触发订阅失效、冻结快照失效。
  2. 打开父任务详情页和对应子任务详情页，观察等待状态摘要和时间线。
  3. 继续观察系统是否自动恢复；若不能恢复，检查是否进入人工介入状态。
- 预期结果：
  - 任务详情中可查看等待异常原因、当前受阻子任务、最近一次重评估结果、命中的等待阈值依据和建议动作。
  - 时间线出现与等待异常或人工介入对应的正式事件或等价状态变更，不会保持无反馈静默等待。
  - 若系统无法自动恢复，父任务或子任务进入明确人工介入状态，用户可执行继续等待、重新安排或手动关闭等后续动作。

**用例 T-N-02.4-03：等待阈值逐级超限触发异常检测并路由至人工介入**（验证需求：N-02.4，N-14）
- 前置条件：系统已启动；等待订阅阈值配置（`maxWaitTimeMs`、`anomalyThresholdMs`）已设为可测试的短时值；存在一个 `queued` 或 `scheduled` 等待态任务。
- 步骤：
  1. 创建一个 `queued` 等待态任务，并为其关联订阅配置较短的 `anomalyThresholdMs` 和 `maxWaitTimeMs`。
  2. 让任务等待超过 `anomalyThresholdMs` 但未超过 `maxWaitTimeMs`，查看任务详情和时间线。
  3. 继续等待直至超过 `maxWaitTimeMs`，再查看任务详情和时间线。
  4. 检查任务最终状态和建议动作。
- 预期结果：
  - 超过 `anomalyThresholdMs` 时，任务详情出现等待异常标记，时间线记录 `WaitAnomalyDetected` 或等价事件，写入 `waitingAnomalyCode`、`waitingAnomalySummary`。
  - 超过 `maxWaitTimeMs` 时，任务进入 `TaskInterventionRequired` 或等价终态，时间线记录对应事件。
  - 任务详情中可查看 `waitingThresholdMode`、`waitingThresholdBasisSummary`、`interventionRequiredReason` 和 `suggestedActions`。
  - 阈值配置支持按 `queued`、`scheduled`、`event_triggered` 分别配置，且三类模式使用各自的阈值而不是共用全局值。

**用例 T-N-02.4-04：事件触发子任务返回区间估算而非伪精确 ETA**（验证需求：N-02.4，N-02.2，N-14）
- 前置条件：系统已启动；存在一个父任务，其子任务之一为 `event_triggered` 且触发条件依赖外部不可控事件；安排完成后父任务已生成初始 ETA。
- 步骤：
  1. 发起一个复合任务，使其中一个 blocking 子任务以 `event_triggered` 模式等待外部事件。
  2. 查询父任务的 `estimatedCompletionAt` 或 `estimatedDurationMinutes` 和 `etaBasisSummary`。
  3. 检查 ETA 返回是否为区间估算（如"15–120 分钟"）或含不确定性说明，而不是精确到分钟的单点值。
  4. 再发起一个全部子任务为 `immediate` 模式的父任务，对比其 ETA 格式差异。
- 预期结果：
  - 包含 `event_triggered` 子任务的父任务 ETA 返回区间估算或"等待外部条件"摘要，而不是伪精确的单点时间。
  - `etaBasisSummary` 中明确说明不确定性来源，如"子任务 X 依赖外部事件触发，无法确定完成时间"。
  - 全部 `immediate` 子任务的父任务可返回较精确的单点 ETA，与区间估算格式有明确区分。
  - 任务详情页和安排通知中展示的 ETA 与接口返回的格式一致，不会在界面上将区间估算渲染为单点时间。
  - `estimatedCompletionAt` 为 `null` 或区间类型时，接口不会回退到返回当前时间加固定偏移量的伪值。

### N-02.5 降级交付判定摘要

**用例 T-N-02.5-01：non-blocking 子任务失败时父任务可降级交付**（验证需求：N-02.5，N-03，N-04，N-14）
- 前置条件：存在一个包含至少 1 个 blocking 子任务和 1 个 non-blocking 子任务的父任务。
- 步骤：
  1. 让全部 blocking 子任务成功完成。
  2. 让 non-blocking 子任务失败、超时或被显式降级跳过。
  3. 查看父任务详情、结果页和最终通知记录。
- 预期结果：
  - 父任务不会因 non-blocking 子任务失败而被一律判定为主链路失败。
  - 父任务可进入部分完成或降级交付状态，并产出最终结果。
  - 结果页和通知中明确标记未完成的 non-blocking 子任务及其对最终交付的影响范围。
  - 系统会生成结构化降级交付判定摘要，至少列出已完成的 blocking 子任务、失败或被忽略的 non-blocking 子任务、影响范围以及本次仍可交付的理由，并能在结果页、通知或事件流中追溯。

## N-03 从步骤执行到输出物生成

### N-03.1 子任务执行记录与结果回传

**用例 T-N-03.1-01：步骤执行产生中间与最终输出**（验证需求：N-03.1）
- 前置条件：已存在处于 `running` 状态的任务，已生成 TaskPlan。
- 步骤：
  1. 等待任务执行完成，或在任务详情页观测执行过程。
  2. 查看步骤列表和执行报告。
- 预期结果：
  - 每个步骤都有对应的执行结果记录（成功/失败、耗时、摘要）。
  - 至少一个步骤调用了工具或 Skill，并在执行报告中可见。
  - TaskOutputs 中存在与本任务相关的主输出物记录。

**用例 T-N-03.1-02：运行中任务取消后状态清理与事件完整**（验证需求：N-03.1，N-14）
- 前置条件：系统已启动；存在一个处于 `running` 状态的任务，该任务至少有一个步骤正在执行或等待中；若该任务为延迟任务，还存在活跃等待订阅。
- 步骤：
  1. 调用 `POST /api/tasks/:id/cancel`，提交 `cancelReason` 和可选的 `expectedAggregateVersion`。
  2. 查看任务详情、步骤列表、时间线和等待订阅状态。
  3. 在取消后再次提交同一任务的写操作或重试请求。
- 预期结果：
  - 任务状态从 `running` 变为 `cancelled`，时间线出现 `TaskCancelled` 事件，记录 `cancelReason` 和 `cancelledBy`。
  - 正在执行的步骤被中止或标记为 `cancelled`，不会继续产生新的步骤事件。
  - 若存在活跃等待订阅，同步记录 `TaskTriggerInvalidated` 并将订阅置为 `cancelled`。
  - 取消后对同一任务的写操作被拒绝，返回任务已终态相关错误。
  - 取消后的任务仍可通过详情页和时间线查看完整历史，支持复盘。

**用例 T-N-03.1-03：失败步骤重试与整任务重试遵守版本控制**（验证需求：N-03.1，N-14）
- 前置条件：系统已启动；存在一个因某步骤失败而处于 `failed` 或 `partial_failed` 状态的任务；任务详情已知当前 `aggregateVersion`。
- 步骤：
  1. 调用 `POST /api/tasks/:id/retry`，使用 `retryMode=failed_step_only`、`failedStepId` 和正确的 `expectedAggregateVersion`。
  2. 观察失败步骤是否被重新执行，其他已成功步骤是否保留。
  3. 在第一次重试还未完成时，使用旧 `expectedAggregateVersion` 再次提交整任务重试（`retryMode=full`）。
  4. 等待步骤重试完成后，使用最新版本号重新提交整任务重试。
- 预期结果：
  - `failed_step_only` 模式下，仅重试指定失败步骤，已成功步骤不被重复执行。
  - 时间线出现新的 `StepStarted` 事件，且与原始失败事件可区分（`retryCount` 递增）。
  - 使用过期版本号提交的重试请求被拒绝，返回 `TASK_WRITE_CONFLICT`，不产生错误的状态推进。
  - 使用最新版本号重新提交后，重试请求被接受并正确执行。
  - `preserveMemoryScope=true` 时，重试任务沿用原始记忆范围而不重新加载。

### N-03.2 blocking 完成后的最终输出生成

**用例 T-N-03.2-01：全部 Domain 子任务完成后由 Leader 汇总并通知用户**（验证需求：N-03.2，N-07.2，N-14）
- 前置条件：已存在一个父任务及多个 Domain 子任务，且父任务已完成安排确认并向用户发送过 ETA。
- 步骤：
  1. 等待全部 blocking 子任务执行完成并回传结果，可选的 non-blocking 子任务允许失败、超时或被降级跳过。
  2. 观察父任务时间线、输出结果和用户通知记录。
- 预期结果：
  - 每个子任务完成后都会把结果回传父任务聚合链路。
  - 当全部 blocking 子任务完成后，父任务生成统一输出物和汇总报告。
  - Leader Agent 向用户发送最终完成通知，且该通知发生在“已安排完成”通知之后。

### N-03.3 non-blocking 差异对主链路结果的影响表达

**用例 T-N-03.3-01：non-blocking 子任务差异会被显式标记而不影响主链路成立**（验证需求：N-03.3，N-04，N-14）

  1. 查看父任务时间线、输出结果、执行摘要和结果页。
  2. 核对失败或跳过的 non-blocking 子任务在交付结果中的表达方式。
- 预期结果：
  - 对于失败、超时或被降级跳过的 non-blocking 子任务，最终输出或执行摘要中会明确标记其影响范围。

- 前置条件：系统已启动；模型配置可用；系统具备错误分类机制，区分可重试错误（`MODEL_TIMEOUT`、`MODEL_RATE_LIMITED`、`NETWORK_ERROR`）和不可重试错误（`MODEL_RESPONSE_INVALID`、`PERMISSION_DENIED`、`RESOURCE_NOT_FOUND`）。
- 步骤：
  1. 发起一个需要模型推理的任务，人为使模型返回不可解析的响应内容（触发 `MODEL_RESPONSE_INVALID`）。

  3. 再发起一个任务，人为使模型返回超时错误（`MODEL_TIMEOUT`），检查系统是否按退避策略自动重试。
  4. 对比两类错误的处理路径和最终状态。

  - 不可重试错误的模型调用日志 `retry_count=0`，`final_status=failed`，`error_classification=non_retryable`。
  - 可重试错误的模型调用日志记录每次重试的时间戳和退避间隔，`retry_count` 与实际重试次数一致。
  - 时间线中不可重试错误直接出现 `TaskInterventionRequired` 事件；可重试错误先出现若干 `ModelRetryAttempted` 事件，全部重试失败后才出现 `TaskInterventionRequired` 或 `TaskFailed`。
  - 错误分类字段口径与 05-详细设计错误分类表和 06-功能函数文档错误码定义一致。

  - 日摘要整理作业的执行与任务主时间线完全隔离，整理作业失败不会写入任何任务的 `task_events` 表，不会阻塞或影响正在执行的任务。
  - 整理作业的日志和状态记录在独立的 `memory_consolidation_jobs` 表或等价审计链路中，可通过 `jobId` 独立追溯。

- 前置条件：系统已启动；模型配置可用；存在可触发 Prompt 构建和裁剪的任务样本；模型调用日志或调试接口可查看实际发送的 Prompt 结构；Agent Prompt Profile 已配置。

  1. 发起一个需要模型推理的任务，在模型调用日志或调试接口中检查实际发送的 Prompt 结构。
  2. 验证 Prompt 是否包含五层结构：`system`（系统指令）、`developer`（开发者模板）、`user`（用户输入）、`context`（知识/记忆上下文）、`tool-result`（工具观察结果）。
  3. 检查每层内容是否来自正确的输入源：用户输入在 `user` 层、知识检索结果在 `context` 层、工具调用结果在 `tool-result` 层，且 Agent 角色内容来自 Agent Prompt Profile 插槽填充。
  4. 构造一个超长上下文任务，使 Prompt 触发裁剪策略，检查裁剪后五层结构是否仍然完整。
  5. 检查裁剪后的 Prompt 中 `system` 和 `developer` 层是否始终保留，不被优先裁减。
- 预期结果：
  - 正常场景下 Prompt 包含完整五层结构，各层之间有明确的分隔标记或结构化字段区分。
  - 每层内容来源与其角色定义一致，不出现用户输入混入 `system` 层或知识文档混入 `user` 层。
  - 裁剪后五层结构仍然完整，`system` 和 `developer` 层始终保留不被裁减。
  - 裁剪优先级遵循"优先裁旧步骤、次裁知识、再裁输入"的策略，`context` 层中较旧或相关度较低的内容先被移除。
  - 裁剪后记录 `promptTruncated=true` 和 `truncatedLayers` 字段，可在模型调用日志中追溯。

  - 使用最新 `confirmation_version` 重新提交后，操作可正常执行。

  1. 打开权限策略管理页，确认页面可正常渲染且展示已有策略列表。
  2. 通过 `POST /api/policies` 创建一条新策略，指定作用域（如 Agent 级 + 写入操作 = `ask`），保存后检查列表。
  3. 通过 `GET /api/policies/:id` 查询策略详情，核对作用域层级、动作类型和决策值与创建时一致。
  4. 通过 `PATCH /api/policies/:id` 将该策略的决策从 `ask` 修改为 `deny`，检查更新结果。
  5. 发起一个会命中该策略的任务工具调用，确认修改后的策略已在权限裁决中生效。
  6. 通过 `DELETE /api/policies/:id` 删除该策略，确认列表中不再包含。
  7. 再次发起相同工具调用，确认删除后权限裁决回退到上层默认策略。
- 预期结果：
  - 权限策略管理页可正常列出已有策略，包含策略名称、作用域、动作类型、决策值和关联 Agent/Skill 信息。
  - 创建后的策略可在列表和详情接口中稳定查询，字段与提交时一致。
  - 修改策略后，后续权限裁决立即使用新决策值，不需要重启服务。
  - 删除策略后，依赖该策略的权限裁决回退到更高层级的默认策略（如全局默认），不会出现裁决空档或错误放行。
  - 所有策略变更（创建、修改、删除）在审计日志中可追溯，包含变更前后值和操作人。

**用例 T-N-14-02：基于 `sequence/cursor` 的断线补拉与重复事件幂等处理**（验证需求：N-14）
- 前置条件：任务执行过程中启用 WebSocket 事件推送与 HTTP 时间线补拉接口，事件对象包含 `sequence` 和 `cursor`。
- 步骤：
  1. 启动一个会持续产生多条步骤事件的任务，并记录当前已接收的最后一个 `sequence` 或 `cursor`。
  2. 在任务继续执行时，主动断开前端事件连接。
  3. 使用断开前记录的 `sequence` 或 `cursor` 调用时间线补拉接口，获取断线期间遗漏的事件。
  4. 模拟服务端重复投递其中一条已补拉事件，再恢复实时订阅。
- 预期结果：

  - 前端能基于 `cursor` 或 `sequence` 恢复完整时间线，不因断线造成步骤状态缺失或乱序。
  - 当同一事件被重复投递时，前端或消费端按 `eventId` 或 `sequence` 做幂等处理，不重复追加时间线，也不重复推进任务状态。

  - 直接插入重复 `sequence` 的数据库操作被唯一索引拒绝，不会产生重复事件记录。

  - `tasks.last_event_sequence` 始终等于该任务 `task_events` 表中最大 `sequence` 值，不出现不一致。

**用例 T-N-14-13：系统重启后恢复执行中任务继续执行**（验证需求：N-14，N-03.1）
- 前置条件：系统已启动；存在至少一个处于 `running` 状态的任务（非等待态），且该任务已完成若干步骤但尚未到达终态。
- 步骤：
  1. 记录当前 `running` 任务的 `taskId`、已完成步骤数和最新 `last_event_sequence`。
  2. 正常关闭后端服务，再重新启动。
  3. 启动完成后，查询该任务状态，确认任务是否被自动检测并恢复执行。
  4. 等待任务继续执行完成，检查恢复后的步骤是否从中断位置继续，而非从头重新开始。
  5. 查看任务事件流，确认存在 `TaskResumedAfterRestart` 或等价恢复事件。
- 预期结果：
  - 系统重启后自动检测处于 `running` 状态的中断任务，并触发恢复流程。
  - 恢复后任务从中断位置继续执行，已完成的步骤不会被重复执行。
  - 事件流中记录恢复事件，包含恢复时间、中断位置和恢复策略。
  - 恢复后的 `last_event_sequence` 严格递增，不与中断前的事件序号冲突。
  - 若任务因中断时间过长导致上下文失效，系统进入降级或人工介入流程，而非静默失败。

**用例 T-N-14-14：任务详情展示步骤级 Agent 使用摘要与 per-Agent 耗时统计**（验证需求：N-14，N-07.2）
- 前置条件：系统已启动；存在一个已完成的复合任务，该任务涉及 Leader Agent 和至少两个 Domain Agent 协作执行。
- 步骤：
  1. 打开该复合任务的详情页，查看步骤级执行摘要区域。
  2. 核对每个步骤是否展示执行该步骤的 Agent 标识、所调用的工具名称、知识检索命中情况和记忆使用情况。
  3. 查看任务详情中是否展示 per-Agent 执行统计区，包括每个参与 Agent 的总耗时和调用次数。
  4. 通过任务详情接口查询同一任务，核对接口返回是否包含 `agentExecutionSummary` 或等价字段。
  5. 对比页面展示的 per-Agent 耗时与调用次数与接口返回数据的一致性。
- 预期结果：
  - 任务详情页在步骤区展示每个步骤对应的 Agent、工具、知识和记忆使用摘要，而不是只展示事件类型列表。
  - 任务详情页展示 per-Agent 统计区，至少包含每个参与 Agent 的总执行耗时和工具调用次数。
  - 接口返回的 per-Agent 统计数据与页面展示一致，不出现遗漏或偏差。
  - 未参与执行的 Agent 不会出现在统计区，避免展示误导性零值记录。

## N-16 安全与凭证管理

  3. 调整一条脱敏规则后，再重复执行一次相同流程。
- 预期结果：
  - 日志、审计和导出结果都对敏感字段应用统一脱敏规则，而不是只在某一处生效。
  - 调整脱敏规则后，后续生成的日志与导出结果按新规则输出。
  - 不会出现导出报告保留明文、而界面或日志已脱敏的不一致情况。

**用例 T-N-16.4-02：HTTPS 场景下 WebSocket 加密连接、鉴权心跳与默认监听边界符合基线**（验证需求：N-16.4）
- 前置条件：系统已启用 HTTPS；实时事件通道已开启；已准备合法 Token 和无效 Token；服务按默认监听配置启动。
- 步骤：
  1. 通过 `wss://` 地址连接实时事件通道，携带合法 Token，确认连接建立并可接收心跳或事件数据。
  2. 使用无 Token 或无效 Token 重复连接，观察服务端是否拒绝握手或在鉴权阶段主动断开。
  3. 保持已鉴权连接空闲，验证客户端按 30 秒心跳窗口发送保活时连接稳定；停止发送心跳后，验证连接会在超时后断开或被标记为失活。

- 预期结果：
  - 启用 HTTPS 后，实时事件通道使用 `wss://` 加密链路，合法 Token 可建立连接。
  - 无 Token 或无效 Token 不可绕过鉴权接入实时事件通道。
  - 30 秒心跳窗口内连接保持稳定，心跳超时后连接会被正确回收，不出现长期悬挂连接。
  - 未显式放开网络边界时，服务默认仅允许本机访问，不会静默开放公网监听。

**用例 T-N-17-04：健康检查接口覆盖度与响应结构**（验证需求：N-17.1）
